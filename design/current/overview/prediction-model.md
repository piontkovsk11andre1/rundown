# Prediction model

The framework is shaped by one observation: **a plan is a prediction of the future**, and predictions can be cheap to evaluate if you stay at the right level of abstraction.

## Two prediction tasks

Given work that flows `A → B → C`:

- **Task 1 — fill in `B`**: we know start `A` and goal `C`. AI can predict the path at a high level, but the actual *materialization* requires steps small enough to be optimized and reused. The plan/build boundary is where rundown lives — it is the loop where each step is cheap, deterministic and reusable.
- **Task 2 — predict `C`**: knowing `A`, `A → B`, and `B → C`, ask the model to imagine the resulting state. This is cheaper than running the world. The risk is that the prediction drifts from reality, so we need a protocol that re-anchors prediction to verified fact.

`rundown` is the protocol that re-anchors prediction:

- **`design/`** = intent (what we want).
- **`migrations/`** = pending intent records generated by planner convergence.
- **`materialize`** = collapse the prediction onto reality.
- **`test`** = run quality gates against the materialized workspace.

## Why deterministic automation matters

Each contact with reality (file write, commit, deploy) splits the world into *before* and *after*. The framework needs to:

- guarantee ordering (one task at a time, parent before children),
- collect evidence of each contact (artifacts, traces, commits),
- and let humans review the bridge (PRs at design-release and materialization steps).

This is why the core loop is single-threaded, verification-gated, lock-protected, and fully traced.

## How this maps to commands

| Conceptual step | Command |
|---|---|
| Edit intent | (manual edits in `design/current/`) |
| Freeze a revision | `rd design release` |
| Propose migrations | `rundown migrate` |
| Touch reality | `rundown materialize` |
| Test reality | `rundown test` (default materialized mode) |
| Reverse a touch | `rundown undo`, `rundown revert` |

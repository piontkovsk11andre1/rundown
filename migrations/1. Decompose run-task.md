# Decompose `run-task.ts`

## Goal

Break `src/application/run-task.ts` (currently ~3,060 lines) into focused, single-responsibility modules. The file is the largest in the codebase — roughly 4× the next largest application module (`revert-task.ts` at 759 lines) — and contains multiple orthogonal concerns entangled inside one closure. The target state is a top-level orchestrator of ≤400–500 lines that delegates to extracted modules, each independently testable.

## Current state

`run-task.ts` exports:

| Export | Kind | Purpose |
|---|---|---|
| `createRunTask` | factory → `(options) → Promise<number>` | Main entry point: resolves sources → selects task → executes → verifies → checks checkbox → commits/hooks |
| `RunTaskDependencies` | interface (21 ports) | Dependency bundle injected at construction |
| `RunTaskOptions` | interface (~30 fields) | Runtime configuration from CLI flags |
| `getAutomationWorkerCommand` | pure function | Normalizes opencode TUI worker commands |
| `finalizeRunArtifacts` | side-effectful function | Persists run artifacts via `ArtifactStore` |
| `toRuntimeTaskMetadata` | pure function | Serializes `Task` to artifact-storable shape |
| `isOpenCodeWorkerCommand` | pure function | Detects opencode executables in worker command |
| `RuntimeTaskMetadata` | type | Shape of serialized task metadata |
| `TaskSelectionResult` | re-export | Port type alias |
| `RunnerMode`, `PromptTransport` | re-exports | Port type aliases |

Internally the file also defines:

- **Trace orchestration** (~600 lines): `startTraceRun`, `beginPhaseTrace`, `completePhaseTrace`, `emitPromptMetrics`, `emitTraceTaskOutcome`, `emitTraceTimingWaterfall`, `emitTraceRunCompleted`, `emitResetPhaseTrace`, `runTraceEnrichment`, `emitDeferredTraceEvents`. These functions form an internal state machine (`activeTraceRun`) that tracks phase timings, output volumes, agent signals, and analysis summaries.
- **Trace artifact collection and formatting** (~350 lines): `collectTracePhaseArtifacts`, `summarizePhaseAnalyses`, `formatPhaseTimingsForTrace`, `formatPhaseOutputsForTrace`, `formatAgentSignalsForTrace`, `formatThinkingBlocksForTrace`, `formatToolUsageForTrace`, `parseAnalysisSummaryFromWorkerOutput`.
- **Trace-only enrichment flow** (`runTraceOnlyEnrichment`, ~130 lines): Standalone sub-flow reusing trace artifacts from a previous run.
- **CLI block expansion wiring** (~120 lines): Composable `withCommandExecutionHandler` / `withCliTrace` / `withSourceCliFailureWarning` / `withTemplateCliFailureAbort` handler chain, plus `TemplateCliBlockExecutionError` class.
- **Template loading** (`loadProjectTemplatesFromPorts`, ~30 lines): Reads or defaults six project templates.
- **Git operations** (~100 lines): `isGitRepoWithGitClient`, `isWorkingDirectoryClean`, `commitCheckedTaskWithGitClient`, `resolveGitArtifactAndLockExcludes`, `isPathInsideRepo`, `buildCommitMessage`.
- **Checkbox mutations** (~50 lines): `checkTaskUsingFileSystem`, `resetFileCheckboxes`, `maybeResetFileCheckboxes`, `countCheckedTasks`.
- **Rundown task delegation** (~100 lines): `parseRundownTaskArgs`, `buildDelegatedRundownArgs`, `resolveDelegatedRundownTargetArg`, `delegatedTargetExists`, `isSameFilePath`, `normalizeLegacyRetryArgs`, `hasLongOption`, `hasLongOptionVariant`.
- **Run lifecycle and hook helpers** (~80 lines): `afterTaskComplete`, `afterTaskFailed`, `runOnCompleteHookWithProcessRunner`.
- **Task context resolution** (~80 lines): `resolveTaskContextFromRuntimeMetadata`, `findTaskByFallback`, `computeTaskContextMetrics`, `hasDescendantTasks`, `validateRuntimeTaskMetadata`, `resolveLatestCompletedRun`, `hasReverifiableTask`, `isCompletedArtifactRun`.
- **Generic utilities** (~30 lines): `parseJson`, `asStringArray`, `asNonNegativeInt`, `asEnum`, `countTraceLines`, `formatTaskLabel`, `computeDurationMs`.

## Why this matters

1. **Readability**: 3,060 lines in a single closure makes navigation and comprehension difficult. Concerns are discovered by scrolling, not by module boundaries.
2. **Testability**: The test file (`run-task.test.ts`, ~3,500 lines) must mock 21 dependencies and reason about interleaved concerns. Extracted modules would have focused, smaller test suites.
3. **Reuse friction**: `reverify-task.ts` and `revert-task.ts` already duplicate some patterns (template loading, task context resolution, artifact formatting) because they cannot import focused sub-modules from `run-task.ts` without pulling in the entire dependency graph.
4. **Change risk**: Any modification to trace formatting, git commit logic, or CLI block expansion requires editing a 3k-line file where unrelated state is in scope.

## Existing architecture context

The project follows **clean architecture** with three layers:

- **Domain** (`src/domain/`): Pure types, parsers, template rendering, trace event constructors, port interfaces (`src/domain/ports/`). No I/O.
- **Application** (`src/application/`): Use-case orchestrators. Each file exports a `createXxx(deps)` factory returning an async function. Dependencies are injected via port interfaces.
- **Infrastructure** (`src/infrastructure/`): Port implementations (file system, git, process runner, adapters). Wired in `src/create-app.ts`.

`run-task.ts` sits in the application layer. Extracted modules should remain in the same layer (or move to domain if they are pure). The composition root (`src/create-app.ts`) should not need changes — it injects ports into `createRunTask`, and the internal decomposition stays behind the same public API.

## Identified extraction candidates

The groups below are ordered by independence — earlier extractions have fewer internal dependencies and unblock later ones.

### 1. Pure utility functions → `src/domain/` or `src/application/`

Functions with no port dependencies and no state:

- `parseJson`, `asStringArray`, `asNonNegativeInt`, `asEnum` — generic parsing helpers.
- `countTraceLines`, `computeDurationMs` — trace-related pure computations.
- `formatTaskLabel` — task display formatting.
- `isSameFilePath`, `hasLongOption`, `hasLongOptionVariant` — string/path utilities.

These are candidates for a shared domain utility module or can be co-located with their primary consumer.

### 2. Rundown task delegation logic

Self-contained argument-parsing and delegation logic:

- `parseRundownTaskArgs`, `buildDelegatedRundownArgs`, `resolveDelegatedRundownTargetArg`
- `delegatedTargetExists`, `normalizeLegacyRetryArgs`

Depends only on `FileSystem` and `PathOperationsPort` (port interfaces). Could become `src/application/rundown-delegation.ts` or `src/domain/rundown-delegation.ts`.

### 3. Git operations

- `isGitRepoWithGitClient`, `isWorkingDirectoryClean`, `commitCheckedTaskWithGitClient`
- `resolveGitArtifactAndLockExcludes`, `isPathInsideRepo`, `buildCommitMessage`

Depends on `GitClient`, `PathOperationsPort`, `ConfigDirResult`. Could become `src/application/git-operations.ts`. Already partially reused by `revert-task.ts` (which has its own inline git helpers that could converge).

### 4. Template loading

- `loadProjectTemplatesFromPorts`, `ProjectTemplates` interface

Depends on `TemplateLoader`, `PathOperationsPort`, `ConfigDirResult`. Used identically in `reverify-task.ts`, `revert-task.ts`, `discuss-task.ts`. A shared `src/application/project-templates.ts` would eliminate duplication across all four use cases.

### 5. Task context resolution

- `resolveTaskContextFromRuntimeMetadata`, `findTaskByFallback`, `validateRuntimeTaskMetadata`
- `resolveLatestCompletedRun`, `hasReverifiableTask`, `isCompletedArtifactRun`
- `computeTaskContextMetrics`, `hasDescendantTasks`
- `ResolvedTaskContext` interface

Used by both `run-task.ts` (trace-only enrichment) and `reverify-task.ts`. Could become `src/application/task-context-resolution.ts`.

### 6. Checkbox filesystem operations

- `checkTaskUsingFileSystem`, `resetFileCheckboxes`, `maybeResetFileCheckboxes`, `countCheckedTasks`

Thin wrappers over domain `markChecked`/`resetAllCheckboxes` + `FileSystem`. Could become `src/application/checkbox-operations.ts` or remain as inline helpers if the orchestrator is small enough after other extractions.

### 7. CLI block expansion wiring

- `TemplateCliBlockExecutionError` class
- `withCommandExecutionHandler`, `withCliTrace`, `withSourceCliFailureWarning`, `withTemplateCliFailureAbort`
- `handleTemplateCliFailure`

Composable handler chain for `CommandExecutionOptions`. Could become `src/application/cli-block-handlers.ts`. The error class could move to domain if other modules need to catch it.

### 8. Trace orchestration state machine

The largest and most coupled extraction. The `activeTraceRun` mutable state and all functions that read/write it:

- `startTraceRun`, `beginPhaseTrace`, `completePhaseTrace`, `emitPromptMetrics`
- `emitTraceTaskOutcome`, `emitTraceTimingWaterfall`, `emitTraceRunCompleted`
- `emitResetPhaseTrace`, `runTraceEnrichment`, `emitDeferredTraceEvents`

These form a cohesive state machine that could be encapsulated as a class or module: `src/application/trace-run-session.ts`. The public surface would be a small set of lifecycle methods (`start`, `beginPhase`, `completePhase`, `emitOutcome`, `finalize`), hiding the internal `activeTraceRun` state.

### 9. Trace artifact formatting

- `collectTracePhaseArtifacts`, `summarizePhaseAnalyses`
- `formatPhaseTimingsForTrace`, `formatPhaseOutputsForTrace`
- `formatAgentSignalsForTrace`, `formatThinkingBlocksForTrace`, `formatToolUsageForTrace`
- `parseAnalysisSummaryFromWorkerOutput`
- Related interfaces: `TracePhaseMetadata`, `TracePhaseArtifact`, `PhaseAnalysisSummary`

Pure functions that read artifact files and format trace data. Could become `src/application/trace-artifacts.ts` (or `src/domain/trace-artifacts.ts` if FileSystem access is inverted).

### 10. Run lifecycle helpers

- `afterTaskComplete`, `afterTaskFailed`, `runOnCompleteHookWithProcessRunner`
- `finalizeRunArtifacts` (already exported)
- `failRun`, `finishRun`, `resetArtifacts`, `finalizeArtifacts` (closure-internal)

These are tightly coupled to the run state (`artifactContext`, `traceWriter`, `activeTraceRun`). They would naturally follow the trace state machine extraction — once trace orchestration is behind an interface, lifecycle helpers become thin wrappers.

### 11. Trace-only enrichment sub-flow

`runTraceOnlyEnrichment` is an independent code path (~130 lines) that short-circuits the main flow. After extracting trace artifacts and task context resolution, this could become a separate use case or a clearly separated internal function.

## Constraints and invariants

- **Public API stability**: `createRunTask`, `RunTaskDependencies`, `RunTaskOptions`, and the exported helper functions must remain importable from the same path (or be re-exported). The composition root (`src/create-app.ts`) and CLI layer (`src/presentation/cli.ts`) are the primary consumers.
- **Exit code contract**: The function returns `0` (success), `1` (execution/config error), `2` (verification failure), or `3` (no tasks found). This must be preserved.
- **Test coverage**: `run-task.test.ts` (~3,500 lines) currently covers all paths. Extracted modules should get their own test files, and `run-task.test.ts` should shrink to focus on orchestration flow. Existing tests should not break during extraction.
- **No behavioral changes**: This is a pure structural refactoring. No feature additions, no bug fixes, no changes to user-visible output or exit codes.

## Relationship to other modules

| Module | Shared pattern | Current state |
|---|---|---|
| `reverify-task.ts` (711 lines) | Template loading, task context resolution, trace artifact formatting, CLI block expansion | Inline reimplementation |
| `revert-task.ts` (759 lines) | Template loading, git operations, task context resolution | Inline reimplementation |
| `discuss-task.ts` (556 lines) | Template loading, CLI block expansion | Inline reimplementation |
| `plan-task.ts` (729 lines) | Template loading, CLI block expansion, trace writing | Inline reimplementation |

After extracting shared modules from `run-task.ts`, follow-up work can converge the other use cases onto the shared implementations. That convergence is out of scope for this decomposition — it would be a separate migration step.

migrate: rename docs command to design

## Goal

Rename the top-level revision lifecycle command from `rundown docs` to `rundown design` so command naming matches the canonical workspace model (`design/current/**`, `design/rev.*/**`) introduced by earlier migrations, while preserving existing revision behavior and linked-workspace resolution semantics.

## Why this migration now

The project already completed path-level renaming from legacy docs paths to canonical design paths:

- Canonical workspace defaults are `design/current/Target.md` and `design/rev.N/`.
- Legacy fallbacks remain supported for `docs/current/Design.md`, `docs/rev.*/`, and root `Design.md`.
- Revision lifecycle is currently exposed under `rundown docs` (`release`, `diff`) even though underlying behavior is design-centric.

This creates naming drift between command surface and actual domain model.

## Current implementation context

### CLI surface

- `src/presentation/cli.ts` currently defines:
  - `rundown docs release` as canonical revision snapshot command.
  - `rundown docs publish` as deprecated alias.
  - `rundown docs save` as removed alias with actionable error.
  - `rundown docs diff` for revision comparison.

### Command routing

- `src/presentation/cli-command-actions.ts` routes docs lifecycle via `docsTask` with action union `"release" | "diff"`.
- Compatibility behavior exists for deprecated/removed aliases and should be considered during rename.

### Application boundary

- `src/application/docs-revision-task.ts` contains the actual revision lifecycle logic:
  - Workspace resolution (`--workspace`, linked workspace handling).
  - Snapshot save via `saveDesignRevisionSnapshot(...)`.
  - Diff preview via `prepareDesignRevisionDiffContext(...)`.
  - Canonical-first design workspace checks with legacy fallback support.

### Domain logic

- `src/application/design-context.ts` already encodes design-first semantics and revision model:
  - Canonical workspace resolution first, legacy fallback second.
  - Immutable `rev.N` snapshot handling.
  - Deterministic diff computation and source-reference collection.

## Rename target and expected UX

Primary command family should become:

- `rundown design release`
- `rundown design diff`

Expected UX intent:

- Users manage design revision lifecycle under `design` command.
- `migrate` remains migration lifecycle only.
- Command naming aligns with filesystem naming and domain vocabulary.

## Scope

### In scope

- Rename command group from `docs` to `design` in CLI definitions/help.
- Rename action creators and references as needed for consistency at presentation layer boundaries.
- Update docs/examples/help text to make `design` the canonical lifecycle command namespace.
- Update tests that assert CLI command names, warnings, and error messaging.
- Preserve runtime behavior of release/diff workflows.

### Out of scope

- Changing snapshot/diff algorithms.
- Changing revision metadata schema or storage format.
- Removing legacy design-context fallbacks.
- Redesigning migrate flow or satellite generation behavior.

## Compatibility strategy considerations

A clear compatibility decision is required:

- Keep `rundown docs ...` as deprecated alias for one transition window, with warnings that point to `rundown design ...`.
- Or remove `rundown docs ...` immediately with explicit migration errors.

Given existing precedent (`publish` deprecated, `save` removed), a staged transition is likely lower risk:
- Introduce `design` as canonical now.
- Keep `docs` alias temporarily with warning.
- Remove alias in a later migration once docs/scripts/test fixtures are updated downstream.

## Integration touchpoints likely affected

- `src/presentation/cli.ts`:
  - Command group declaration (`docs` -> `design` canonical).
  - Help examples and description text.
- `src/presentation/cli-command-actions.ts`:
  - Command action factory names and messaging.
  - Error/warn text currently hardcoded with `docs` command wording.
- `src/create-app.ts` and exported app contracts:
  - Naming alignment (`docsTask` interface naming may need canonical `designTask` naming, with compatibility bridge if needed).
- `src/application/docs-task.ts`:
  - Currently an alias wrapper around docs-revision task; evaluate whether this becomes `design-task.ts` with compatibility exports.
- User-facing docs:
  - `docs/cli.md`, `docs/examples.md`, `docs/overview.md`, `README.md`.

## Behavioral constraints

- `release` behavior remains identical:
  - No-op when current draft matches latest revision.
  - Immutable snapshot creation and monotonic revision indexing.
  - Optional metadata label support.
- `diff` behavior remains identical:
  - Existing shorthand and selector constraints preserved.
  - Existing deterministic summary and file-level change output preserved.
- Linked workspace semantics remain unchanged:
  - Same ambiguity handling.
  - Same `--workspace` override behavior.

## Edge cases to preserve during rename

- Multi-record `.rundown/workspace.link` without default should still fail with candidate guidance.
- Configured non-default workspace directories (`designDir`, `migrationsDir`, etc.) must still route correctly.
- Deprecated/removed alias messaging should remain actionable and explicit.
- Integration tests that parse exact warning/error text may require synchronized updates.

## Acceptance criteria outline

- `rundown design release` executes the same snapshot lifecycle as current canonical `docs release`.
- `rundown design diff` executes the same diff lifecycle as current `docs diff`.
- CLI help and examples present `design` command group as canonical.
- User docs consistently describe revision lifecycle under `rundown design`.
- Existing snapshot/diff behavior remains unchanged at application/domain layers.
- Linked-workspace resolution and configured-directory behavior remain unchanged.
- Test coverage reflects canonical `design` naming plus any chosen compatibility path for legacy `docs` command invocations.

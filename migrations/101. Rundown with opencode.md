We want to add command `with <harness>`. It's like installation, you do

```bash
rundown with opencode
```

And preset for opencode will be configured (`opencode run $boostrap`, etc).
Same for other popular harnesses including pi.

This is a one-command onboarding flow: choose a known harness name and get a working `.rundown/config.json` worker setup without manually editing JSON.

## Intent

`rundown with <harness>` should be the fastest way to make a repo runnable with a known external worker harness, using project-safe defaults for:

- non-interactive execution (`run`/`plan`/`research`/`reverify`) via `... run`
- interactive discussion (`discuss`) via base CLI when applicable
- prompt transport via staged prompt files (`$file` + `$bootstrap`) for Windows/PowerShell reliability

## Existing architecture context

Current worker execution already supports everything needed for presets; the missing part is a UX command that writes the right config in one step.

Relevant existing boundaries:

- CLI command registration: `src/presentation/cli.ts`
- CLI action wiring: `src/presentation/cli-command-actions.ts`
- Worker resolution and profile/command mapping: `src/application/resolve-worker.ts`, `src/domain/worker-config.ts`
- Worker token expansion (`$file`, `$bootstrap`): `src/domain/worker-pattern.ts`
- Config mutation APIs (`config set/unset/get/list/path`): `src/application/config-mutation.ts` + worker config adapter
- Config schema/source of truth: `workers.default`, `workers.tui`, `workers.fallbacks`, `commands.<name>`

This means `with` should be mostly a preset-selection + config-write flow, not a new execution engine.

## Preset shape (what gets configured)

For each harness preset, configure at least:

- `workers.default`: deterministic non-interactive command
- `workers.tui` (when harness supports interactive mode): interactive command
- `commands.discuss` override when discuss should use non-`run` form
- prompt-transport-safe command pattern using `$file`/`$bootstrap` where required

OpenCode baseline should align with existing docs and defaults:

- deterministic flows: `opencode run`
- interactive discuss flow: `opencode`
- prompt transport: prefer file-based staging (`$file` + `$bootstrap`) for robustness

Note: existing docs consistently treat deterministic flows as `opencode run` and interactive discuss as `opencode`.

First-pass preset catalog:

This satisfies the requirement to include `pi` while covering common agent/harness choices teams currently adopt for terminal workflows.

## Behavioral contract

`rundown with <harness>` should:

- validate `<harness>` against known preset names (case-insensitive aliases are fine, canonical key persisted)
- create `.rundown/` and config file if missing (without clobbering unrelated existing config keys)
- merge preset keys into existing config object deterministically
- keep non-worker config intact (workspace placement, trace config, run defaults, toolDirs, etc.)
- print exactly what was configured in concise user-facing output
- fail with actionable message when harness is unknown, including supported list

## Scope boundaries

In scope:

- preset mapping for 7 harnesses
- command wiring and docs
- deterministic merge/write behavior
- tests for happy path + unknown harness + merge safety

Out of scope (this iteration):

- auto-installing external CLIs
- network detection/version probing of harness binaries
- dynamic online preset discovery
- per-provider auth/bootstrap wizard flows

## Edge cases and compatibility constraints

- Existing custom worker config must not be erased wholesale; only targeted preset keys are written.
- If user already has `workers.fallbacks`, preserve unless explicitly replaced by preset policy.
- If harness lacks distinct interactive mode, `workers.tui` can mirror default or be omitted intentionally.
- Keep prompt transport file-first friendly for Windows shells; avoid fragile inline mega-prompts.
- Preserve deterministic/interactive split conventions already used in docs and templates.
- Keep output/noise style aligned with current CLI behavior (timestamped output layer, actionable errors).

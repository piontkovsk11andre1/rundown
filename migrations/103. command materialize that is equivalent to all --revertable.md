migrate: command materialize that is equivalent to all --revertable.

## Intent

Add a dedicated `materialize` command that behaves like a full-run invocation with
revertable defaults, matching `run --all --revertable` while preserving the normal
run-style option surface.

## Current CLI context

- The CLI already has a first-class `run` command with shared run-like options
  registered through `configureRunLikeCommandOptions(...)`.
- The legacy `all` shortcut is implemented as argv rewrite (`all` -> `run --all`)
  in `src/presentation/cli-argv.ts`, not as an independent command action.
- Revertable behavior is normalized in `createRunCommandAction(...)`:
  `--revertable` implies `commitAfterComplete: true` and `keepArtifacts: true`,
  with explicit CLI flags taking precedence over config defaults.
- `materialize` is conceptually a deterministic flow and should align with the
  same worker conventions used by other deterministic commands (`opencode run`).

## Behavioral specification

`materialize <source>` should be equivalent to invoking:

`run <source> --all --revertable`

with these semantics:

- Always execute in run-all mode (`runAll: true`).
- Always enable revertable mode (`revertable: true`), which in turn keeps
  artifacts and enables auto-commit defaults.
- Accept the same run-like options as `run` (mode/sort/verify/repair/trace/
  commit options/worker options/etc.) unless explicitly constrained.
- Preserve separator-passed worker commands (`-- <worker command>`) and
  existing worker resolution rules.

## Option and precedence details

- `materialize` should reuse `createRunCommandAction(...)` plumbing instead of
  duplicating run-task request shaping logic.
- To guarantee revertable expansion consistently (including config-aware code
  paths that inspect raw argv), invocation argv should include `--revertable`
  before the `--` separator when missing.
- Explicit CLI values should still override config defaults where applicable;
  equivalence means the command defaults are forced to the same effective state
  as `run --all --revertable`, not that unrelated flags are ignored.

## Integration boundaries

- CLI registration lives in `src/presentation/cli.ts`.
- Run/materialize action normalization lives in
  `src/presentation/cli-command-actions.ts`.
- Alias rewriting behavior lives in `src/presentation/cli-argv.ts`.
- Coverage expectations span unit tests in:
  - `__tests__/presentation/cli-command-actions.test.ts`
  - `__tests__/presentation/cli-options.test.ts`

## Acceptance criteria outline

- A `materialize` command exists in CLI help with description indicating
  equivalence to `run --all --revertable`.
- Invoking `materialize <source>` produces the same effective run-task payload
  as `run <source> --all --revertable` for core fields (`runAll`,
  `commitAfterComplete`, `keepArtifacts`).
- Run-like options remain available on `materialize` via shared option
  configuration.
- Tests assert action-level semantics and CLI-level parsing/expansion behavior.

## Out of scope

- Changing migration workflow semantics under `migrate` actions (`up/down/...`).
- Introducing new commit policies beyond existing run/revertable rules.
- Altering interactive `discuss` flow behavior (`opencode` without `run`).

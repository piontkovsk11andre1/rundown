## Context

Planning currently post-processes worker edits in `src/application/plan-task.ts` via
`relocateInsertedTodosToEnd(beforeSource, afterSource)` from
`src/domain/planner.ts`.

That relocation step exists to keep add-only planning stable: tasks should not be
inserted between narrative paragraphs or mixed into earlier sections when the
worker adds new `- [ ]` items.

The intent of this migration is to make placement deterministic but less blunt:

- when the document truly ends with a TODO list block, append newly added TODO
  items to that trailing block;
- when there is non-list content after existing TODO items, append new TODO
  items at the document end (after that content), instead of moving unrelated
  blocks or forcing insertion into earlier TODO groups.

## Existing behavior and implementation surface

- Relocation hook: `src/application/plan-task.ts` (scan loop, after
  `validatePlanEdit(...)`, before convergence checks).
- Relocation implementation: `src/domain/planner.ts`,
  `relocateInsertedTodosToEnd(...)`.
- Parsing conventions used by relocation:
  - checkbox regex allows `- [ ]` and `- [x]` style items;
  - fenced code blocks are tracked and ignored for TODO detection;
  - line identity uses normalized checkbox text counting so only truly new TODO
    lines are candidates for movement.
- Coverage location: `__tests__/domain/planner.test.ts` under
  `describe("relocateInsertedTodosToEnd", ...)`.

## Behavioral specification

For each planning scan edit, compare document state before and after worker
output and relocate only newly introduced TODO checkbox lines.

Placement policy:

1. If, after removing newly inserted TODO lines, the resulting document ends in
   a TODO checkbox block, append new TODO lines directly to that block.
2. Otherwise, append new TODO lines at the very end of the document.
3. Preserve existing TODO item order and placement.
4. Preserve all non-TODO content and relative order (prose, headings, examples,
   code fences).
5. Preserve newline style (`LF`/`CRLF`) and final newline behavior.

Non-goals:

- No rewriting, rewording, or reformatting existing TODO text.
- No movement of existing checked/unchecked items.
- No changes to plan validation rules (checked item protection, add-only safety,
  prefix-fix handling).

## Edge cases to explicitly handle

- Checkbox-looking lines inside fenced blocks must not be treated as real TODO
  items for relocation.
- Duplicate TODO text must be handled by count, not by naive string lookup, so
  only newly added occurrences are moved.
- Mixed checked/unchecked trailing task groups still count as a trailing TODO
  block.
- Documents with trailing blank lines should keep stable final formatting after
  relocation.
- If no new TODOs were introduced by the worker, relocation must be a no-op.

## Integration notes

- Keep relocation in the same post-validation stage inside the planning scan
  loop so downstream convergence logic sees the canonicalized document.
- Keep the change isolated to planner relocation logic and associated tests;
  avoid touching unrelated run/research/discuss flows.
- This migration is marked `fast:` and should stay implementation-focused
  without introducing cross-command behavior changes.

## Acceptance criteria outline

- New TODO items are appended to an existing trailing TODO block when that block
  is the last meaningful content in the document.
- New TODO items are appended to document end when non-TODO content appears
  after existing TODO sections.
- Existing content order is preserved; only newly added TODO lines may move.
- Fenced code block content does not participate in TODO relocation.
- Unit tests in `__tests__/domain/planner.test.ts` cover both primary placement

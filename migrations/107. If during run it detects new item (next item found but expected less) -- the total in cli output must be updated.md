## Problem context

During `run --all` style execution, the CLI emits per-task progress counters via `group-start` events (rendered as `[current/total]` in the terminal output).  
Today, the total is initialized from unchecked task count at loop start and can become stale if task shape changes while the run is in progress.

A common case is when an executed task appends a new unchecked TODO item. The next selected item is valid, but the previously computed total is too small, so progress output can show an inconsistent counter (for example, a later task being reported against an outdated total).

## Current behavior and relevant code paths

- `src/application/run-task-execution.ts`
  - `countUncheckedTasks()` computes unchecked totals from parsed files.
  - `totalTasks` is initialized before the run loop.
  - `totalTasks` is currently refreshed on force-retry paths (`force:` handling), but not as a general pre-iteration refresh.
  - `currentTaskIndex` increments as tasks complete and is passed into iteration context.
- `src/application/run-task-iteration.ts`
  - Emits `group-start` with `counter: { current: taskIndex + 1, total: totalTasks }`.
  - This is the user-visible source of per-task progress totals.
- `src/presentation/output-port.ts`
  - Renders counters directly from event payload; it does not reconcile stale totals.
  - Therefore correctness must be handled in application-layer counter computation.
- Existing tests already validate related behavior:
  - force-retry total recalculation in `__tests__/application/run-task-execution.test.ts`.
  - completion messaging in integration tests (for example, `All tasks completed (N tasks total)`).

## Intended behavior

When run-time task discovery changes unchecked task count (especially when it increases), the next emitted task counter must reflect the updated total, not the original snapshot.

This applies to deterministic non-interactive and TUI flows, including parallel-sibling sequencing where applicable.

## Scope

In scope:
- Keep per-task CLI counters synchronized with the latest unchecked task total during an active run loop.
- Ensure newly discovered tasks do not cause stale denominator values in progress output.
- Preserve existing retry/failover/parallel behavior and message formats unless strictly required for correctness.

Out of scope:
- Redesigning output formatting, counter styling, or timestamp rendering.
- Changing task-selection ordering semantics.
- Introducing new execution strategies beyond keeping totals accurate.

## Acceptance criteria outline

- If a task creates a new unchecked task during execution, subsequent `group-start` events use the increased total.
- No stale denominator appears when the next runnable task is selected after mid-run document mutation.
- Existing force-retry counter behavior remains valid.
- Existing success summary behavior remains intact (`All tasks completed (N tasks total)` still reports actual completed count).
- No regressions in parallel batch handling, TUI sequential fallback, or non-interactive execution.

## Compatibility and implementation notes

- The safest place to refresh total is in the run-loop flow before emitting each task’s `group-start` (or immediately before assembling iteration execution context), so every iteration uses fresh totals.

- Ensure recalculation is based on unchecked tasks across active files, consistent with existing `countUncheckedTasks()` semantics.

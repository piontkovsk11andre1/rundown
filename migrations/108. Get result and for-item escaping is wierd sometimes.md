## Intent

Stabilize how extraction-style built-ins (`get:` and `for:`/`each:`/`foreach:`) consume model output and persist metadata lines, so reruns are deterministic and human-readable, and markdown rendering/parsing does not produce surprising escaping artifacts.

The core pain here is that current escaping and prefix expectations are partly different between `get-result:` and `for-item:` flows, which can make output look inconsistent or “over-escaped” even when runtime behavior is technically correct.

## Why this is showing up now

Recent migrations introduced both capabilities with explicit round-trip safety:

- `get:` extraction writes `get-result:` sub-items.
- loop extraction writes `for-item:` sub-items and uses `for-current:` cursoring.
- both flows parse model text and normalize line formats.
- both flows already have tests for escaping/round-trip behavior.

As usage increases, the UX issue becomes less about correctness and more about output shape consistency and predictability.

## Current implementation context

### `get:` behavior today

In `src/domain/builtin-tools/get.ts`:

- Model prompt currently requests JSON output (`{"results":[...]}`), with fallback plain-text line parsing.
- Text parsing accepts bullets/ordered lists and strips optional `get-result:` prefix.
- Persisted `get-result:` values go through `renderGetResultValue(...)`.
- `renderGetResultValue(...)` wraps values containing markdown-sensitive chars in inline code fences (dynamic backtick length).
- Existing persisted values are read via `decodeRenderedGetResultValue(...)` + normalization, enabling rerun reuse/refresh.
- Fenced blocks are intentionally ignored during child scanning and parsing to avoid accidental mutation.

Observed implication: `get-result:` often uses inline-code wrapping instead of backslash escapes, which is safe but visually different from loop metadata.

### `for-item:` behavior today

In `src/domain/for-loop.ts` and `src/domain/builtin-tools/for-loop.ts`:

- Loop output parsing accepts markdown bullets/ordered lines and strips optional `for-item:` prefix.
- Persisted `for-item:` lines are generated via `formatForLoopItemMetadataLine(...)`.
- Escaping uses backslash strategy (`escapeForLoopMetadataValue`) for `\`, `` ` ``, `*`, `_`, `[`, `]`, `<`, `>`.
- Metadata read path unescapes those characters for runtime use.
- Existing `for-item:` metadata is reused when present (trust metadata policy).

Observed implication: loop metadata favors explicit backslash escaping, not inline code wrapping.

## Problem statement (expanded)

The issue is not just “escaping is weird”; it is a contract mismatch across two similar data-capture tools:

- `get-result:` persistence style differs from `for-item:` persistence style.
- prompts still ask model for prefixed lines in some places, while parser can already infer and normalize without strict prefixes.
- user-facing metadata lines can become noisier than needed (extra code-fence wrappers or escape sequences).
- inconsistent style increases cognitive load when scanning task docs.

## Proposed direction for planning

### 1) Normalize model-output contract

Canonical model-output contract (final) for both `get:` and loop extraction (`for:`/`each:`/`foreach:`):

- model should return one logical value per line,
- markdown bullets and ordered-list lines are valid and equivalent to plain lines,
- literal metadata prefixes (`get-result:` / `for-item:`) are not required in model output,
- each non-empty parsed line is treated as markdown text content for logical-value extraction,
- output order is preserved exactly as returned,
- duplicate values are allowed and preserved,
- if no values are extracted, existing empty-result policy behavior remains unchanged per tool.

Backward-compatible parser tolerances (explicit):

- shared tolerances for both flows:
  - accept plain one-line items, markdown bullets (`-`, `*`, `+`), and ordered lines (`1.`, `1)`),
  - accept and strip optional leading metadata prefixes (`get-result:` or `for-item:`) when present,
  - normalize surrounding whitespace around extracted logical values,
  - ignore empty list items/blank lines after normalization,
  - preserve item order and duplicates after normalization.
- `get:`-specific tolerance:
  - continue accepting JSON object output with `results` array as a legacy-compatible input shape,
  - if JSON parsing fails or JSON shape is invalid, fall back to markdown/text line parsing without failing solely due to shape.
- loop-specific tolerance:
  - continue accepting prefixed or non-prefixed list/text items identically,
  - continue treating trusted existing `for-item:` metadata as authoritative when present, without changing loop trust semantics.

This aligns with existing parser resilience already present in both handlers.

### 2) Separate model-output parsing from persistence rendering

Keep two explicit phases:

- parse phase: robustly extract logical values from JSON or markdown bullets/lines.
- render phase: apply one deliberate metadata rendering policy per tool.

This makes behavior explainable and testable even if prompts evolve.

### 3) Decide and document one escaping policy family

Choose whether metadata rendering should converge:

- either both tools use inline-code wrapping when needed,
- or both tools use minimal backslash escaping,
- or allow tool-specific policy but document why and when.

Current mismatch is functional but surprising; this migration should define whether consistency is required or only compatibility is required.

Decision (canonical policy): use one shared minimal backslash-escaping renderer for both `get-result:` and `for-item:` metadata lines.

- Canonical persistence format:
  - `get-result: <escaped-value>`
  - `for-item: <escaped-value>`
- Canonical escaping set (same in both tools): escape literal `\\`, `` ` ``, `*`, `_`, `[`, `]`, `<`, `>` using leading backslash.
- Canonical rendering rule: persist plain text unless a character from the escaping set appears; do not introduce inline-code wrappers for newly persisted metadata.

Rationale:

- Visual consistency: both extraction families render with the same style, reducing scan-time cognitive load.
- Deterministic round-trip: one encoding family simplifies decode/rehydration rules and test expectations across reruns.
- Markdown safety with low ceremony: targeted escaping prevents accidental markdown interpretation while keeping lines mostly readable.
- Migration safety: this converges on behavior already proven in loop metadata tests, minimizing new surface area.

Concrete examples (canonical persisted form):

| Logical value | `get-result:` persisted line | `for-item:` persisted line |
| --- | --- | --- |
| `simple value` | `get-result: simple value` | `for-item: simple value` |
| `use `code`` | `get-result: use \`code\`` | `for-item: use \`code\`` |
| `*wild* _emph_` | `get-result: \*wild\* \_emph\_` | `for-item: \*wild\* \_emph\_` |
| `[label](x) <tag>` | `get-result: \[label\](x) \<tag\>` | `for-item: \[label\](x) \<tag\>` |
| `path\\name` | `get-result: path\\\\name` | `for-item: path\\\\name` |

Compatibility notes for this decision:

- Read-path compatibility remains required for both tools:
  - parse canonical backslash-escaped lines,
  - parse legacy `get-result:` inline-code wrapped values,
  - parse already-plain legacy lines,
  - normalize all of the above to the same logical runtime value.
- Write-path compatibility policy:
  - on refresh/new persistence, emit canonical shared backslash-escaped format,
  - on reuse flows that trust existing metadata, preserve trusted lines unless policy explicitly refreshes them.

## Design constraints and compatibility requirements

Any change should preserve the following guarantees:

- No mutation of fenced code blocks.
- Metadata lines remain non-checkbox `subItems`.
- Existing documents with current escaped forms continue to parse and rerun deterministically.
- `get` rerun policies (`reuse` / `refresh`) and empty-result policies (`marker` / `fail`) remain unchanged unless explicitly expanded.
- Loop metadata trust behavior for existing manual `for-item:` lines remains deterministic.
- Prefix-chain compatibility with other built-ins remains unaffected.

## Edge cases to keep explicit

- Values containing backticks, unmatched backticks, or repeated backtick runs.
- Values containing markdown syntax markers (`*`, `_`, `[]`, `<...>`, `\`).
- Values containing `:` immediately after metadata prefix labels.
- Mixed outputs (JSON object + markdown text fallback failure paths).
- Duplicate values and order preservation.
- Empty outputs under each policy mode.
- Existing legacy persisted lines that are already inline-coded or backslash-escaped.

### Edge-case expected outcomes (parse/render/round-trip)

This matrix defines explicit expected behavior for both extraction families under the canonical shared backslash-escaping policy.

| Edge-case category | Parse outcome | Render/persist outcome | Round-trip outcome |
| --- | --- | --- | --- |
| Backticks (single, unmatched, repeated runs) | Treat each parsed line as literal text content after list/prefix normalization; do not treat backticks as syntax delimiters during logical-value extraction. | Persist with canonical escapes for literal backticks and backslashes only as needed (for example, `` ` `` becomes `\``; existing `\\` remains escaped). Do not emit inline-code wrappers for new writes. | Reading persisted metadata decodes escapes back to the original literal backtick content; rerun reuse/refresh resolves to the same logical value. |
| Markdown markers (`*`, `_`, `[`, `]`, `<`, `>`, `\\`) | Parse preserves literal marker characters in logical values (after optional bullet/ordered marker and optional metadata-prefix stripping). | Persist with minimal leading-backslash escaping for canonical set (`\\`, `` ` ``, `*`, `_`, `[`, `]`, `<`, `>`). | Decode/unescape restores the same literal logical value; repeated reruns remain stable (no progressive double-escaping). |
| Colons after labels (for example `get-result: value:with:colons`) | Strip only the first recognized optional metadata prefix token (`get-result:` or `for-item:`) at line start; preserve all remaining colons in the extracted value exactly. | Persist as canonical `get-result: <escaped-value>` / `for-item: <escaped-value>` with colons in value unchanged unless another escapable character is present. | Decoding preserves colon placement exactly; reparsing persisted lines produces the same logical value. |
| Mixed JSON/text fallback (`get:` path) | If input is valid JSON object with `results` array, use array items in order. If JSON parsing fails or shape is invalid, fall back to markdown/text line parsing instead of failing due to shape alone. | Persist extracted values using canonical escaping policy and `get-result:` metadata lines. | Rerun parsing yields the same ordered values regardless of whether source came from JSON or fallback text; metadata form is deterministic after persistence. |
| Duplicates and order preservation | Keep non-empty parsed items in original order; do not deduplicate. | Emit one metadata sub-item per parsed value, preserving original sequence, including repeated identical values. | Reading metadata rehydrates identical sequence length and order; reruns keep duplicates intact. |
| Empty outputs | After normalization, if no non-empty values remain, treat as empty extraction result and defer to existing per-tool empty-result policy (unchanged by this migration). | Do not invent placeholder values beyond existing policy behavior; persistence follows current tool policy (`get` marker/fail semantics and existing loop semantics unchanged). | Repeated runs preserve current deterministic empty-result behavior for each tool mode without introducing new metadata-shape side effects. |
| Legacy persisted formats (inline-coded `get-result` and prior escaped lines) | Parse accepts canonical backslash-escaped lines, legacy inline-code-wrapped `get-result` values, and already-plain legacy values; normalize to the same logical value model. | On refresh/new writes, emit canonical shared backslash-escaped format; on trusted reuse paths, preserve existing trusted metadata lines per current trust policy. | Old and new persisted forms resolve to identical runtime values; refresh converges representation without changing logical content. |

Feature planning is complete when the document clearly defines:

- canonical expected model output shape for `get:` and `for:` flows,
- canonical persisted metadata rendering policy (and rationale),
- deterministic round-trip behavior for markdown-sensitive content,
- compatibility behavior for existing persisted docs,
- unchanged treatment of metadata as `subItems`,
- unchanged fenced-block safety behavior,
- required test updates for all edge-case categories above.

## Scope boundaries

In scope:

- clarification/unification of extraction output contracts,
- escaping/rendering consistency decisions for `get-result:` and `for-item:`,
- parser/renderer compatibility notes and regression expectations.

Out of scope:

- introducing new markdown syntax beyond current list/sub-item conventions,
- changing loop orchestration semantics unrelated to metadata rendering,
- changing broader CLI/task execution architecture outside extraction metadata behavior.

# Feature: Invert agent output visibility default

## Summary

Invert the `--hide-agent-output` CLI flag so that worker/agent stdout and stderr are **hidden by default**. Replace the opt-out `--hide-agent-output` flag with an opt-in `--show-agent-output` flag. When the flag is enabled, only produce agent output for the **execute**, **verify**, and **plan** stages — all other stages remain silent.

## Current behaviour

### The flag

The `--hide-agent-output` flag exists on two CLI commands:

- `run` — [src/presentation/cli.ts](src/presentation/cli.ts) line 115
- `discuss` — [src/presentation/cli.ts](src/presentation/cli.ts) line 147

Its default is `false`, meaning **agent output is shown by default**. Users must explicitly pass `--hide-agent-output` to suppress it.

### Where the flag takes effect

Worker stdout/stderr is emitted through an `emit({ kind: "text" | "stderr", text })` pattern. The `hideAgentOutput` boolean gates these emissions in two places:

1. **Task execution dispatch** ([src/application/task-execution-dispatch.ts](src/application/task-execution-dispatch.ts)) — the `emitExecutionWorkerOutput` helper returns early when `hideAgentOutput` is true. This covers three execution paths:
   - Inline CLI execution (e.g. `` ```sh `` blocks)
   - Rundown delegation (nested `rundown run` calls)
   - Standard worker execution in `"wait"` mode

2. **Discuss task** ([src/application/discuss-task.ts](src/application/discuss-task.ts)) — output is suppressed when `hideAgentOutput` is true **or** `keepArtifacts` is true (line 390).

### Where the flag does NOT take effect

- **Verify/repair loop** ([src/application/verify-repair-loop.ts](src/application/verify-repair-loop.ts)) — has no `hideAgentOutput` check. Verification and repair phases only emit `info`, `warn`, `error`, and `success` status messages (never raw worker stdout/stderr to the console), so the flag is irrelevant here.

- **Plan task** ([src/application/plan-task.ts](src/application/plan-task.ts)) — `PlanTaskOptions` has no `hideAgentOutput` field. Plan always emits stderr from the planner worker when in `"wait"` mode.

- **Reverify task** ([src/application/reverify-task.ts](src/application/reverify-task.ts)) — no `hideAgentOutput` field in `ReverifyTaskOptions`.

### Propagation to child processes

When running nested rundown processes (delegation), the parent's `hideAgentOutput` value is forwarded via `--hide-agent-output` in the child's argument list ([src/application/rundown-delegation.ts](src/application/rundown-delegation.ts) lines 54-55, [src/infrastructure/inline-rundown.ts](src/infrastructure/inline-rundown.ts) lines 171-172).

### Separate concept: hideHookOutput

`hideHookOutput` (default `false`) controls output from `--on-fail` and `--on-complete` lifecycle hooks in [src/application/run-lifecycle.ts](src/application/run-lifecycle.ts). This is a separate concern and is **not affected by this feature**.

### Execution stages in the codebase

The codebase tracks these execution phases via `traceRunSession`:

| Stage | Worker output emitted? | hideAgentOutput checked? |
|---|---|---|
| **execute** | Yes — stdout/stderr via `emitExecutionWorkerOutput` | Yes |
| **verify** | No — only status messages (info/warn/success/error) | N/A |
| **repair** | No — only status messages | N/A |
| **plan** | Partially — stderr always emitted in wait mode | No |
| **discuss** | Yes — stdout/stderr gated by two conditions | Yes |
| **on-fail / on-complete hooks** | Yes — gated by `hideHookOutput` | No (separate flag) |
| **rundown-delegate** | Yes — via `emitExecutionWorkerOutput` | Yes (inherited) |

### CLI rendering

- `"text"` events → `console.log()` (stdout)
- `"stderr"` events → `process.stderr.write()` (stderr)
- `"info"`, `"warn"`, `"success"` → `console.log()` with colored symbols (always shown, not affected by this flag)
- `"error"` → `console.error()` (always shown)

Only `"text"` and `"stderr"` events from worker processes are gated by `hideAgentOutput`. Status messages are always visible regardless of the flag.

## Desired behaviour

### 1. Invert the flag

- Remove `--hide-agent-output` from both `run` and `discuss` commands.
- Add `--show-agent-output` with default `false`.
- Internally rename `hideAgentOutput: boolean` to `showAgentOutput: boolean` throughout the option interfaces and threading path.
- Invert the logic: output is emitted when `showAgentOutput` is **true** (currently emitted when `hideAgentOutput` is **false**).

### 2. Stage-specific output visibility

When `--show-agent-output` is passed, only produce agent output for these stages:

| Stage | Show output when flag is set? |
|---|---|
| **execute** | Yes |
| **verify** | Yes (currently no raw output emitted — but if it were, it should be visible) |
| **plan** | Yes |
| **discuss** | No — remains silent for agent output |
| **repair** | Same as verify |
| **hooks** | No change — controlled by `hideHookOutput` separately |

Without `--show-agent-output`, all worker stdout/stderr is suppressed across all stages (status messages remain visible as always).

### 3. Propagation update

Child process delegation must forward `--show-agent-output` instead of `--hide-agent-output`. The inheritance logic in `rundown-delegation.ts` and `inline-rundown.ts` needs to be updated accordingly.

### 4. Affected commands

The flag should be available on the same commands that currently have `--hide-agent-output`:

- `run` / `all`
- `discuss`

And potentially extended to:

- `plan` (currently has no agent output control — stderr is always emitted)

## Design notes

- The verify/repair loop currently emits only status messages (`info`, `warn`, `success`, `error`) and never raw worker stdout/stderr. No changes are needed there unless we want to start surfacing raw verification worker output, which is a separate concern.
- The discuss command's dual condition (`!hideAgentOutput && !keepArtifacts`) complicates the inversion. With the new flag, discuss output should remain suppressed regardless of `--show-agent-output` — so the discuss path should not check the new flag at all, or should always suppress worker output.
- Plan task currently always emits stderr. The new behaviour should gate plan stderr behind `showAgentOutput` as well, making plan silent by default.

We have verify-repair loops, through config we need to be able to define different workers, default first time, second repair might be with stronger model, repair after resolution (might be other model) too and resolver itself. Also, we might after last repair (after resolver) move to a separate worker instead of full stop. But this is not the same as technical fallback (health). Which means -- if we are using default worker or tui -- use fallbacks, if this is specific worker for repair on level 2 -- this won't fallback. On reset (instead of stop) we want to revert commits (rundown revert them) and try again on separate worker (not default with default-fallbacks).

## intent and problem framing

Current run behavior already has verify/repair/resolve phases, worker fallback health logic, and retry loops, but worker selection is mostly single-chain per iteration. This migration defines phase-aware worker routing so model strength can escalate across verification/repair levels without conflating that with infra failover.

We need to separate two different mechanisms:

- semantic routing: "for this phase/attempt, use this specific worker/model"
- technical failover: "this worker is unhealthy/unavailable, try fallback candidate"

Those must not be treated as the same thing.

- revert/reset semantics and commit artifact requirements are documented in `docs/cli-revert.md` and `docs/overview.md`

Important baseline constraints from existing behavior:

- fallback candidates come from `workers.fallbacks`
- health failover currently reacts to retryable failure classes (usage-limit, transport-unavailable)
- resolve-informed repair already exists (`--resolve-repair-attempts`)
- `reverify` replays verify/repair from artifacts, without mutating source checkboxes
- git revertability currently depends on runs executed with commit metadata/artifacts

## required behavior changes

## 1) phase-level worker selection

Add explicit worker-selection points for the full verify/repair lifecycle, not just one resolved command per task iteration. Selection points must include at least:

- verify (initial verification)
- repair attempt N (pre-resolve loop)
- resolve diagnosis phase
- resolve-informed repair attempt N (post-resolve loop)
- optional post-exhaustion reset worker (semantic retry path)

Expected outcome: first pass can stay on default model, later attempts can route to stronger/specialized workers.

## 2) separation of semantic routing vs health failover

Rules to enforce:

- when phase worker is inherited from default/tui resolution, normal health failover remains enabled
- when phase worker is explicitly configured for that phase/attempt, do not auto-append `workers.fallbacks`
- explicit phase workers are deterministic: run the configured command only, fail as-is if that command fails
- this non-fallback behavior applies even when global fallback policy exists

This creates a strict distinction between "escalation by design" and "fallback by health."

## 3) level-aware repair escalation

Config must support attempt-sensitive escalation (example: attempt 1 = baseline, attempt 2+ = stronger model). This includes both:

- standard repair attempts before resolve
- resolve-informed repair attempts after diagnosis

Escalation should be controllable without changing task text or requiring per-task CLI overrides.

## 4) reset-after-exhaustion flow with dedicated worker

After all configured repair and resolve-informed attempts are exhausted, allow optional reset strategy instead of immediate hard stop:

- undo task-side git effects for this failed cycle (revert/reset contract must be explicit and safe)
- re-run the task through a dedicated reset worker path
- reset worker path is semantic retry and must not inherit default fallback chain unless explicitly requested

This is conceptually closer to "start fresh with another strategy" than to technical failover.

## proposed config surface (planning shape)

Keep existing config precedence/cascade, but introduce a phase-scoped extension under run semantics. One workable shape:

- phase worker map for run lifecycle (`execute`, `verify`, `repair`, `resolve`, `resolveRepair`, `reset`)
- optional attempt selectors for repair phases (exact attempt number and/or range semantics)
- explicit fallback policy flag per phase entry (default off for explicit entries, inherited on for default/tui-derived entries)

Compatibility expectation:

- if new phase config is absent, behavior remains identical to current run/reverify behavior
- existing `workers.default`, `workers.tui`, `workers.fallbacks`, `commands.*`, and `profiles.*` keep current meaning

## integration boundaries and likely touchpoints

Domain/config:

- extend worker config types and validation in `src/domain/worker-config.ts`
- keep current merge semantics (global -> local, array replace behavior)

Resolution:

- extend `resolveWorkerPatternForInvocation`/selection pipeline in `src/application/resolve-worker.ts` to accept phase context and attempt index
- ensure candidate construction can skip configured fallbacks when phase route is explicit

Execution loop:

- thread phase route context into `src/application/verify-repair-loop.ts` for each verify/repair/resolve callsite
- preserve existing verification-store and trace behavior while changing worker source per phase
- maintain current usage-limit detection behavior independent of model routing changes

Run orchestration:

- align failover budgets in `src/application/run-task-execution.ts` with new semantic reset path so health failover limits and semantic retries are independently counted
- ensure retry-boundary git-state handling remains safe with added reset cycles

Reverify:

- apply the same phase-routing rules to `reverify` in `src/application/reverify-task.ts` so replayed verification reflects configured escalation strategy

Docs/CLI:

- update `docs/configuration.md`, `docs/cli-run.md`, `docs/cli-reverify.md`, and `docs/overview.md` to document phase routing and fallback distinction
- keep CLI flags backward-compatible; new behavior should be mostly config-driven

## reset/revert design constraints

Because "reset and try again" touches git history, define hard boundaries:

- reset strategy must be explicit and opt-in
- must not silently perform destructive history rewrite beyond already documented revert/reset contracts
- behavior must be deterministic and auditable in artifacts
- if commit metadata is unavailable for the targeted cycle, reset strategy must fail with actionable guidance rather than guessing

If reset uses existing revert infrastructure, clearly document whether it reverts per-task commit, run-level commit (`file-done`), or retry-boundary stash path.

## edge cases to plan for

- phase worker configured but command missing/invalid
- explicit phase worker points to unavailable backend and has no fallback by policy
- mixed mode runs (`tui` execution but wait-mode verify/repair) with phase overrides
- reverify of historical runs after config changes (determinism vs current-config behavior must be defined)
- interaction with verify-only / fast-execution / tool-expansion intents
- lock and artifact behavior when reset introduces additional attempt cycles

## scope and non-goals

In scope:

- phase-aware worker selection across verify/repair/resolve/reset lifecycle
- explicit policy separation between semantic phase routing and technical failover
- configuration and docs updates needed to make behavior predictable

Out of scope (for this migration step):

- redesign of worker health store schema unless strictly required
- new task-text syntax for selecting models inline
- changing core parser/task-selection semantics

## acceptance criteria outline

- can configure different worker/model for verify, repair, resolve, and resolve-informed repair
- can configure stronger worker for later repair attempts without CLI override
- explicit phase-configured worker does not use `workers.fallbacks` unless explicitly enabled
- default/tui inherited workers still use health-based fallback behavior as today
- after final repair exhaustion, optional reset path can revert/reset cycle state and retry on dedicated worker
- reset worker path is treated as semantic retry, not health fallback
- legacy configs (without new keys) produce current behavior unchanged
- docs clearly explain fallback-vs-escalation distinction and reset semantics

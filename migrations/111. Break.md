## Goal

Add explicit terminal-control prefixes for early stop semantics (`quit`, `exit`, `end`, plus `break`/`return`) that behave differently from `optional`/`skip`:

- In normal `run` flow: stop execution after applying full skip behavior for remaining work in scope, then exit cleanly.
- In `rundown loop`: stop the outer loop immediately (no next iteration).
- Keep existing `optional:` / `skip:` semantics unchanged.

## Current behavior baseline (important)

Today, control-flow prefixes are handled through `endHandler` in `src/domain/builtin-tools/end.ts`, and aliases are registered in `src/domain/builtin-tools/index.ts`:

- `optional`, `skip`, `end`, `return`, `quit`, `break` all route to the same handler.
- That handler evaluates condition text via worker and, when true, sets `skipRemainingSiblings`.
- Sibling skipping is applied in `src/application/complete-task-iteration.ts` through `skipRemainingSiblingsUsingFileSystem`.
- This is sibling-scope short-circuiting, not an explicit global stop signal for outer loop command iterations.
- There is currently no built-in `exit` alias in the registry.

So current implementation couples legacy terminal words (`end`/`quit`/`break`/`return`) to optional-skip behavior. This migration asks to separate that.

## Required semantic split

### 1) Keep optional skip semantics unchanged

`optional:` and `skip:` remain conditional sibling short-circuit:

- condition evaluated by worker
- true => skip remaining siblings/descendants in the same parent scope
- false/ambiguous => continue
- no change to existing skip-optional behavior, docs, or compatibility guarantees for this path

### 2) Introduce terminal stop semantics for end-family prefixes

`quit`, `exit`, `end`, `break`, `return` should represent stop-run intent:

- They should not alter `optional/skip` behavior.
- In normal run, they should stop remaining execution flow after applying defined skip behavior.
- In `rundown loop`, they must stop the outer iteration loop (equivalent to graceful “break loop now”).

This likely requires a distinct handler/result contract from `skipRemainingSiblings` alone.

## Design constraints

- Backward compatibility: existing task files using `quit:` / `end:` / `return:` / `break:` should still parse and execute; behavior update must be explicit in docs.
- Non-breaking for `optional`/`skip`: no regression in condition handling, sibling-skip marking, or verification behavior.
- Deterministic state mutation: checked/skip updates should still use existing bottom-up-safe mutation flow.
- Loop stop must be graceful: no hard process kill; preserve lock cleanup, artifact finalization, and summary output.
- Exit code policy should be explicit:
  - graceful intentional stop should normally return success unless configuration says otherwise.

## Terminal interaction contract

- `exit:` with empty payload is allowed and means unconditional terminal stop.
- Terminal prefixes are conditional only when payload is present:
  - non-empty payload is evaluated as the same yes/no condition contract used by current optional/skip condition evaluation
  - condition `true` => emit terminal stop signal
  - condition `false` or ambiguous => no terminal stop signal (continue normally)
- Interaction with `--all` is deterministic:
  - when terminal stop is raised, no new task files are selected after the current lifecycle completes
  - remaining not-yet-started files are left unmodified (not auto-checked, not auto-skipped)
  - run summary reports intentional terminal stop and counts only work that actually started
- Interaction with parallel sibling execution is deterministic:
  - first observed terminal stop signal in the active batch becomes authoritative
  - queued siblings that have not started are not started
  - siblings already running are allowed to finish their current lifecycle; after batch completion, no further sibling/task scheduling occurs
  - skip/check mutations stay confined to items that actually executed, preserving existing deterministic bottom-up mutation order
- Interaction with verify/repair is deterministic:
  - when terminal stop is raised, the current task iteration still completes its integrity/finalization steps
  - verify/repair stages are not started for any future not-yet-started tasks
  - no additional repair retries are scheduled after a terminal stop signal
- Interaction with `--continue-on-error` in loop mode is deterministic:
  - terminal stop intent always has higher precedence than continue-on-error retry/advance policy
  - once terminal stop is surfaced to loop orchestration, the outer loop exits immediately after current iteration finalization
  - loop cooldown/wait logic is skipped for the post-stop path because no next iteration is allowed

## Scope

### In scope

- Add/normalize terminal aliases (`quit`, `exit`, `end`, `break`, `return`) for stop-run/stop-loop intent.
- Preserve `optional/skip` behavior exactly as-is.
- Propagate terminal signal from tool handler to run-loop and loop command controller.
- Update docs/help text to clearly separate “optional skip” from “terminal stop”.

### Out of scope

- Reworking sibling-skip algorithm internals.
- Changing unrelated prefix systems (`verify`, `memory`, `fast`, `include`, `profile`, `force`, `for`).
- Redesigning loop cooldown/retry policy outside terminal-stop integration.

## Acceptance criteria outline

- `optional:` and `skip:` behavior is unchanged from current baseline.
- `quit:` / `exit:` / `end:` / `break:` / `return:` are supported as terminal stop prefixes.
- In normal run, terminal prefix causes remaining execution to stop cleanly after defined skip handling.
- In `rundown loop`, terminal prefix stops outer loop immediately (no next iteration).
- `exit:` is recognized as built-in alias.
- Existing docs and default help text describe two distinct behaviors:
  - conditional sibling short-circuit (`optional/skip`)
  - terminal stop control (`quit/exit/end/break/return`)
- Regression tests cover both paths and prove no optional-skip behavior changes.

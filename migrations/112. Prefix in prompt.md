User prompts in do and make etc should support prefix, if input starts with something: it will try to apply template from .rundown/tools. Profiles are supported here and custom, other build-in tools too if they fit, if they dont it passes it as part of seed text in migration (as usual text)

## intent

Support prefix-aware seed input for bootstrap commands (`make`, `do`, and related seed-driven flows), so a seed that starts with a known tool prefix can be expanded through the existing tool system before writing the initial migration/task document.

## current context

Today `make`/`do` accept `<seed-text>` and write it directly into the target Markdown file (`createSeedMarkdownFile(...)` in `src/presentation/cli-command-actions.ts`) before research/plan/execution phases run.

Relevant existing architecture already available in repo:

- Prefix parsing and chaining: `src/domain/prefix-chain.ts`
  - Supports `profile=<name>` modifier syntax.
  - Supports chained prefixes with `, ` and `; ` boundaries.
  - Falls back to plain text when no known tool is resolved.
- Tool resolution: `src/infrastructure/adapters/tool-resolver-adapter.ts`
  - Resolves tools from configured directories (default `.rundown/tools`), including `.js` and `.md`.
  - Supports built-ins plus project-level custom tools.
  - Applies frontmatter/config overrides.
- Built-in tool registry: `src/domain/builtin-tools/index.ts`
  - Includes handler and modifier tools (`verify`, `include`, `parallel`, `for`, `profile`, `force`, aliases, etc.).
- Template-based tool behavior: `src/domain/builtin-tools/template-tool.ts`
  - Renders `.md` tool templates with template vars and payload for worker-style execution in normal task flow.

The gap is specifically bootstrap seed handling: it does not currently invoke prefix/tool parsing and therefore cannot transform seed text using `.rundown/tools` or built-in prefix semantics at creation time.

## feature specification

### what should happen

When seed text passed to `make`/`do` starts with a recognized prefix expression, bootstrap should attempt prefix resolution before writing the file.

Target behavior:

- Parse seed with the same prefix grammar used by tasks.
- Resolve tools through the same resolver layers (project JS tool, built-in, project MD tool, config overrides).
- Support `profile=<name>` modifier in the seed prefix chain.
- Support custom tools from `.rundown/tools` and configured `toolDirs`.
- If prefix/tool cannot be applied safely in bootstrap context, keep original seed text unchanged (current behavior).
- If prefix is unknown, malformed, or non-applicable, treat it as normal seed text (no regression).

### applicability rule ("if they fit")

Not all runtime task tools are meaningful for bootstrap seed generation.  
Bootstrap should apply only tools that can produce deterministic initial document text without requiring task-iteration side effects.

Examples:

- Applicable:
  - Template-style custom tool producing text content from payload.
  - Lightweight built-ins/custom handlers explicitly designed for seed transformation.
  - `profile=<name>` as context modifier when a downstream applicable handler is used.
- Non-applicable in bootstrap (must gracefully fall back to raw seed):
  - Control-flow/iteration handlers (`parallel`, `for`, terminal flow controls).
  - Handlers that depend on run-task execution loop semantics.
  - Handlers requiring task checkbox context, verification lifecycle, or include recursion semantics not available during file creation.

## scope boundaries

### in scope

- Prefix-aware preprocessing for seed text in `make` and `do`.
- Reuse of existing prefix parser/tool resolver contracts.
- Support for profile modifiers and custom `.rundown/tools` templates/handlers where bootstrap-compatible.
- Fallback-to-raw-seed behavior for unsupported/unknown prefixes.

### out of scope

- Changing task runtime prefix semantics in `run`.
- Redesigning tool grammar.
- Forcing all built-ins to work in bootstrap context.
- Altering research/plan logic after file creation, except consuming the transformed seed content.

## integration notes

Primary integration seam is before `createSeedMarkdownFile(...)` in `runMakeBootstrapPhases(...)` (and therefore inherited by `do`, which composes `make`).

A clean implementation boundary is a small bootstrap seed resolver helper that:

1. accepts raw seed text and current runtime/config context,
2. attempts prefix-chain parse and tool resolution,
3. returns either transformed seed text or original seed text with reasoned fallback.

This keeps CLI command wiring stable and avoids coupling bootstrap creation logic directly to run-task dispatch internals.

## compatibility and edge cases

- No-prefix seed remains unchanged.
- Unknown prefix remains unchanged.
- Legacy text that contains `:` but is not a valid tool prefix remains unchanged.
- `profile=<name>` without an applicable terminal handler should not break seed creation; either no-op or fallback to raw seed text.
- Chained prefixes should preserve current delimiter semantics and case-insensitive tool matching.
- Errors in tool loading/frontmatter for seed preprocessing should not block file creation unless explicitly configured; default should prefer safe fallback to original seed.
- Existing `make`/`do` UX must remain backward compatible for plain seed strings.

## acceptance criteria outline

- Seed starting with a known bootstrap-applicable prefix is expanded before initial file write.
- Custom tool templates in `.rundown/tools` can be used from seed text in `make`/`do`.
- `profile=<name>` is accepted in seed prefix chains and works with applicable handlers.
- Non-applicable built-ins do not crash bootstrap; seed text falls back to plain content.
- Unknown or unsupported prefixes preserve old behavior (written as-is).
- `do` inherits identical seed prefix behavior through shared bootstrap path.
- Regression coverage confirms unchanged behavior for non-prefixed seed input.

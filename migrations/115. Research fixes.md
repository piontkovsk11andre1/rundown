## Intent

This migration clarifies the expected behavior of the `research` phase so it behaves like enrichment work, not a brittle string-preservation test.

The current behavior is perceived as unnatural because the worker is asked to enrich content, but runtime validation still effectively requires original text retention in-order. That creates tension between "rewrite the document body" and "preserve every meaningful original line."

## Current implementation context

`research` currently runs as a single execution pass in `src/application/research-task.ts`:

- it renders `DEFAULT_RESEARCH_TEMPLATE` (or project override),
- executes the worker once,
- rejects output when checkbox state changes,
- rejects output when original content is not preserved in sequence (`detectResearchOutputContractViolation` + `preservesOriginalDocumentContent`),
- strips newly introduced unchecked TODO items,
- writes cleaned output when constraints pass.

Unlike `run`, this path does not currently invoke `src/application/verify-repair-loop.ts`. The verify/repair/resolve lifecycle is implemented for task execution, but not for research enrichment.

## Problem statement

The research contract is currently over-coupled to textual continuity instead of semantic continuity:

- "preserve existing author intent" is implemented as line-presence ordering,
- meaningful rewrites can fail even when intent is preserved,
- workers may overfit to verbatim retention instead of producing stronger structure and planning context.

This makes research output quality worse and increases false negatives.

## Desired behavior

Research should validate outcome quality through a dedicated verification-repair loop, not through strict original-text echo constraints.

Expected model:

- execution phase: produce enriched full-document markdown,
- verification phase: evaluate structural and semantic constraints,
- repair phase: retry when verification fails with concrete failure reason,
- optional resolve phase parity with existing verify/repair architecture when repeated failures persist.

## Verification goals for research

Research verification should check outcome-level rules:

- no checkbox state mutation for existing tasks,
- no newly introduced unchecked TODO items,
- original intent preserved (semantic, not strict line-order copy),
- document remains valid markdown and usable for planning,
- enrichment quality present (context, constraints, boundaries, acceptance framing).

Verifier output should follow the same strict verdict style already used in task verification (`OK` or `NOT_OK: ...`) to keep orchestration deterministic.

## Integration notes

This change should align with existing architecture patterns:

- orchestration entry remains `createResearchTask`,
- verification/repair mechanics should reuse or mirror `runVerifyRepairLoop`,
- research-specific verify/repair templates may be added similarly to `verify.md` / `repair.md`,
- runtime artifacts should record research verify/repair attempts consistently with existing phase metadata conventions.

## Scope boundaries

In scope:

- replacing strict "preserve all original text in order" as primary success gate,
- introducing research-phase verification and repair retries,
- preserving existing hard safety rules (checkbox state protection, no unchecked TODO additions).

Out of scope:

- changing `plan` behavior,
- changing execute-task checkbox completion lifecycle,
- introducing implementation-code generation in research phase.

## Compatibility and risk considerations

Compatibility:

- existing users relying on strict textual preservation may see more flexible rewrites,
- existing template override behavior should remain supported.

Risks to control:

- semantic-preservation checks can become too weak if under-specified,
- verification prompts must avoid vague pass criteria,
- repair attempts must not regress safety constraints.

Mitigations:

- keep deterministic verifier verdict contract,
- keep hard post-processing guards for checkbox/TODO safety,
- store verification failure reasons for repair guidance, as already done in run workflows.

## Acceptance criteria outline

A correct implementation will demonstrate:

- research can pass with semantically equivalent rewrites that do not retain exact original line order,
- research fails with actionable `NOT_OK` when intent is dropped or structure is degraded,
- failed research verification triggers repair retries,
- checkbox states remain unchanged across all research outcomes,
- newly introduced unchecked TODO lines are still rejected/removed,
- artifact logs clearly show research execution, verification, and repair outcomes for diagnosis.

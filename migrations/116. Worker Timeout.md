We need to add configurable value for worker timeout. If it doesnt give any respond in this timeout we kill it.

## Context

Worker execution currently goes through:

- `src/application/*` task flows -> `workerExecutor.runWorker(...)`
- `src/infrastructure/adapters/worker-executor-adapter.ts` -> `runWorker(...)`
- `src/infrastructure/runner.ts` -> `executeCommand(...)` with `cross-spawn`

Today there is no timeout enforcement in `runWorker`/`runner` path, so a worker process can run forever.

There is already timeout behavior in other parts of the codebase we can align with:

- `cli` fenced block execution supports `timeoutMs` and kills with `SIGTERM` on timeout (`src/infrastructure/cli-block-executor.ts`).
- process-runner adapter also has optional timeout and `SIGTERM` behavior (`src/infrastructure/adapters/crossspawn-process-runner.ts`).

This migration is to bring equivalent timeout control to worker process execution.

## Problem statement

When a worker hangs, stalls, or never exits, command execution can block indefinitely. We need a configurable timeout that terminates the worker process and returns a deterministic failure so failover/repair logic can continue.

## Proposed configuration surface

Primary goal is one configurable timeout value for worker execution.

Recommended shape for planning:

- Add optional worker-timeout key in config schema (for example `workers.timeoutMs` or top-level `workerTimeoutMs`).
- Value type should be non-negative integer milliseconds.
- `0` should mean disabled timeout (preserves current unlimited behavior when explicitly set).
- If key is omitted, default behavior should remain backward compatible (no timeout unless we explicitly define a safe default in this migration).

Notes for planner:

- `src/domain/worker-config.ts` and `src/infrastructure/adapters/worker-config-adapter.ts` are the schema + validation boundaries.
- Any new key must participate in existing local/global merge behavior and `rundown config get/list/set/unset` dotted-path support.

## Execution behavior specification

Timeout target:

- Applies to worker command lifecycle started by `runWorker` (all commands that use worker execution).

Timeout semantics:

- Start timer when child process is spawned.
- If process is still alive after timeout window, terminate it.
- Return a non-success result with clear stderr message including timeout duration.

Kill behavior:

- Follow existing project pattern: send `SIGTERM` first.
- Ensure timer is cleared on normal exit/error to avoid post-close races.
- Treat timeout as execution failure for upstream handling.

Important interpretation detail:

- Requirement wording says "doesnt give any respond". We need to decide whether this means:
  - total runtime timeout (no exit within N ms), or
  - inactivity timeout (no stdout/stderr activity within N ms).
- Existing code patterns use total runtime timeout; using same semantics reduces risk and complexity.

## Integration points

Expected touchpoints for implementation planning:

- `src/domain/worker-config.ts`: add config type field for worker timeout.
- `src/infrastructure/adapters/worker-config-adapter.ts`: parse/validate/merge/clone new field.
- `src/domain/ports/worker-executor-port.ts`: add optional timeout in worker execution options.
- `src/infrastructure/adapters/worker-executor-adapter.ts`: forward timeout to runner.
- `src/infrastructure/runner.ts`: enforce timeout in `executeCommand` for wait/tui capture modes (and define behavior for tui inherit/detached).
- Application callsites that construct `runWorker` options may need to pass resolved config value.
- CLI/docs/config docs (`docs/configuration.md`, `docs/cli-config.md`) need schema + behavior updates.

## Failure classification and failover considerations

Worker timeout should be visible to existing failure/failover flow:

- timeout should produce a recognizable stderr/error string so classification can treat it as retriable or transport/execution failure consistently.
- failover selection in `src/application/resolve-worker.ts` and run lifecycle in `src/application/run-task-execution.ts` should continue to work without special-casing unless required.
- runtime artifacts should capture timeout outcome clearly (phase notes/exit context).

## Compatibility and constraints

- Backward compatibility: existing configs without timeout must continue to load.
- Config validation errors should match current style (path-specific, actionable messages).
- Cross-platform behavior should rely on existing spawn/kill patterns already used by project.
- No changes to worker command resolution precedence are required; timeout is execution policy, not routing policy.

## Scope

In scope:

- Configurable worker timeout value.
- Worker process termination when timeout is exceeded.
- Clear surfaced failure message.
- Docs update for new config key and behavior.

Out of scope (for this migration unless explicitly expanded):

- Per-command/per-phase timeout overrides.
- Adaptive timeout by task type.
- Separate inactivity timeout vs total runtime timeout dual-mode support.
- Multi-stage terminate escalation policy (`SIGTERM` then `SIGKILL`) beyond current project conventions.

## Acceptance criteria outline

- Worker timeout is configurable via config and validated.
- Worker execution path enforces timeout and terminates overdue process.
- Timeout failure is surfaced with deterministic message and non-success outcome.
- Existing commands continue working when timeout is unset/disabled.
- Documentation reflects new config key, defaults, and `0` behavior.

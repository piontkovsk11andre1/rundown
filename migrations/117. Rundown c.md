bare rundown command (to open agent) will get `-c` / `--continue` parameter and pass this to worker (`-c` or `--continue`).

## Context and intent

When users run bare `rundown` (no subcommand), the CLI routes to the root help/agent path (`createHelpCommandAction`). That path may launch an interactive worker-backed help session (`helpTask`) or fall back to static help output depending on environment and flags.

The change is to support continuation-style OpenCode arguments at this root entry path so resume intent is not dropped:

- accepted forms: `-c`, `--continue`
- behavior: pass continuation intent through to worker invocation used by bare root help/agent flow
- non-goal: changing behavior of task-oriented commands (`run`, `plan`, `research`, `make`, `do`, etc.)

## Existing behavior summary

- Root command/options are defined in `src/presentation/cli.ts`.
- Bare invocation detection and dispatch happen via invocation parsing (`src/presentation/cli-argv.ts`) and `createHelpCommandAction` (`src/presentation/cli-command-actions.ts`).
- Help path invokes `getApp().helpTask({ ... })`.
- `helpTask` resolves worker config and calls worker executor in TUI mode (`src/application/help-task.ts`).
- Worker argument sources already include explicit worker pattern/command and separator payload (`-- ...`).
- `plan` already has explicit clean-session validation for continuation-related args, showing continuation semantics are intentionally controlled per command.

## Problem statement

Bare `rundown` should behave like “open agent/help session.” If user supplies `-c` or `--continue`, this intent should propagate into worker argv in the help flow. Today, root/global option handling does not reliably normalize and forward this flag in that path.

## Scope

### In scope

- Bare-root invocation behavior (`rundown` with no subcommand).
- Accepting `-c` and `--continue` in root path.
- Forwarding continuation intent to worker invocation in interactive help/agent flow.
- Test coverage for parsing/forwarding and non-regressions.

### Out of scope

- Semantics changes for `run`, `call`, `loop`, `plan`, `research`, `explore`, `make`, `do`, `discuss`, etc.
- New continuation aliases beyond `-c` / `--continue`.
- Relaxing `plan` clean-session restrictions.
- Broad worker-pattern refactors.

## Design constraints

- Additive change only; preserve existing root/help behavior.
- Do not break `--agents` priority behavior.
- Do not alter separator (`--`) semantics.
- Keep non-TTY fallback to static help unchanged.
- Preserve trace/artifact behavior in help flow.
- Avoid unintended coupling to unrelated commands.

## Implementation planning guidance

### 1) CLI contract update

Define root-level continuation contract in presentation layer:

- root accepts `-c`, `--continue`
- applies to bare root help/agent flow
- explicitly document precedence with:
  - separator worker args (`-- ...`)
  - explicit `--worker` pattern
  - config-default worker command

### 2) Normalization and propagation

Use a single deterministic path (likely in root action processing):

- detect continuation flag before separator payload
- map to worker arg form to pass forward (`--continue` is canonical)
- dedupe if equivalent continuation arg already exists in worker args
- preserve existing worker command ordering conventions

### 3) Integration boundaries

Likely touchpoints:

- `src/presentation/cli.ts` (root option declaration)
- `src/presentation/cli-command-actions.ts` (bare-root action shaping, helpTask request)
- possibly `src/application/help-task.ts` only if explicit contract extension is cleaner than mutating worker pattern/argv in presentation
- tests in `__tests__/presentation` and/or `__tests__/application` based on final boundary

### 4) Edge cases to validate

- `rundown -c` in non-TTY: should still fall back to static help without breakage
- `rundown --continue --agents`: `--agents` output remains authoritative
- `rundown -- --continue`: separator-provided continuation should not be duplicated
- duplicated flags (`-c --continue`): idempotent/harmless
- unknown-command cases with `-c`: error behavior unchanged where applicable
- no accidental carryover into `plan` clean-session constrained behavior

## Acceptance criteria

- Bare invocation accepts `-c` and `--continue` without parse errors.
- Interactive root help/agent flow forwards continuation intent to worker invocation.
- Existing root behavior (`--agents`, static help fallback, no-worker-configured messaging) remains unchanged.
- `plan` clean-session validation remains unchanged.
- Tests cover positive forwarding and key non-regressions.

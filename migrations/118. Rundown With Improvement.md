## expected opencode preset shape

for `rundown with opencode`, effective local config mutation should produce this worker split (without clobbering unrelated keys):

```json
{
  "workers": {
    "default": ["opencode", "run", "--file", "$file", "$bootstrap"],
    "tui": ["opencode"]
  },
  "commands": {
    "discuss": ["opencode"]
  }
}
```

important behavioral constraint: `with` updates only targeted keys and preserves all other sections (`run`, `workspace`, trace config, custom command overrides not targeted by preset write).

## config and discovery constraints

`with` writes to local scope (`<config-dir>/config.json`) using config-dir resolution rules:

- if `--config-dir` is passed, use it directly
- else discover `.rundown/` upward from invocation directory
- create config file when missing

global config is a separate layer and must not be modified by this command.

the merge model across layers remains `built-in -> global -> local`, so validating correctness should check effective resolution, not only raw file text.

## discrepancy investigation scope

investigate mismatch between:

- documented opencode convention (`opencode run --file $file $bootstrap` for deterministic commands, `opencode` for discuss/tui)
- runtime preset payload in registry
- actual local config after running `rundown with opencode` in current directory

this includes checking whether aliases (for example `OpenCode`, `open-code`) resolve to canonical `opencode` and remain idempotent.

## acceptance outline

success criteria for this improvement:

1. running `rundown with opencode` writes the canonical opencode split exactly once and is idempotent on repeat runs.
2. local config keeps unrelated keys untouched.
3. docs and implementation describe the same contract for deterministic vs interactive usage.
4. alias inputs still resolve to canonical key and produce canonical persisted command arrays.
5. if no change is required, command reports no-op semantics rather than rewriting config noise.

## compatibility and edge cases

- unknown harness names must continue to follow interactive custom mapping flow (prompt for default worker, optionally prompt for tui/discuss command).
- preset application must not delete `workers.fallbacks` unless preset explicitly owns fallback policy.
- output and behavior must stay compatible with existing test expectations around `changed`, `source`, and `configuredKeys`.
- windows and shell quoting behavior must remain unchanged because worker arrays are stored as json string arrays, not raw shell script.

Add add command add <seed-text> <file> like make -- but it appends the seed text and run plan. Add extra newline before append

## Intent

Introduce a new composition command `add` that mirrors the `make` workflow shape and worker resolution, but targets an existing Markdown file and runs only the planning phase after appending new seed content.

Command form:

```bash
rundown add "<seed-text>" "<markdown-file>" [options] -- <command>
rundown add "<seed-text>" "<markdown-file>" [options] --worker <pattern>
```

Behavior summary:

1. validate target Markdown file path,
2. append seed text to the existing file with an extra newline separator before appended content,
3. run `plan` on that same file using the same runtime/worker option conventions used by `make`.

## Existing patterns to follow

Current `make` command provides the closest baseline:

- CLI registration and options live in `src/presentation/cli.ts` (`.command("make")`).
- Command action orchestration lives in `src/presentation/cli-command-actions.ts`:
  - `createMakeCommandAction(...)`
  - `parseMakeBootstrapCliOptions(...)`
  - `runMakeBootstrapPhases(...)`
  - `createSeedMarkdownFile(...)`
- Path validation for make-style single-target Markdown input lives in `src/presentation/cli-options.ts`:
  - `resolveMakeMarkdownFile(...)`
  - shared `resolveSingleMarkdownFile(...)`.
- User-facing command docs follow `docs/cli-<command>.md` plus index mention in `docs/cli.md`.

`make` semantics that should remain consistent for `add` where applicable:

- sequential fail-fast orchestration,
- `--mode` restricted to `wait` for deterministic chaining,
- shared worker/runtime options (`--worker`, separator `-- <command>`, `--dry-run`, `--print-prompt`, `--trace`, vars options, CLI block options),
- lock behavior delegated to phase command (`plan` lock lifecycle),
- no destructive overwrite behavior.

## Proposed `add` semantics

## Scope

`add` is for iterative expansion of an existing planning document: append new intent, then synthesize additional TODO coverage via `plan`.

## Input and validation rules

- Positional args: exactly two required, `<seed-text>` and `<markdown-file>`.
- File extension must be `.md` or `.markdown` (reuse single-file validator conventions).
- Target must resolve to exactly one non-directory file path.
- Target file must already exist (difference vs `make`).
- Glob or directory inputs are rejected with command-specific error wording, consistent with existing style in `cli-options.ts`.

## Append behavior

- Append operation occurs before `plan`.
- Add an extra newline before appended seed text so new material is visually separated from prior content.
- Separator behavior should be deterministic:
  - preserve existing content,
  - ensure appended block starts after a blank-line boundary,
  - avoid rewriting prior sections.
- Append is text-only; no automatic checkbox transformation or markdown templating.

## Phase behavior

- Single phase execution after append: `plan`.
- No `research` phase in `add`.
- If append fails, `plan` does not run.
- If `plan` fails, exit non-zero and keep appended content/artifacts (same preservation principle as other commands).

## Option surface

Start from make-style shared runtime options, then keep only options relevant to a single `plan` phase:

- keep: `--mode` (wait only), `--scan-count`, `--max-items`, `--dry-run`, `--print-prompt`, `--keep-artifacts`, `--show-agent-output`, `--trace`, `--force-unlock`, `--vars-file`, `--var`, `--ignore-cli-block`, `--cli-block-timeout`, `--worker`, and separator `-- <command>`.
- include `--deep` for parity with direct `plan`; forward it unchanged to `plan` so depth behavior matches standalone planning.
- no `research`-specific wording in descriptions.

## Compatibility and edge cases

- Works with relative and absolute paths, matching existing path handling tests around make.
- Preserves lock semantics by relying on `plan` command lock acquisition for the target source.
- Worker resolution should remain identical to existing composition commands:
  - explicit `--worker`,
  - separator command `-- <command>`,
  - fallback resolver behavior when neither is provided.
- `--dry-run` / `--print-prompt` still perform the append before `plan` for consistency with `make`'s pre-phase file mutation behavior.
  - rationale: composition commands apply input shaping first, then run phase execution mode controls.
  - implication: dry-run/print-prompt skip planner execution effects but do not skip the append side effect.

## Acceptance criteria outline

- CLI accepts `rundown add "<seed>" "<file.md>"` and routes to `plan` after append.
- Appended content is separated by an extra newline boundary before seed text.
- Existing file content is preserved; only append is performed.
- Nonexistent path, directory path, glob input, and non-markdown extension fail with clear add-specific errors.
- `--mode tui` (or any non-wait mode) is rejected consistently.
- Shared runtime/worker options are forwarded to `plan` consistently with make/explore conventions.
- Exit code and fail-fast behavior match established command patterns.

## Out of scope

- Running `research` as part of `add`.
- Automatic restructuring of existing markdown content.
- Changing planner merge rules or TODO normalization behavior.
- Multi-file append/planning in one invocation.

# Feature: Rounds for --clean

## Summary

Add a `--rounds <n>` option that repeats the full `--clean` cycle (reset → execute all → verify all → reset) multiple times. This enables deterministic multi-pass execution where a task file is run to completion N times in a row, useful for confidence checks, flaky-task detection, and iterative stabilization workflows.

## Current behaviour

### How --clean works today

`--clean` is a shorthand for `--redo --reset-after` ([src/presentation/cli.ts](src/presentation/cli.ts) lines 117-119). It implies `--all`. The execution flow in `createRunTaskExecution` ([src/application/run-task-execution.ts](src/application/run-task-execution.ts)) is:

1. **Pre-run reset** (line ~413): `if (redo)` → iterates all resolved source files, calls `maybeResetFileCheckboxes(file, ..., "pre-run")` to uncheck every checked task.
2. **Task loop** (line ~435): `while (true)` selects the next unchecked task via `dependencies.taskSelector.selectNextTask(files, sortMode)`, calls `runTaskIteration(...)`, and checks `iterationResult.continueLoop`. Loop ends when no unchecked tasks remain (`state.runCompleted = true`) or an iteration fails.
3. **Post-run reset** (line ~513): In the `finally` block, `if (resetAfter && state.runCompleted)` → calls `maybeResetFileCheckboxes(file, ..., "post-run")` on every source file.
4. **Deferred commit** (line ~527): If `--commit` + `--reset-after`, the commit is deferred until after the post-run reset.

### Key structural constraints

- The `while (true)` task loop lives inside `runTask()` in [run-task-execution.ts](src/application/run-task-execution.ts). There is no outer round abstraction.
- `state.runCompleted` is a boolean — it marks that the inner loop exhausted all tasks. It is checked by the post-run reset guard and has no concept of "current round".
- `effectiveRunAll` is derived once at line ~224: `runAll || redo || clean`. It does not vary per-round.
- Lock acquisition, artifact initialization, and template variable resolution all happen once before the task loop — they should remain once-per-run, not once-per-round.
- `finalizeArtifacts()` and `finishRun()` are designed as single-shot terminal operations (guarded by `artifactsFinalized`).

### Task iteration control flow

`runTaskIteration` ([src/application/run-task-iteration.ts](src/application/run-task-iteration.ts)) returns `{ continueLoop: boolean; exitCode?: number }`. When `continueLoop` is false, the task loop breaks and the run ends. The iteration delegates finalization to `completeTaskIteration` ([src/application/complete-task-iteration.ts](src/application/complete-task-iteration.ts)), which handles verify/repair, checkbox marking, completion hooks, and the continue-or-stop decision.

### Checkbox operations

`maybeResetFileCheckboxes` ([src/application/checkbox-operations.ts](src/application/checkbox-operations.ts)) reads the file, counts checked tasks, and rewrites the file with all checkboxes unchecked. It emits an info message and supports dry-run mode. The domain function `resetAllCheckboxes` ([src/domain/checkbox.ts](src/domain/checkbox.ts)) does the actual string transformation.

## Why this matters

1. **Confidence testing**: Running a task file through multiple clean cycles catches flaky tasks or environment-dependent failures that pass once but fail on retry.
2. **Iterative training data**: Some workflows generate different outputs each round, and repeated execution accumulates artifacts across runs.
3. **Natural user expectation**: `--clean` already implies "start fresh", so `--clean --rounds 3` is the intuitive way to say "do that 3 times".

## Implementation approach

### New CLI option

Add `--rounds <n>` to the `run` command in [src/presentation/cli.ts](src/presentation/cli.ts), alongside the existing `--clean`/`--redo`/`--reset-after` group. Default value: `1` (current behavior, no change). Constrain to a positive integer. `--rounds` requires `--clean` (or `--redo --reset-after`) — emit an error if used without both reset phases, since a round without pre-run reset would re-run already-checked tasks without unchecking them first.

### Option threading

Add `rounds: number` to `RunTaskOptions` in [run-task-execution.ts](src/application/run-task-execution.ts). Parse and pass from `cli-command-actions.ts` ([src/presentation/cli-command-actions.ts](src/presentation/cli-command-actions.ts)) in the same block where `clean`, `redo`, and `resetAfter` are resolved (line ~108).

### Outer round loop

Wrap the existing `while (true)` task loop in a `for (let round = 0; round < rounds; round++)` outer loop inside `runTask()`. Each round:

1. **Reset checkboxes** (re-use the existing `redo` pre-run reset logic). For round 0 this is the normal `--redo` path. For rounds 1+ the reset is needed again since the previous round checked tasks and the post-round reset unchecked them — but we need the pre-run reset to happen at the start of each round.
2. **Run the inner task loop** unchanged — `selectNextTask` / `runTaskIteration` / `continueLoop`.
3. **Post-round reset**: After the inner loop completes (`state.runCompleted`), reset checkboxes for the next round (unless this is the final round, where the existing `resetAfter` in the `finally` block handles it). Between rounds, re-set `state.runCompleted = false` and `state.tasksCompleted = 0` so the next round starts fresh.

The cleanest structural approach is to extract the inner task loop (lines ~435–510) into a helper function, then call it N times from the outer round loop. This avoids deeply nesting while keeping the try/finally structure intact for lock cleanup and artifact finalization.

### State between rounds

- `state.runCompleted` must be reset to `false` at the start of each round > 0.
- `state.tasksCompleted` resets to `0` per round to accurately count tasks per round; alternatively, keep accumulating and emit a per-round summary.
- Artifact context (`state.artifactContext`) persists across rounds — all rounds belong to the same run session. The run artifact's metadata could include `currentRound` and `totalRounds`.
- Trace events should emit round boundaries: a `round-started` and `round-completed` event per round via `traceRunSession`.

### Error semantics

If a task fails verification in any round, the run fails immediately (same as today). Partial-round results are preserved in artifacts. The exit code is the same as a single-round failure.

### Dry run and print-prompt

`--dry-run --rounds 3` should report what would happen in round 1 only — dry-run already terminates after printing the first prompt/plan. No special handling needed.

### Interaction with --commit

`--commit --clean --rounds 3` should commit after the final round only (same as the existing deferred commit for `--commit --reset-after`). Between rounds, no commit occurs since intermediate checkbox state is transient.

### Trace and artifact metadata

The run artifact (`run.json`) should include `rounds` and `currentRound` in its metadata so downstream tooling (traces, log-runs) can distinguish multi-round runs. Trace events should bracket each round with `round-started` / `round-completed` events carrying the round index.

### Output

Emit a round header at the start of each round when `rounds > 1`:

```
Round 1/3 — resetting checkboxes and running all tasks...
```

After the final round, the existing "All tasks completed" message suffices. For multi-round runs, add a summary:

```
All 3 rounds completed successfully (9 tasks total).
```

## Edge cases

- `--rounds 1` is exactly current `--clean` behavior — no visible difference.
- `--rounds 0` or negative values should be rejected at CLI parse time with a clear error.
- `--rounds N` without `--clean` (or without both `--redo` and `--reset-after`) should emit an error explaining the requirement.
- `--rounds N` with `--only-verify` is rejected by the existing `onlyVerify && (redo || resetAfter || clean)` guard.
- `--rounds N` with `--dry-run` runs exactly one dry-run pass regardless of N.
- If a source file has 0 unchecked tasks at the start of a round (could happen if reset failed), emit "No unchecked tasks found" and stop — do not spin through empty rounds.

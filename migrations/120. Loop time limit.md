Loop should have time limit, after timeout it will finish and exit.

## Context

The `loop` command currently supports:

- cooldown between iterations (`--cooldown`)
- bounded loop count (`--iterations`)
- continue-on-error behavior (`--continue-on-error`)
- terminal stop handling from task prefixes (`quit:`, `exit:`, `end:`, `break:`, `return:`)

Current behavior is implemented in:

- `src/presentation/cli.ts` (`loop` option surface)
- `src/presentation/cli-command-actions.ts` (`createLoopCommandAction` execution loop)
- `docs/cli-loop.md` (user-facing contract)
- `__tests__/presentation/cli-command-actions.test.ts` and `__tests__/presentation/cli-options.test.ts` (behavior and option parsing)

Today, if `--iterations` is omitted, loop is unlimited and only stops on interrupt, terminal stop intent from task execution, or non-zero iteration exit when `--continue-on-error` is not set. There is no explicit elapsed-time cap for the whole loop session.

## Problem statement

The loop runner needs a time-based upper bound for the full command lifecycle. Without this, long-running or continuously succeeding workloads can run forever, which makes automation jobs hard to control and can block pipelines.

The requested behavior is: after configured timeout, the loop should finish cleanly and exit.

## Feature goal

Add an optional total loop runtime limit (wall-clock), so users can run unlimited or high-iteration loops with a deterministic stop boundary.

This timeout should apply to the outer loop lifecycle, not only to individual cooldown sleeps.

## Proposed CLI and config surface

Recommended public option:

- `--time-limit <seconds>` (or `--timeout <seconds>` if naming should align with existing CLI wording)

Recommended semantics:

- positive integer seconds
- omitted = disabled (backward compatible)
- timeout starts when loop execution begins (before first iteration)
- timeout is a global budget for the entire `loop` invocation

If config support is added, keep CLI precedence consistent with existing option layering (CLI overrides config defaults). If config is not added in this migration, keep scope to CLI-only.

## Execution semantics

Timeout behavior should be deterministic and easy to reason about:

1. Record loop start time once, before entering iteration cycle.
2. Before starting each new iteration, check whether time budget is exhausted.
3. While waiting in cooldown, stop early when timeout is reached.
4. Do not forcibly kill an in-flight iteration solely due to outer loop timeout unless explicitly designed; default to graceful boundary at iteration/cooldown checkpoints.
5. On timeout, emit a clear info/warn message and exit the loop with a success-like graceful stop code (same class as bounded completion), unless product policy requires non-zero timeout code.

The requirement text says “after timeout it will finish and exit”, which aligns better with graceful stop than hard-fail.

## Interaction with existing loop controls

Precedence and interactions should be explicit:

- `--iterations` and time limit are both active bounds; whichever is reached first ends loop.
- terminal stop intent from executed task still ends loop immediately after current iteration finalization.
- `--continue-on-error` only affects handling of failed iteration results; it must not bypass time limit.
- `Ctrl+C` during cooldown remains graceful and should continue to work unchanged.

## Observability and output

Add user-visible output lines consistent with existing style:

- start message can include configured time limit
- timeout exit message should include elapsed and/or configured duration
- final loop summary should still print total/succeeded/failed counts

Message format should be deterministic enough for test assertions.

## Documentation touchpoints

Expected docs update:

- `docs/cli-loop.md` option table and behavior notes
- `README.md` command examples only if this option is considered core enough for top-level docs
- `docs/cli.md` loop command section if it mirrors option list

Clarify that time limit is global loop runtime, not per-iteration execution timeout.

## Testing scope

Add/adjust tests in:

- `__tests__/presentation/cli-options.test.ts` for option parsing/validation
- `__tests__/presentation/cli-command-actions.test.ts` for runtime behavior

Key cases:

- loop exits when time limit is reached in cooldown
- loop exits before launching next iteration when budget already exhausted
- bounded iterations still stop correctly before timeout
- unlimited loop with continue-on-error still respects timeout
- invalid values (`0`, negative, non-integer) are rejected if positive-int semantics are chosen

Use fake timers where possible to avoid flaky wall-clock tests.

## Scope boundaries

In scope:

- outer loop runtime cap
- deterministic stop and exit behavior
- docs and tests for new option

Out of scope (unless explicitly expanded):

- per-iteration worker timeout (already covered by worker timeout mechanisms)
- inactivity-based timeout based on stdout/stderr silence
- force-killing currently executing iteration when global loop time budget expires mid-iteration
- introducing new non-zero exit code classes for timeout unless product decision requires it

## Acceptance criteria outline

- loop command accepts a time-limit option with validated numeric input
- loop execution tracks global elapsed time and stops when limit is reached
- stop is graceful: loop finalizes and exits predictably
- existing loop features (`--iterations`, cooldown, continue-on-error, terminal stop) remain compatible
- docs and tests reflect the new behavior and constraints

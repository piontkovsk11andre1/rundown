When running any command that requires a worker, rundown should warn about unhealthy workers at startup and never silently get stuck because every candidate is blocked by persisted health state. If all candidates are in cooldown, the run should wait for the nearest cooldown to elapse and then retry instead of failing instantly.

## Context

Worker health is persisted in `.rundown/worker-health.json`. Each worker candidate gets a health entry with one of three statuses:

- `healthy` — eligible for selection.
- `cooling_down` — temporarily blocked, becomes eligible after `cooldownUntil` timestamp.
- `unavailable` — permanently blocked until manual intervention (delete the file or reset the entry).

The current resolution flow in `src/application/resolve-worker.ts` (`resolveWorkerSelectionForInvocation`) evaluates all candidates (primary + fallbacks), picks the first eligible one, and returns an empty command array when none are eligible.

The caller in `src/application/run-task-iteration.ts` then hits the `requiresWorkerCommand` guard which emits a generic error: _"No worker command available: .rundown/config.json has no configured worker"_ — even though the config does have a worker, it is just health-blocked. This is confusing and leaves no recovery path except manually deleting `worker-health.json`.

### Key code paths

- `src/domain/worker-health.ts` — `evaluateWorkerHealthEligibility()`: returns `eligible: false` for `unavailable`, and for `cooling_down` when `cooldownUntil > now`.
- `src/application/resolve-worker.ts` — `resolveWorkerSelectionForInvocation()`: iterates candidates, finds first eligible, returns empty array when none found.
- `src/application/run-task-execution.ts` — `updateWorkerHealthForAttemptOutcome()`: sets status to `unavailable` (permanent block) for `transport_unavailable` when `unavailableReevaluation.mode` is `"manual"` (the default).
- `src/application/run-task-iteration.ts` — `requiresWorkerCommand` guard emits the no-worker error at line ~427.
- `src/presentation/cli-command-actions.ts` — `createLoopCommandAction()`: calls `app.runTask()` per iteration.
- Default `healthPolicy.unavailableReevaluation.mode` is `"manual"`, and default `cooldownSecondsByFailureClass.transport_unavailable` is `0` — which means transport failures go straight to permanent `unavailable` with no auto-recovery.

## Problem statement

Three gaps:

1. **No startup warning** — the user gets no visibility into worker health problems until a task fails mid-run with a misleading "no configured worker" message.
2. **No cooldown wait** — when all candidates are in `cooling_down` state, the run fails instantly instead of waiting for the nearest cooldown to elapse and retrying.
3. **Permanent `unavailable` is a trap** — transport failures set workers to `unavailable` with no auto-recovery by default. The default `unavailableReevaluation.mode` of `"manual"` means the only recovery is deleting `worker-health.json`. This should default to `"cooldown"` so workers get a probe retry after some time.

## Feature goals

1. Emit a warning at run startup about any workers that are not healthy (cooling down or unavailable) so the user sees the problem early.
2. When all worker candidates are ineligible but at least one is in `cooling_down` (not permanently `unavailable`), wait for the nearest cooldown to expire and re-evaluate instead of failing instantly.
3. Change the default `unavailableReevaluation.mode` from `"manual"` to `"cooldown"` with a sensible default probe cooldown (e.g. 300s) so `transport_unavailable` failures get automatic cooldown-based recovery instead of permanent blocking.
4. Improve the error message when no worker is available to distinguish between "no worker configured" and "all workers blocked by health policy".

## Proposed behavior

### 1. Startup health warnings

At the beginning of `runTask` (in `run-task-execution.ts`), after loading worker health entries and before the task iteration loop, emit warnings for any entries that are not healthy:

```
⚠ Worker "opencode run $bootstrap" is unavailable (transport_unavailable since 2026-04-16T13:00:48Z).
⚠ Worker "opencode run $bootstrap --model opus" is cooling down until 2026-04-16T03:39:20Z.
```

This applies to `run`, `loop`, `call`, `all`, `materialize` — any path that goes through `runTask`.

### 2. Cooldown wait before giving up

In `resolveWorkerSelectionForInvocation` (or its caller), when no candidate is eligible:

- Check if any candidate has `cooling_down` status with a `nextEligibleAt` timestamp.
- If yes, compute wait time = `max(0, nextEligibleAt - now)`.
- Emit info: `ℹ All workers cooling down. Waiting Xs for "worker label" to become eligible...`
- Sleep for that duration (use cancellable sleep compatible with SIGINT).
- Re-evaluate candidates after waking up.
- If still no eligible candidate after re-evaluation, fail with the improved error message.

This wait loop should have a reasonable upper bound (e.g. respect worker timeout or a max wait cap) to prevent indefinite hangs.

The wait should happen at the `run-task-iteration` level (around the `requiresWorkerCommand` check), not deep inside `resolveWorkerSelectionForInvocation`, so the caller controls the retry policy.

### 3. Change default `unavailableReevaluation.mode` to `"cooldown"`

In `updateWorkerHealthForAttemptOutcome` (in `run-task-execution.ts`), the default mode for `transport_unavailable` failures should be `"cooldown"` instead of `"manual"`:

```typescript
const unavailableMode = healthPolicy?.unavailableReevaluation?.mode ?? "cooldown";  // was "manual"
```

This means transport failures will get a `cooling_down` status with `cooldownUntil` set to `now + probeCooldownSeconds` (default 300s) instead of being permanently `unavailable`.

Users who want the old sticky-unavailable behavior can explicitly set `"manual"` in their config.

### 4. Better error message when health-blocked

Change the error in `run-task-iteration.ts` from the generic "no configured worker" message to differentiate:

- If config has no worker at all: keep current message.
- If config has workers but all are health-blocked: emit something like _"All configured workers are blocked by health policy. Use `rundown worker-health` to inspect status or delete `.rundown/worker-health.json` to reset."_

## Execution semantics

- Startup warnings are emitted once per `runTask` invocation, not per task iteration.
- Cooldown wait applies per task iteration (each task re-evaluates health before worker selection).
- Wait is cancellable via SIGINT (same as existing cooldown mechanisms in loop).
- After waiting, a single re-evaluation attempt occurs. If still blocked, fail the task (no infinite retry).
- The default 300s probe cooldown for transport_unavailable is configurable via `healthPolicy.unavailableReevaluation.probeCooldownSeconds`.

## Scope boundaries

In scope:
- Startup health warnings in run-task-execution
- Cooldown wait before failing in run-task-iteration
- Default mode change from "manual" to "cooldown"
- Improved error messages
- Tests for new behavior
- Docs update for changed defaults

Out of scope:
- New CLI command to reset health entries (can be done via config or file deletion)
- Changes to loop cooldown behavior (loop has its own cooldown between iterations)
- UI/TUI health dashboard

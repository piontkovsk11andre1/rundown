# Translate command

`rundown translate <what> <how> <output>` is a new command that re-expresses one document in the vocabulary defined by another.

## The core idea

Two markdown files go in, one comes out.

`<what>` is a document containing an idea, requirement, task, or description written in whatever language the author happened to use — product language, user-story language, plain English, informal notes.

`<how>` is a know-how document written in Q&A style. Each entry pairs a common-language phrasing with its corresponding meaning inside a specific framework, system, or domain. Think of it as a rosetta stone: "When someone says X, here is what it means in terms of our system." The questions are ordinary language; the answers explain the concept using the domain's own vocabulary, structure, and mental models.

The output is a redescription of `<what>` as though it had originally been written by someone fluent in the domain described by `<how>`. Nothing is added or invented — the same idea is expressed, but now using the domain's natural terms, patterns, and distinctions.

## The prompt structure

The underlying prompt tells the model three things:

1. Here is a document that describes something (`<what>`).
2. Here is a know-how reference that defines how common language maps to domain concepts (`<how>`).
3. Rewrite the first document using only the vocabulary and concepts from the second.

The model's job is translation in the linguistic sense — fidelity of meaning, not literal word substitution. It should match the intent of `<what>` to the closest equivalent concepts in `<how>`, produce natural prose in the target domain's idiom, and signal uncertainty only when a concept in `<what>` has no clear analog in `<how>`.

## The know-how document format

The `<how>` document is typically structured as a series of Q&A pairs. A question poses a situation or intent in everyday language. The answer explains what that looks like inside the framework — which concept applies, which steps are involved, which terms belong to the domain.

This format is well-suited for the translation task because it is already written as a mapping between two vocabularies. The model can use it as a lookup reference rather than having to infer the domain's structure from formal specification. Informal know-how documents are often easier to write accurately than formal glossaries, and they naturally include cues about intent and typical usage that a glossary omits.

The format does not have to be strictly Q&A. Any document that explains the domain through examples — worked cases, annotated decisions, annotated patterns — serves the same purpose. What matters is that the document teaches meaning through contrast: here is how something sounds in ordinary language, here is what it actually means in the domain.

## Motivation

Many projects have a translation problem. Requirements arrive in product or business language. Developers work in domain or framework language. The gap between the two causes misreading, wasted clarification, and subtle mistranslations that only surface late.

The `translate` command externalizes the translation layer. Rather than relying on each person to hold the mapping in their head, the know-how document makes the mapping explicit and reusable. Once written, it can be applied consistently to any incoming document.

It also separates authoring from domain fluency. A product manager can write `<what>` in natural language. A developer can write `<how>` once. The command produces output that reads as though a domain expert authored `<what>` from the start — suitable for planning, design reviews, or feeding into subsequent rundown commands that expect domain-aware input.

## Relationship to other commands

`translate` is purely a transformation step. It does not run tasks, does not produce a plan, and does not modify any checked-off state. It is closest in spirit to `make`, but narrower: instead of producing a document from scratch based on an instruction, it re-expresses an existing document into a different vocabulary.

The output of `translate` is intended to become input to other commands. A translated requirement can feed `plan`. A translated design note can feed `make`. The command is a pre-processing step that makes subsequent downstream prompts more accurate by ensuring they operate in the domain's own language rather than a generic one.

## Command contract and CLI behavior

`translate` should follow the same deterministic CLI contract style as `plan`, `research`, and `make`:

- Exactly three positional arguments are required: `<what>`, `<how>`, `<output>`.
- `<what>` and `<how>` must be existing Markdown files (`.md` or `.markdown`), not directories or globs.
- `<output>` must be a Markdown file path and may be new; parent directory must already exist.
- Global `--config-dir <path>` should work the same way it works on every other command.
- Worker invocation should support both `--worker <pattern>` and separator form `-- <command>`.
- Runtime flags should match existing command conventions where applicable (`--mode`, `--dry-run`, `--print-prompt`, `--keep-artifacts`, `--show-agent-output`, `--trace`, `--vars-file`, `--var`, `--ignore-cli-block`, `--cli-block-timeout`).

The command should be single-pass (no convergence loop). It runs one translation turn and writes one output document.

## Translation semantics and output guarantees

The translation result is a full-document rewrite of `<what>` into the idiom of `<how>`, with these guarantees:

- Meaning preservation: preserve intent, scope, and constraints from `<what>`.
- Vocabulary alignment: prefer terms, distinctions, and framing established by `<how>`.
- No invention: do not add new requirements, tasks, or implementation commitments.
- Uncertainty signaling: when no analog exists in `<how>`, keep the original concept explicit and mark the mismatch clearly.
- Natural target prose: output should read like native domain writing, not dictionary substitution.
- Markdown preservation: produce valid Markdown document text suitable for downstream rundown commands.

## Scope and non-goals

In scope:

- Controlled linguistic/domain translation of one source document using one know-how reference.
- Explicitly reusable output for planning/execution flows.
- Deterministic command behavior and artifacting consistent with existing CLI commands.

Out of scope:

- Executing tasks from the translated content.
- Generating TODOs directly (that remains `plan`).
- Verifying implementation correctness (that remains `run`/`reverify`/`test`).
- Maintaining any checkbox/task state in source documents.

## Existing codebase alignment and module boundaries

The command fits current architecture with a clear split across presentation and application layers:

- CLI registration pattern lives in `src/presentation/cli.ts`, where commands define arguments/options and attach action creators.
- Command-option parsing and input validation helpers live in `src/presentation/cli-options.ts`.
- CLI-to-application request mapping lives in `src/presentation/cli-command-actions.ts`.
- Worker-backed command orchestration lives in application modules such as `src/application/research-task.ts` and `src/application/plan-task.ts`.
- Template loading conventions live in `src/application/project-templates.ts` using config-dir override first, built-in defaults second.

`translate` should follow this same layering rather than introducing a custom path.

## Prompt/template integration expectations

Current commands load templates through `loadProjectTemplatesFromPorts(...)` and support project overrides inside `.rundown/`. `translate` should adopt the same mechanism, with a dedicated template file (for example `.rundown/translate.md`) and a built-in fallback default in domain defaults.

Prompt composition should include:

- Full content of `<what>`.
- Full content of `<how>`.
- Explicit translation rules (meaning fidelity, no invention, uncertainty policy).
- Output contract requiring full translated Markdown document body (no wrapper text).

CLI fenced-block handling should match existing behavior: expand by default, support `--ignore-cli-block`, and enforce timeout with `--cli-block-timeout`.

## File safety, locking, and artifact behavior

Following established patterns in `research` and `plan`:

- Use source-relative lock discipline to prevent concurrent writes to the target output document.
- If output path collides with active lock holder, fail with lock metadata and stale-lock recovery guidance.
- Preserve artifacts on failure; preserve on success only when `--keep-artifacts` is enabled.
- Emit trace/output events consistent with existing command telemetry and logging expectations.
- Keep read-only inputs (`<what>`, `<how>`) untouched.

If `<output>` is the same path as `<what>`, behavior should still remain safe and deterministic (single locked write). If `<output>` equals `<how>`, reject to avoid mutating the translation reference.

## Edge cases and compatibility considerations

- Missing or unreadable `<what>`/`<how>` should fail fast with clear path-specific errors.
- Non-Markdown extension should fail consistently with existing path validators.
- Empty `<how>` or structurally weak `<how>` should not fail parsing by default; translation prompt should instruct the model to surface ambiguity explicitly.
- Very large inputs should still run through normal worker execution/artifact flow without special-case behavior changes.
- Downstream compatibility matters: output should remain suitable for `plan`, `make`, `research`, and `query` pipelines.

## Integration with current workflows

Typical flow this command unlocks:

1. Author writes requirement in plain language (`<what>`).
2. Domain owner maintains mapping reference (`<how>`).
3. `translate` produces domain-native redescription (`<output>`).
4. `plan` consumes translated output for TODO synthesis.
5. `run` or `do` executes from domain-aligned tasks.

This keeps translation explicit and reusable, instead of implicit and person-dependent.

## Acceptance criteria outline

Functional outcomes:

- Command accepts three positional Markdown paths and executes one translation pass.
- Output document reflects `<what>` intent in `<how>` vocabulary without invented requirements.
- Input files remain unchanged; output file is written deterministically.
- Dry-run/print-prompt modes show translation intent without performing worker execution.

Safety and UX outcomes:

- Path validation errors are explicit and command-consistent.
- Locking and stale-lock recovery match existing source-lock behavior.
- Artifact and trace behavior matches current command conventions.

Quality outcomes:

- Translated prose is domain-natural, not literal substitution.
- Ambiguous/no-analog concepts are called out rather than hallucinated.
- Output remains valid Markdown and usable by downstream commands.

# Feature: Research command

## Summary

Add a `rundown research <source>` command that enriches a Markdown document with context, background knowledge, implementation guidance, and structural scaffolding — without producing TODO items. Research is a preparatory phase: it reads the document, understands its intent, gathers relevant project context, and rewrites/augments the document body so that a subsequent `plan` pass (and then `run`) can operate on a well-informed foundation.

## Motivation

The current workflow assumes that the human author writes a feature document with enough context for `plan` to propose actionable TODOs and for `run` to execute them. In practice, early-stage documents are often thin sketches — a title, a sentence or two of intent, maybe a rough scope note. Running `plan` against such documents produces shallow, generic TODOs because the planner lacks the surrounding context it needs.

Research fills this gap. It sits before `plan` in the lifecycle:

```
author drafts thin document → research enriches it → plan proposes TODOs → run executes them
```

After research, the document should contain enough detail that `plan` can produce specific, grounded TODOs without hallucinating scope or missing constraints.

## What research produces

Research updates the source Markdown file in-place. The output is **prose and structured context**, not task items. Specifically, the worker should:

- **Expand feature descriptions** — flesh out what the feature does, why it matters, how it fits into the existing system.
- **Add technical context** — identify relevant existing code, patterns, modules, and conventions that constrain or inform implementation.
- **Surface design constraints** — call out edge cases, compatibility requirements, security considerations, performance implications.
- **Propose structure** — add section headings, acceptance criteria outlines, scope boundaries, and out-of-scope notes.
- **Preserve author intent** — never remove or contradict what the author wrote; only augment and elaborate.

Research must **not** produce `- [ ]` task items. That is `plan`'s job. Research must **not** change any existing checkbox state.

## How it works

Research follows the same application pattern as `plan` and `discuss`:

1. **Lock** the source file (exclusive, same lockfile strategy as other commands).
2. **Resolve worker config** via the standard cascade: `config.defaults` → `config.commands.research` → file frontmatter profile → CLI `--worker` override.
3. **Load template** — custom `.rundown/research.md` if present, otherwise the built-in `DEFAULT_RESEARCH_TEMPLATE`.
4. **Render template** — inject standard template variables (`{{context}}`, `{{source}}`, `{{file}}`, etc.) plus any `--var` / `--vars-file` overrides.
5. **Expand CLI blocks** — process `` ```cli ``` `` fenced blocks in the template to gather live project state (e.g., directory listings, dependency versions, test output).
6. **Invoke worker** — send the rendered prompt via the configured transport (`file` or `arg`), in `wait` mode by default.
7. **Apply output** — the worker's stdout is the updated document content. Replace the source file's body with the worker output (or a parsed subset of it).
8. **Artifact persistence** — save prompt, raw output, and metadata under `.rundown/runs/<runId>/` with phase label `research`.
9. **Release lock**.

### Convergence / scan behavior

Unlike `plan`, research does **not** loop by default. A single pass should be sufficient for most documents. However, supporting `--scan-count` for iterative deepening (where each pass reads the freshly updated file) is a reasonable extension for later.

### Diagnostic modes

- `--dry-run` — show what would be sent to the worker without executing.
- `--print-prompt` — render the full research prompt to stdout and exit.

## Suggested default prompt

The built-in `DEFAULT_RESEARCH_TEMPLATE` should follow the shared-prefix convention for KV-cache efficiency. Proposed template:

```markdown
{{context}}

---

The Markdown above is the source document up to but not including the selected unchecked task.

## Source file

`{{file}}` (line {{taskLine}})

## Selected task

{{task}}

## Full document

{{source}}

## Phase

Research and enrich the source document with implementation context.

Read the full document above and enrich it with the context, background, and
structural detail needed for a planning agent to later propose concrete TODO
items.

Your output replaces the document body. Write the complete updated Markdown
document.

Guidelines:

- Expand thin descriptions into clear feature specifications.
- Add relevant technical context: existing code patterns, module boundaries,
  API surfaces, and conventions discovered via CLI blocks or project knowledge.
- Surface design constraints, edge cases, and compatibility considerations.
- Propose document structure: add section headings, acceptance criteria
  outlines, scope/out-of-scope boundaries, and integration notes.
- Preserve everything the author already wrote — do not remove or contradict
  existing content, only augment it.
- Do NOT add TODO items (`- [ ]` lines). Task decomposition is a separate
  phase.
- Do NOT change any checkbox state in the document.
- Do NOT add implementation code. This phase is about context and planning
  guidance, not execution.
- Write in the same voice and style as the existing document.

Return the full updated Markdown document and nothing else.
{{traceInstructions}}
```

### Why this prompt works

- **Full document access via `{{source}}`** — research needs the entire file to reason about structure, not just the prefix context. This is a key difference from execute/verify which operate on a single task.
- **"Your output replaces the document body"** — gives the worker a clear contract. The application layer can then do a full-file replacement, which is simpler than trying to merge partial edits.
- **Explicit prohibition of TODOs and checkboxes** — prevents scope creep into `plan` territory and avoids accidental task mutation.
- **CLI blocks in the template** — users can add `` ```cli ``` `` blocks to their custom `.rundown/research.md` to inject live project state (e.g., `tree src/`, `cat package.json`, `git log --oneline -10`) into the prompt. This is where the most valuable context comes from.

## CLI interface

```
rundown research <source>
```

**Arguments:**
- `<source>` — file path (single Markdown file). Globs/directories are out of scope for the first iteration; research operates on one document at a time.

**Options** (reuse the standard option set):
- `--mode <mode>` — execution mode: `wait` (default), `tui`.
- `--transport <transport>` — prompt transport: `file` (default), `arg`.
- `--dry-run` — render prompt, show plan, exit without executing.
- `--print-prompt` — print rendered prompt to stdout and exit.
- `--worker <command...>` / `-- <command...>` — worker override.
- `--show-agent-output` — display worker stderr during execution.
- `--trace` — enable structured trace output.
- `--keep-artifacts` — preserve `.rundown/runs/` artifacts after completion.
- `--var <key=value>` — template variable (repeatable).
- `--vars-file <path>` — load template variables from JSON file.
- `--ignore-cli-block` — skip CLI block expansion in template.
- `--cli-block-timeout <ms>` — timeout for CLI block commands.
- `--force-unlock` — break stale lockfile.
- `--config-dir <path>` — explicit `.rundown/` directory.

## Configuration

Research participates in the standard config cascade:

```json
{
  "commands": {
    "research": {
      "worker": ["opencode", "run"],
      "workerArgs": ["--model", "opus-4.6"]
    }
  }
}
```

This allows teams to use a more capable (potentially slower/more expensive) model for the research phase, since it runs once per document and quality matters more than speed.

Custom templates go in `.rundown/research.md`.

## Output parsing and file update

The worker's stdout is treated as the complete updated Markdown document. The application layer:

1. Reads the current source file content.
2. Takes a checkbox snapshot (before).
3. Writes the worker output as the new file content.
4. Takes a checkbox snapshot (after).
5. **Rejects the update** if any checkbox state changed (same guard as `discuss`).
6. **Rejects the update** if new `- [ ]` items were introduced (research must not produce TODOs).

If rejected, the original content is restored and the command exits with a non-zero code and a clear error message explaining the violation.

## Relationship to other commands

| Command    | Input                  | Output                      | Modifies file? |
|------------|------------------------|-----------------------------|----------------|
| `research` | Thin feature document  | Enriched context document   | Yes (body)     |
| `plan`     | Enriched document      | Additive TODO items         | Yes (appends)  |
| `discuss`  | Task + user            | Refined task text           | Yes (task)     |
| `run`      | Single TODO item       | Project implementation      | Yes (checkbox) |

Research is the only command whose primary output is **document prose** rather than task items or project changes.

## Scope boundaries

**In scope for first iteration:**
- Single-file research with built-in and custom templates.
- Full config cascade support (defaults, commands.research, profiles, CLI).
- Standard diagnostic modes (dry-run, print-prompt).
- Checkbox and TODO-item mutation guards.
- Artifact persistence.

**Out of scope / future work:**
- Multi-file or glob-based research.
- Iterative scan-count convergence (single pass only for now).
- Automatic chaining with `plan` (user runs them separately).
- Research-specific verification (no verify/repair loop — research is best-effort).
- Diff-based partial update (full replacement is simpler and sufficient initially).

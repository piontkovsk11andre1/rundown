# 139. Migrate Walks Released Revision Pairs Not Current

## Problem

`rundown migrate` today targets `design/current/` as the diff endpoint (see `runMigrateLoop` in `implementation/src/application/migrate-task.ts` calling `prepareDesignRevisionDiffContext({ target: "current" })`). This is wrong by the intended model:

- `design/current/` is a **fluid scratchpad** for in-progress design edits. It must never drive migration generation.
- Released revisions `design/rev.0/`, `design/rev.1/`, ŌĆ” are the **immutable target states**. `rev.0` is the baseline (mirrors current implementation, no migrations needed). Each subsequent `rev.N` represents a desired next state.
- `migrate` should walk **revision-to-revision pairs** in order: `(rev.0 ŌåÆ rev.1)`, `(rev.1 ŌåÆ rev.2)`, ŌĆ” catching up the implementation one released revision at a time. It must stop at the highest released revision and never look at `current/`.

Today, with `rev.1` already released and `current/` byte-identical to `rev.1`, the planner sees an empty diff and either says `DONE` or returns prose, when in fact it should be migrating `rev.0 ŌåÆ rev.1`.

## Model (the contract this migration installs)

1. `design/rev.0/` is the baseline. By definition: **already reached**, no migrations.
2. For each subsequent released `rev.N`, migrations exist (or will be generated) that take the implementation from `rev.(N-1)` to `rev.N`. When all such migrations have been executed, `rev.N` is **reached**.
3. Track "reached" via a field in `design/rev.N.meta.json`:
   - `migratedAt: <ISO timestamp>` ŌĆö set when migrate finishes the up-pass for the diff `(rev.(N-1) ŌåÆ rev.N)`.
   - Absent / null ŌåÆ not reached yet.
4. `migrate` (no action) iterates: find the lowest unreached `rev.N` (where `N >= 1`), run the planner with `from = rev.(N-1), to = rev.N`, create migration files, run `migrate up`, mark `rev.N.meta.json.migratedAt`, then continue to `rev.(N+1)`. Stop when no unreached released revision remains.
5. `migrate` **never** reads `design/current/`. `current` only matters at `design release` time.

## Seeding for the existing repo

The current workspace has:
- `design/rev.0` (baseline, by definition reached ŌåÆ write `migratedAt` to its meta).
- `design/rev.1` (released but no targeted migrations exist for it; the existing 138 migrations were authored against `current` before this model). Decision: **treat `rev.1` as already reached** by stamping `migratedAt` on its meta as a one-time backfill ŌĆö the existing 138 migrations are accepted as covering the `rev.0 ŌåÆ rev.1` jump.
- `design/current` matches `rev.1` byte-for-byte.

After this migration ships, the next `design release` will create `rev.2`, and `migrate` will generate fresh migrations for `(rev.1 ŌåÆ rev.2)`.

## Implementation boundaries

Files expected to change:
- `implementation/src/application/migrate-task.ts`:
  - In `runMigrateLoop`, replace the single `prepareDesignRevisionDiffContext({ target: "current" })` with a loop over unreached released revisions; for each, call with `target: "rev.N"` (the existing function already accepts `"current" | string | number`).
  - After a successful `runMigrateUp` for a revision, write `migratedAt` to its meta file.
  - Stop when no unreached released revision remains; emit info: `"Migrations are caught up to rev.N (highest released)."`
- `implementation/src/application/design-context.ts`:
  - Extend `DesignRevisionMetadata` (and the read/write helpers ŌĆö likely in `design-revision-task.ts` or a shared revision-meta helper) with optional `migratedAt: string | null`.
  - Add `findLowestUnreachedRevision(workspaceRoot, fileSystem): RevisionDescriptor | null` ŌĆö returns the lowest `rev.N` (N >= 1) whose meta has no `migratedAt`. Returns `null` when all released revisions are reached.
  - Add `markRevisionMigrated(workspaceRoot, fileSystem, revisionName)` ŌĆö writes `migratedAt: new Date().toISOString()` to the meta file (preserves other fields).
- One-time backfill: as part of this migration's TODO, write `migratedAt` to `design/rev.0.meta.json` and `design/rev.1.meta.json` directly (not via code) so the new logic starts in a sane state on this workspace.

## Out of scope

- Per-migration frontmatter recording target revision (rejected ŌĆö meta-on-revision is simpler).
- Changing `design release` semantics.
- Changing `migrate up` / `migrate down` behavior beyond the new "stamp meta after up" step.
- Reordering or rewriting the existing 138 migrations.
- Multi-workspace / linked-workspace edge cases ŌĆö the existing workspace resolution upstream of `runMigrateLoop` continues to apply.

## Edge cases

- No released revisions yet (`design/rev.0/` doesn't exist): emit a clear info and exit success ŌĆö nothing to migrate.
- Only `rev.0` exists: caught-up state. Emit info and exit.
- `rev.N.meta.json` missing (manually created revision dir without metadata): treat as unreached and emit a warn; still attempt to migrate `(rev.(N-1) ŌåÆ rev.N)`. After success, create the meta file from scratch with `migratedAt`.
- A revision is unreached but its diff is empty (e.g., `rev.N == rev.(N-1)` byte-for-byte from a no-op release): planner gets empty input, returns DONE ŌåÆ still stamp `migratedAt` and continue. A no-op release is allowed.
- Existing tests in `implementation/__tests__/integration/migrate-task.integration.test.ts` that rely on `target: "current"` need updating to use a released revision target. Most of them already build a `rev.N` directory; update fixtures rather than asserting the old behavior.

# 139. Migrate Walks Released Revision Pairs Not Current

## Problem

`rundown migrate` today targets `design/current/` as the diff endpoint (see `runMigrateLoop` in `implementation/src/application/migrate-task.ts` calling `prepareDesignRevisionDiffContext({ target: "current" })`). This is wrong by the intended model:

- `design/current/` is a **fluid scratchpad** for in-progress design edits. It must never drive migration generation.
- Released revisions `design/rev.0/`, `design/rev.1/`, … are the **immutable target states**. `rev.0` is the baseline (mirrors current implementation, no migrations needed). Each subsequent `rev.N` represents a desired next state.
- `migrate` should walk **revision-to-revision pairs** in order: `(rev.0 → rev.1)`, `(rev.1 → rev.2)`, … catching up the implementation one released revision at a time. It must stop at the highest released revision and never look at `current/`.

There is also no record on disk of *which revision* a given migration was generated to satisfy. That makes "did we already plan migrations for rev.N?" unanswerable, so today the planner is asked the same question on every run.

## Model (the contract this migration installs)

1. `design/rev.0/` is the baseline. By definition: **already planned**, no migrations needed.
2. For each subsequent released `rev.N`, migrations are generated for the diff `(rev.(N-1) → rev.N)` and recorded on `rev.N.meta.json`.
3. `rev.N.meta.json` gains two optional fields:
   - `plannedAt: string | null` — ISO timestamp when migrate finished generating migrations for this revision's diff. `null` (or absent) = not planned yet.
   - `migrations: string[]` — list of migration filenames (basename only, e.g. `"139. Foo bar.md"`) generated for this revision. Empty array is valid (means: no diff to act on, planner returned DONE).
4. The cursor is computed: `lastPlannedRevision = max N where rev.N.meta.json.plannedAt != null` (and `rev.0` is always considered planned).
5. `migrate` (no action) iterates: for each released `rev.N` from `lastPlannedRevision + 1` up to the highest released revision:
   - Run the planner with `from = rev.(N-1)`, `to = rev.N`.
   - Create migration files. Record their basenames in `rev.N.meta.json.migrations`.
   - Set `rev.N.meta.json.plannedAt = nowIso()`.
   - Run `migrate up` to execute the new migration files' TODOs.
   - Continue to `rev.(N+1)`.
6. `migrate` **never** reads `design/current/`. `current` only matters at `design release` time.
7. **Important**: `plannedAt` tracks **planning**, not execution. If `migrate up` partially fails (some TODOs remain unchecked), `plannedAt` stays set — the migration files exist with their checkbox state, and any subsequent `rundown run migrations/` or `rundown migrate` will pick up the unchecked TODOs without re-asking the planner. We never want to re-plan the same diff.

## Seeding for the existing repo

The current workspace has:
- `design/rev.0` (baseline). Backfill: `plannedAt = createdAt`, `migrations: []`.
- `design/rev.1` (released; the existing ~128 numbered migrations under `migrations/` were authored against `current` before this model existed). Decision: **treat `rev.1` as already planned** by stamping `plannedAt = rev.1.meta.json.createdAt` and listing **all currently existing numbered migration files in `migrations/`** in `rev.1.meta.json.migrations`. The existing migrations are accepted as covering the `rev.0 → rev.1` jump.
- `design/current` matches `rev.1` byte-for-byte today.

After this migration ships, the next `design release` will create `rev.2`, and `migrate` will generate fresh migrations for `(rev.1 → rev.2)` and record them on `rev.2.meta.json`.

## Implementation boundaries

Creates the file if missing (with `revision`, `index`, `createdAt: <ISO now>`).

The `rev.1` `migrations` array must be the basenames of all `migrations/*.md` files matching the pattern `^\d+\. ` at the moment this migration runs (currently ~128 files, e.g. `1. ## Out of scope

- Per-migration frontmatter recording target revision (rejected — meta-on-revision is simpler and provides the provenance link).
- Changing `design release` semantics.
- Changing `migrate down` behavior — that's migration 140's concern.
- Reordering or rewriting the existing migration files.
- Multi-workspace / linked-workspace edge cases — the existing workspace resolution upstream of `runMigrateLoop` continues to apply unchanged.
- Validating that the implementation actually matches `rev.N` after migrations run. The cursor tracks "planned," not "implementation matches."
- Detecting drift between `rev.N.meta.json.migrations` and the actual files in `migrations/` (e.g., someone deleted a listed file). Worth a future migration but not this one.

## Edge cases

- No released revisions yet (`design/rev.0/` doesn't exist): `findLowestUnplannedRevision` returns `null`. Migrate emits `"No released design revisions yet. Run `rundown design release` to create rev.0."` and exits 0.
- Only `rev.0` exists (after backfill): caught-up state. Emit `"Migrations are caught up to rev.0 (highest released revision)."` and exit 0.
- `rev.N.meta.json` missing entirely on disk (manually created revision dir without metadata): treat as unplanned; on success of the planner, `markRevisionPlanned` creates the meta from scratch.
- Diff between `rev.(N-1)` and `rev.N` is empty (a no-op release): planner gets empty input and returns DONE. Still call `markRevisionPlanned` with `migrations: []`. Continue to next revision.
- Planner produces no parseable names: do **not** stamp `plannedAt`. Emit warn (today's existing message), exit 0. Re-running `migrate` will re-attempt the same revision.
- `runMigrateUp` partially fails (some TODOs unchecked): `plannedAt` is **already** stamped (it was set after file creation, before up). This is intentional — the planning is done. Emit error and return non-zero so the user sees the failure; the next `rundown run migrations/` or `rundown migrate` (which still calls runMigrateUp on entry per migration-task structure — verify and preserve) will pick up the unchecked TODOs without re-planning.
- Stamp ordering: `markRevisionPlanned` must be called **after** all migration files are written but **before** `runMigrateUp` is called. This guarantees that a crash mid-up doesn't cause re-planning on the next invocation.

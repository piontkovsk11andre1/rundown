# 140. Migrate Down Clears MigratedAt On Reverted Revisions

## Problem

Migration 139 introduces `migratedAt` on `design/rev.N.meta.json` to track which released revisions have been reached by the migration history. `migrate up` stamps it after a successful pass. But `migrate down [n]` reverts the last n migration runs without un-stamping the corresponding revisions, leaving the cursor inconsistent: the system would believe `rev.N` is reached when in fact the migrations that took it there have been undone.

This migration closes that loop.

## Model

- Each released revision `rev.N` (N >= 1) is "reached" iff its meta has a non-null `migratedAt`.
- `migrate up` of the diff `(rev.(N-1) → rev.N)` sets `rev.N.meta.json.migratedAt`.
- `migrate down` that reverts work belonging to the diff `(rev.(N-1) → rev.N)` clears `rev.N.meta.json.migratedAt` (sets it back to `null`, or removes the key — pick one and stick with it; **decision: set to `null` for round-trip stability**).
- `rev.0` is the baseline; `migrate down` never modifies its meta.

## How "belonging to revision N" is determined

Migration 139 plans by revision pair, but does **not** record per-migration revision attribution in the file itself. To avoid adding frontmatter, attribution is reconstructed from artifact runs:

- Each `migrate up` records its revision target in the artifact `run.json` (e.g., add `extra.targetRevision: "rev.2"`).
- `migrate down` already chooses runs to revert by `--run <id|latest>` / count. After reverting, group the reverted runs by `extra.targetRevision`. For each group, if **all** migrations belonging to that revision are now in a reverted state, clear `migratedAt` on that revision's meta.

A migration is "in a reverted state" when its file no longer exists OR its TODOs are all unchecked AND no more-recent un-reverted run touched it. Conservatively, this migration uses a simpler rule: **clear `migratedAt` on a revision iff `migrate down` reverted at least one run whose `extra.targetRevision` equals that revision name**. This errs on the side of marking unreached, which is the safe direction (re-running `migrate up` will re-stamp).

## Implementation boundaries

Files expected to change:
- `implementation/src/application/migrate-task.ts`:
  - In the `runMigrateUp` artifact-finalize step (or wherever `extra` is composed), include `targetRevision: <rev name>`. The revision is known because migration 139 made `runMigrateUp` callable per-revision.
  - In `runMigrateDown`, after the existing revert path, collect the set of distinct `extra.targetRevision` values from the reverted runs. For each, call `markRevisionUnmigrated(fileSystem, projectRoot, revisionName)`.
- `implementation/src/application/design-context.ts` (or the revision-meta helper introduced in 139):
  - Add and export `markRevisionUnmigrated(fileSystem, workspaceRoot, revisionName)` that writes `migratedAt: null` to the named revision's meta file, preserving all other fields. No-op if the file is missing.
- Tests in `implementation/__tests__/integration/migrate-task.integration.test.ts`:
  - Add coverage for the up→down→meta-cleared flow.

## Out of scope

- Re-running migrations automatically after a down. Down stays manual.
- Cross-revision dependency tracking. If a single artifact run somehow spans two revisions (it shouldn't after 139), that's a bug in 139, not 140's concern.
- Backfilling `extra.targetRevision` on the existing 138 migrations' historical runs — those runs are out of `migrate down` scope already (the existing flow only reverts recent runs).
- Per-migration frontmatter (still rejected).

## Edge cases

- `migrate down` reverts runs that have no `extra.targetRevision` (legacy / pre-139 runs): skip them silently. Don't touch any meta.
- `migrate down` reverts runs across multiple revisions (e.g., `--run` count of 5 spanning `rev.2` and `rev.3` runs): clear `migratedAt` on both `rev.2` and `rev.3`.
- The reverted-run set is empty (down found nothing to do): no meta changes.
- Meta file missing for a revision named in a reverted run: log warn and skip.

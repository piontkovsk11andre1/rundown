# 140. Migrate Down Rewinds By Revision Boundary

## Problem

Migration 139 introduces `plannedAt` and `migrations` fields on `design/rev.N.meta.json`, recording which migration files were generated for each revision's diff. With this provenance link in place, `migrate down` can be redefined from "undo the last n artifact runs" (today's behavior, which has no relationship to design revisions) to a cleaner model: **rewind to a revision boundary**.

The existing `migrate down [n]` semantics — counting artifact runs — is fragile: it has no awareness of revision boundaries, can split a logically atomic revision rewind across multiple invocations, and leaves the new `plannedAt` cursor on `rev.N.meta.json` lying about state.

## Model

`migrate down` rewinds the implementation to the boundary at `rev.K`, where `K` is `lastPlannedRevision - 1` by default (or chosen explicitly). To rewind to `rev.K`:

1. Compute the set of revisions `R = {rev.K+1, rev.K+2, …, lastPlannedRevision}` to drop.
2. For each revision in `R` (descending order), revert the migration files listed in `rev.N.meta.json.migrations`:
   - Find artifact runs whose source file matches one of those migrations and revert them via the existing git-revert path.
   - When all listed migrations for `rev.N` have been successfully reverted, clear that revision's cursor: `plannedAt = null`, `migrations = []`.
3. Optionally delete the migration files themselves and (if `--backlog` is on, default true) push their content to `migrations/Backlog.md`. This is the existing "backlog promotion" behavior, just gated by the revision boundary instead of run count.
4. After the loop, the cursor naturally returns to `rev.K`.

Behavior matrix:

| Invocation | Effect |
| --- | --- |
| `migrate down` | Rewind one revision: drop `lastPlannedRevision` |
| `migrate down 2` | Rewind two revisions |
| `migrate down --to rev.0` | Rewind all the way to baseline (drops everything from rev.1 onward — disallowed on this workspace by default; requires `--allow-baseline-rewind`) |

## Why this is cleaner

- "Down" matches the mental model of design revisions, not artifact run ids.
- The `plannedAt` cursor stays consistent automatically (cleared as part of the rewind).
- Artifact runs without revision attribution (legacy / non-migrate runs) are no longer in scope for `migrate down` — they are reachable via existing `rundown revert` / `rundown undo` commands.
- The `--run <id|latest>` option becomes unnecessary in the new model. Keep as a deprecated escape hatch for one release, then remove.

## Implementation boundaries

Files expected to change:
- `implementation/src/application/migrate-task.ts`:
  - Rewrite `runMigrateDown` so that it:
    1. Reads all released revision metas (via the helpers added in 139).
    2. Resolves a target `rev.K` from CLI input (`down` count, or `--to <revName>`).
    3. Walks revisions descending from `lastPlannedRevision` to `rev.K+1`, reverting their listed migrations and clearing each meta as it goes.
  - Replace the existing run-count loop. Keep error handling (revert failure stops the rewind and leaves meta in a partially-cleared state, with a clear error message naming the revision that failed).
  - Remove or deprecate `--run <id|latest>` (decision: deprecate with a warn, log it as removed-next-release).
- `implementation/src/presentation/cli.ts`:
  - Add `--to <revName>` option to `migrate down`.
  - Update `--no-backlog` help text only if needed (semantics unchanged).
- Tests in `implementation/__tests__/integration/migrate-task.integration.test.ts`:
  - Replace existing run-count down tests with revision-boundary down tests.

## Out of scope

- Auto-running `migrate up` after a successful down. Up stays manual.
- Cross-revision dependency analysis. If migrations from `rev.2` depend on state created by `rev.3` (they shouldn't, by model), that's a planner bug, not down's concern.
- Backfilling pre-139 artifact runs with `targetRevision`. Those are out of scope for down.
- Any change to `rundown revert` / `rundown undo` (general-purpose run reverters).

## Edge cases

- `migrate down` when `lastPlannedRevision == "rev.0"`: nothing to rewind. Emit info `"Already at baseline (rev.0). Nothing to rewind."` and exit 0.
- `migrate down N` when `N > (lastPlannedRevision - 0)`: rewinds as far as possible (down to rev.0 + 1 by default, requires `--allow-baseline-rewind` to clear rev.0+1 too — actually rev.0 is never cleared since it's the baseline; this clarifies to: rewinds all the way through rev.1's migrations, leaving `lastPlannedRevision = rev.0`). Emit info naming the actual stopping point.
- `migrate down --to rev.5` when `rev.5` is already the highest planned: no-op. Emit info and exit 0.
- `migrate down --to rev.7` when `rev.7` is unplanned (`plannedAt == null`): error. The target must be a planned revision (or `rev.0` for the baseline case).
- A revision's `migrations: []` (no-op release): clearing its meta is a metadata-only operation; no files to revert, no backlog promotion. Just clear `plannedAt`.
- A migration listed in `rev.N.meta.json.migrations` no longer exists on disk (someone deleted it manually): emit warn, treat as already reverted, continue. Don't fail the down.
- A migration listed in `rev.N.meta.json.migrations` has unchecked TODOs (was never fully run): revert is still attempted via the existing artifact-run path; if no artifact run exists, simply delete the file (and push to backlog if applicable). This is consistent with the existing partial-down behavior in today's code.

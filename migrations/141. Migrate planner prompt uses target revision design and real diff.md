# 141. Migrate Planner Prompt Uses Target Revision Design And Real Diff

## Problem

When `rundown migrate` plans migrations for an unplanned released revision (e.g. `rev.2`), today's behavior is broken in two user-visible ways:

1. **Opaque CLI message.** `runMigrateLoop` in `implementation/src/application/migrate-task.ts` emits:

   ```
   ℹ Planning migrations from design revision diff to rev.2 (position 140)...
   ```

   The user cannot tell **from which revision** the diff is being computed. With the model installed by migration 139 (`rev.(N-1) → rev.N` walks), the source revision is well-defined (`prepareDesignRevisionDiffContext` already resolves it as `revisionDiff.fromRevision`). It must appear in the message.

2. **Planner prompt does not actually contain the diff and points at the wrong design.** `DEFAULT_MIGRATE_TEMPLATE` in `implementation/src/domain/defaults.ts` is rendered with these substitutions (see `buildTemplateVars` in `migrate-task.ts`):

   - `{{design}}` is filled by `resolveDesignContext(fileSystem, projectRoot, …).design`, which always reads from `design/current/` — the **fluid scratchpad**, never the target revision. Migration 139 explicitly removed `design/current/` from migrate's data path, but this leak still exists through `{{design}}`.
   - `{{designRevisionDiffFiles}}` is only a list of `kind: relativePath` lines (added/modified/removed). The planner has **no per-file content** to inspect — neither the prior contents, nor the new contents, nor a unified diff. So the planner cannot reason about what actually changed.
   - `{{designRevisionDiffSources}}` is just a list of source directory paths. Useful for traceability but not actionable for an agent that must decide what migrations to write.

   Net effect: when planning for `rev.2`, the planner sees the contents of `design/current/` (which usually equals `rev.2` here, but conceptually shouldn't be consulted), and a bare list of file names labelled "modified" — with no way to see *what* was modified.

The user expectation (per migration 139's contract) is: the planner should be focused on the **diff** between `rev.(N-1)` and `rev.N`, with the **target revision's design** as the authoritative source-of-truth context. `design/current/` must not be read.

## Model

For each unplanned revision `rev.N`:

1. **Info message** must include both endpoints. New format (no trailing comma artifacts):

   ```
   ℹ Planning migrations for rev.<N-1> → rev.N (position <pos>)...
   ```

   When there is no previous revision (initial diff against nothing — i.e. first revision after `rev.0` is missing, or `rev.0` itself if it ever became unplanned), use `nothing` as the source name to match the existing `formatInitialDesignDiffSummary` wording:

   ```
   ℹ Planning migrations for nothing → rev.N (position <pos>)...
   ```

2. **Planner prompt design content** must come from the **target revision directory** (`revisionDiff.toTarget.absolutePath`), not from `design/current/`. Reuse the existing `formatDesignWorkspaceContext` formatter so the rendered shape matches what the rest of the system produces; just point it at the target revision dir instead of `current/`.

3. **Planner prompt diff content** must include the actual content changes. Add a new section to `DEFAULT_MIGRATE_TEMPLATE` titled `### Diff` that contains a unified-style diff per changed file (added / modified / removed). Format per file:

   ```
   #### <relativePath> (added|modified|removed)

   ```diff
   <unified-diff body, 3 lines of context, no headers>
   ```
   ```

   For `added` files: emit only `+` lines for the entire new content. For `removed`: only `-` lines for the entire old content. For `modified`: a real line-level unified diff (use a small in-repo line diff, no external dependency — node `diff` already in `package.json` if present, otherwise a minimal LCS-based hunk producer is acceptable; prefer reusing whatever `prepareDesignRevisionDiffContext` already has access to via `collectDirectoryFileMap` so we don't re-read files).

   Keep the `### Changed files` summary (file-kind list) — it's still useful as an at-a-glance index. Keep `### Diff source references` for traceability.

4. **`design/current/` is never read by migrate.** Drop the `resolveDesignContext(... )` call from migrate's `buildTemplateVars` entirely. Replace `{{design}}` with the target-revision-derived content described in (2). `resolveDesignContextSourceReferences` may stay (or be repointed at the target revision) since it only contributes the `{{designContextSourceReferences}}` listing — but it must not pull from `current/` either. Simplest: derive both from the target revision dir and keep both vars filled.

## Implementation boundaries

Files expected to change:

- `implementation/src/application/migrate-task.ts`
  - `runMigrateLoop`: change the info `emit({ kind: "info", message: ... })` to include `revisionDiff.fromRevision?.name ?? "nothing"`. To do this without re-running the diff, hoist `prepareDesignRevisionDiffContext({ target: targetRevision.name })` above the `emit` call (or compute it inside `buildTemplateVars` and surface the source revision back). Cleanest: call `prepareDesignRevisionDiffContext` once at the top of the loop iteration, pass the resulting context into `buildTemplateVars` (new optional input) so it isn't recomputed.
  - `buildTemplateVars`: accept an optional `revisionDiff: DesignRevisionDiffContext` input (compute internally if not provided, to keep other callers untouched). When the target is a revision (not `current`), fill `{{design}}` from the target revision directory using the existing formatting helper, **not** from `resolveDesignContext`. When target is `current` (no other live caller today, but keep behavior intact for safety), preserve today's behavior.
  - Add new template var `designRevisionDiffContent` containing the per-file unified diffs (joined with blank lines). Empty string when there are no changes.

- `implementation/src/application/design-context.ts`
  - Export a small helper to format a single target revision directory as design context, e.g. `formatRevisionDesignContext(fileSystem, revisionAbsolutePath, primaryFileName?): string`. Reuse the existing private `collectDesignFiles` + `formatDesignWorkspaceContext` (export a thin wrapper if needed; do not duplicate the formatting logic). Default `primaryFileName` to `Target.md` (the `CANONICAL_PRIMARY_FILE` constant).
  - Add a helper to compute per-file unified diff text from a `DesignRevisionDiffContext` (or directly from the two revision dirs). Signature: `formatDesignRevisionUnifiedDiff(fileSystem, diff: DesignRevisionDiffContext): string`. Implementation may piggy-back on the existing `collectDirectoryFileMap` to read once. Keep the diff algorithm minimal and self-contained — no new runtime dependencies. If `diff` package is already a transitive dep verifiable via `implementation/package.json`, prefer it; otherwise implement a tiny Myers/LCS-based hunk generator inline (≤ ~120 LOC).

- `implementation/src/domain/defaults.ts`
  - Update `DEFAULT_MIGRATE_TEMPLATE`:
    - Replace the `## Design content` section heading body to reflect that the content is **the target revision's design** (e.g. `## Target revision design ({{designRevisionToTarget}})`).
    - Add a new top-level `## Design diff ({{designRevisionFromRevision}} → {{designRevisionToTarget}})` section containing:
      - The existing summary line.
      - The existing `### Changed files` list.
      - A new `### Diff` block that interpolates `{{designRevisionDiffContent}}`.
      - The existing `### Diff source references` list.
    - Remove `{{design}}`'s old framing comment that suggested it was the current scratchpad. Keep the Latest snapshot, Backlog, Migration history sections unchanged.

- `implementation/__tests__/application/migrate-task.test.ts` (or wherever planner-prompt rendering is unit-tested today — find it; if no dedicated test exists, add one)
  - Assert that for a setup with `rev.0`, `rev.1` planned, and `rev.2` unplanned with one modified file and one added file vs `rev.1`, the rendered prompt:
    - Contains the new info message format (assert via captured output port).
    - Contains the target-revision design content (a unique sentinel string placed in `rev.2/Target.md`).
    - Does **not** contain a sentinel string placed only in `design/current/Target.md`.
    - Contains a `### Diff` section with `+` and `-` lines reflecting the modification.
    - Contains the added file's full content under a `+` block.

- `implementation/__tests__/application/design-context.test.ts`
  - Add unit tests for the new helpers: `formatRevisionDesignContext` returns the same shape as `resolveDesignContext` would for an equivalent `current/` layout, and `formatDesignRevisionUnifiedDiff` produces a stable, snapshot-friendly diff string for a known small input.

## Out of scope

- Changing the `DONE` sentinel or the slug-list output format the planner produces.
- Changing how the planner is invoked (worker selection, timeouts, artifact phases).
- Refactoring `resolveDesignContext` itself; only its callers in migrate change.
- Adjusting `migrate-snapshot` (`DEFAULT_MIGRATE_SNAPSHOT_TEMPLATE`) — that template runs **after** migrations exist and may want a different framing; out of scope here. (Open question for a follow-up: should it also drop `{{design}}` in favor of the target revision? Don't decide here.)
- Internationalizing the new info message (treat as English-only for now, matching surrounding messages).

## Edge cases

- **No previous revision (initial diff against `nothing`):** info message uses `nothing → rev.N`. The `### Diff` block lists every file as `added` with full `+` content.
- **Empty diff (no-op release):** the `### Diff` block is empty (or contains a single line `(no content changes)`). The planner template's existing instruction to return `DONE` when there's nothing to do covers this.
- **Binary or very large files:** revisions are intended to hold Markdown / small text. Out of scope. If a non-UTF8 file slips in, fall back to `(binary or non-text content; <bytes> bytes)` placeholder rather than dumping bytes.
- **Target revision dir missing files but metadata exists:** `formatRevisionDesignContext` returns an empty string; the prompt still renders, planner sees diff section only.
- **Linked workspace:** workspace resolution is upstream of `runMigrateLoop`; no change needed here. The target revision path already comes from `revisionDiff.toTarget.absolutePath` which respects the resolved workspace root.

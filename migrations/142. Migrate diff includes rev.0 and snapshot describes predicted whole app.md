# 142. Migrate Diff Includes Rev.0 And Snapshot Describes Predicted Whole App

## Problem

After migration 141 shipped, `rundown migrate` against this workspace (with `rev.1.meta.json.plannedAt` cleared to retest) exhibits four related defects, all rooted in the migrate prompt path still mis-framing the diff and the snapshot prompt being too vague.

1. **`rev.0` is invisible as a diff source.** `runMigrateLoop` emits `Planning migrations for nothing → rev.1 (...)` even though `design/rev.0/` is a real directory with content. Cause: `findNearestLowerDiscoveredRevision` in `implementation/src/application/design-context.ts` filters with `revision.index > 0`, so it never returns `rev.0` as a previous revision. `prepareDesignRevisionDiffContext` then takes the "initial diff" branch (`previousRevision == null` → `computeInitialDirectoryDiff`) and synthesizes a fake "all-added" diff, hiding the real per-file changes between `rev.0` and `rev.1`.

2. **No way (in product) to inspect the diff that was sent to the planner.** Today the only path is `--show-agent-output --keep-artifacts`, then dig under `.rundown/runs/<id>/` for the worker prompt. There is no first-class command to print "what is the design diff `rev.K → rev.K+1`?". Users hit this immediately when debugging "why did the planner say DONE?".

3. **Planner returns `DONE` for legitimate diffs.** Combined effect of (1) plus the migration history block in the prompt: the planner sees `Compared nothing -> rev.1: N added 0 modified 0 removed` plus 138 existing migrations and concludes "all already covered, return DONE". Fix (1) and the input becomes meaningful (real `+/-` lines for the one file that actually changed), so the planner can reason correctly.

4. **Snapshot prompt describes the delta, not the predicted whole-app state.** Two issues:
   - **Wrong design source.** `generateSatellite` calls `buildTemplateVars` **without `designRevisionTarget`** (see `implementation/src/application/migrate-task.ts` line ~1605), so `prepareDesignRevisionDiffContext` falls back to `target: "current"` and `{{design}}` is filled from `design/current/` (the scratchpad). The snapshot generator should be told "the just-completed migrations brought the implementation up to `rev.N`; describe the predicted whole-app state at `rev.N`", which means the design content must come from `design/rev.N/`, not from `current/`.
   - **Vague instruction.** `DEFAULT_MIGRATE_SNAPSHOT_TEMPLATE` says: *"Update the snapshot to reflect changes introduced by the new migrations. Keep concise - capture decisions and structure, not exhaustive detail."* This produces a delta-style note. A snapshot is meant to be the **predicted post-migration state of the whole application** (modules, public surfaces, invariants, key flows), not a changelog.

## Model

After this migration:

1. `rev.0` participates in diffs like any other revision. `findNearestLowerDiscoveredRevision` returns the highest revision strictly below the target index, with no special-casing of `rev.0`.
2. A new read-only command `rundown design diff [<target>]` prints the diff that migrate would compute, in the same unified-diff format the planner sees. Default target is the lowest unplanned revision (matching `findLowestUnplannedRevision`); when none, falls back to the highest released revision against its predecessor. Explicit target accepts `rev.N` and computes against `rev.(N-1)` (or `nothing` when no predecessor exists).
3. With (1) fixed, the planner receives a real diff and can answer correctly. No code change in the planner itself; this falls out of (1).
4. Snapshot generation:
   - Threads `designRevisionTarget` (the just-migrated revision) into `buildTemplateVars`, so `{{design}}` resolves via `formatRevisionDesignContext(rev.N)` (introduced in 141), not via `resolveDesignContext` reading `current/`.
   - `DEFAULT_MIGRATE_SNAPSHOT_TEMPLATE` is rewritten to ask for the **predicted whole-app state at the target revision**: an enumeration of modules / packages / public surfaces / invariants / key flows after the migrations are applied. The delta itself is still referenced (via `{{newMigrations}}`) but is positioned as evidence, not as the deliverable.

## Out of scope

- Changing the planner template itself (was the focus of 141; any further tweaks belong in a separate migration).
- Adding `rundown design diff --from <revA> --to <revB>` with arbitrary pairs beyond the predecessor relation. Single positional + optional `--from` is enough; arbitrary pairs are a future enhancement.
- Persisting the snapshot diff in machine-readable form (JSON). The snapshot stays Markdown-only.
- Introducing new template vars on the snapshot template. Keep var surface stable to not break user-customized `migrate-snapshot.md`.
- Backfilling old snapshots in this workspace to the new format. Only forward-looking.
- Any change to `migrate down` (139/140 territory).

## Edge cases

- **Target = `rev.0`.** No predecessor exists. `prepareDesignRevisionDiffContext({ target: "rev.0" })` continues to return the existing initial-diff form (`fromRevision: null`, summary `Compared nothing -> rev.0`). The CLI `rundown design diff rev.0` prints accordingly; the planner path is irrelevant since `rev.0` is always considered planned (per 139).
- **`design/rev.0/` and `design/rev.1/` byte-identical.** Diff is empty; `### Diff` block is empty; planner sees zero changes and returns `DONE`. `markRevisionPlanned` is called with `migrations: []`. Same shape as the existing no-op-release case from 139.
- **`rev.N` exists but its directory is empty (corruption).** `prepareDesignRevisionDiffContext` already returns the "target directory does not exist" branch when missing; for empty-but-existent, the comparison succeeds with `removedCount > 0` (everything in `rev.(N-1)` is "removed"). This is correct and matches user intent if they truly emptied the revision.
- **Snapshot generation when `targetRevision` is absent (legacy callers):** `generateSatellite` continues to work; `{{design}}` falls back through the existing `resolveDesignContext` path. We don't break the legacy path; we just extend the satellite call inside `runMigrateUp` to thread the target.
- **`rundown design diff` on a workspace with no released revisions:** error with `"No released design revisions yet. Run rundown design release to create rev.0."` (matches existing migrate idle message).

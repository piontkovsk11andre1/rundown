# 144. Add Prediction Bucket To Workspace Directories

## Problem

The framework's mental model is shifting toward four parallel tracks:

1. `design/` — target, domain language (already a workspace bucket).
2. `specs/` — assertion specs (already a workspace bucket).
3. `migrations/` — numbered steps (already a workspace bucket).
4. `prediction/` — predicted-implementation tree, sibling to `implementation/` (**not yet a bucket**).

Today the workspace config in `implementation/src/application/workspace-paths.ts` (renamed from `prediction-workspace-paths.ts` in migration 143) exposes only the first three buckets. The `prediction` directory has no canonical home, no config key, no resolution helper, no template var. Any future migration that wants to read or write to `prediction/` has to hard-code the path, which contradicts how every other tracked directory works.

This migration introduces the `prediction` bucket as a peer of the existing three. It is **plumbing only**: nothing reads or writes `prediction/` after this migration. Subsequent migrations (145+) will hook readers/writers onto it.

## Model

After this migration:

- `WorkspaceDirectories` has a fourth field: `prediction: string`.
- `WorkspacePlacementMap` has a fourth field: `prediction: WorkspacePlacement`.
- `WorkspacePaths` has a fourth field: `prediction: string`.
- `WorkspaceBucket` includes `"prediction"` (it is `keyof WorkspaceDirectories`, so this happens automatically once the field is added).
- `DEFAULT_WORKSPACE_DIRECTORIES.prediction === "prediction"`.
- `DEFAULT_WORKSPACE_PLACEMENT.prediction === "sourcedir"`.
- `resolveWorkspaceConfig`, `resolveWorkspacePaths`, and `resolveWorkspacePath` all resolve the new bucket symmetrically with the existing three.
- Conflict validation (`validateDirectoryConflicts`, `validateResolvedBucketConflicts`) covers the new bucket.
- `WORKSPACE_CONTEXT_TEMPLATE_VAR_KEYS` includes `"workspacePredictionDir"`, `"workspacePredictionPlacement"`, `"workspacePredictionPath"`.
- `buildWorkspaceContextTemplateVars` emits those three new keys.
- `runtime-workspace-context.ts`'s default-paths fallback computes the prediction path the same way as the others.
- `start-project.ts` (workspace scaffolder) creates `prediction/` alongside `design/`, `specs/`, `migrations/`.

No consumer of the `prediction` bucket beyond config plumbing and template vars exists yet. `migrate up`, `materialize`, `test --future`, satellite generation are unchanged in this migration.

## Out of scope

- Wiring any reader or writer onto the `prediction` bucket. Migrations 145+ do that.
- Updating `design/current/**` or `prediction/**` text content. This migration deliberately leaves docs out of sync; doc reconciliation is a later phase.
- Adding a CLI flag (`--prediction-dir <path>`) to override the bucket. CLI option surface stays unchanged.
- Touching the AGENTS.md / harness preset templates.
- Modifying `.rundown/config.json` in the live workspace. Config keys stay implicit defaults.

## Edge cases

- **Existing project without `prediction` configured.** Defaults to `sourcedir` placement and `prediction/` directory name. The directory may not exist on disk; `resolveWorkspacePaths` returns the path regardless (consistent with how `design`/`specs`/`migrations` work today — there is no `exists` check, only normalization and conflict detection).
- **`prediction` configured to clash with another bucket.** Same conflict error as today's design-vs-migrations clash. `validateDirectoryConflicts` iterates pairs, so it picks up the new bucket once added.
- **`prediction` placement set to `workdir`.** Symmetrical with the existing `workdir` story for design/specs/migrations: resolved path roots at invocation root, not workspace root. Validation rules unchanged.
- **`start-project` on a workspace where `prediction/` already exists.** Idempotent: scaffolding should not overwrite. Match the existing behavior of the other three buckets.
- **Templates referencing the new vars don't exist yet.** That is fine: vars are exposed but no template uses them. No runtime regression.

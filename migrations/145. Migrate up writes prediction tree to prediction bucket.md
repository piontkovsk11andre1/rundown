# 145. Migrate Up Writes Prediction Tree To Prediction Bucket

## Problem

Today `migrate up` (and the materialize/snapshot variants in `migrate-task.ts`) persists the predicted-implementation state as **satellite files attached to a migration**. After every successful migration, `persistPredictionBaselineSnapshot` reads the existing satellite files (`*.snapshot.md`, `*.context.md`, `*.backlog.md`) and reconciles a prediction baseline. The baseline lives inside `migrations/` as opaque per-file satellites — it is *not* a tree that mirrors `implementation/`.

Migration 144 introduced a real `prediction` bucket (a directory sibling to `implementation/`). The new mental model says: after `migrate up`, the `prediction/` directory **is** the predicted implementation — file for file, mirroring `implementation/`'s layout. Anyone (a planner prompt, a diff tool, a human) can read `prediction/foo/bar.ts` and see what the predicted code looks like.

This migration switches the writer in `migrate up`. Instead of (or in addition to — see Out of scope) writing satellite files, a successful `migrate up` writes each predicted file into `prediction/<relative-path>` using the bucket resolved by `resolveWorkspacePath({ bucket: "prediction", ... })`. Reading from satellites continues to work for now (back-compat). Migration 148 removes the satellite reader and writer once nothing depends on them.

## Model

After this migration:

- After a successful `migrate up` of migration N, every predicted file produced by the planner/snapshot stage for migration N is written to `prediction/<relative-path>`. Path is relative to the project root, identical to where the same file would land if materialized into `implementation/`.
- The predicted file's content is the latest predicted text for that file (the same content that previously fed `latestSnapshot`/`latestContext` satellites).
- Files that were predicted in migration N-1 but no longer exist in N's predicted state are deleted from `prediction/`. The `prediction/` tree always reflects the predicted state at the latest executed migration — it is not append-only.
- The `migrations/` directory still contains its own `.md` migration files, but no longer accumulates `*.snapshot.md` / `*.context.md` / `*.backlog.md` satellites for files written into `prediction/` (back-compat reading paths remain — see Out of scope).
- `migrate down` (rewind) restores the `prediction/` tree to match the executed-up-to migration after rewind. Walking back to revision boundary R rewrites `prediction/` to the predicted state at R.
- `migrate up` of a fresh workspace (no executed migrations) leaves `prediction/` empty.
- The reconciliation domain (`prediction-reconciliation.ts`) keeps its existing `PredictionBaseline` shape; only the *persistence target* changes from satellites to the `prediction/` bucket. The in-memory baseline is computed by reading the `prediction/` tree directly when the bucket is non-empty; otherwise it falls back to reading satellites (back-compat).

## Out of scope

- **Removing satellite writes.** Satellite files (`*.snapshot.md`, `*.context.md`, `*.backlog.md`) continue to be authored by `generateSatellite` for back-compat with any reader that has not yet migrated. Migration 148 removes the writer once the reader is gone.
- **Removing satellite readers.** The fallback that reads satellites when `prediction/` is empty stays put — that's how upgrades work. Migration 148 deletes the fallback after a confirmation pass.
- **Changing the `PredictionTrackedFileKind` taxonomy.** The `"snapshot" | "context" | "backlog" | "executed"` distinctions stay for this migration. Collapsing them belongs in 148.
- **Materialize.** `materialize` does not write to `prediction/` — see migration 146 for the contract.
- **Init/start scaffolding.** Migration 147 owns the case where `prediction/` does not yet exist on disk for a fresh project.
- **Planner / snapshot prompt template changes.** Prompts continue to reference the satellite files where they did before. Migration 149 cleans up the prompt copy.

## Edge cases

- **`prediction/` directory does not exist.** `writePredictionTree` creates it (`fileSystem.mkdirp` or equivalent) before the first write. Empty baseline ⇒ leave dir empty; do not error.
- **`prediction/` contains stale files from a hand-edit.** Treated as not-in-baseline ⇒ deleted on next `migrate up`. Document this in the inline comment of `writePredictionTree`.
- **File path collision between `prediction/` and `implementation/` config.** If `directories.prediction === directories.<other>`, that conflict is already caught by `validateDirectoryConflicts` (added in 144).
- **`workdir` placement for `prediction`.** `resolveWorkspacePath` already handles this; no special case in the writer.
- **Read-only filesystems / write failures.** Surface the error from `fileSystem.writeText`. No silent fallback.
- **Concurrent `migrate up` invocations.** Out of scope (no locking exists today). Behavior matches today's satellite write behavior — last writer wins.

# 146. Materialize Does Not Write To Prediction Bucket

## Problem

After migration 145, the `prediction/` bucket is the canonical persisted form of the predicted-implementation tree, written by `migrate up`. The `materialize` command (CLI: `rundown materialize <migration>`) is conceptually different: it applies a migration to the **real** workspace by running the worker against `implementation/` (or the linked target). It does not, and should not, also rewrite `prediction/`.

Today `materialize`, by routing through the same `migrate-task.ts` machinery as `migrate up`, calls `persistPredictionBaselineSnapshot` (lines ~996/1010/1041 of `migrate-task.ts`). With migration 145 in place, that means `materialize` ends up rewriting `prediction/` as a side-effect of running. That is wrong by design:

- `prediction/` is meant to be the *predicted* state (what `migrate up` thinks the code should look like).
- After `materialize`, `implementation/` is what the code actually looks like, but `prediction/` should remain the prediction made by the most recent `migrate up`. Re-running `migrate up` is the explicit way to refresh `prediction/`.
- Mirroring on materialize would erase the prediction-vs-reality gap that the new model is built around.

This migration carves out `materialize` so it never writes to the `prediction/` bucket. It still writes to `implementation/` (or the linked target) the same way it does today.

## Model

After this migration:

- The `materialize` command does not call any writer that targets the `prediction/` bucket. `prediction/` is read-only from materialize's perspective.
- `materialize` continues to: run the worker against the real implementation, mutate checkboxes inside the migration `.md`, and (today) write satellite files. Satellite writes from `materialize` stay until 148 removes them globally.
- `migrate up` is the only producer of `prediction/`. Re-running `rundown migrate up` after `rundown materialize <X>` is the supported way to refresh `prediction/`.
- The contract is documented inline at the `materialize` action site in `cli-command-actions.ts` and in the `materialize` command help in `cli.ts`: a one-line note "writes implementation/ only; refresh prediction/ with `rundown migrate up`".

## Implementation boundaries

Files expected to change:

- `implementation/src/application/migrate-task.ts`
  - The `persistPredictionBaselineSnapshot` calls inside the materialize-related code paths (currently around lines ~996, ~1010, ~1041 — the materialize/run-with-snapshot entry points) are gated by a new boolean parameter `writePredictionBucket: boolean`. When `false`, the helper still computes the baseline (for return value / logging) but skips the `writePredictionTree` step added in 145. Satellite writes (back-compat) stay regardless.
  - Add the parameter to the relevant function signatures with a default of `true`. The materialize-specific call sites pass `false`.
  - Alternative if cleaner: split `persistPredictionBaselineSnapshot` into two — `computePredictionBaseline` (pure) and `writePredictionBaselineToBucket` — and have materialize call only the first.

- `implementation/src/presentation/cli-command-actions.ts`
  - In `createMaterializeCommandAction` (line ~549), thread the `writePredictionBucket: false` flag (or the equivalent split) into the migrate-task entry point.

- `implementation/src/presentation/cli.ts`
  - Update the `materialize` command's `.description(...)` (around line ~228) to add: `Writes implementation/ only; refresh prediction/ with \`rundown migrate up\`.`

- Tests
  - `implementation/__tests__/application/migrate-task.test.ts` (or the materialize-specific test file):
    - Add a test that runs `materialize` on a workspace with a non-empty `prediction/` directory and asserts `prediction/` is **byte-for-byte unchanged** afterward.
    - Add a test asserting that `materialize` still writes to `implementation/` correctly (regression guard).
  - `implementation/__tests__/presentation/cli.test.ts` (if it asserts help/description text): update the materialize description assertion to include the new sentence.

## Out of scope

- Removing satellite writes from the materialize path. That belongs to 148.
- Changing the materialize CLI surface (flags, args). No new flag for "also refresh prediction".
- Auto-running `migrate up` after `materialize`. Explicitly out of scope — keep the operations decoupled.
- Refreshing `prediction/` based on the diff that materialize produced. Out of scope by design (per the project's "prediction/ is populated only by migrate up" rule).
- Init/scaffolding behavior when `prediction/` does not exist — that's migration 147.

## Edge cases

- **`prediction/` does not exist when materialize runs.** Materialize neither creates nor deletes `prediction/`. If `prediction/` is missing, materialize does not bring it into existence.
- **Materialize fails partway.** No rollback obligation for `prediction/` — it was never written. `implementation/` rollback story is unchanged.
- **Linked-target workspaces.** Materialize already redirects writes to the linked target's `implementation/`. The `prediction/` bucket lives in the source workspace and remains untouched there too.
- **Test mode `--future` / `materialized` modes.** These already do not affect `prediction/` today; verify by inspection and add an assertion in the test if a path could regress.

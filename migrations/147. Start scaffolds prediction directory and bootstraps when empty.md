# 147. Start Scaffolds Prediction Directory And Bootstraps When Empty

## Problem

After migrations 144–146, `prediction/` is a real workspace bucket and `migrate up` populates it. Two project-lifecycle gaps remain:

1. **Fresh `rundown start` does not create `prediction/`.** `start-project.ts` scaffolds `design/`, `specs/`, `migrations/` for a brand-new project. After 144 the bucket exists in config but the directory is not created on disk. Users see config that mentions a `prediction/` directory that does not exist until the first `migrate up`. That is confusing and inconsistent with the other three buckets, which `start` materializes immediately.

2. **No bootstrap when `design/` and `prediction/` are both empty but the workspace is non-empty.** A user can land in a workspace where `implementation/` (or some target) already has code but the rundown surface (`design/`, `prediction/`) is empty. Today nothing helps — `rundown migrate up` has nothing to predict from. The new model needs an explicit "init from existing implementation" path: when `design/` and `prediction/` are both empty (or absent) but the workspace is not (i.e. there are tracked files outside rundown's surface), `rundown start` (or a dedicated init action) seeds:
   - `design/current/Target.md` with a placeholder describing "target = the existing implementation, in domain language; refine this".
   - `prediction/` mirrored from `implementation/` (or the linked target) so the first prediction baseline is exactly the current state.

This migration adds (1) unconditionally and (2) as an opt-in path triggered automatically by `rundown start` when the heuristic fires.

## Model

After this migration:

- `rundown start <description>` always creates `prediction/` (empty) alongside `design/`, `specs/`, `migrations/`. The directory is tracked by the same idempotent scaffolding code path as the other three.
- A new helper `bootstrapFromExistingImplementation({ fileSystem, workspaceRoot, paths })` runs at the end of `start-project.ts` *only* when:
  - `design/` is empty (no files; subdirs treated as empty if they contain no files), AND
  - `prediction/` is empty, AND
  - There exists at least one file under the resolved implementation root (or linked target) that is not under `design/`, `specs/`, `migrations/`, `prediction/`, `.rundown/`, `.git/`, `node_modules/`, build outputs.
- When the heuristic fires:
  - Seed `design/current/Target.md` with a deterministic placeholder containing the user's `<description>` plus a stock note: "Bootstrapped from existing implementation. Replace with target description in domain language; do not list implementation details."
  - Mirror the implementation tree (every file passing the same exclusion list as above) into `prediction/<relative-path>` with byte-for-byte content. This makes the initial prediction baseline equal to the current real state — exactly what the user means by "we start from here".
- When the heuristic does not fire (workspace already has design content, or is fully empty), `start` falls back to today's behavior plus the empty-`prediction/` directory.
- The bootstrap is idempotent: re-running `rundown start` on a workspace where bootstrap already ran is a no-op (since `design/` is no longer empty).
- A CLI flag `--no-bootstrap` skips the bootstrap step even when the heuristic fires (used by tests and by users who want to start with a clean slate).

## Out of scope

- Bootstrapping a non-empty `design/` from existing implementation. Out of scope: bootstrap fires only when `design/` is empty.
- Auto-running `migrate up` immediately after bootstrap. The bootstrap leaves `prediction/` populated; the user can run `rundown migrate up` themselves to author the first migration.
- Smart inference of "domain language" target text from the implementation. The seeded `Target.md` is a placeholder, not a generated summary.
- Linked-target / multi-workspace edge cases beyond what `resolveWorkspacePaths` already handles.
- Removing the legacy `docs/current/Design.md` fallback. That stays untouched here.
- Touching `init-project.ts`. `init` is a different command (configures `.rundown/`); only `start` does workspace scaffolding.

## Edge cases

- **`implementation/` resolves to the workspace root itself (no separate dir).** The exclusion list (`design/`, `specs/`, `migrations/`, `prediction/`, `.rundown/`, `.git/`, `node_modules/`, common build outputs like `dist/`, `coverage/`) prevents the bootstrap from copying its own scaffolding into `prediction/`. Be explicit about the exclusion set in code with a single `BOOTSTRAP_EXCLUDED_DIRS` constant.
- **Linked target.** When a `workspace.link` is configured, the source-of-truth for "existing implementation" is the linked target's tree, not the source workspace. The mirror writes into `prediction/` on the source workspace (where rundown lives). If the linked target is not yet linked, fall back to the source workspace tree.
- **Symlinks / very large trees.** Mirror only regular files. Skip symlinks (or follow once and stop, no recursion). Reasonable cap (e.g. 5000 files / 50MB total) — error out with a clear message if exceeded so the user can intervene.
- **Files with different line endings or encoding.** Copy raw bytes. Do not transcode.
- **Existing `prediction/` content (not empty).** Bootstrap heuristic does not fire; bootstrap is skipped. No deletions.
- **`design/current/` exists but is empty.** Treat as empty for the heuristic.

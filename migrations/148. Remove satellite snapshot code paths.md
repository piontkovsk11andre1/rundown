# 148. Remove Satellite Snapshot Code Paths

## Problem

Migrations 145–147 made the `prediction/` bucket the canonical persisted form of the predicted-implementation tree. The legacy satellite mechanism (per-migration `*.snapshot.md`, `*.context.md`, `*.backlog.md` files in `migrations/`) is now redundant:

- The writer in `migrate up` writes to `prediction/` (145).
- The reader prefers `prediction/` and only falls back to satellites if `prediction/` is empty (145, back-compat).
- `materialize` no longer rewrites prediction artifacts (146).
- Fresh projects start with `prediction/` populated, either empty or bootstrapped (147).

Once 145–147 are materialized and any in-flight workspace has had a `rundown migrate up` to populate `prediction/`, satellite files are no longer read by anyone. They continue to be **written** today (the back-compat write was deliberately preserved in 145), and the **back-compat read fallback** in the reader is also still present. Both are now dead code.

This migration removes:

1. The satellite **writer**: `generateSatellite` calls inside `migrate-task.ts` for kinds that the prediction bucket now owns (`snapshot`, `context`, `backlog`). Satellite writing for any other kinds (if any survive) stays.
2. The satellite **reader fallback**: the `if (predictionDir is empty) read satellites` branch added in 145.
3. The satellite parsing helpers in `migration-parser.ts` and types in `migration-types.ts` that exist solely to support the kinds being removed (e.g. `formatSatelliteFilename` if it has no remaining caller). Helpers used by other features stay.
4. The migration files' satellite files on disk: an `--all` clean step is **out of scope**; users are expected to delete `migrations/*.snapshot.md` etc. by hand or via a one-shot script. Document the manual step in this migration's notes; do not automate destructive cleanup.

## Model

After this migration:

- `migrate-task.ts` no longer calls `generateSatellite` for `snapshot`/`context`/`backlog` kinds. The `for (const satellite of migration.satellites)` loop (around line ~503) is removed; tracked files are read exclusively from `prediction/`.
- `prediction-reconciliation.ts` no longer exposes the satellite-input branch. `PredictionInputs` no longer contains satellite-derived fields.
- `migration-parser.ts` exports `formatSatelliteFilename` only if other callers exist (search before deleting). The `isSatellite` parsed flag remains if migration-name parsing still needs it for some other reason; otherwise it is removed.
- `SatelliteType` in `migration-types.ts` is narrowed: `"snapshot" | "context" | "backlog"` members are removed; remaining members (if any) stay. If no members remain, the type and its file are deleted.
- `PredictionTrackedFileKind` is collapsed: `"snapshot" | "context" | "backlog" | "executed"` becomes just `"file"` (or stays multi-kind only if a meaningful semantic distinction remains for `executed`). Tracked files in the new model are uniformly "files in `prediction/`".
- All tests asserting satellite behavior are deleted or rewritten against the prediction-bucket model.
- README / inline docs referencing satellites are updated or removed.
- Existing satellite files on disk remain untouched. Users who want to reclaim disk run `git rm migrations/*.snapshot.md migrations/*.context.md migrations/*.backlog.md` themselves.

## Out of scope

- Deleting on-disk satellite files (`migrations/*.snapshot.md` etc.). Document the manual `git rm` step in this migration's commit message; do not automate.
- Cleaning up satellite mentions in old migration markdown files under `migrations/N. *.md`. Those are historical record.
- CLI help text changes for the `migrate` / `materialize` commands. Migration 149 owns prompt and help-text cleanup.
- Changes to `design/rev.N/` snapshot logic. That is an entirely separate concept and stays.

## Edge cases

- **A user runs `migrate up` after this migration on a workspace that still has only satellites.** The fallback is gone, so `migrate up` reads an empty `prediction/`, treats it as an empty baseline, and produces a fresh prediction. The user must run `migrate up` once to repopulate `prediction/` — that's the upgrade path. Document it.
- **Stale satellite files on disk after this migration.** Inert. They are markdown files under `migrations/`; nothing reads them. The migration parser may or may not stumble on them depending on whether the satellite-name pattern still gets recognized. Verify the parser ignores unrecognized markdown filenames gracefully — if not, add a permissive skip for any `*.snapshot.md` / `*.context.md` / `*.backlog.md` filename so the migration list is not corrupted by leftover files.
- **A removed type (e.g. `SatelliteType`) is referenced from a file we missed.** TypeScript compile error during `npm run lint` will catch it. Fix at the report site.

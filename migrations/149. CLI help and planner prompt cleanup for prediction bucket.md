# 149. CLI Help And Planner Prompt Cleanup For Prediction Bucket

## Problem

Migrations 143–148 reshape the internal model: the `prediction/` bucket is canonical, satellites are gone, materialize no longer touches prediction. But user-visible copy still reflects the old model:

- `cli.ts` `start` description says "Scaffold a new prediction project with a design/current/Target.md workspace."  No mention of `prediction/`.
- `cli.ts` `migrate up` / `materialize` descriptions and the multi-line help notes (around lines 388–390, 438) talk about "design snapshots" interchangeably with the (now-removed) satellite snapshots, which was the original source of confusion.
- Planner prompts and migration-snapshot prompts (under `implementation/src/infrastructure/templates/...` or wherever they live) reference satellite filenames (`*.snapshot.md`) and the "predicted state from the previous migration's satellites". After 145–148 these phrases are wrong.
- The `runtime-workspace-context.ts` template variables `workspacePredictionDir` / `workspacePredictionPath` (added in 144) are exposed but no template uses them. Templates that previously hardcoded `migrations/<n>.snapshot.md` should reference `{{workspacePredictionPath}}/<relative>` (or read from a richer prediction-tree variable if introduced).

This migration is a copy / template-only pass. No control flow changes. No new types. Update the user-visible strings and prompt templates so they describe the new model accurately.

## Model

After this migration:

- `cli.ts` command descriptions and help text read consistently with the new model:
  - `start` mentions `design/`, `prediction/`, `migrations/` as the four primary workspace tracks (with `specs/` if material).
  - `migrate up` description mentions that it produces the `prediction/` tree.
  - `materialize` description mentions it writes only `implementation/` and that `prediction/` is refreshed by `migrate up` (already added in 146 — verify wording is consistent).
  - The "Historical snapshots ... design/rev.N/" verbiage is preserved (those are *design revision* snapshots, a separate concept).
  - Any line that says "snapshot" without "design revision" qualifier is reviewed: if it referred to the satellite (now-removed), it is rewritten or deleted.
- Planner prompt templates that mentioned satellite files are rewritten to reference `prediction/<relative>` or the new template variables.
- Migration-snapshot prompt templates (used to author the predicted state) consume the same prediction tree as input — wording reflects this.
- A new entry in `WORKSPACE_CONTEXT_TEMPLATE_VAR_KEYS` (if not added in 144) for `workspacePredictionTree` (if useful) is **out of scope** unless a template clearly demands it; the existing `workspacePredictionDir` / `workspacePredictionPath` from 144 should suffice.

## Implementation boundaries

Files expected to change:

- `implementation/src/presentation/cli.ts`
  - Line ~357: rewrite `start` description to mention `design/`, `prediction/`, `migrations/` workspace tracks. Drop "Scaffold a new prediction project" (the word "prediction" there refers to the old project taxonomy, not the new bucket — confusing).
  - Lines ~388–390: review the multi-line help block. The phrase "Historical snapshots are stored under design/rev.N/ as immutable revisions" stays (design rev story). If any sibling line refers to the satellite-style "predicted snapshot", rewrite or remove it.
  - Line ~438: similar review.
  - `migrate up` command: add a one-line `Writes the predicted-implementation tree into prediction/.` to the description.
  - `materialize` command: verify the line added in 146 is present and consistent.

- Planner prompt templates (under `implementation/src/infrastructure/templates/` — locate via grep on `satellite`, `snapshot`, `*.snapshot.md`):
  - For each template that referenced satellites, rewrite the prose to reference `prediction/`. Use the template variables `{{workspacePredictionDir}}` / `{{workspacePredictionPath}}` where the path needs to be substituted.
  - Where the template instructed an agent to "read the previous migration's satellite", rewrite to "read the current prediction tree at `{{workspacePredictionPath}}`".
  - Where the template instructed the agent to "write a snapshot satellite", rewrite to "produce the predicted file content for the migration; rundown will persist it into `prediction/` automatically".

- `implementation/src/application/migrate-task.ts`
  - Inside template-var construction (around line ~1767, identified in migration 144), confirm `workspacePredictionDir` and `workspacePredictionPath` are passed to planner / snapshot prompts. If not, add them.

- Tests
  - `implementation/__tests__/presentation/cli.test.ts` (or equivalent help-text test): update assertions on the rewritten descriptions.
  - Snapshot/golden tests on planner prompts must be regenerated / updated to reflect the new wording. Each prompt template should have at least one test asserting the new copy contains "prediction/" and does not contain "satellite" / `*.snapshot.md`.

## Out of scope

- Functional changes to planner / snapshot agents. Same inputs (prediction tree + migration TODO), same outputs (predicted file contents). Only wording changes.
- New template variables beyond what 144 already added.
- Translation of the new strings into other locales (`localize-project.ts`). The localization migrations (134–135) handle the locale story; this migration only updates English source.
- README / external docs updates beyond what 148 already touched.
- Any change to design-revision (`design/rev.N/`) language. Those snapshots stay as a separate concept with their existing wording.

## Edge cases

- **A template references both the old and new model.** Rewrite to the new model only; the old model no longer exists after 148.
- **A help string is constructed from multiple template literals.** Update each piece; do a final read-through to ensure the assembled help reads naturally.
- **Localized strings in `localize-project.ts` defaults.** English fallbacks in `DEFAULT_*_TEMPLATE` constants are updated here. Translated copies are out of scope.
- **Tests assert exact help-string equality.** Update the expected strings; do not relax assertions to substring match.

I want to add 'make' command that will actually do that:

rundown make 'please do something' '8. Do something.md'

Create file 8. Do something.md, save 'please do something' there, then run `research` on this file, after that `plan` on the same file.

## Summary

Add a new top-level CLI command that automates the authoring bootstrap flow:

1. create a Markdown file from a short prompt,
2. run `research` on that file,
3. run `plan` on that same file.

This is a composition command that should reuse existing `research` and `plan` behavior rather than introducing a parallel implementation of those phases.

## Why this exists

The current flow works, but it is manual and easy to do inconsistently:

- create a thin file,
- run `research`,
- then run `plan`.

`make` should be the one-step entry point for this exact flow so users can go from idea text to an enriched, actionable TODO document with one command.

## Target command shape

```bash
rundown make "<seed text>" "<markdown-file>"
```

Example:

```bash
rundown make "please do something" "8. Do something.md"
```

Expected high-level behavior:

- Create `8. Do something.md`.
- Write `please do something` into that file as initial document body.
- Execute `rundown research 8. Do something.md ...`.
- If research succeeds, execute `rundown plan 8. Do something.md ...`.
- Return success only if the full chain succeeds.

## Existing code and architecture context

The current CLI is organized with clear command-level wiring in `src/presentation/cli.ts` and action handlers in `src/presentation/cli-command-actions.ts`.

Relevant existing pieces:

- `research` and `plan` are already first-class commands in `src/presentation/cli.ts`.
- Option normalization and validation utilities are centralized in `src/presentation/cli-options.ts`.
- `research` currently requires exactly one existing Markdown file path and rejects directory/glob input.
- `plan` requires exactly one Markdown file path and has scan behavior (`--scan-count`, default `3`).
- The app boundary (`src/create-app.ts`) exposes `researchTask` and `planTask`.

This means `make` can be implemented as orchestration on top of mature command flows rather than new domain logic for research/planning itself.

## Proposed behavior details

### File creation phase

- The second argument must be `.md` or `.markdown`.
- The file should be created with the first argument as contents.
- If parent directories are missing, fail with a clear error (or explicitly define `mkdir -p` behavior if desired, but it must be consistent).
- Prefer safe behavior on collisions: do not silently overwrite existing files unless an explicit overwrite option is later introduced.

### Research phase

- Run against the newly created file.
- Reuse current research validations and guardrails:
  - checkbox state must not be changed by research output,
  - new unchecked TODO items are not allowed in research output,
  - single-pass behavior remains unchanged.

### Plan phase

- Run only after successful research completion.
- Reuse existing plan scan/convergence behavior and defaults.
- Preserve current default output visibility behavior (agent output hidden unless explicitly enabled).

## Option and forwarding strategy

For v1, `make` should forward shared worker/runtime options to both sub-commands where meaningful, especially:

- `--worker`
- `--transport`
- `--keep-artifacts`
- `--show-agent-output`
- `--trace`
- `--vars-file`
- `--var`
- `--ignore-cli-block`
- `--cli-block-timeout`
- `--force-unlock`
- `--config-dir` (global option already exists)

Mode handling needs explicit rule:

- `plan` only supports `wait`.
- `research` supports `wait` and `tui`.
- `make` should default to deterministic automation (`wait`) and reject/avoid combinations that break non-interactive chaining.

## Failure semantics

`make` is sequential and fail-fast:

- If file creation fails: stop immediately.
- If `research` fails: do not run `plan`.
- If `plan` fails: return non-zero after preserving research output and any generated artifacts.
- Exit code should align with existing command conventions (execution/config errors as `1`, validation-specific failures where existing subcommand behavior returns `2` or `3`).

## Locking and concurrency notes

`research` and `plan` each already acquire and release source locks. Chaining them in one command should preserve that behavior. No new lock protocol is required, but sequence correctness matters:

- run `research` to completion,
- then run `plan`.

No concurrent execution between those two phases.

## Integration notes

Two implementation directions fit current structure:

1. Add a dedicated `make` use case at the app boundary and call existing `researchTask`/`planTask` from there.
2. Add a CLI-level composition action that performs file write + delegates into existing app calls.

Either is valid, but reusing existing action/use-case contracts is preferable to duplicating prompt/render/worker logic.

There is also related context in `src/infrastructure/inline-rundown.ts`, which currently delegates inline `rundown:` tasks to nested `rundown run`. If future scope includes inline `rundown: make ...`, delegation may need to support subcommands beyond `run`.

## Scope boundaries

In scope for this change:

- top-level `rundown make` command,
- file bootstrap from seed text,
- automatic research then plan on the same file,
- docs/help updates for the new command,
- tests covering happy path and fail-fast behavior.

Out of scope for this iteration:

- multi-file make operations,
- custom make pipelines beyond `create -> research -> plan`,
- implicit overwrite of existing files without explicit option,
- changing research/plan core semantics.

## Edge cases to account for

- Seed text containing quotes/newlines/special shell characters.
- Relative vs absolute file paths.
- Target file already exists.
- Target path has non-Markdown extension.
- Target directory path passed instead of file path.
- Missing worker command when config does not resolve one.
- Research succeeds but plan fails (partial-chain state remains understandable and recoverable).

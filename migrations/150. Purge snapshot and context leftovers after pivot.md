# 150. Purge Snapshot And Context Leftovers After Pivot

## Problem

Migrations 143–149 pivoted the predicted-implementation persistence from per-migration satellite files (`*.snapshot.md`, `*.context.md`, `*.backlog.md`) to the `prediction/` workspace bucket. The pivot was deliberately not back-compat: there is no upgrade path from old workspaces, no fallback reader, no dual-write. Yet several leftover surfaces from the old model survived 148/149:

1. **`PredictionReconciliationEntryPoint` carries dead fields.** [implementation/src/domain/prediction-reconciliation.ts](implementation/src/domain/prediction-reconciliation.ts#L62-L67) still declares `latestExecutedMigration`, `latestContext`, `latestSnapshot`, `latestBacklog` — and the constructor at line ~304 hard-codes `latestContext: null` and `latestBacklog: null` because the old satellite readers are gone. The `latestSnapshot` field is still populated, but populated by `findLatestFileAtOrBefore` matching `*.snapshot.md` files inside the migrations tree (`isSnapshotTrackedFile` regex `\.(?:snapshot|review)\.md$`) — exactly the legacy satellite pattern that 148 said was inert.

2. **`buildPendingReResolutionPrompt` emits dead sections.** [implementation/src/domain/prediction-reconciliation.ts](implementation/src/domain/prediction-reconciliation.ts#L554-L582) renders "Latest context", "Latest snapshot", "Latest backlog" sections that are always `(none)` (or, for snapshot, picked from legacy `.snapshot.md` files that should not exist). The prompt also instructs the agent to "use the latest executed migration/context/snapshot/backlog as the source of truth" — wrong contract under the new model.

3. **`ReResolvedPredictionItem.snapshotContent`** and the JSON `snapshot` output field at [prediction-reconciliation.ts](implementation/src/domain/prediction-reconciliation.ts#L73), [#L622](implementation/src/domain/prediction-reconciliation.ts#L622), [#L632](implementation/src/domain/prediction-reconciliation.ts#L632). The reconciler still asks the worker to emit `<snapshot>` content per pending migration and writes those into `<n>.1 Snapshot.md` satellite files alongside the migration markdown. This contradicts 148 (no satellite writes) and 145 (predicted code lives in `prediction/`).

4. **`isSnapshotTrackedFile`, `formatSnapshotFilename`, `LEGACY_SUFFIX_PATTERN`, `ReviewFilenameType = "snapshot"`** in [prediction-reconciliation.ts](implementation/src/domain/prediction-reconciliation.ts#L881) and [migration-parser.ts](implementation/src/domain/migration-parser.ts#L4-L23). All exist solely to recognise or generate satellite filenames. The `LEGACY_SUFFIX_PATTERN` was retained in 148 as "ignore legacy filenames gracefully"; the user has confirmed there is no back-compat mode and stale on-disk satellite files are out of scope here, so the pattern can go.

5. **`MigrationState.latestSnapshot`** in [migration-types.ts](implementation/src/domain/migration-types.ts#L22) is set to `null` everywhere by [migration-parser.ts](implementation/src/domain/migration-parser.ts#L186), then read by [migrate-task.ts](implementation/src/application/migrate-task.ts#L1633) and [#L1678](implementation/src/application/migrate-task.ts#L1678) where its content gets fed into the planner template variable `{{latestSnapshot}}`. With the field always null, the var is always empty — dead plumbing.

6. **`{{latestSnapshot}}`, `{{latestContext}}`, `{{backlog}}` template variables** are still consumed by three default templates in [defaults.ts](implementation/src/domain/defaults.ts#L730), [#L775](implementation/src/domain/defaults.ts#L775), [#L842](implementation/src/domain/defaults.ts#L842), [#L846](implementation/src/domain/defaults.ts#L846) (the migrate planner, the migrate-snapshot batch checkpoint, and the test verify template). They render empty strings or pull from the legacy field. The prompts mislead the agent about what inputs are available.

7. **`DEFAULT_MIGRATE_SNAPSHOT_TEMPLATE` and the `migrate-snapshot.md` template file** in [defaults.ts](implementation/src/domain/defaults.ts#L756), [project-templates.ts](implementation/src/application/project-templates.ts#L72), [init-project.ts](implementation/src/application/init-project.ts#L8), [localize-project.ts](implementation/src/application/localize-project.ts#L10). The whole "batch-end checkpoint snapshot" flow is the old satellite-snapshot mechanism in template form. Under the new model the predicted state *is* the `prediction/` tree; there is no separate snapshot document to render.

8. **`TestContext.latestContext` / `TestContext.latestSnapshot` and `latestContextPath`** in [test-specs.ts](implementation/src/application/test-specs.ts#L49-L50), [#L238-L239](implementation/src/application/test-specs.ts#L238-L239), [#L426](implementation/src/application/test-specs.ts#L426), [#L528](implementation/src/application/test-specs.ts#L528), [#L552](implementation/src/application/test-specs.ts#L552). The future-mode test path still tries to read a `latestContextPath` (always `null`) and feeds the empty string into the verify template's `{{latestContext}}` / `{{latestSnapshot}}` slots.

9. **Tests still asserting dead fields:** [migration-parser.test.ts](implementation/__tests__/domain/migration-parser.test.ts#L55), [migration-adapter.test.ts](implementation/__tests__/infrastructure/adapters/migration-adapter.test.ts#L62), [project-templates.test.ts](implementation/__tests__/application/project-templates.test.ts#L59), [migrate-task.integration.test.ts](implementation/__tests__/integration/migrate-task.integration.test.ts#L1530).

The pivot was a hard cut. This migration finishes it: every surface tied to the old satellite/snapshot/context/backlog persistence is removed, with no replacement legacy adapter and no rename. After this migration, the codebase mentions "snapshot" only where it refers to genuinely separate concepts (worker health snapshots, design-revision `design/rev.N/` snapshots) — never the deleted satellite mechanism.

## Model

After this migration:

- `PredictionReconciliationEntryPoint` declares only fields the new model needs:
  - `lastCompletedMigrationNumber`
  - `reconciliationStartMigrationNumber`
  - `preservedCompletedMigrationNumbers`
  - `pendingPredictionMigrationNumbers`
  - `latestExecutedMigration` (still meaningful: the migration markdown of the last applied migration)
  - **No** `latestContext`, `latestSnapshot`, `latestBacklog`.
- `ReResolvedPredictionItem` declares:
  - `migrationNumber`
  - `migrationName`
  - `migrationContent`
  - **No** `snapshotContent`. The reconciler writes only the migration markdown; predicted file content lives in `prediction/` and is owned by `migrate up`'s prediction-tree writer added in 145.
- `buildPendingReResolutionPrompt` emits no "Latest context", "Latest snapshot", "Latest backlog" sections. It exposes the design, the latest executed migration, the current pending migration drafts, and the instruction to read `{{workspacePredictionPath}}` directly when the agent needs the current predicted tree.
- The reconciler's atomic patch (`PendingPredictionAtomicPatch.writeFiles`) contains only migration markdown files. No `<n>.1 Snapshot.md` writes.
- `isSnapshotTrackedFile`, `formatSnapshotFilename`, `LEGACY_SUFFIX_PATTERN`, `MigrationReviewType = "snapshot"`, the dotted aux index/label entries for `snapshot`, `RECOGNIZED_AUX_TYPES.snapshot`, and the entire snapshot branch of `getReviewTypeFromLabel` / `getReviewTypeFromDottedIndex` are removed. `MigrationReviewType` collapses to `"review"` only.
- `MigrationState` has no `latestSnapshot` field. `parseMigrationDirectory` no longer emits one.
- `migrate-task.ts` no longer reads `state.latestSnapshot`, no longer constructs the planner variable `latestSnapshot`. The variable is removed from every call site.
- `defaults.ts`:
  - `DEFAULT_MIGRATE_SNAPSHOT_TEMPLATE` removed.
  - `{{latestSnapshot}}` / `{{latestContext}}` / `{{backlog}}` placeholders removed from `DEFAULT_MIGRATE_TEMPLATE` and `DEFAULT_TEST_VERIFY_TEMPLATE`. The "Current prediction tree" / "Latest context" / "Backlog" sections are removed; agents are pointed to `{{workspacePredictionPath}}` and asked to read it directly.
- `project-templates.ts`:
  - `migrateSnapshot: string` field removed from `ProjectTemplates`.
  - `loadTemplateWithFallback("migrate-snapshot.md", ...)` call removed.
- `init-project.ts`, `localize-project.ts`:
  - `migrate-snapshot.md` is not written and not in the localized template list.
- `test-specs.ts`:
  - `TestContext` has no `latestContext` / `latestSnapshot`.
  - `FutureScope` has no `latestContextPath`. The future-mode builder no longer attempts to find one.
  - The verify-template `TemplateVars` literal omits `latestContext` / `latestSnapshot`.
- Tests:
  - `migration-parser.test.ts` and `migration-adapter.test.ts`: assertions on `state.latestSnapshot` removed (the field no longer exists).
  - `project-templates.test.ts`: the `{{latestSnapshot}}` placeholder is removed from the asserted default template body.
  - `migrate-task.integration.test.ts`: the test that writes `migrate.md` containing `{{design}}\n{{latestSnapshot}}\n{{backlog}}\n{{migrationHistory}}\n` is rewritten to use only the surviving placeholders. If the test was specifically asserting on the (now removed) batch-snapshot flow, it is deleted.
- After this migration, a workspace search across `implementation/src/**` and `implementation/__tests__/**` for the regex `latestSnapshot|latestContext|latestBacklog|MIGRATE_SNAPSHOT|migrate-snapshot|formatSnapshotFilename|isSnapshotTrackedFile|snapshotContent|\\.snapshot\\.md|\\.context\\.md|\\.backlog\\.md|LEGACY_SUFFIX_PATTERN` returns **zero hits**.

## Out of scope

- Renaming the surviving `MigrationReview` / `MigrationReviewType = "review"` concept. Reviews are a separate, still-supported feature.
- Removing the design-revision `design/rev.N/` snapshot mechanism. That is a different concept from the deleted satellite snapshots; per migration 148/149 it stays.
- Removing worker-health "snapshot" terminology in `worker-health-store.ts`. Different concept.
- Deleting on-disk `*.snapshot.md`, `*.context.md`, `*.backlog.md` files in real workspaces. Users handle that themselves; this migration just stops the parser from special-casing them.
- Adding new template variables or new sections to `DEFAULT_MIGRATE_TEMPLATE` / `DEFAULT_TEST_VERIFY_TEMPLATE` beyond the one-line "read the prediction tree directly" instruction. Prompt redesign is a separate effort.
- Localization of the modified default templates into other locales. The English source is updated here; locale propagation follows the existing localization workflow.

## Edge cases

- **A workspace's user-customised `migrate.md` still contains `{{latestSnapshot}}` / `{{backlog}}`.** The renderer leaves unknown placeholders as-is or substitutes empty strings depending on the template engine's default. Verify the existing behavior (search for the renderer's handling of unknown vars in `template.ts` / `renderTemplate`) and document it inline. Do not add a deprecation warning — the pivot is a hard cut.
- **A test fixture writes a template containing `{{latestSnapshot}}` to disk.** Update the fixture; do not preserve the placeholder for back-compat.
- **`MigrationReviewType` collapses to a single member.** TypeScript treats `type Foo = "review"` as a literal type. The existing usages compile unchanged. Verify by lint.
- **A consumer relied on `entryPoint.latestExecutedMigration` being non-null.** Behavior unchanged: that field is preserved.
- **The reconciliation prompt now lacks a "previous predicted state" section.** Acceptable: the prompt instructs the agent to read `{{workspacePredictionPath}}` directly. Worker filesystem access is the supported channel for that information under the new model.

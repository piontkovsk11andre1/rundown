# 151. Remove Migrate Up Command And Prediction Bucket Writer

## Problem

The user has decided that *prediction* will live in a separate project that consumes rundown as an API. Rundown itself stops trying to materialise predicted state. The first concrete step is to delete `rundown migrate up` and everything it writes:

- The `up` action of the `migrate` command, including the inline `runMigrateUp` calls that the convergence loop triggers after each newly-authored migration ([implementation/src/application/migrate-task.ts](implementation/src/application/migrate-task.ts#L269), [#L800](implementation/src/application/migrate-task.ts#L800), [#L896](implementation/src/application/migrate-task.ts#L896)).
- The `runMigrateUp` function ([migrate-task.ts](implementation/src/application/migrate-task.ts#L924)) and its prediction-tree writer (`writePredictionTree`, `persistPredictionBaselineSnapshot`, [#L339](implementation/src/application/migrate-task.ts#L339)–[#L420](implementation/src/application/migrate-task.ts#L420)).
- The `writePredictionBucket` option that plumbs through `MigrateCommandOptions`, `migrateTask`, the materialize callsite ([cli-command-actions.ts](implementation/src/presentation/cli-command-actions.ts#L578)–[#L582](implementation/src/presentation/cli-command-actions.ts#L582)), and downstream call chains.
- The CLI's `up` action wiring in [cli.ts](implementation/src/presentation/cli.ts).

After this migration `rundown migrate` only runs the planner convergence loop (author next migration file → loop until `DONE`). It never executes pending migrations, never reads or writes the `prediction/` workspace bucket, never invokes `prediction-reconciliation` to fill predicted files.

`migrate down` and `test --future` still use `prediction-reconciliation` and `prediction-workspace-paths`, so those domain modules survive this migration. They are deleted by 153 once their last consumers are gone.

## Model

After this migration:

- `MigrateAction` no longer accepts `"up"`. The action enum collapses to `"down"` (until 152 removes that too) plus the no-action default. Calling `rundown migrate up` errors with "unknown action: up".
- `MigrateCommandOptions` and the underlying `migrateTask` no longer expose `writePredictionBucket`. The flag is gone everywhere.
- The convergence loop body, after authoring a new migration file, returns to the planner immediately. There is no embedded "now run that migration" step.
- `runMigrateUp`, `writePredictionTree`, `persistPredictionBaselineSnapshot` are deleted. Helpers exclusive to those functions (e.g. atomic-patch application paths that were only invoked by `up`) are deleted along with them.
- `materialize` no longer needs to pass `writePredictionBucket: false` to `migrateTask`. The `materialize` action stays exactly as before in observable behaviour.
- `prediction-reconciliation.ts`'s `createPredictionReconciliationEntryPoint` and `reReconcilePendingPredictionSequence` are still exported and still called by `migrate down` and `test --future` — they are not removed here.
- The `workspacePredictionPath` runtime variable still exists ([runtime-workspace-context.ts](implementation/src/application/runtime-workspace-context.ts#L121)) because the `start` command and other commands reference it; only the *writer* and the `migrate up` consumer go.
- After this migration, `rundown migrate up` must produce a clear "unknown action" error (Commander's default behaviour), and `rundown materialize <source>` continues to work unchanged.

## Out of scope

- Deleting `prediction-reconciliation.ts` or `prediction-workspace-paths.ts`. Their last consumer (`test --future`) goes away in 153.
- Removing `migrate down` or `Backlog.md` handling. That's 152.
- Removing `test --future`. That's 153.
- Reshaping `design/current/prediction/`. That's 154.
- Adding any deprecation message or back-compat shim. The pivot is a hard cut.
- Cleaning up an existing project's `prediction/` workspace bucket on disk. Users delete it themselves; rundown simply stops touching it.
- Changing `runtime-workspace-context.ts`'s `workspacePredictionPath` export. Other commands and templates still reference the path; only the writer goes here.

## Edge cases

- **A user invokes `rundown migrate up`.** Commander emits its standard "unknown action" error. No graceful fallback.
- **A user invokes `rundown materialize <source>`.** Unchanged: `materialize` is still equivalent to `run --all --revertable`. The internal call to `migrateTask` no longer threads `writePredictionBucket: false`, but `migrateTask` no longer accepts it either.
- **The convergence loop authors a migration that itself contains a `migrate up`-style instruction.** The inner step that previously ran is gone; the planner is responsible for adapting its prompt (covered by template updates downstream — out of scope here, just verify no crash).
- **A workspace already has a `prediction/` bucket from a previous `migrate up`.** Rundown ignores it. `rundown migrate` does not read it, write it, or prune it. The new prediction project will own that bucket.
- **A planner template still references `{{workspacePredictionPath}}`.** The variable still resolves (the runtime context still produces it). The directory may be empty or stale — that's fine; the agent is expected to read it directly when present.

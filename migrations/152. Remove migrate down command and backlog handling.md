# 152. Remove Migrate Down Command And Backlog Handling

## Problem

With `migrate up` removed by 151, the `down` action and the `Backlog.md` singleton it maintains are the next prediction-only surfaces to retire. They exist solely to undo predicted-state effects:

- `migrate down [n]` rewinds the migration tail by `n` migrations or to a target revision boundary, regenerates the now-current predicted snapshot, and (by default) appends rewound items to `migrations/Backlog.md`.
- `--no-backlog`, `--to <revName>`, and the optional `[count]` argument are exclusive to `down`.
- `appendMigrationsToBacklog`, `removePromotedBacklogItems`, `stripBacklogEntries`, `parseBacklogEntryName`, `extractBacklogEntryFencedContent`, `normalizeBacklogEntryContent`, and the `MigrationState.backlogPath` / `backlog` reads exist solely to support backlog promotion/demotion across `migrate down` ⇄ `migrate` (create) cycles.

Per the user's decision, the new prediction project will own any rewind UX. Users of the trimmed rundown delete migration files manually (e.g. `git revert`, `rm`).

After this migration the only `migrate` action is the no-action planner convergence loop. The CLI's argv-munging special case for `migrate down` ([cli-argv.ts](implementation/src/presentation/cli-argv.ts#L65)–[#L95](implementation/src/presentation/cli-argv.ts#L95)) is gone, the `--no-backlog` and `--to <revName>` options are gone, and `Backlog.md` is no longer read or written by rundown.

`prediction-reconciliation.ts` and `prediction-workspace-paths.ts` survive this migration because `test --future` still uses them. They are deleted by 153.

## Model

After this migration:

- `MigrateAction` collapses to a single member representing the no-action default. Calling `rundown migrate down` errors as an unknown action.
- `MigrateCommandOptions` no longer exposes `noBacklog`, `count`, or `targetRevisionName` (or whatever the `--to` option is parsed into).
- `runMigrateDown` is deleted. The convergence-loop callsites that used to remove promoted backlog items via `removePromotedBacklogItems` are deleted along with the helpers.
- `MigrationState` no longer carries `backlogPath` or `backlog`. `parseMigrationDirectory` no longer scans for `migrations/Backlog.md`.
- The planner template variable `{{backlog}}`, if any survives in default templates after 150, is removed from every default template and every consumer site. (150 removed it from `DEFAULT_MIGRATE_TEMPLATE`; verify no other template re-introduced it and no test still injects it.)
- The CLI no longer exposes `--no-backlog` or `--to <revName>`. They are removed from the `migrate` command's option list in [cli.ts](implementation/src/presentation/cli.ts#L416)–[#L417](implementation/src/presentation/cli.ts#L417).
- The argv munger that rewrites `migrate down [n]` into a normalised form ([cli-argv.ts](implementation/src/presentation/cli-argv.ts#L65)–[#L95](implementation/src/presentation/cli-argv.ts#L95)) is deleted.
- `rundown migrate down` returns Commander's "unknown action" error.
- An existing `migrations/Backlog.md` on disk is ignored: never read, never written, never pruned.
- Tests that exercised down-flow behaviour, backlog promotion, backlog rollback, `--no-backlog`, and revision-bounded down are deleted. Tests covering the planner convergence loop (still active) stay.

## Out of scope

- Deleting `prediction-reconciliation.ts` or `prediction-workspace-paths.ts`. Their last user (`test --future`) goes away in 153.
- Removing `test --future`. That's 153.
- Reshaping `design/current/prediction/`. That's 154.
- Deleting an existing project's `migrations/Backlog.md`. Users handle that themselves.
- Adding a top-level rewind/undo command to replace `migrate down`. The new prediction project will own that.
- Changing `MigrationReview` (the still-supported `*.review.md` satellite). Reviews are unrelated to backlog/down.

## Edge cases

- **A workspace has a `migrations/Backlog.md` from a previous run.** Rundown does not read or write it. `rundown migrate` (planner loop) does not see it as a migration file because the parser already filtered named singletons.
- **A user-customised `migrate.md` template still references `{{backlog}}`.** Renderer leaves the unknown placeholder as-is or substitutes empty. Same hard-cut policy as 150 — no deprecation warning.
- **A user invokes `rundown migrate down 3`.** Commander returns an "unknown action" error.
- **A user invokes `rundown migrate --to rev.5`.** Commander returns an "unknown option" error.
- **A planner-loop test that fed a synthetic `MigrationState` with `backlog` set.** Update the fixture; the field no longer exists.

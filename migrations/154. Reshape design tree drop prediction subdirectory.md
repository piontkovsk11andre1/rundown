# 154. Reshape Design Tree Drop Prediction Subdirectory

## Problem

The design specification at [design/current/prediction/](design/current/prediction/) (and its mirror at [prediction/prediction/](prediction/prediction/) inside the predicted-implementation bucket) describes a workflow that no longer exists in rundown after 151–153:

- [design-revisions.md](design/current/prediction/design-revisions.md) — still relevant (design release semantics).
- [migrations.md](design/current/prediction/migrations.md) — describes the planner convergence loop **and** `migrate up`/`migrate down`/snapshot writing.
- [snapshots-and-backlog.md](design/current/prediction/snapshots-and-backlog.md) — entirely about removed concepts.
- [test-modes.md](design/current/prediction/test-modes.md) — describes both materialised and future modes; future mode is gone.
- [materialize.md](design/current/prediction/materialize.md) — still relevant; not actually a prediction concern.
- [undo-and-revert.md](design/current/prediction/undo-and-revert.md) — still relevant; not actually a prediction concern.

The `prediction/` design subdirectory was always a category mismatch: it bundled real lifecycle commands (materialize, undo, revert) with prediction-specific docs. With prediction moving to a separate project, the subdirectory must go and the surviving docs need a coherent home.

This migration also updates the CLI command-surface docs ([design/current/cli/prediction-commands.md](design/current/cli/prediction-commands.md)) to drop `migrate up`, `migrate down`, `--no-backlog`, `--to`, `--future`, and the snapshot/backlog plumbing, and reshapes them into a "lifecycle commands" section.

## Model

After this migration:

- `design/current/prediction/` does not exist.
- A new `design/current/lifecycle/` subdirectory hosts the surviving lifecycle docs:
  - `lifecycle/README.md` — index.
  - `lifecycle/design-revisions.md` — verbatim move from `prediction/design-revisions.md` (release semantics, baseline rules, file organisation note).
  - `lifecycle/migrate.md` — rewritten from `prediction/migrations.md`. Describes only the planner convergence loop (no `up`, no `down`, no snapshots, no backlog). Mentions that pending migrations are intent records consumed by external tooling (the new prediction project) and by humans reading history.
  - `lifecycle/materialize.md` — verbatim move from `prediction/materialize.md` (already prediction-free in content).
  - `lifecycle/undo-and-revert.md` — verbatim move from `prediction/undo-and-revert.md`.
  - `lifecycle/test.md` — rewritten from `prediction/test-modes.md`. Materialised mode only; one source argument; no `--future`.
- `design/current/cli/prediction-commands.md` is renamed to `design/current/cli/lifecycle-commands.md` and updated:
  - `migrate` describes only the no-action planner loop; the `up`/`down` rows of the action table are deleted.
  - `test` table loses the `--future` and `--future <n>` rows.
  - The "memory-related operations" cross-reference and the "design release"/"design diff" sections stay as-is.
  - Cross-references that pointed into `../prediction/...` retarget to `../lifecycle/...`.
- `design/current/README.md` index table:
  - The `[prediction/](prediction/README.md)` row is replaced with `[lifecycle/](lifecycle/README.md)` — "design release, migrate (planner loop), materialize, undo/revert, test".
  - Any other table row that referenced prediction is rewritten or dropped.
- `design/current/cli/README.md` index updates `prediction-commands.md` → `lifecycle-commands.md`.
- A short note is added to `design/current/overview/README.md` (or the most-appropriate overview file) recording that prediction has moved to a separate project; this is the only place that mentions the pivot.
- Cross-references throughout `design/current/**` that link to `prediction/...` are updated to `lifecycle/...` or removed if the target was a deleted file (`snapshots-and-backlog.md`, the future-mode portion of `test-modes.md`).
- `prediction/prediction/` (the predicted-implementation mirror) is left untouched. After 151 removes `migrate up`, that mirror is no longer regenerated; it becomes stale and is ignored. Cleanup of the predicted-implementation bucket itself is out of scope (a one-shot delete the user can do manually).

## Implementation boundaries

Files expected to change:

- `design/current/prediction/` — entire subdirectory deleted (all six files plus the `README.md`).
- `design/current/lifecycle/` — new subdirectory created with:
  - `README.md` (new index file).
  - `design-revisions.md` (move).
  - `migrate.md` (new, derived from old `migrations.md`).
  - `materialize.md` (move).
  - `undo-and-revert.md` (move).
  - `test.md` (new, derived from old `test-modes.md`).
- `design/current/README.md` — index row update.
- `design/current/cli/prediction-commands.md` — renamed to `design/current/cli/lifecycle-commands.md`, content updated to drop removed actions/options.
- `design/current/cli/README.md` — index entry update.
- `design/current/cli/maintenance-commands.md` — verify the cross-reference to `prediction-commands.md` (if any) is retargeted.
- `design/current/cli/command-surface.md` — drop `migrate up`/`migrate down`/`--future`/`--no-backlog`/`--to` from the surface table.
- `design/current/overview/**` — short note about the prediction pivot.
- `design/current/architecture/**` — verify no architectural file references the deleted modules (`prediction-reconciliation`, `prediction-workspace-paths`); update any survivor.
- `design/current/configuration/**` — verify no `Backlog.md` or snapshot references survive in configuration docs.
- `design/current/ci/**` — `agent-design-release.md` still relevant. `agent-materialize.md` still relevant. Any CI doc referencing `migrate up` in a workflow step is updated to drop it.
- `implementation/README.md`, `implementation/roadmap.md`, top-level `README.md` — paragraphs referencing the deleted commands/flows are trimmed.

Out of scope for this migration:

- Renumbering or rewriting historical migrations in `migrations/`. They stay as the audit trail.
- Deleting the `prediction/prediction/` mirror or any other content in the predicted-implementation bucket.
- Adding a `lifecycle/` mirror in any rev.N — `design release` will create one organically the next time it runs.
- Reshaping `design/current/execution/`. The run-loop docs are unaffected.
- Rewriting CI workflows under `.github/`. They are referenced by the design tree but live elsewhere.

## Edge cases

- **A migration in `migrations/` (e.g. 100, 145–150) links into `design/current/prediction/...`.** Those links break after this migration. That is acceptable: the migrations are historical records and the broken link is a marker of when the surface was removed. Do not rewrite migration history.
- **A `rev.N/prediction/` directory exists in a frozen revision.** Untouched; revisions are immutable.
- **A user has a workspace whose `.rundown/config.json` configures a non-default design directory name.** All edits are inside the configured design directory; the rename/move is local to whatever path the user has. The migration assumes the canonical `design/current/` shape; users with custom layouts apply the same edits in their tree.
- **A README cross-reference uses `../../prediction/...`.** The verifier (workspace search) catches the broken link.

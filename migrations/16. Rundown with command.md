Please implement Change rundown shortcut task from Features

## Summary

Update inline `rundown:` task delegation so task text can explicitly choose the nested command (`run` or `make`) instead of always being forced through implicit `run`.

The feature request in `migrations/Features.md` currently expresses the intended shape as:

- `rundown: make "Scan count 3 by default" "3. Scan"`
- `rundown: run "3. Scan"`

## Why this change exists

Today, inline rundown delegation is hard-wired to execute nested `rundown run ...` calls. That means any inline task starting with `rundown:` is treated as `run` arguments, not as a true subcommand invocation. This blocks command composition from task lists (especially `make`) and makes inline delegation inconsistent with top-level CLI usage.

Now that `make` exists as a first-class command, inline shortcuts should be able to trigger both:

- direct delegated execution (`run`)
- bootstrap flow creation + research + planning (`make`)

from within Markdown TODO items.

## Current behavior (as implemented)

Relevant code paths:

- Parser marks tasks prefixed with `rundown:` as delegated rundown tasks in `src/domain/parser.ts`.
- Delegation args are parsed in `src/application/rundown-delegation.ts`.
- Runtime dispatch delegates through `workerExecutor.executeRundownTask(...)` in `src/application/task-execution-dispatch.ts`.
- Infrastructure invocation currently prepends `"run"` unconditionally in `src/infrastructure/inline-rundown.ts`.

Current implied contract is effectively:

- `rundown: <target> [run-flags...]` -> nested `rundown run <target> [run-flags...]`

and an error is emitted if there is no target-file operand before flags.

## Desired behavior

Inline `rundown:` tasks should accept explicit subcommand-first syntax:

- `rundown: run <source> [run options...]`
- `rundown: make "<seed text>" "<markdown-file>" [shared options...]`

This should map to nested invocation of the same top-level command shape users can type in the shell, while preserving parent-to-child option inheritance behavior where appropriate.

## Command grammar for inline rundown tasks

Treat delegated task arguments as command-style tokens:

- first token is the delegated subcommand when it is a supported command name (`run`, `make`)
- if first token is not a supported subcommand, treat as legacy implicit-`run` form for compatibility

Practical interpretation:

- `rundown: run Child.md --verify` -> explicit `run`
- `rundown: make "Feature text" "7. Feature.md"` -> explicit `make`
- `rundown: Child.md --verify` -> legacy implicit `run` (backward compatible)

## Compatibility and migration expectations

Backward compatibility is important because existing task files likely use the old shorthand:

- keep legacy implicit `run` support
- add explicit `run`/`make` support
- avoid breaking existing delegated `run` tasks already in repositories

If future cleanup is desired, explicit `run` can become preferred style in docs while legacy remains supported.

## Validation and argument rules

Validation should be subcommand-aware:

- `run` delegated tasks continue to require a target/source operand semantics consistent with `rundown run`.
- `make` delegated tasks require both positional operands (`<seed-text>` and `<markdown-file>`) and should reuse existing `make` validation behavior.
- unsupported delegated subcommands should fail with a clear error listing allowed inline subcommands for this iteration.
- do not reinterpret worker separator (`--`) behavior in surprising ways; keep token handling aligned with existing parser/split logic.

## Execution semantics and inheritance

Delegated tasks should remain sequential, fail-fast, and return child exit status to the parent run pipeline.

Parent run settings currently forwarded for delegated execution include worker/runtime controls (`--worker`, `--transport`, `--keep-artifacts`, output visibility, CLI-block behavior, verify/repair controls). This forwarding should stay explicit-override-safe:

- child explicit args win
- parent defaults fill only missing compatible options

Forwarding compatibility should be checked for `make` because `make` and `run` do not expose identical option sets. Unsupported forwarded options must not corrupt delegated command behavior.

## Output and trace expectations

No change in core output policy is intended:

- delegated stdout/stderr remains hidden unless agent output visibility is enabled
- trace/artifact recording for delegated phase remains intact (`rundown-delegate` phase context)
- failure reporting remains consistent with existing run-task dispatch error patterns

## Integration notes

Primary touchpoints for this migration:

- `src/application/rundown-delegation.ts` (delegated arg parsing/building rules)
- `src/application/run-task-iteration.ts` (missing-target and early validation assumptions)
- `src/application/task-execution-dispatch.ts` (delegation routing and user-facing messaging)
- `src/infrastructure/inline-rundown.ts` (remove unconditional `run` prefix, invoke resolved subcommand)
- `src/domain/ports/worker-executor-port.ts` and `src/infrastructure/adapters/worker-executor-adapter.ts` (if option contract needs widening for command-aware forwarding)

Tests to extend/update:

- `__tests__/application/rundown-delegation.test.ts`
- `__tests__/infrastructure/inline-rundown.test.ts`
- relevant `run-task` integration tests covering delegated tasks

## Scope boundaries

In scope:

- inline `rundown:` delegation supports explicit `run` and `make`
- legacy implicit `run` remains functional
- subcommand-aware validation, forwarding, and error messages
- tests covering parse/build/invoke behavior and fail paths

Out of scope for this iteration:

- full arbitrary subcommand delegation (`init`, `research`, `plan`, etc.)
- redesign of Markdown task parser syntax beyond `rundown:` prefix behavior
- changing top-level `run`/`make` command semantics themselves

## Edge cases to cover

- quoted arguments and spaces in `make` seed text and filenames
- Windows vs POSIX path token handling in delegated targets
- explicit `run` with flags-only input (missing source)
- `make` collisions/invalid extension/missing parent directory surfaced through delegated path
- parent option forwarding when child explicitly sets conflicting flags
- delegated command failure propagation (`run` or `make`) into parent run exit handling

## Acceptance criteria outline

- Inline task `rundown: run "<source>"` invokes nested `rundown run ...` (not `rundown run run ...`).
- Inline task `rundown: make "<seed>" "<file>.md"` invokes nested `rundown make ...` successfully.
- Legacy inline task form `rundown: <source> [flags]` continues to work as implicit `run`.
- Invalid or unsupported delegated command forms fail with clear user-facing errors.
- Parent-to-child option inheritance remains stable and explicit child overrides still take precedence.
- Delegated execution trace/artifact behavior and output visibility behavior remain consistent with current run pipeline.

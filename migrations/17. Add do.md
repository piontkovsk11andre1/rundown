add do command that make + all

## Summary

Add a new top-level `do` command that composes the existing `make` flow with full execution flow (`all`) for the same Markdown file.

The intent is one-step automation:

1. bootstrap a task document from seed text (`make`),
2. enrich + plan it (`research` + `plan` via `make`),
3. execute all generated TODO items (`all` / `run --all`).

## Why this change exists

Current usage can already chain these steps manually, but it is easy to run them inconsistently or forget a phase.  
`do` should be the canonical "idea -> planned doc -> fully executed checklist" command for automation-first workflows.

This is especially useful for delegated inline tasks and migration workflows where command composition is expected directly from Markdown task lists.

## Target command shape

```bash
rundown do "<seed-text>" "<markdown-file>"
```

Example:

```bash
rundown do "add do command that make + all" "17. Add do.md"
```

High-level behavior:

- Execute `make` semantics on `<seed-text>` + `<markdown-file>`:
  - create file,
  - write seed text,
  - run `research`,
  - run `plan`.
- If `make` succeeds, execute `all` semantics on the same file:
  - run all unchecked tasks sequentially to completion/failure stop.
- Return success only when the full chain succeeds.

## Existing architecture context

Current command surface and behavior already define almost all required building blocks:

- `make` is implemented as composition (`create -> research -> plan`).
- `all` exists as shorthand of `run --all`.
- Inline `rundown:` delegation is command-aware and currently supports explicit delegated subcommands (at least `run` and `make`).
- Option normalization and worker-resolution behavior are centralized in CLI-layer option modules and command actions.
- Locking semantics are already defined per phase (`research`, `plan`, `run`), so `do` should reuse sequential lock acquisition/release rather than introduce a new lock model.

So `do` should be orchestration-only, reusing existing command/use-case behavior instead of duplicating logic.

## Proposed semantics

### Phase sequence

`do` runs strict ordered phases:

1. **bootstrap phase**: perform `make` with provided operands,
2. **execution phase**: perform `all` (equivalent to `run --all`) against the same markdown file.

No parallelism between phases.

### Fail-fast rules

- If bootstrap file creation fails: stop immediately.
- If `research` fails: stop immediately (no `plan`, no `all`).
- If `plan` fails: stop immediately (no `all`).
- If `all` fails on any task: command exits non-zero with existing run/all error semantics.
- Exit code behavior should stay aligned with existing command contracts (`0/1/2/3` model).

### Data and file behavior

- Same file path validations as `make` for `<markdown-file>` (`.md`/`.markdown`, non-directory, collision-safe create).
- Seed text is written exactly as provided (no transformation).
- `all` phase consumes the post-plan document state from disk.

## Option and forwarding model

`do` should preserve predictable forwarding while staying command-aware across two different downstream surfaces (`make` then `run --all`).

Shared options expected to flow across both phases where supported:

- worker/runtime transport and invocation controls,
- artifact/trace and output visibility controls,
- variable injection controls,
- cli-block behavior controls,
- lock recovery flags (`--force-unlock`),
- config resolution (`--config-dir` global).

`do`-specific normalization should avoid forwarding incompatible options into the wrong subcommand.  
When both parent and child-equivalent values exist, explicit child-equivalent inputs should win (same precedence model used elsewhere).

## Interaction with `all` semantics

The execution phase should behave exactly like existing `rundown all <source>` / `rundown run <source> --all`:

- task-by-task sequential execution,
- stop on first failure,
- completion marking only after verification success,
- existing verify/repair behavior unchanged,
- existing hook/commit behavior unchanged when those options are in scope for `run`.

No special behavior should be introduced only for `do`.

## Locking and concurrency expectations

`do` is a composed sequential command, so lock behavior should remain phase-native:

- `make` internally runs `research` then `plan` with their existing lock lifecycle,
- once bootstrap is complete, `all` acquires run lock behavior and processes full task execution loop.

No shared super-lock protocol is required if existing phase-level locking already guarantees source safety.

## Output and observability expectations

`do` should keep the same output policy as existing commands:

- agent output hidden by default unless enabled,
- trace/artifacts follow existing command behavior,
- delegated/phase error reporting remains clear and phase-specific (bootstrap vs all execution).

Given existing note that `make` logging visibility can feel thin, `do` should preserve phase boundary messages clearly (start/success/failure per phase) so users can see where failure happened without changing core logging model.

## Integration touchpoints

Likely primary touchpoints:

- `src/presentation/cli.ts` (register top-level `do` command and help text),
- `src/presentation/cli-options.ts` (option schema/normalization for `do`),
- `src/presentation/cli-command-actions.ts` (orchestration action: `make` then `all`),
- app boundary wiring where command actions call existing use cases,
- docs updates (`docs/cli.md`) for syntax/options/examples.

If delegated Markdown execution scope includes explicit `rundown: do ...`, delegation parsing/allowed-subcommand lists may also need extension in delegated command modules and their tests.

## Scope boundaries

In scope:

- top-level `rundown do <seed-text> <markdown-file>`,
- strict `make` then `all` composition,
- command-aware option forwarding and validation,
- docs/help updates,
- tests for happy path and fail-fast propagation.

Out of scope:

- redefining `make` internals,
- changing `all`/`run` core task lifecycle semantics,
- arbitrary pipeline composition beyond this fixed two-stage chain,
- introducing overwrite-by-default behavior for target files.

## Edge cases to cover

- seed text with quotes/newlines/special chars,
- relative and absolute markdown target paths,
- target collision and missing parent directories,
- invalid extensions / directory path input,
- `make` success + `all` failure propagation clarity,
- no actionable tasks after planning (`all` returns no-actionable semantics),
- worker resolution failures at either phase,
- option compatibility when a flag is valid for one phase but not the other.

## Acceptance criteria outline

- `rundown do "<seed>" "<file>.md"` performs `make` then `all` in that exact order.
- `all` phase does not start unless `make` fully succeeds.
- File/path validation and collision safety match existing `make` behavior.
- Execution semantics in second phase match existing `all` behavior (including verify/repair/stop-on-failure).
- Option forwarding is stable and command-aware; incompatible forwarding does not break downstream phases.
- CLI documentation includes `do` usage, behavior, and examples.
- Tests cover success path, phase-failure propagation, validation failures, and option-forwarding precedence.

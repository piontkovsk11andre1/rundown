Add a memory feature that will persist a text file in .rundown local dir for md file. We will write to that file using memory: or remember: task prefix. We will address writing later, in this task we want implement logic of storing/indexing this memory and injecting it into the prompts. We don't want to inject the content, but path to memory file and short description should be there.

## Summary

Introduce a per-Markdown memory context channel that is discoverable and prompt-visible across `run`, `discuss`, `plan`, `research`, and `reverify` flows, without inlining memory file contents into prompts.

This phase is focused on memory storage shape, identity/indexing, and prompt injection contract. Authoring/updating memory entries from `memory:` / `remember:` task prefixes is intentionally deferred.

## Why this matters

Current prompt rendering already carries rich task/document context (`task`, `file`, `context`, `source`), but it has no durable per-document "working memory" pointer. That makes long-running documents and iterative workflows rely on task text repetition or user recall.

A memory pointer + short description in prompt context gives workers stable continuity without blowing up token usage or changing template behavior substantially.

## Scope for this task

### In scope

- Define how memory is persisted for a Markdown source in a local `.rundown` directory.
- Define deterministic indexing/lookup so any command can resolve the right memory record for a source file.
- Inject memory reference metadata into prompts (path + short description), not file contents.
- Ensure behavior is safe when memory does not exist yet.

### Out of scope (explicitly deferred)

- Parsing `memory:` / `remember:` tasks and writing new memory content entries.
- Editing memory file content during task execution.
- Any automatic summarization pipeline that reads and rewrites memory text.

## Existing architecture context

Prompt rendering is currently command-specific but follows a shared pattern:

- `run`/`reverify` execution prompt assembly goes through `src/application/prepare-task-prompts.ts` + template vars in `src/domain/template.ts`.
- `discuss` renders via `src/application/discuss-task.ts`.
- `plan` renders via `src/application/plan-task.ts`.
- `research` renders via `src/application/research-task.ts`.
- Default templates live in `src/domain/defaults.ts` and rely on `{{...}}` interpolation.
- Unknown template placeholders are preserved by `renderTemplate(...)` (not silently stripped), so adding new vars is backward-safe for templates that do not use them.
- Lockfiles are source-relative (`<source-dir>/.rundown/<basename>.lock`), even when config discovery points elsewhere. This is a useful precedent for per-source memory locality.

## Proposed memory model

### Storage location (source-local)

Memory should be source-relative, not global-config-relative:

- Memory file path: `<source-dir>/.rundown/<source-basename>.memory.md`
- Optional metadata/index file: `<source-dir>/.rundown/memory-index.json`

Rationale:

- Matches "local dir for md file" requirement.
- Avoids collisions between same-named files in different directories.
- Aligns with existing source-relative lockfile strategy.
- Keeps memory portable with the document directory.

### Identity/indexing contract

Each memory record should be keyed by canonical source file identity:

- Absolute normalized source path (primary runtime key).
- Stored relative display path for human-readable prompt output.
- Stable basename-derived memory filename for on-disk persistence.
- A short description field (human summary) persisted separately from memory content body.

Index responsibilities:

- Resolve source -> memory file path.
- Resolve source -> short description (empty/undefined when not available).
- Create deterministic record even when memory file exists but description is missing.

## Prompt injection contract

### What to inject

Inject only:

- Memory file path
- Short description
- Availability flag (present/missing)

Do not inject:

- Memory file contents
- Large excerpts/snippets

### Where to inject

Inject across all worker-facing prompts:

- execute/verify/repair paths (`run`, `reverify`)
- `discuss`
- `plan`
- `research`

This should be command-agnostic at the application layer, not dependent on one template only.

### Suggested prompt shape

A compact section should be included in rendered prompts, for example:

- memory status (available/unavailable)
- memory file path (absolute or stable workspace-relative path)
- memory summary (single short line)

The exact wording can vary by template style, but fields should remain structurally stable for tooling/tests.

If implemented as a new port, keep dependency direction consistent: domain port -> infrastructure adapter -> wired in `create-app.ts`.

## Compatibility and constraints

- Must not break existing templates that do not yet reference memory vars.
- Must work with both `file` and `arg` prompt transport.
- Must remain path-safe on Windows and POSIX.
- Must not alter checkbox behavior, task selection, verification semantics, or lock lifecycle.
- Must not require `--keep-artifacts`; memory is independent from runtime artifact retention.
- Must handle missing `.rundown` directory by creating minimal local structure only when needed for memory persistence/index presence.

## Edge cases

- Source file renamed/moved after memory file exists.
- Same basename in different directories.
- Source paths resolved via relative, absolute, directory, and glob flows.
- `plan`/`research` on a file with no memory yet.
- `reverify` on historical runs where source still exists but memory entry is absent.
- Description exists but memory file missing (and inverse case).
- Unicode/special characters in source filename affecting derived memory filename.

## Observability and testing guidance

Validation should cover:

- Deterministic source -> memory path/index resolution.
- Prompt output includes memory path + short description fields in each command flow.
- Prompt output does not include memory file content.
- Graceful behavior when memory metadata is absent.
- Cross-platform path normalization expectations.
- No regression in existing run/discuss/plan/research/reverify exit-code behavior.

## Acceptance criteria outline

- Every prompt-producing command can resolve memory metadata for its source file.
- Rendered prompts include memory path and short description (or explicit "not available" equivalent).
- Rendered prompts never inline memory file content.
- Memory resolution is source-local and collision-safe across directories.
- Existing command semantics remain unchanged aside from added prompt context fields.
- The design leaves a clean extension point for later `memory:` / `remember:` write support.

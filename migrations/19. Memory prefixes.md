Add memory: memorize: remember: inventory: prefix that will execute prompt and save memory. Templates should have memory map

## Summary

Extend the task-intent system with memory-writing prefixes (`memory:`, `memorize:`, `remember:`, `inventory:`) so a selected task can run a worker prompt and persist the resulting knowledge into source-local memory artifacts.

This builds on the existing memory metadata foundation already present across `run`, `discuss`, `plan`, `research`, and `reverify` prompts (`memoryStatus`, `memoryFilePath`, `memorySummary`) and adds the missing authoring/write path.

## Why this matters

Memory metadata is already injected into prompts, but there is currently no first-class workflow for creating/updating that memory from task text.

Adding explicit memory prefixes gives a deterministic, auditable, Markdown-native way to:
- capture durable context during execution,
- keep memory tied to the source file (`<source-dir>/.rundown/...`),
- and improve multi-step continuity without bloating templates with raw memory content.

## Existing architecture context

Relevant current behavior and boundaries:

- Task intent classification currently supports only verify-only prefixes via `classifyTaskIntent(...)` in `src/domain/task-intent.ts` (`verify:`, `confirm:`, `check:`).
- Execution routing in `src/application/task-execution-dispatch.ts` already has specialized branches for:
  - inline CLI tasks (`cli:`),
  - delegated rundown tasks (`rundown:`),
  - default worker execution.
- Memory read metadata is already implemented via `MemoryResolverPort` and `createMemoryResolverAdapter(...)`:
  - memory file path resolution (source-local),
  - optional summary from `.rundown/memory-index.json`,
  - no inlining of memory body into prompts.
- Default templates already include a compact memory context section in `src/domain/defaults.ts`.
- Template variable interpolation is backward-safe for unknown placeholders, so adding new memory-map variables is compatible with existing custom templates.

## Proposed prefix semantics

### Supported prefixes (aliases)

Treat these as equivalent memory-capture intents:

- `memory:`
- `memorize:`
- `remember:`
- `inventory:`

### Normalization rules

- Prefix matching should be case-insensitive and whitespace-tolerant around `:`.
- The task payload is the text after the first matched prefix marker.
- Empty payload is invalid and should fail with a clear user-facing message (same exit semantics as other execution failures).
- Prefix parsing should happen before default execute routing so these tasks are deterministic and do not rely on template interpretation.

### Intent model

Add a memory-capture intent alongside existing intent handling, without changing verify-only behavior.

Conceptually:
- verify-only prefixes still mean "skip execute, run verify";
- memory prefixes mean "execute prompt payload, then persist memory";
- all other tasks remain execute-and-verify as today.

## Execution and persistence contract

### Execution behavior

For memory-prefixed tasks:

1. Resolve source-local memory target for the selected Markdown file.
2. Execute worker with prompt payload (post-prefix text).
3. Persist worker output into memory storage.
4. Continue with existing completion/verification lifecycle rules (no checkbox shortcuts).

### Persistence behavior

Memory writes should remain source-local and deterministic:

- Memory body file: `<source-dir>/.rundown/<source-basename>.memory.md` (current resolver naming, including source extension in basename).
- Memory metadata index: `<source-dir>/.rundown/memory-index.json`.

Suggested persisted fields in index for each canonical source path:

- `summary`: short one-line description of current memory state.
- optional extensible metadata (for example `updatedAt`, `lastPrefix`, `entryCount`) if needed for diagnostics and template rendering.

### Write strategy expectations

- Ensure parent `.rundown` directory exists only when a write is needed.
- Prefer append-safe memory updates (do not destroy existing memory content by default).
- Keep writes atomic enough to avoid partial index corruption on interrupted runs.
- If memory body write succeeds but index update fails, surface warning/error clearly and preserve recoverable state.

## Template memory map requirement

Templates should expose a memory map section, not only status/path/summary lines.

### Template shape goal

Add stable variables for a compact map-style block, for example:

- memory status
- memory file path
- memory summary
- memory index path (or equivalent)
- small machine-readable/human-readable map payload (`memoryMap`)

### Scope of template updates

- Update built-in defaults in `src/domain/defaults.ts` to include the map block in worker-facing templates.
- Keep custom-template compatibility: projects not referencing new variables should continue to work unchanged.
- Update docs in `docs/templates.md` and `docs/cli.md` so prefix semantics and memory map placeholders are discoverable.

## Integration touchpoints

Likely modules to extend:

- `src/domain/task-intent.ts` (memory-prefix classification).
- `src/application/task-execution-dispatch.ts` (memory-capture dispatch branch).
- `src/domain/ports` (memory write/index port definitions if introducing explicit writer abstraction).
- `src/infrastructure/adapters/memory-resolver-adapter.ts` and/or new writer adapter for body/index persistence.
- `src/create-app.ts` (wiring memory writer dependencies).
- `src/domain/template.ts` (new template vars for memory map section).
- `src/domain/defaults.ts` (default template memory map block).
- `src/presentation` CLI docs/help surfaces where task prefix behavior is documented.

## Compatibility and constraints

- Do not break existing `verify:` / `confirm:` / `check:` semantics.
- Do not change `cli:` and `rundown:` behavior.
- Keep run/reverify/discuss/plan/research prompt contracts stable aside from additive memory-map fields.
- Maintain cross-platform path behavior (Windows/POSIX).
- Preserve current exit code model (`0/1/2/3`) and verification lifecycle.
- Do not inline full memory body into prompt context by default.

## Edge cases and failure handling

- Prefix with no payload text.
- Worker returns empty output for memory capture.
- Memory file exists but index missing (and inverse).
- Malformed `memory-index.json` during update path.
- Same basename in different directories.
- Source file rename/move after memory was created.
- Concurrent memory-prefixed tasks on same source (respect source lock semantics).
- Very large worker output written as memory content.

## Observability and testing guidance

Validation should cover:

- Prefix recognition and intent routing for all aliases.
- Correct payload extraction after prefix normalization.
- Worker output persistence to source-local memory file.
- Index summary updates keyed by canonical absolute source path.
- Prompt/template rendering includes memory map fields where expected.
- No regression in non-memory tasks, verify-only tasks, and delegated task flows.
- Clear failure messages for empty payload, write failures, and malformed index recovery paths.

## Acceptance criteria outline

- Memory prefixes (`memory:`, `memorize:`, `remember:`, `inventory:`) are recognized as first-class execution intents.
- Memory-prefixed tasks execute prompt payload and persist output into source-local memory artifacts.
- Memory persistence updates both body and map/index metadata in a deterministic, collision-safe way.
- Default templates include a compact memory map section using stable template variables.
- Existing command/task semantics (`verify:`, `cli:`, `rundown:`, run lifecycle, exit codes) remain unchanged outside the new feature scope.
- Documentation clearly describes memory prefix behavior, storage shape, and template memory map variables.

# 199. Move Materialize And Revert To Implementation Snapshot History

## Problem

Migration 198 introduces a filesystem-backed implementation snapshot system and a dedicated `snapshot` command.

But the surrounding implementation lifecycle still points at the older git-backed history model:

- `materialize` is still defined as `run --all --revertable`,
- `revert` still requires `extra.commitSha` from saved run metadata,
- docs still describe reversal in terms of git commits,
- and operators still need git-specific preconditions to undo materialized state.

That leaves the product model split across two incompatible history systems:

- filesystem snapshots for implementation state,
- git commits for materialize/revert behavior.

If implementation snapshots are the new canonical history layer, `materialize` and `revert` need to move onto that layer too.

## Desired behavior

After this migration:

- `materialize` uses implementation snapshots as its history/reversal foundation,
- `revert` restores implementation state from saved implementation snapshots instead of reverting git commits,
- and git metadata is no longer required for materialize/revert flows.

This does not remove git support from unrelated commands or tooling. It only removes git as the canonical backing store for implementation-state history.

## Behavior contract

### 1. `materialize` saves implementation snapshot history, not git-backed revert history

`materialize` should stop being defined primarily as `run --all --revertable`.

Instead:

- it still performs the same materialization work over the chosen source,
- it still preserves runtime artifacts when needed for inspection,
- but its durable history output is an implementation snapshot at the completed migration boundary,
- and it must no longer require commit creation to be considered reversible.

Recommended shape:

- materialize completes the run,
- determines the completed migration boundaries,
- and records the corresponding implementation snapshot(s) using the snapshot system from migration 198.

### 2. `revert` restores implementation state from snapshots

`revert` should stop resolving git commits from `extra.commitSha` and stop performing `git revert` / `git reset` as its core mechanism.

Instead, `revert` should:

- identify the target implementation snapshot boundary from saved runtime metadata and/or snapshot history,
- restore the live `implementation/` tree from that saved snapshot,
- and record a new artifact entry describing the restoration.

From the user point of view, `revert` remains the command that undoes prior materialized implementation state. The mechanism changes from git history replay to filesystem snapshot restore.

### 3. Revert target eligibility is snapshot-based, not commit-based

The current revert contract filters runs by presence of `extra.commitSha`.

After this migration, eligible revert targets should instead be runs or snapshot records that:

- completed successfully,
- are associated with a recorded implementation snapshot boundary,
- and still have a corresponding snapshot payload on disk.

The old message "No revertable runs found ... run.json contains extra.commitSha" must be replaced with snapshot-based guidance.

### 4. Restores operate on the live implementation tree only

`revert` in this new model restores the current `implementation/` tree from a chosen implementation snapshot.

It should not:

- modify git history,
- require a clean git worktree,
- or depend on repository ancestry checks.

It should, however, continue to use rundown file-locking and artifact bookkeeping so restores do not race with other commands.

### 5. Snapshot numbering remains tied to completed migrations

This migration does not loosen the numbering rule from migration 198.

Materialize and revert must both respect the same contract:

- implementation snapshots are indexed by completed migration boundaries,
- no ad hoc between-migrations snapshot ids are invented,
- and restore targets map to those recorded lane-aware boundaries.

### 6. `materialize` and `snapshot` serve different roles

Even after migration:

- `snapshot` remains the explicit command for saving implementation state history on demand,
- `materialize` remains the command that executes and applies work,
- and `revert` remains the command that restores a prior implementation snapshot.

So the command family becomes:

- `predict` -> update future state
- `snapshot` -> save present implementation state boundary
- `materialize` -> apply work and record implementation snapshot history
- `revert` -> restore implementation state from recorded snapshots

## Scope

In scope:

- move `materialize` off `--revertable`/commit-based semantics,
- move `revert` off `extra.commitSha` and git undo operations,
- redefine revert target eligibility in terms of implementation snapshots,
- and update docs/help/messages for the new snapshot-based lifecycle.

Out of scope:

- deleting all git integration from the repository,
- changing unrelated commands that still optionally use git,
- introducing snapshot diff UX,
- or changing prediction snapshot behavior.

## Implementation plan

1. Introduce snapshot-history metadata for implementation runs so completed materialize actions can be mapped to implementation snapshot boundaries.
2. Update `materialize` command action so it no longer enforces `--revertable` as the history mechanism.
3. After successful materialize execution, persist or confirm the corresponding implementation snapshot(s).
4. Rewrite `revert-task.ts` to resolve snapshot restore targets instead of git commit targets.
5. Replace git-specific validation and user-facing errors with snapshot-based validation and guidance.
6. Update CLI/docs/tests for the new materialize/revert contract.

## Acceptance criteria

- `materialize` no longer depends on git commit metadata to participate in reversal/history flows.
- `revert` can restore a prior implementation state using recorded implementation snapshots.
- `revert` no longer requires `extra.commitSha`, a clean git worktree, or git reset/revert operations.
- User-facing revert eligibility/help text refers to implementation snapshots rather than commit SHAs.
- `log --revertable` or the equivalent eligibility view is updated so snapshot-backed implementation history is visible as revertable.
- A successful materialize followed by revert returns the live `implementation/` tree to the earlier recorded snapshot state.

## Current implementation context (research baseline)

- `materialize` is currently documented as a convenience wrapper over `run --all --revertable` in [implementation/docs/cli-materialize.md](implementation/docs/cli-materialize.md).
- `revert` is currently documented and implemented as a git-commit undo command and explicitly requires `extra.commitSha` in [implementation/docs/cli-revert.md](implementation/docs/cli-revert.md) and [implementation/src/application/revert-task.ts](implementation/src/application/revert-task.ts#L1).
- The snapshot model introduced in migration 198 creates the missing filesystem history substrate needed to replace git-backed implementation-state restores.

## Edge cases

- If a run is marked as snapshot-restorable but the target snapshot payload is missing on disk, `revert` should fail with a snapshot-missing error rather than falling back to git.
- If multiple lane snapshots correspond to the same effective implementation state boundary, restore logic must deterministically choose or compose the correct restore source according to the snapshot contract introduced in migration 198.
- If a snapshot restore is requested while another rundown operation holds relevant locks, `revert` should continue failing fast rather than restoring concurrently.
- Historical git-backed revert runs may still exist in artifacts; compatibility handling can be additive, but new revert flows must prefer snapshot-backed targets.

# Converge sibling use cases onto shared modules

## Goal

Replace inline reimplementations in `plan-task.ts`, `reverify-task.ts`, `discuss-task.ts`, and `revert-task.ts` with imports from shared modules extracted during step 1. This is a pure mechanical refactoring — no behavioral changes, no new features.

## Background

Step 1 ([migrations/1. Decompose run-task.md](1.%20Decompose%20run-task.md)) decomposed `run-task.ts` (~3,060 lines) into focused modules. Several of those modules export functions that are **also defined inline** in the four sibling use cases listed above. The duplication exists because the original `run-task.ts` was a monolith — siblings could not import individual concerns without pulling in the entire dependency graph, so each reimplemented what it needed.

Now that shared modules exist, the siblings should converge onto them.

## Approach per module

### General pattern

For each module:

1. Add imports from the shared module.
2. Delete the inline reimplementation.
3. If the inline version had a slightly different signature (e.g. `formatTaskLabel` in `revert-task.ts` takes `ArtifactRunMetadata` instead of `Task`), either:
   - Adapt the call site to match the shared signature, or
   - Add a thin adapter function at the call site (not a new shared export).
4. Run the module's test suite to verify no behavioral change.

### Module-specific notes

**`plan-task.ts`**: The `isOpenCodeCommand()` helper has a slightly different shape than `isOpenCodeWorkerCommand()` in `run-task-execution.ts` — `isOpenCodeCommand` takes a single string while `isOpenCodeWorkerCommand` takes `string[]`. Evaluate whether to generalize or keep the plan-specific version with a one-line wrapper. The `finalizePlanArtifacts` function has plan-specific artifact metadata (convergence stats), so it is not a direct substitute for `finalizeRunArtifacts` — retain as plan-specific but consider extracting the common artifact finalization pattern.

**`reverify-task.ts`**: This module has the most overlap with `task-context-resolution.ts`. The `resolveTaskContextFromMetadata` and `findTaskByFallback` functions are functionally identical. The `validateTaskMetadata` function differs only in error message wording. The shared `validateRuntimeTaskMetadata` returns the same error semantics (null = valid, string = error message), so the switch is safe. The `resolveTargetRuns` / `resolveTargetRunMetadata` functions are reverify-specific and should stay.

**`discuss-task.ts`**: The checkbox mutation detection logic (`captureCheckboxState`, `detectCheckboxMutations`) is discuss-specific and should stay. Only the shared patterns (CLI block handlers, template loading, task label formatting) are convergence targets.

**`revert-task.ts`**: The git operations overlap is exact for `isGitRepoWithGitClient`, `isWorkingDirectoryClean`, `resolveGitArtifactAndLockExcludes`, and `isPathInsideRepo`. The revert-specific git logic (`git revert`, `git reset`, commit validation, contiguous-HEAD checks) stays. The `formatTaskLabel` takes a different shape (`ArtifactRunMetadata` with optional task field) — keep the revert-specific version.

## Constraints

- **No behavioral changes**: Exit codes, output messages, and side effects must remain identical.
- **Test suites must pass as-is**: The existing test files (`__tests__/application/plan-task.test.ts`, `reverify-task.test.ts`, `discuss-task.test.ts`, `revert-task.test.ts`) should not need modifications beyond updating import paths if any types moved.
- **No new shared modules**: Use only the modules extracted in step 1. If a would-be-shared function doesn't quite fit, keep the module-specific version.
- **Order of convergence**: Modules can be converged independently. A suggested order (by risk, lowest first): `discuss-task.ts` → `plan-task.ts` → `revert-task.ts` → `reverify-task.ts`. `reverify-task.ts` is last because it has the most overlap with `task-context-resolution.ts` and requires the most careful validation.

## Verification

After each module convergence:

1. Run the module's test suite (e.g. `npx vitest run __tests__/application/discuss-task.test.ts`).
2. Run the full application test suite (`npx vitest run __tests__/application/`).
3. Confirm no compile errors (`npx tsc --noEmit`).

## Context and current behavior

The empty-feeling CLI experience is reproducible with delegated `rundown:` tasks and also appears in other worker-backed flows when output is intentionally buffered or hidden.

Today the run pipeline emits high-level status, but most command output is suppressed unless explicitly enabled:

- `run` iteration emits `Next task: ...` in `src/application/run-task-iteration.ts`.
- Delegated rundown emits `Delegating to rundown: rundown ...` in `src/application/task-execution-dispatch.ts`.
- Worker/delegated stdout+stderr forwarding is gated by `showAgentOutput` (`emitExecutionWorkerOutput` in `src/application/task-execution-dispatch.ts`).
- `--show-agent-output` defaults to false in CLI option handling (`src/presentation/cli.ts`, `src/presentation/cli-command-actions.ts`).
- Nested `rundown:` delegation forwards `--show-agent-output` only when parent requested it (`src/application/rundown-delegation.ts`, `src/infrastructure/inline-rundown.ts`).

Result: users see a start message, then silence, then final success/failure. For long-running delegated `run`/`make`, this feels like “no output” even though work is happening.

## Relevant module boundaries

- `src/application/run-task-iteration.ts`  
  Owns per-task lifecycle messaging (`Next task`), mode decisions, prompt prep, dispatch callouts.

- `src/application/task-execution-dispatch.ts`  
  Owns execution branch selection (`cli:`, `rundown:`, worker), emits delegation message, controls whether stdout/stderr is surfaced, returns structured failure states.

- `src/application/rundown-delegation.ts` and `src/infrastructure/inline-rundown.ts`  
  Own parsing/validation/forwarding and nested process execution for inline `rundown:` tasks.

- `src/domain/ports/output-port.ts` + `src/presentation/output-port.ts`  
  Define renderable output events (`info`, `warn`, `error`, `success`, `task`, `text`, `stderr`). No dedicated progress event exists yet.

- `src/application/plan-task.ts`, `src/application/research-task.ts`, `src/application/discuss-task.ts`  
  Other long-running command flows with similar “quiet until done” risk depending on `showAgentOutput` and emitted lifecycle status.

## Problem shape to solve

Primary issue is not command correctness; it is runtime observability and perceived responsiveness.

We need to improve UX for long-running task execution by making progress visible without forcing full raw agent output. The UX should be consistent across:

- delegated `rundown:` tasks,
- normal `run` worker execution,
- long-running `plan` / `research` / `discuss` where applicable,
- both top-level and nested invocations.

## Desired behavior (feature intent)

1. Users should always see meaningful lifecycle updates while work is in progress, even when `--show-agent-output` is false.
2. Delegated `rundown:` tasks should produce visible “in progress” feedback, not only start and end lines.
3. Progress indicators should be used where phase boundaries are known (e.g., execute vs verify vs repair loops, delegated subcommand start/finish).
4. Raw worker stdout/stderr remains opt-in via `--show-agent-output`; improved status logging is separate and always-on.
5. Nested delegation should preserve clarity (parent and child context visible without noisy duplication).

## Logging/progress model direction

Keep current event model compatible, then layer richer progress semantics:

- Baseline (low-risk): add more `info` events at phase transitions and attempt boundaries.
  - examples: `Starting delegated rundown run ...`, `Delegated rundown run still running...`, `Delegated rundown run finished (exit 0)`.
  - for verify/repair: emit per-attempt start/result lines.

- Optional enhancement: add a first-class progress event kind in `ApplicationOutputEvent`.
  - enables progress bar/spinner in terminal renderer (`src/presentation/output-port.ts`) while preserving plain logs in adapters that do not support animation.
  - fallback rendering should degrade to plain text (`info`) in non-TTY contexts.

- Progress bars should be reserved for deterministic bounded flows (known step counts).
  - good fit: repair attempts (`attempt i of N`), multi-round clean cycles.
  - less deterministic execution should use spinner or periodic status ticks instead of fake percentages.

## Compatibility and constraints

- Do not break existing `--show-agent-output` contract; users who want full raw output still opt in.
- Avoid changing delegated command semantics (`rundown: run ...` / `rundown: make ...`) and existing flag inheritance behavior.
- Keep output structured and parse-safe for global logging (`src/presentation/logged-output-port.ts`).
- Ensure non-interactive environments (CI, redirected stdout) do not receive unstable ANSI animation streams unless explicitly supported.
- Maintain current error payload behavior in `task-execution-dispatch` so run exit/status logic remains unchanged.

## Acceptance criteria outline

- Delegated `rundown:` tasks visibly report start, active progress, and completion/failure in CLI without requiring `--show-agent-output`.
- Long-running `run` paths show non-silent phase updates (execute/verify/repair/delegation) with clear task context.
- Other long commands (`plan`, `research`, `discuss`) are reviewed for silence gaps; each has at least minimal lifecycle logging for long waits.
- Existing tests covering delegation behavior continue to pass; output-focused tests are updated/added for new status lines.
- `--show-agent-output` still controls raw worker stream forwarding exactly as before.
- Behavior remains readable in both TTY and non-TTY environments.

## Scope boundaries

In scope:

- CLI-facing logging and progress UX improvements in application/presentation layers.
- Delegated rundown visibility improvements.
- Cross-command consistency pass for long-running flows.

Out of scope:

- Rewriting worker transport or agent protocol.
- Altering task parsing/delegation validation semantics.
- Inventing synthetic percentage metrics for unbounded agent tasks.

## Integration notes for planning phase

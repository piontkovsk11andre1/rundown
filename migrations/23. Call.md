## Feature: `call` command with CLI block caching

<details>
<summary>Context</summary>

### What `call` is

A new top-level CLI command that is a convenience alias for:

```
rundown run --clean --all --cache-cli-blocks <source>
```

It combines three existing behaviors plus one new capability:
- `--clean` (= `--redo` + `--reset-after`): reset all checkboxes before run, reset again after.
- `--all` (implied by `--clean`): run every task sequentially instead of stopping after one.
- `--cache-cli-blocks` (**new**): cache the output of CLI fenced blocks so identical commands are not re-executed across tasks within the same run session.

### Why caching matters

During a `--clean --all` run, every task's source markdown and every template is expanded via `expandCliBlocks` (defined in `src/domain/cli-block.ts:227`). Each expansion spawns a fresh shell process per command (`src/infrastructure/cli-block-executor.ts:26`). For files with expensive or repeated CLI blocks (e.g. `git log`, `tree`, test-suite output), the same commands execute once per task — multiplied again across source, task-template, and verify-template phases (`src/application/prepare-task-prompts.ts:187,249,278`).

Caching eliminates redundant executions by storing `CommandResult` keyed on `(command, cwd)` and returning the cached result on subsequent hits within the same session.

### Cache lifecycle

The cache is **scoped to a single `call` execution** — it lives only in-memory for the duration of the process:

1. **Startup**: any stale on-disk cache (from a previously interrupted run) is deleted.
2. **During execution**: `CommandResult` values are stored in a `Map<string, CommandResult>` keyed on `(command, cwd)`. Every `expandCliBlocks` call hits the cache first; on miss it delegates to the real executor and stores the result.
3. **Completion**: the cache is discarded when `call` finishes (success or failure). No cleanup in `list` or any other command is needed.

### Implementation approach

1. **Caching executor decorator** — wrap `CommandExecutor` with an in-memory caching layer keyed on `(command, cwd)`. On cache hit, return the stored `CommandResult` without spawning. On miss, delegate to the real executor and store the result. The decorator is a plain `Map` — no disk persistence needed.

2. **New `--cache-cli-blocks` flag** — add to `run` command options. When enabled, the app wraps the injected `cliBlockExecutor` with the caching decorator for the lifetime of that `runTask` call.

3. **`call` command** — new CLI command in `cli.ts` that accepts the same `<source>` argument and all `run` options, but hardcodes `clean: true`, `runAll: true`, and `cacheCliBlocks: true`. Delegates to `app.runTask(...)` like `do` delegates after its bootstrap phase.

4. **Stale cache cleanup** — on startup, if a previous run left an on-disk cache (future-proofing for crash recovery), delete it before proceeding. Since the cache is in-memory, this is only relevant if disk persistence is added later.

5. **No `list` changes** — `list` remains read-only with no side-effects.

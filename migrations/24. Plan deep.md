# Feature: `rundown plan --deep <n>` — Nested Sub-task Generation

`rundown plan` is getting a `--deep <n>` mode that forces the model to create tasks **and subtasks**. 

## Example output

For `--deep 2` (one layer of sub-tasks):

```markdown
- [ ] Connect cliResource to Something
  - [ ] Do this.
  - [ ] Do that.
  - [ ] verify: This.
  - [ ] memorize: How cliResource connected
```

For `--deep 3` the hierarchy should be one level deeper still — each sub-task may gain its own children.

## Desired behavior

The process is **iterative**, layered on top of the existing multi-scan loop:

1. **Phase 1 — Top-level TODO generation** (unchanged from today).
   Run the existing plan scan loop (`scanCount` passes, convergence-aware) to produce flat `- [ ]` items. This uses `DEFAULT_PLAN_TEMPLATE` and `buildPlanScanPrompt()` in `src/application/plan-task.ts` (lines ~476-577).

2. **Phase 2..N — Deep passes** (`--deep <n>` adds `n` additional depth passes).
   For each pass, iterate over every leaf task that has no children yet, and invoke a **dedicated deep-plan prompt** that asks the model to return child sub-items for that single task. Insert results using the existing `insertSubitems()` helper from `src/domain/planner.ts` which already handles correct child indentation via `computeChildIndent()`.

Each deep pass should be a **clean independent worker session** (same constraint as scan passes — see `validatePlanCleanSessionWorkerCommand()` in `plan-task.ts` lines ~735-753).

## Current architecture anchors

### CLI wiring (`src/presentation/cli.ts`)
The `plan` command is defined around line 239 with options including `--scan-count <n>`, `--mode`, `--transport`, `--dry-run`, `--print-prompt`, etc. The action delegates to `getApp().planTask()` via `createPlanCommandAction` in `src/presentation/cli-command-actions.ts` (line ~411).

The new `--deep <n>` option slots in alongside `--scan-count`.

### Application layer (`src/application/plan-task.ts`)
- `createPlanTask()` orchestrates the scan loop: for each scan index it builds a prompt via `buildPlanScanPrompt()`, runs a worker, and applies output via `applyPlannerOutput()` → `insertPlannerTodos()`.
- Convergence tracking (`PlanConvergenceOutcome`) breaks the loop early when the worker returns nothing new.
- Template variables already include `scanCount`, `existingTodoCount`, `hasExistingTodos`, `task`, `source`, etc.

### Domain layer (`src/domain/planner.ts`)
- `parsePlannerOutput()` — extracts `- [ ]` lines from worker stdout.
- `insertPlannerTodos()` — validates additive-only rules, deduplicates, appends flat TODOs after the last checkbox line **or** at end-of-document.
- `insertSubitems()` — inserts child list items beneath a specific `Task` node with correct two-space depth indentation. **This is the function that will insert deep-pass output.**
- `computeChildIndent()` — derives `"  "` × depth from a parent line's leading whitespace.

### Domain model (`src/domain/parser.ts`)
`Task` already supports nested hierarchies:
- `depth: number` — nesting depth (0 = top-level).
- `children: Task[]` — nested checkbox child tasks.
- `subItems: SubItem[]` — nested non-checkbox list items.

So the parser can re-parse the file between deep passes and discover which tasks still lack children.

### Infrastructure (`src/infrastructure/planner-io.ts`)
`applyPlannerOutput(task, plannerOutput)` reads/writes the task file and calls `insertSubitems()`. This is used for per-task sub-item insertion today and can be reused or adapted for the deep pass.

### Default plan template (`src/domain/defaults.ts`, line ~211)
The current template asks for flat `- [ ]` output only. A **new deep-plan template** is needed for "add next layer" passes, scoped to a single parent task.


## Design notes

- Each deep pass operates on the **freshly re-parsed** file so it sees previously inserted children — this prevents duplicate child generation across passes.
- Clean worker sessions per task per pass avoids cross-task contamination (same rationale as scan-level isolation).
- The `--scan-count` and `--deep` options are orthogonal: scans control top-level coverage breadth; deep controls hierarchy depth.
- When `--deep 0` (default), behavior is identical to today — no migration risk.

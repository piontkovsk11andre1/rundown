## Feature: Extensible tool prefixes via `.rundown/tools/`

<details>
<summary>Context</summary>

### Current prefix system

Rundown recognizes built-in prefixes on TODO items that change execution behavior:

| Prefix | Intent | Behavior |
|--------|--------|----------|
| `verify:` / `confirm:` / `check:` | `verify-only` | Skip execution, run only the verification phase |
| `memory:` / `memorize:` / `remember:` / `inventory:` | `memory-capture` | Execute task, persist worker output to `.rundown/<file>.memory.md` |
| `cli:` | `inline-cli` | Execute shell command directly (no worker), detected at parse time |
| _(none)_ | `execute-and-verify` | Standard worker execution + verify/repair loop |

Prefixes are classified in `src/domain/task-intent.ts` via `classifyTaskIntent()`, which returns a `TaskIntentDecision` containing `intent`, `normalizedTaskText`, and optional `memoryCapturePrefix`. The `inline-cli` intent is detected separately at parse time in `src/domain/parser.ts`.

Dispatch happens in `src/application/task-execution-dispatch.ts`, which routes to one of four branches based on intent: verify-only, inline-cli, memory-capture, or standard execute-and-verify.

### Desired behavior

Allow users to define **custom tool prefixes** by placing prompt template files in `.rundown/tools/`:

```
.rundown/tools/
  post-on-gitea.md      ← tool definition (prompt template)
  summarize.md
  create-migration.md
```

A task using a tool prefix:

```markdown
- [ ] post-on-gitea: Please post an issue about broken auth flow
```

Execution flow:
1. Parse TODO → detect prefix `post-on-gitea` → find `.rundown/tools/post-on-gitea.md`
2. Load tool template, render with `{{payload}}` = `"Please post an issue about broken auth flow"` plus standard template vars (`{{file}}`, `{{context}}`, `{{source}}`, etc.)
3. Send rendered prompt to worker
4. Parse worker output for `- [ ]` TODO lines (same contract as planner output)
5. Insert returned TODOs as children under the tool task (or after it)
6. Execute the first inserted TODO (continue the normal run loop)
7. Check off the tool task when all inserted children are done (or immediately if no TODOs returned)

This makes tools act like **inline plan expansions** — they are prompts that dynamically produce work items.

### Existing infrastructure to leverage

| Component | How it helps |
|-----------|-------------|
| `classifyTaskIntent()` in `task-intent.ts` | Add a new `"tool-expansion"` intent with the resolved tool name |
| `dispatchTaskExecution()` in `task-execution-dispatch.ts` | Add a tool-expansion branch that loads tool template, runs worker, inserts results |
| `parsePlannerOutput()` in `planner.ts` | Reuse to parse `- [ ]` lines from tool worker output |
| `insertSubitems()` in `planner.ts` | Reuse to insert TODO children under the tool task with proper indentation |
| `loadProjectTemplatesFromPorts()` in `project-templates.ts` | Extend or parallel pattern for loading tool templates from `.rundown/tools/` |
| `renderTemplate()` in `template.ts` | Reuse for `{{placeholder}}` interpolation in tool templates |
| `ConfigDirResult` / config discovery | `.rundown/tools/` lives inside the already-discovered config directory |
| SubItem directive system in `parser.ts` | Could carry tool-specific configuration (e.g., `- profile: fast` under a tool task) |
| Deep plan child insertion | Proven pattern for inserting generated TODOs as children of a parent task |

### Example tool template (`.rundown/tools/post-on-gitea.md`)

```markdown
You are a project management assistant.

The user wants to post an issue on Gitea.

**Request:** {{payload}}

**Source file:** {{file}}
**Context:**
{{context}}

Based on the request, determine what needs to happen and return a TODO list.
Each item must be an unchecked Markdown checkbox: `- [ ] description`.

Return ONLY the TODO items, nothing else.
```

### How tool resolution differs from other prefixes

Built-in prefixes are detected via hardcoded regex patterns. Tool prefixes are **dynamic** — they exist only when a matching `.md` file is present in `.rundown/tools/`. Resolution order:

1. Check built-in prefixes (verify, memory, cli) — these always win
2. Extract candidate prefix: text before the first `:`
3. Check if `.rundown/tools/<candidate>.md` exists
4. If yes → `tool-expansion` intent with tool name and payload
5. If no → standard `execute-and-verify` intent (colon is treated as regular text)

This means tool names cannot collide with built-in prefixes, and unknown prefixes degrade gracefully to normal tasks.

Variables should be visible to prompts, so we probably want to dump them in main rundown prompts.

Also we want to pass variables as ENVs to cli-blocks and cli:. It should be multiplatform.

## Context

Template variables (`--var key=value`, `--vars-file`) are currently:
- Merged in `run-task-execution.ts` → `extraTemplateVars` (file vars + CLI vars, CLI wins).
- Interpolated into prompt templates via `renderTemplate()` in `prepare-task-prompts.ts`.
- Invisible to shell processes spawned by CLI blocks (`expandCliBlocks`) and inline CLI tasks (`cli:` prefix).

Three process-spawning paths exist, none of which inject user-defined variables as environment variables:

| Spawner | File | Used by |
|---|---|---|
| `createCliBlockExecutor` | `src/infrastructure/cli-block-executor.ts` | `expandCliBlocks` — fenced ` ```cli ` blocks in source & templates |
| `executeInlineCli` | `src/infrastructure/inline-cli.ts` | `cli:` prefix tasks dispatched via `task-execution-dispatch.ts` |
| `runWorker` / `executeCommand` | `src/infrastructure/runner.ts` | Main agent worker processes |

Lifecycle hooks (`run-lifecycle.ts`) already set `RUNDOWN_TASK`, `RUNDOWN_FILE`, etc. via `process.env` spread — a precedent for the pattern we need.

Prompts never show the user-defined variables to the agent; the agent only sees them via `{{placeholder}}` expansion.

## Goals

1. **Prompt visibility**: Add a dedicated section to the shared template prefix that dumps all extra template variables so the agent can reference them by name/value.
2. **CLI block environment**: Pass extra template variables as `RUNDOWN_VAR_<NAME>=<value>` environment variables to every child process spawned by `createCliBlockExecutor`.
3. **Inline CLI environment**: Same `RUNDOWN_VAR_` prefix for `executeInlineCli`.
4. **Worker environment**: Same `RUNDOWN_VAR_` prefix for `runWorker` / `executeCommand` in `runner.ts`.
5. **Cross-platform**: Use `cross-spawn` (already a dependency) or `shell: true` (already used) — env vars work identically on Windows/macOS/Linux via Node `spawn({ env })`.

## Design decisions

### Environment variable naming

Prefix: `RUNDOWN_VAR_` + uppercase key. Example: `--var db_host=localhost` → `RUNDOWN_VAR_DB_HOST=localhost`.

Keys are already validated against `/^[A-Za-z_]\w*$/` in `parseCliTemplateVars`, so uppercasing is always safe for env var names.

### Template section for variables

Add a new `DEFAULT_TEMPLATE_VARS_SECTION` to `src/domain/defaults.ts` that renders a `## Variables` block. When no extra vars are present, the section should be omitted or show `(none)` to avoid noise.

The section uses a `{{userVariables}}` placeholder. The value is built at `renderTemplate` call sites — a simple key=value dump. This keeps template rendering stateless.

### Environment builder utility

A single pure function `buildRundownVarEnv(vars: ExtraTemplateVars): Record<string, string>` in `src/domain/template-vars.ts` converts extra vars to env entries. All spawners call it.

### Prompt variable dump utility

A single pure function `formatTemplateVarsForPrompt(vars: ExtraTemplateVars): string` in `src/domain/template-vars.ts` produces the human-readable text block (or `"(none)"`).

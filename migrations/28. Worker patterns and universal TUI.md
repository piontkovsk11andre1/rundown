# Feature: Pattern-based worker invocation and universal TUI

<details>
<summary>Context</summary>

### Current worker system

Today, worker invocation uses two separate concepts that must be coordinated:

1. **`--worker [command...]`** ŌĆö A command array like `--worker opencode run` or `--worker claude -p`.
2. **`--transport file|arg`** ŌĆö Controls *how* the prompt is appended:
   - `file`: writes prompt to a `.rundown` temp file and appends the **path** as the last argument.
   - `arg`: appends the **prompt text** directly as the last argument.

The runner (`src/infrastructure/runner.ts`) then has special-case logic:
- `isOpenCodeCommand()` detects opencode executables and routes to `buildOpenCodeArgs()` which handles `run` vs TUI sub-commands, `--file` flags, `--prompt=` argument shaping, and bootstrap prompts.
- Generic workers just get the prompt/path blindly appended as the final argument.

Additionally, the `normalizeWorkerOptionArgs()` hack in `cli-argv.ts` exists solely to work around Commander's inability to parse `--worker claude -p` (where `-p` looks like a rundown flag).

### TUI mode on Windows

The `executeCommand()` function in `runner.ts` special-cases Windows TUI mode by launching the worker in a new console window via `cmd /c start /wait "" <cmd> <args>`. This was added to avoid input-buffer issues but creates a disjointed UX ŌĆö the worker opens in a separate window outside the user's terminal.

### Problems

- Two options (`--worker` + `--transport`) for what is conceptually one thing: "how to invoke the worker".
- OpenCode-specific code in the generic runner that doesn't scale to other workers.
- The `normalizeWorkerOptionArgs` workaround for Commander parsing conflicts.
- Windows TUI opening in a separate window is surprising and inconsistent with non-Windows behavior.
- Adding a new worker type (Claude, Aider, custom scripts) means either adding more special cases or forcing users to remember the right `--transport` value.

</details>

## Proposed design

### Always-write prompt file

The runner **always** writes the rendered prompt to a `.rundown` runtime file, regardless of which substitution variables the pattern uses. This is the single source of truth for the full task prompt.

Three substitution variables are always available:
- **`$file`** ŌĆö the path to the prompt file on disk (containing the full rendered prompt).
- **`$bootstrap`** ŌĆö a short **bootstrap instruction** text, e.g. `"Read the task prompt file at .rundown/runs/.../prompt.md and follow the instructions."` This is NOT the full prompt ŌĆö it's a pointer that tells the worker where to find the real content.
- **`$prompt`** ŌĆö reserved for future use. Will contain the full rendered prompt text when needed (e.g. for workers that accept inline prompts without file support).

This mirrors what OpenCode TUI mode already does today: the full prompt is staged in a file, and the worker receives a bootstrap message telling it to read that file first.

### Pattern-based `--worker`

Replace `--worker [command...]` + `--transport file|arg` with a single **pattern string** containing substitution variables:

```bash
# Bootstrap text tells Claude to read the prompt file
rundown plan '.\tasks.md' --worker 'claude -p $bootstrap'

# File path passed directly (worker reads the file itself)
rundown run '.\tasks.md' --worker 'opencode run --file $file'

# OpenCode TUI with bootstrap instruction
rundown run '.\tasks.md' --mode tui --worker 'opencode --prompt=$bootstrap'

# No substitution variable ŌåÆ append $file as last arg (backward compat default)
rundown run '.\tasks.md' --worker 'my-agent'
```

If no substitution variable (`$bootstrap`, `$file`) appears in the pattern, `$file` is appended as the last argument (preserving current default behavior).

The pattern is a **single string** (not a variadic argument). This eliminates the Commander parsing conflict entirely ŌĆö no need for `normalizeWorkerOptionArgs`.

### Unified prompt delivery

Because the prompt file is always created and `$bootstrap` is always a pointer to it, every worker has the same contract:

| Pattern | What worker receives | How it gets the full prompt |
|---------|--------------------|-----------------------------|
| `claude -p $bootstrap` | Bootstrap text via `-p` | Claude reads the file path mentioned in the bootstrap text |
| `opencode run --file $file` | File path via `--file` | OpenCode reads the file directly |
| `opencode --prompt=$bootstrap` | Bootstrap text via `--prompt=` | OpenCode reads the file path mentioned in the bootstrap text |
| `my-agent` | File path as last arg | Agent reads the file directly |

No worker-specific code in the runner. The bootstrap text generation is a single, universal function.

### Universal TUI

TUI mode should work identically on all platforms: spawn the worker with `stdio: "inherit"` in the current terminal. Remove the Windows `cmd /c start /wait` code path.

This makes `--mode tui` available for any worker on any OS ŌĆö the user gets an interactive session in their own terminal.

### Remove OpenCode special-casing

With patterns and the always-file approach, users express OpenCode-specific arguments directly in the pattern. The runner no longer needs `isOpenCodeCommand()`, `buildOpenCodeArgs()`, `buildOpenCodeRunBootstrapPrompt()`, `buildOpenCodeTuiPromptArg()`, or `buildOpenCodeTuiBootstrapPrompt()`.

### `--transport` removal

`--transport` is fully removed. The pattern itself determines how the prompt is delivered. There is no transition period ŌĆö the concept is replaced entirely by `$file` / `$bootstrap` in patterns.

## Architecture anchors

### Domain ports

- `src/domain/ports/worker-executor-port.ts`: `PromptTransport = "file" | "arg"` ŌĆö **remove entirely**. Transport is no longer a domain concept; it's derived from the pattern (`$file` vs `$bootstrap`).
- `src/domain/ports/process-runner.ts`: `ProcessRunMode = "wait" | "tui" | "detached"` ŌĆö unchanged.

### Infrastructure

- `src/infrastructure/runner.ts` (430 lines):
  - `runWorker()` ŌĆö always create the prompt file (already does for `file` transport; now unconditional).
  - `buildWorkerArgs()` (line ~158) ŌĆö replace with `expandWorkerPattern()` that substitutes `$file`, `$bootstrap`, and (future) `$prompt` into the parsed pattern tokens.
  - Add `buildBootstrapPrompt(promptFilePath, cwd)` ŌĆö generates the universal bootstrap instruction text (e.g. `"Read the task prompt file at <relative-path> and follow the instructions."`). This replaces all OpenCode-specific bootstrap functions.
  - `isOpenCodeCommand()` (line ~187) ŌĆö **remove**.
  - `buildOpenCodeArgs()` (line ~198) ŌĆö **remove**.
  - `buildOpenCodeRunBootstrapPrompt()`, `buildOpenCodeTuiBootstrapPrompt()`, `buildOpenCodeTuiPromptArg()`, `hasOpenCodeThinkingArg()` ŌĆö **remove**.
  - `executeCommand()` TUI branch (line ~276ŌĆō305) ŌĆö remove Windows `cmd /c start /wait` path, use `stdio: "inherit"` universally.

### Presentation

- `src/presentation/cli.ts`:
  - All commands currently declare `--worker [command...]` (variadic) ŌĆö change to `--worker <pattern>` (single string).
  - **Remove** `--transport` option from all commands entirely.
  - `configureRunLikeCommandOptions()` (line ~610) ŌĆö update `--worker`, remove `--transport`.
  - Remove per-command `--transport` declarations on `plan`, `discuss`, `research`, `reverify`, `make`, `do`.
- `src/presentation/cli-argv.ts`:
  - `normalizeWorkerOptionArgs()` ŌĆö **remove** (no longer needed when `--worker` is a single string).
  - `KNOWN_CLI_OPTION_TOKENS`, `isKnownCliOptionToken()` ŌĆö **remove** if only used by the above.
- `src/presentation/cli-command-actions.ts`:
  - `resolveWorkerCommand()` (line ~78) ŌĆö simplify: accepts a pattern string, returns parsed pattern (command tokens + substitution variables).

### Application layer

All use cases that accept `workerCommand: string[]` and `transport: PromptTransport` must be updated to accept a parsed pattern representation instead. Key files:
- `src/application/run-task.ts`
- `src/application/plan-task.ts`
- `src/application/research-task.ts`
- `src/application/discuss-task.ts`
- `src/application/reverify-task.ts`

### `.rundown/config.json` and `vars.json`

The default worker pattern could be stored in config:

```json
{
  "worker": "opencode run --file $file"
}
```

This is out of scope for the first iteration but should be considered in the design.

## Background

The profile system currently allows configuring the worker command (model, tools, flags) at several levels:

- **Global defaults** (`config.json` ‚Üí `defaults`) ‚Äî applies to all commands
- **Command-specific overrides** (`config.json` ‚Üí `commands.{run|plan|discuss|research|reverify}`) ‚Äî targets a specific top-level command
- **Named profiles** (`config.json` ‚Üí `profiles.{name}`) ‚Äî reusable profile fragments
- **File-level frontmatter** (`profile: profileName` in YAML frontmatter) ‚Äî applies to all tasks in a file
- **Directive-level profile** (`profile: profileName` as a sub-item of a parent list item) ‚Äî applies to tasks nested under that directive

Each `WorkerProfile` currently supports two fields: `worker` (the command tokens) and `workerArgs` (extra arguments). These accumulate additively in resolution order, so a directive profile can append `--model claude-opus-4` on top of defaults.

## The Gap: Prefix-based Tasks Don't Have Their Own Profile Slot

Tasks with the `verify:` (or `confirm:`, `check:`) prefix are classified at parse time as `task.intent = "verify-only"`. They are not a separate command type ‚Äî they are standard checkboxes that skip the execution phase and proceed directly to the verification phase.

Because they are not a command type (unlike `run`, `plan`, `discuss`, `research`, `reverify`), there is no `commands.verify` slot in `WorkerConfig`. The verification phase for these tasks uses the worker that was resolved for the parent `run` invocation. There is no mechanism today to specify a different model, tools, or flags specifically for `verify:` tasks.

The same limitation applies to **tool-expansion prefix tasks** (e.g., `- post-on-gitea: payload`) and **memory-capture prefix tasks** (e.g., `- memory: note text`). All intent-classified prefix tasks inherit the enclosing context's worker; none can carry their own profile configuration.

### Scope of prefix types affected

| Prefix pattern | Intent classification | Has own profile slot? |
|---|---|---|
| `verify:` / `confirm:` / `check:` | `verify-only` | No |
| `memory:` / `memorize:` / `remember:` / `inventory:` | `memory-capture` | No |
| `<custom-tool>:` (tool expansion) | `tool-expansion` | No |
| `cli:` | inline-cli (shell, no worker) | N/A |

## What "Including the Tools" Means

Tool expansion tasks resolve a named tool template from `.rundown/tools/` and render a task body from it. The rendered template is then passed to the currently-resolved worker. Today, `WorkerProfile` has no concept of per-tool worker overrides ‚Äî every tool expansion uses the same worker as the enclosing `run` context.

Being able to set a profile "for the tools" means: when a task with a specific tool prefix is dispatched, a different `worker` / `workerArgs` may apply ‚Äî for example, routing `post-on-gitea:` tasks to a lightweight model, or giving a research tool access to web-search flags not present in the default worker.

## Design Constraints and Considerations

### Where to express the configuration

There are two natural places to extend:

1. **`config.json` `commands` map** ‚Äî extend `WorkerConfigCommandName` to include intent-based keys such as `verify`, `memory`, and per-tool names (e.g., `tools.post-on-gitea`). Allows central, file-agnostic defaults.

2. **Inline directive sub-item** ‚Äî allow a task to carry `profile: X` as a child bullet, the same way parent directives already do. Resolution logic already handles this for regular tasks; extending it to intent-classified tasks would give per-task granularity.

Both may be needed. A `config.json` slot gives project-wide defaults; a directive sub-item allows per-task overrides.

### Verification phase vs. execution phase

For `verify:` tasks the execution phase is entirely skipped. The relevant worker is the one passed to the verifier (`task-verification-port`). Changing the profile for `verify:` tasks means passing a different worker command when constructing the verification invocation ‚Äî it must not fall back to the enclosing `run` worker silently.

### Profile resolution order

For parity with regular tasks, the resolution precedence should remain consistent:

1. CLI `--worker` flag (overrides all)
2. `config.json` `defaults`
3. `config.json` `commands.{intent-or-tool-name}`
4. File frontmatter profile
5. Directive-level profile
6. Task-level `profile:` sub-item (if supported)
7. Runtime fallback worker

### Backwards compatibility

Tasks that currently rely on the inherited worker must continue to work unchanged. Adding new config keys is additive; a missing key means "inherit from enclosing context" as today.

## Acceptance Criteria Outline

- A `verify:` prefixed task can use a different worker (model, flags) than the surrounding `run` context.
- Tool-expansion tasks can have a per-tool worker configuration.
- The configuration can be expressed in `config.json` (project-wide) and overridden per-directive or per-task inline.
- When no intent-specific or tool-specific profile is set, behaviour is identical to today (no regression).
- Profile resolution for prefix tasks follows the same precedence order as for regular tasks.

## Out of Scope

- Changing how `cli:` inline tasks work (they bypass the worker entirely by design).
- Altering the `WorkerProfile` field structure (adding new profile fields beyond `worker`/`workerArgs`) is a separate concern.
- UI or CLI flags for selecting intent profiles at runtime (profiles remain a config/document-level concern).

# Decompose `run-task-execution.ts`

## Goal

Reduce `src/application/run-task-execution.ts` (currently ~1,460 lines) to a concise orchestrator of ~400–500 lines by extracting the repeated execution paths and shared setup/teardown logic into focused internal modules. The file is the single largest module in the codebase after step 1 and 2 refactorings, and its main structural problem is now internal duplication, not tangled concerns.

## Background

Steps 1 and 2 ([migrations/1. Decompose run-task.md](1.%20Decompose%20run-task.md), [migrations/2. Converge shared modules.md](2.%20Converge%20shared%20modules.md)) extracted cross-cutting concerns (trace, git, templates, CLI block handlers, task context, etc.) into shared modules and converged sibling use cases onto them. What remains in `run-task-execution.ts` is the orchestrator for `rundown run` — the core command.

The orchestrator's structure is:

1. **Options destructuring and early returns** (~60 lines) — flag validation, trace-only path.
2. **Run behavior + template vars + worker config resolution** (~40 lines).
3. **Mutable run state initialization** (~50 lines) — artifact context, trace session, pending reset events.
4. **Trace enrichment closure** (`runTraceEnrichment`, ~70 lines) — renders trace template, invokes worker, queues analysis summary.
5. **Run lifecycle closures** (`failRun`, `finalizeArtifacts`, `finishRun`, `resetArtifacts`, ~50 lines).
6. **Source resolution, locking, git pre-checks, redo reset** (~80 lines).
7. **Main loop — 4 execution paths** (~800 lines, the bulk of the file):
   - Only-verify path (~100 lines)
   - Inline CLI path (~120 lines)
   - Rundown task path (~170 lines)
   - Normal worker path (~100 lines)
   - Each path includes: dry-run/print-prompt early returns, execution, verification, checkbox check, commit/hooks, artifact finalization, loop control.
   - **Post-execution block repeated 4×** — verification call, failure handling, `checkTaskUsingFileSystem`, `afterTaskComplete`, `finishRun`, `tasksCompleted++`, `resetArtifacts`.
8. **Finally block** (~50 lines) — post-run reset, deferred commit, lock release, error artifact finalization.
9. **Helper functions outside the closure** (~100 lines) — `runVerification`, `getAutomationWorkerCommand`, `isOpenCodeWorkerCommand`.

## Primary problem: repeated post-execution sequence

The 4 execution paths (only-verify, inline-CLI, rundown-task, normal-worker) diverge only in **how the task is executed** (which executor is called, what arguments are passed). Everything before execution (template expansion, prompt rendering) and everything after execution (verification, checkbox, hooks, artifact finalization, loop control) is structurally identical. The current code repeats the post-execution block 4 times (~60 lines each = ~240 lines of near-identical code).

A unified flow would:
1. Prepare the execution context (source expansion, template rendering, CLI block expansion).
2. Dispatch to the appropriate executor based on task type.
3. Run the shared post-execution sequence once.

## Identified extraction candidates

### 1. Prompt preparation pipeline

The section from source CLI block expansion through template rendering and verification prompt expansion (~150 lines, roughly L650–L810 of the current file) can be extracted into an internal function or module. It takes a task, source, templates, template vars, and CLI expansion options, and returns:
- Expanded source and expanded task
- Rendered task prompt (with CLI blocks expanded)
- Rendered verification prompt (with CLI blocks expanded)
- CLI block count for dry-run reporting

This is a pure data transformation (with side effects limited to CLI block execution). Extracting it would simplify the main loop body and make the prompt pipeline independently testable.

### 2. Dry-run / print-prompt dispatch

The dry-run and print-prompt early returns (~80 lines, roughly L810–L890) are a self-contained decision tree based on `(printPrompt|dryRun) × (onlyVerify|inlineCli|rundownTask|normal)`. Extracting this into a function that returns either an exit code (if an early return applies) or `null` (to continue execution) would remove 8 conditional blocks from the main loop.

### 3. Task execution dispatch

The actual execution calls are:
- `runVerification(...)` for only-verify
- `workerExecutor.executeInlineCli(...)` for inline CLI
- `workerExecutor.executeRundownTask(...)` for rundown tasks 
- `workerExecutor.runWorker(...)` for normal tasks

Each returns `{ exitCode, stdout, stderr }` (or a verification result). A dispatcher function could normalize these into a common result shape, letting the main loop handle one code path after execution.

### 4. Post-execution sequence (the biggest win)

After execution, all 4 paths do the same thing:
1. If verification is enabled: call `runVerification(...)`, handle CLI block failure, check result.
2. On verification failure: emit error, call `afterTaskFailed`, return `failRun(2, ...)`.
3. On success: `checkTaskUsingFileSystem`, emit success, optionally defer commit, call `afterTaskComplete`, call `finishRun`, increment counter, break or continue loop.

Extracting this into a `completeTaskExecution(...)` function would eliminate ~240 lines of duplication (4 copies → 1).

### 5. Trace enrichment

`runTraceEnrichment` (~70 lines) is a closure that captures `artifactContext`, `traceRunSession`, `traceEnrichmentContext`, and dependencies. After the trace session was extracted to `trace-run-session.ts` in step 1, this function's remaining logic is: render trace template → invoke worker → parse analysis summary → queue event. It could become an exported function in `trace-run-session.ts` or a new `trace-enrichment.ts`, reducing the closure surface of the main orchestrator.

### 6. Pre-execution setup

Source resolution, file locking, git pre-checks, and redo reset (~80 lines) are a sequential setup phase. They could be extracted into a `prepareRunContext(...)` function returning the resolved files and lock handles, or they can remain inline since they run once and are already linear.

## Approach

The recommended extraction order (by impact):

1. **Post-execution sequence** → `completeTaskIteration` or similar internal function. This is the highest-impact change (~240 lines of duplication → ~60 lines + one function call per path).

2. **Prompt preparation pipeline** → `prepareTaskPrompts` internal function. Reduces the main loop body by ~150 lines.

3. **Dry-run / print-prompt dispatch** → `handleDryRunOrPrintPrompt` internal function. Removes the 8-way conditional tree from the main loop.

4. **Task execution dispatch** → normalize all 4 execution calls to return a common `{ exitCode, stdout, stderr }` shape. The main loop becomes: prepare → dispatch → post-execute.

5. **Trace enrichment** → move to a separate module or into trace-run-session. Lower priority since it's already a cohesive closure.

After these extractions, the main loop body would look approximately like:

```
while (true) {
  select next task
  classify intent, resolve worker
  prepare prompts (source expansion, template rendering)
  handle dry-run/print-prompt → early return if applicable
  create artifact context + trace session
  dispatch execution (only-verify | inline-cli | rundown | worker)
  complete task iteration (verify → check → hooks → finalize)
  continue or break
}
```

## Constraints

- **Public API stability**: `createRunTaskExecution`, `RunTaskDependencies`, `RunTaskOptions`, and exported helpers (`getAutomationWorkerCommand`, `isOpenCodeWorkerCommand`, re-exports) must remain at the same import paths.
- **Exit code contract**: Returns `0` (success), `1` (execution/config error), `2` (verification failure), `3` (no tasks found). Must be preserved.
- **Test coverage**: `__tests__/application/run-task.test.ts` (~3,500 lines) covers all paths. Extracted internal functions do not need separate test files if they remain private to the module — the existing integration-level tests cover them. If any function is exported for reuse, it should get its own tests.
- **No behavioral changes**: Pure structural refactoring. No feature additions, no bug fix side effects.
- **Extracted functions stay in `run-task-execution.ts` or become sibling modules**: The goal is readability, not a proliferation of tiny files. Functions used only by the run-task orchestrator can remain as module-private functions in the same file or in a closely related `run-task-*.ts` file.

# Plan: Unified Tool Prefix System

## TL;DR

Replace the hardcoded intent system (`verify-only`, `memory-capture`, `tool-expansion`) with a unified tool resolution pipeline. Every `prefix:` on a task line resolves to a **tool** — either a `.md` template, a `.js` handler, or a package-level built-in. Tools declare themselves as **modifiers** (context changers like `profile`) or **handlers** (terminal executors like `verify`, `memory`, `include`). Multiple prefixes compose via `, ` or `; ` delimiters: `- [ ] profile: fast, verify: tests pass`.

---

## Design Decisions

- **`cli:` stays special-cased** — it's a parser-level flag (`isInlineCli`) that bypasses the worker entirely. Not unified.
- **`profile:` becomes a modifier tool** — instead of a non-checkbox directive, it becomes a composable prefix modifier.
- **JS tools loaded via `import()`** — in-process, fast, structured return values.
- **Project tools override built-ins** — `.rundown/tools/verify.js` takes priority over built-in `verify`.
- **Include copies to run folder** — cloned to artifacts/run directory, executed from there, original file untouched.
- **Include auto-checks** — when all tasks in cloned file complete, include task is auto-checked (no separate verify).
- **Tool behavior config** — YAML frontmatter in `.md` files, `config.json` can override. Layered.
- **Modifier composition** — `- [ ] profile: fast, verify: tests pass` splits at `, ` or `; ` boundaries when the next segment starts with a known tool name.

## Phase 1: Domain Model — Tool Resolution Port & Types

**Goal**: Define the unified tool abstraction that replaces `TaskIntent`.

### Step 1.1: Define core tool types

Create `src/domain/ports/tool-handler-port.ts`:

```
ToolKind = "modifier" | "handler"

ToolHandlerResult:
  - skipExecution?: boolean
  - shouldVerify?: boolean
  - childTasks?: string[]       // TODOs to insert
  - contextModifications?: {    // For modifiers
      profile?: string
      workerArgs?: string[]
      templateVars?: Record<string, string>
    }
  - exitCode?: number

ToolHandlerContext:
  - task, payload, source, contextBefore
  - workerExecutor, fileSystem, templateVars
  - artifactContext, emit, configDir
  - resolvedWorkerConfig (may be modified by prior modifiers)

ToolDefinition (extend existing):
  - name: string
  - kind: ToolKind
  - templatePath?: string        // .md tools
  - handlerPath?: string         // .js tools
  - template?: string            // raw .md content
  - handler?: (ctx: ToolHandlerContext) => Promise<ToolHandlerResult>
  - frontmatter?: ToolFrontmatter
```

### Step 1.2: Define tool frontmatter schema

For `.md` tools:
```yaml
---
kind: handler          # "modifier" | "handler", default "handler"
skipExecution: false    # Skip worker execution phase
shouldVerify: false     # Run verification after execution
insertChildren: true    # Parse output for child TODOs
profile: null           # Default profile override
---
```

### Step 1.3: Define prefix chain parser

Create `src/domain/prefix-chain.ts`:
- `parsePrefixChain(taskText, toolResolver)` → `PrefixChain`
- `PrefixChain = { modifiers: Array<{tool, payload}>, handler?: {tool, payload}, remainingText: string }`
- Scanning algorithm:
  1. Split at `, ` or `; ` boundaries where next segment starts with `knownToolName:`
  2. Classify each segment: if tool.kind === "modifier" → accumulate; if "handler" → terminal
  3. Text after last tool prefix = payload of that tool (or plain task text if no handler)

**Files to create/modify:**
- Create `src/domain/ports/tool-handler-port.ts`
- Create `src/domain/prefix-chain.ts`
- Modify `src/domain/ports/tool-resolver-port.ts` — extend `ToolDefinition`

## Phase 2: Tool Resolver — Unified Resolution Cascade

**Goal**: Resolve any prefix to a tool from project dir → built-ins.

### Step 2.1: Extend tool resolver adapter for `.js` files

Modify `src/infrastructure/adapters/tool-resolver-adapter.ts`:
- Look for `<name>.js` in addition to `<name>.md`
- `.js` priority over `.md` if both exist
- For `.js`: dynamic `import()` the file, extract default export as handler
- For `.md`: parse YAML frontmatter for behavioral flags, keep template

### Step 2.2: Create built-in tool registry

Create `src/domain/builtin-tools/` directory with:
- `verify.ts` — handler: skipExecution=true, shouldVerify=true (uses current verify template)
- `memory.ts` — handler: execute + persist to memory files (extract from current dispatch logic)
- `include.ts` — handler: resolve file, copy to run dir, execute tasks from clone, auto-check
- `profile.ts` — modifier: sets profile in context, passes through to next tool
- `index.ts` — registry map: `{ verify: ..., confirm: ..., check: ..., memory: ..., memorize: ..., remember: ..., inventory: ..., include: ..., profile: ... }`

### Step 2.3: Resolution cascade

In the resolver:
1. Try `.rundown/tools/<name>.js` (project JS tool)
2. Try `.rundown/tools/<name>.md` (project MD template)
3. Try built-in registry (package-level)
4. Return undefined → prefix is not a tool (becomes part of task text)

**Files to create/modify:**
- Modify `src/infrastructure/adapters/tool-resolver-adapter.ts`
- Create `src/domain/builtin-tools/verify.ts`
- Create `src/domain/builtin-tools/memory.ts`
- Create `src/domain/builtin-tools/include.ts`
- Create `src/domain/builtin-tools/profile.ts`
- Create `src/domain/builtin-tools/index.ts`

## Phase 3: Execution Pipeline — Replace Intent Dispatch

**Goal**: Replace the if/else intent chain in `task-execution-dispatch.ts` with tool handler calls.

### Step 3.1: Create tool execution orchestrator

Create `src/application/tool-execution.ts`:
- `executeToolChain(chain: PrefixChain, context: ToolHandlerContext)`:
  1. Apply modifiers left-to-right: each modifier's `contextModifications` patches the context
  2. Call handler's `execute(context)` or fall through to default execute-and-verify
  3. Handle `ToolHandlerResult`: insertChildren, shouldVerify, exitCode, etc.

### Step 3.2: Refactor task-execution-dispatch.ts

Replace the current branching:
```
if (isToolExpansionTask) { ... }
else if (isMemoryCaptureTask) { ... }
else if (onlyVerify) { ... }
else { /* execute-and-verify */ }
```

With:
```
const chain = parsePrefixChain(task.text, toolResolver);
if (chain.handler || chain.modifiers.length > 0) {
  return executeToolChain(chain, context);
} else {
  // Default execute-and-verify (unchanged)
}
```

### Step 3.3: Migrate verify logic

Extract current verify-only dispatch into `builtin-tools/verify.ts` handler:
- Returns `{ skipExecution: true, shouldVerify: true }`
- The orchestrator uses this to skip execution and proceed to verification

### Step 3.4: Migrate memory-capture logic

Extract `persistMemoryCaptureOutput` into `builtin-tools/memory.ts` handler:
- Runs worker execution
- Persists output via memoryWriter
- Returns `{ shouldVerify: true }`

### Step 3.5: Deprecate task-intent.ts

- `classifyTaskIntent()` replaced by `parsePrefixChain()` + tool resolution
- `TaskIntent` type replaced by `ToolDefinition.kind`
- `TaskIntentDecision` replaced by `PrefixChain`
- Keep `task-intent.ts` temporarily with deprecation, remove in follow-up

**Files to create/modify:**
- Create `src/application/tool-execution.ts`
- Modify `src/application/task-execution-dispatch.ts` — replace intent branching
- Modify `src/application/iteration-mode.ts` — use PrefixChain instead of TaskIntentDecision
- Modify `src/application/resolve-worker.ts` — accept modifier context
- Deprecate `src/domain/task-intent.ts`

## Phase 4: Include Tool — Markdown File Composition

**Goal**: Implement `- [ ] include: path/to/file.md` as a built-in tool.

### Step 4.1: Implement include handler

In `src/domain/builtin-tools/include.ts`:
1. Resolve payload path relative to task's source file
2. Read the target markdown file
3. Copy to run/artifacts directory: `<artifactsDir>/<runId>/includes/<filename>`
4. Parse the copy for tasks using existing parser
5. Use the run-task-execution loop to execute tasks from the copy
6. When all tasks complete → return `{ skipExecution: true, shouldVerify: false }` (auto-check)
7. If any task fails → propagate failure

### Step 4.2: Wire include into the execution loop

The include handler needs access to the task execution engine (recursive). This requires:
- Passing the `runTaskExecution` function (or a subset) into tools via `ToolHandlerContext`
- Or: include returns `{ childFile: "<path>" }` and the orchestrator handles the recursion

Recommend: orchestrator-level handling — include tool returns the resolved file path, orchestrator manages the sub-execution. This avoids circular dependencies.

**Files to create/modify:**
- Create/complete `src/domain/builtin-tools/include.ts`
- Modify `src/application/tool-execution.ts` — handle `childFile` result
- Modify `src/application/run-task-execution.ts` — support sub-file execution

## Phase 5: Profile Modifier Tool

**Goal**: `profile:` as a composable modifier prefix.

### Step 5.1: Implement profile modifier

In `src/domain/builtin-tools/profile.ts`:
- kind: "modifier"
- `execute()` returns `{ contextModifications: { profile: payload } }`
- The orchestrator applies this by re-resolving worker config with the named profile

### Step 5.2: Deprecate directive-based profile

- Remove `profile:` detection from parser directive scanning
- OR: keep directive parents as a secondary mechanism for backward compat
- Recommend: keep both during migration, document prefix form as preferred

## Phase 6: Config Integration

**Goal**: `config.json` supports tool-specific overrides alongside frontmatter.

### Step 6.1: Extend config schema

In `config.json`:
```json
{
  "tools": {
    "verify": { "skipExecution": true, "shouldVerify": true },
    "my-custom-tool": { "profile": "fast", "shouldVerify": false }
  },
  "toolDirs": [".rundown/tools"]   // Configurable tool directories
}
```

### Step 6.2: Config merge logic

In tool resolver or orchestrator:
1. Load tool (built-in, .js, or .md with frontmatter)
2. Overlay `config.json tools.<name>` properties onto tool definition
3. Config wins over frontmatter (user-level override of tool defaults)

**Files to modify:**
- Modify `src/domain/worker-config.ts` — extend config schema types
- Modify `src/infrastructure/adapters/tool-resolver-adapter.ts` — merge config overrides

## Scope

**Included:**
- Unified tool resolution (built-in + .md + .js)
- Composable modifier prefix chains
- Include tool for markdown composition
- Profile as modifier tool
- Config.json tool overrides

**Excluded:**
- `cli:` prefix (stays special-cased per decision)
- Removing directive-based `profile:` (keep for backward compat, deprecate later)
- Remote/networked tools
- Tool dependency/ordering constraints

- [x] Add typed `ToolFrontmatter` schema parsing/validation and defaulting so malformed `.md` frontmatter fails predictably.
- [x] Update tool resolution to honor configured `toolDirs` (in order) instead of only hardcoded `.rundown/tools`.
- [x] Add include-cycle protection (direct and indirect self-include detection) in include sub-file execution flow.

- [x] Add runtime validation/error handling for `.js` tools so invalid module exports (missing default handler or wrong shape) fail with a clear diagnostic.

- [x] Add integration tests that preserve backward compatibility for directive-based `profile:` while preferring prefix-based `profile:` during migration.

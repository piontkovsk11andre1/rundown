# Fix and improve logging

## Problem

The current `rundown all` output is a wall of flat, noisy, interleaving lines:

```
‚Ñπ Worker: opencode run (from config defaults)
‚Ñπ N‚Ñπ Starting execute phase...        ‚Üê "Next task:" animation collides with next info line
‚Ñπ Running: opencode run [mode=wait]
ext task: C:\Work\md-todo\...         ‚Üê continuation of a clobbered animated line
‚Ñπ Execute phase finished; starting verify/repair phase...
‚Ñπ Verify phase: running initial verification (attempt 1).
‚Ñπ Running verification...
‚úî Verify phase complete: verification passed on initial attempt.
‚úî Verification passed.
‚úî Task checked: Extend tool-resolver-port with...
‚Ñπ Worker: opencode run (from config defaults)       ‚Üê repeats for every task
...                                                   ‚Üê √ó10 tasks = unreadable
```

### Specific defects

1. **Line collision**: "Next task:" uses `typeText` animation which writes char-by-char. Before it finishes, the next synchronous `emit({ kind: "info" })` overwrites the same line ‚Üí garbled "‚Ñπ N‚Ñπ Starting execute phase..."
2. **Extreme verbosity**: Every task prints ~8 info lines for phases that succeed silently. The signal-to-noise ratio is near zero.
3. **No visual grouping**: 10 tasks produce 80+ flat lines with no separation, indentation, or hierarchy.
4. **Redundant repetition**: "Worker: opencode run (from config defaults)" repeats identically for every single task.
5. **Duplicate success**: Both "Verify phase complete: verification passed on initial attempt." AND "Verification passed." print ‚Äî same fact twice.
6. **No task counter**: No way to see progress like [3/10] during a long run.

---

## Design: what clean output looks like

### Happy path (verify passes first try)

```
  ‚îå [1/10] Extend tool-resolver-port with a way to enumerate known tool names...
  ‚îÇ ‚Ñπ opencode run [wait]
  ‚îÇ ‚úî Verified
  ‚îî ‚úî Done

  ‚îå [2/10] Add typed ToolFrontmatter schema parsing/validation...
  ‚îÇ ‚Ñπ opencode run [wait]
  ‚îÇ ‚úî Verified
  ‚îî ‚úî Done
```

### Verify-fail + repair path

```
  ‚îå [4/10] Add include-cycle protection...
  ‚îÇ ‚Ñπ opencode run [wait]
  ‚îÇ ‚öÝ Verification failed
  ‚îÇ   ‚Ñπ Repair attempt 1/1...
  ‚îÇ   ‚úî Repair passed
  ‚îî ‚úî Done
```

### Verify-fail + exhausted repairs

```
  ‚îå [4/10] Add include-cycle protection...
  ‚îÇ ‚Ñπ opencode run [wait]
  ‚îÇ ‚öÝ Verification failed
  ‚îÇ   ‚Ñπ Repair attempt 1/3...
  ‚îÇ   ‚úñ Repair failed
  ‚îÇ   ‚Ñπ Repair attempt 2/3...
  ‚îÇ   ‚úñ Repair failed
  ‚îÇ   ‚Ñπ Repair attempt 3/3...
  ‚îÇ   ‚úñ Repair failed
  ‚îî ‚úñ Failed ‚Äî repairs exhausted
```

### Completion

```
‚úî All tasks completed (10 total).
```

### Key design principles

- **Group per task**: Each task gets a visual box with `‚îå ... ‚îî`.
- **Counter**: `[current/total]` at the start of each group.
- **Collapse noise**: Worker command shown once per task, shortened. Phase transitions hidden ‚Äî only outcomes shown.
- **Indent sub-phases**: Repair attempts nest under the verify failure.
- **No animation on info lines**: Remove typewriter effect from "Next task:" ‚Äî it causes the collision bug and adds no value in batch mode.
- **Deduplicate**: Remove the second "Verification passed." line.

## Implementation plan

### Task 1: Add `group-start` / `group-end` event kinds to `ApplicationOutputEvent`

**File**: `src/domain/ports/output-port.ts`

Add two new event variants:

```ts
| { kind: "group-start"; label: string; counter?: { current: number; total: number } }
| { kind: "group-end"; status: "success" | "failure"; message?: string }
```

These are structurally neutral ‚Äî existing output port implementations that don't handle them will hit the `default:` case in `switch` (which already returns silently).

### Task 2: Emit `group-start` / `group-end` from the run loop

**File**: `src/application/run-task-iteration.ts`

- At the top of `executeRunLoopIteration` (right before "Next task:"), emit `group-start` with the task label and `{ current: taskIndex+1, total: totalTasks }`.
- At the bottom (after "Task checked:" success OR after failure exit), emit `group-end` with appropriate status.

The task index and total must be threaded in. Check if `run-task-execution.ts` already has these (it tracks `tasksCompleted`). Add `taskIndex` and `totalTasks` to the iteration context if not already present.

### Task 3: Remove / demote noisy per-task info lines

**Files**: `src/application/run-task-iteration.ts`, `verify-repair-loop.ts`, `resolve-worker.ts`, `task-execution-dispatch.ts`

Lines to **remove** entirely:
- "Next task: ..." ‚Äî replaced by `group-start` event
- "Starting execute phase..." ‚Äî noise
- "Execute phase finished; starting verify/repair phase..." ‚Äî noise
- "Verify phase: running initial verification (attempt 1)." ‚Äî noise
- "Running verification..." ‚Äî noise
- "Verification passed." (the second duplicate at line 200) ‚Äî keep only one success

Lines to **keep** (but will be rendered indented by the group):
- Worker command (once per task, shortened)
- "Running: ..." ‚Äî keep but simplify
- Verify/repair outcomes (pass/fail)
- "Task checked: ..." ‚Äî becomes the `group-end` success message

Lines to **keep only on failure**:
- "Verify phase complete: initial verification failed." ‚Üí shorten to verification failure warn
- Repair attempt status lines

### Task 4: Render grouped output in `cliOutputPort`

**File**: `src/presentation/output-port.ts`

Add state tracking:
```ts
let groupDepth = 0;
```

**TTY mode** (interactive terminal):
- `group-start`: Print `‚îå [current/total] label`, set `groupDepth = 1`.
- All other events while `groupDepth > 0`: Prefix with `‚îÇ ` + appropriate indent.
- `group-end`: Print `‚îî ‚úî Done` or `‚îî ‚úñ Failed`, reset `groupDepth = 0`.

**Non-TTY / CI mode** (piped output, CI logs):
- `group-start`: Print `[current/total] label` (no box chars).
- All other events while `groupDepth > 0`: Prefix with `  ` (2-space indent, no `‚îÇ`).
- `group-end`: Print `‚úî Done` or `‚úñ Failed` (no `‚îî`).
- Box-drawing chars are hardcoded ‚Äî they're rendering chrome like `‚Ñπ ‚úî ‚öÝ ‚úñ`, not a config surface.

### Task 5: Fix the typewriter animation collision

**File**: `src/presentation/output-port.ts`

The `enqueueAnimatedLine` function is async but the animation queue isn't awaited before the next synchronous `emit()`. This causes the "Next task:" typewriter to collide with "Starting execute phase...".

Fix: Since "Next task:" is being removed (Task 3), this is solved implicitly. But also remove `shouldAnimateInfoMessage` for "Next task:" to prevent future issues. The animation should only be used for final summary-type lines, not per-task status.

### Task 6: Suppress worker repetition

**File**: `src/application/resolve-worker.ts`

The worker line `Worker: opencode run (from config defaults)` is emitted for every task. Options:
- Emit it only once at the start of a multi-task run (in `run-task-execution.ts` before the loop).
- OR let it keep emitting inside the group (it'll be hidden by the `‚îÇ` prefix and shortened form).

Preferred: Keep it inside the group but shorten the message from `Worker: opencode run (from config defaults)` to just the command + mode (the group already contextualizes it).

### Task 7: Update the JSONL log writer

**File**: `src/presentation/logged-output-port.ts`

Add `group-start` and `group-end` to the logged event kinds so the structured log captures group boundaries. The JSONL schema in `src/domain/global-output-log.ts` needs a `kind` union update.

### Task 8: Update tests

Existing tests rely on exact emit sequences. Update:
- `__tests__/application/run-task-iteration-modules.test.ts`
- `__tests__/application/verify-repair-loop.test.ts` (remove assertions for deleted info lines, add group event assertions)
- `__tests__/presentation/output-port.test.ts` (if exists ‚Äî add group rendering tests)
- `__tests__/presentation/logged-output-port.test.ts` (add new event kind coverage)

### Task 9: Add `--verbose` flag for full diagnostic output

**Files**: `src/presentation/cli-options.ts`, `src/presentation/cli-command-actions.ts`, `src/application/run-task-iteration.ts`, `verify-repair-loop.ts`, `task-execution-dispatch.ts`

Add a `--verbose` / `-v` flag to `run` and `all` commands.

When `--verbose` is active:
- The grouped output still applies (box drawing in TTY, flat `[n/m]` prefix in CI).
- **All** info lines are emitted inside the group instead of being suppressed ‚Äî phase transitions, worker source, verification details.
- The verbose lines should still be improved: shortened, deduplicated, consistently formatted. Not a raw dump of the old output ‚Äî just more detail within the same clean structure.

Implementation: Thread a `verbose: boolean` through the iteration context. The emit calls removed in Task 3 become conditional on `verbose`. The output port rendering doesn't change ‚Äî verbosity is controlled at the emit site, not the render site.

---

## Decisions

- **Box chars**: Hardcoded. Same category as `‚Ñπ ‚úî ‚öÝ ‚úñ` ‚Äî terminal chrome, not user config.
- **Non-TTY / CI**: Flat `[1/10]` prefix, 2-space indent, no box-drawing chars. Detected at runtime via `process.stdout.isTTY`.
- **`--verbose`**: Keeps detailed per-phase output but within the clean grouped structure. Improved wording, no duplicates.

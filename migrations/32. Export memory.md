# 32. Export Memory — CLI Commands for Memory Management

## Context

### How memory works today

Memory is a per-Markdown-file local artifact storage system. Each source file gets its own isolated memory stored in a `.rundown/` directory next to the source file:

- **Body file**: `<source-dir>/.rundown/<original-filename>.memory.md` — append-only entries separated by `---`
- **Index file**: `<source-dir>/.rundown/memory-index.json` — JSON map of canonical source paths to metadata (`summary`, `updatedAt`, `lastPrefix`, `entryCount`)

**Writing** (`MemoryWriterPort`): Appends entry to body file, atomically updates index (temp file + rename). Supports prefixes: `memory`, `memorize`, `remember`, `inventory`.

**Reading** (`MemoryResolverPort`): Returns metadata only (`available`, `filePath`, `summary`). Does NOT read body content. Body content can only be read via raw `FileSystem` port.

**Integration**: Memory metadata is injected into worker prompts as template variables (`memoryStatus`, `memoryFilePath`, `memorySummary`, `memoryIndexPath`, `memoryMap`). Content is never inlined — only paths and summaries are exposed.

**Ownership gap**: There is no tracking of which task created which memory entry. The index tracks `entryCount` and `lastPrefix` but not origin tasks.

### CLI command pattern

All commands are root-level (no subcommand grouping). Registration:
```
program.command("name").description("...").argument("<source>").option("--flag", "...").action(withCliAction(createXxxCommandAction({ getApp })))
```
Action factories return `(source, opts) => Promise<number>`, delegate to `getApp().methodName(...)`.

App layer uses 3-step composition: `createAppPorts()` → `createDefaultUseCaseFactories()` → `createAppFromFactories()`.

---

## Goal

Introduce CLI commands to **view**, **validate**, and **clean** memory. Support bidirectional integrity between memory entries and the tasks that created them.

---

## Design

### New CLI commands

All memory commands use `memory-*` naming (flat, consistent with existing style):

| Command | Description |
|---------|-------------|
| `memory-view <source>` | Display memory entries for a source file |
| `memory-validate <source>` | Check memory consistency and report issues |
| `memory-clean <source>` | Remove orphaned, outdated, or invalid memory |

`<source>` — markdown file path, directory, or glob (reuses existing `SourceResolverPort`).

### `memory-view <source>`

**Options**:
- `--json` — Output as structured JSON instead of formatted text
- `--summary` — Show only index summaries (no body content)
- `--all` — When source is a directory/glob, show memory for all matched files

**Behavior**:
1. Resolve source file(s) via `SourceResolverPort`
2. For each source, resolve memory via `MemoryResolverPort`
3. If `--summary`: print index metadata only (summary, updatedAt, entryCount, lastPrefix)
4. Otherwise: read body file via `FileSystem`, parse entries by `---` separator, display with entry numbers
5. If `--json`: output structured object `{ source, memoryFile, entries[], index }`
6. Exit 0 if memory exists, exit 1 if no memory found

### `memory-validate <source>`

**Options**:
- `--fix` — Auto-fix recoverable issues (rebuild index, remove orphans)
- `--json` — Output validation report as JSON

**Checks**:
1. **Index consistency**: Index entry exists ↔ body file exists (detect orphaned index entries or missing index for existing body)
2. **Entry count accuracy**: Actual `---`-separated entry count matches `entryCount` in index
3. **Source existence**: Source file referenced in index still exists on disk
4. **Summary accuracy**: First non-empty line of latest entry matches stored `summary`
5. **Task-memory integrity**: If origin tracking is present (see below), verify originating task still exists and is checked

**Output**: List of issues found, categorized as `error` (broken) or `warning` (stale/drift). Exit 0 if clean, exit 1 if issues found.

**With `--fix`**: Rebuild index from body file (recount entries, regenerate summary, update timestamp). Remove orphaned index entries. Does NOT delete body files (that's `memory-clean`).

### `memory-clean <source>`

**Options**:
- `--dry-run` — Show what would be removed without removing
- `--orphans` — Remove only orphaned memories (source file deleted)
- `--outdated` — Remove memories older than threshold
- `--older-than <duration>` — Age threshold (e.g. `30d`, `6m`) for `--outdated`. Default: `90d`
- `--all` — Remove all memory for matched sources (requires confirmation unless `--force`)
- `--force` — Skip confirmation prompts

**Behavior**:
1. Resolve source(s)
2. Scan `.rundown/` directories for memory files
3. Apply filters (orphans, outdated, all)
4. Show removal plan (file list + sizes)
5. Execute removal: delete body file, remove index entry, update index atomically
6. Report: files removed, space freed

### Task-memory origin tracking (index extension)

Extend `MemoryIndexEntry` with an optional `origin` field:

```typescript
interface MemoryIndexEntry {
  summary: string;
  updatedAt: string;
  lastPrefix: string;
  entryCount: number;
  origin?: {                    // NEW — tracks creating task
    taskText: string;           // Original task checkbox text
    taskLine: number;           // Line number in source file
    sourceHash?: string;        // Optional: hash of source content at capture time
  };
}
```

**Invariants**:
- When a memory-capture task runs, `origin` is written with the task's text and line
- `memory-validate` checks if `origin.taskText` still exists at `origin.taskLine` (fuzzy: allow line drift, match by text)
- If the originating task is removed from the source: validation reports `warning: origin task removed`
- `memory-clean --orphans` treats missing-origin-task memories as orphan candidates

**Bidirectional**: Removing memory does NOT auto-remove the task checkbox (checkboxes are source-of-truth). But `memory-validate` can report "memory exists but origin task was unchecked/removed" so the user can decide.

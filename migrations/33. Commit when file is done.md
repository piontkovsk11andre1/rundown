# 33. Commit when file is done

We have a mode to commit on every task. But we also want a feature to commit on `all` and `run --all` only when file is done.

## Context

Today `--commit` is task-scoped by default: after each successful task completion, rundown stages and commits repository changes with a task-derived message. This behavior is wired through:

- CLI flag surface in `src/presentation/cli.ts` (`--commit`, `--commit-message`, `--all`)
- `all` alias rewrite in `src/presentation/cli-argv.ts` (`all` -> `run --all`)
- run loop orchestration in `src/application/run-task-execution.ts`
- per-iteration completion handling in `src/application/complete-task-iteration.ts`
- git operations in `src/application/run-lifecycle.ts` and `src/application/git-operations.ts`

The docs currently state that in run-all mode, `--commit` applies after each task (`docs/cli.md`), so any file-done behavior must be intentionally introduced as a distinct behavior or compatibility-preserving option.

## Problem to solve

For `run --all` and alias `all`, per-task commit behavior can produce many small commits even when user intent is "commit once when this file/run is fully complete." We need a file-done commit mode that triggers one commit only after the run-all file lifecycle completes successfully.

## Desired behavior

For `run --all` (and `all` alias), support committing only when the file is done, meaning:

- all selected tasks for that source are completed successfully for the current run
- post-run reset actions (if enabled) are already applied before commit
- commit is not attempted if run terminates early due to failure/verification failure/interruption
- commit still uses standard template rendering and exclusion rules (`.rundown/runs`, `.rundown/logs`, lock files)

Single-task `run` behavior remains unchanged (commit after the one completed task).

## Product/API surface options

Two valid designs:

1. Backward-compatible extension (recommended): introduce an explicit commit timing mode, e.g. `--commit-mode per-task|file-done`, defaulting to current behavior.
2. Behavioral change in run-all only: make `--commit` in `--all` imply file-done timing by default.

Given existing docs/tests and revert workflows, explicit mode is safer and avoids breaking existing automation that expects one commit per task.

## Existing implementation hooks that enable this

The code already has deferred commit plumbing for `--reset-after`:

- `run-task-execution.ts` computes `deferCommitUntilPostRun = commitAfterComplete && resetAfter`
- `complete-task-iteration.ts` stores `state.deferredCommitContext`
- final deferred commit runs in `finally` before lock release

This can be generalized so deferred commit is selected for run-all file-done timing, not only for reset-after.

## Integration constraints

- `--commit` currently enforces clean worktree at run start (`run-task-execution.ts`), and this must remain true.
- Non-git directories currently warn and skip commit; preserve this behavior.
- Commit failures are non-fatal warnings today; preserve unless intentionally redesigned.
- Lock lifecycle must remain safe: deferred commit should continue to execute before `releaseAll()`.
- `all` alias must inherit identical semantics because it is argv-rewritten to `run --all`.

## Artifact and revert implications

Revert depends on `extra.commitSha` saved in run artifacts (`revert-task.ts` and related tests).  
If we switch to one commit at file end, artifact metadata strategy must be defined clearly:

- either only the final run artifact gets the commit SHA
- or all completed task artifacts in that run reference the same final SHA
- or run-all/file-done commits are documented as non-task-revertable at per-task granularity

This is a key compatibility decision because current revert UX assumes task-level commit metadata exists for completed runs with `--commit --keep-artifacts`.

## Edge cases to preserve

- `--redo` and `--clean` imply `--all`; file-done commit timing should apply consistently when active.
- Multiple rounds (`--rounds > 1`) need explicit definition: one commit after final successful round vs one commit per round.
- Tool-expansion tasks that insert children already alter loop control; commit timing should still wait until true run completion.
- Dry-run must not create commits.
- Interrupted runs should not create deferred final commits.

## Documentation updates required

If file-done mode is introduced, update:

- `docs/cli.md` run-all section and commit timing notes
- `docs/examples.md` commit examples for `--all`
- any overview/revert guidance that assumes per-task commit granularity

## Acceptance criteria outline

- `run --all` (and `all`) can commit once when file is fully done.
- Existing per-task commit mode remains available (or migration path is explicitly documented).
- Deferred file-done commit occurs after post-run reset and before lock release.
- Failure, verification failure, and interruption paths do not produce final commit.
- Behavior is covered by application and integration tests (run loop, alias, docs-expected semantics, artifact metadata/revert interactions).

# Plan: Direct file-editing plan mode

## Problem
The current plan feature captures model stdout and appends `- [ ]` lines at the end. Review passes can never insert steps in the middle. The entire stdout‚Üíparse‚Üíinsert pipeline is overengineered for what models+harnesses already do well: edit files directly.

## Proposed Approach: Let the worker edit the file, validate the diff

Instead of capturing stdout, unlock the source file, tell the worker "edit this file to add/reorganize TODO items", and validate what changed after the worker exits. Back up the original in the artifacts folder for rollback.

### Steps

#### Phase 1: Domain ‚Äî Post-edit validation
1. Add `validatePlanEdit(beforeSource, afterSource)` in `src/domain/planner.ts`:
   - No `[ ]` ‚Üí `[x]` transitions (model cannot check items off)
   - No removal of `[x]` items (completed work is sacred)
   - Unchecked items may be reordered, inserted anywhere, or removed
   - Return `{ valid, rejectionReason, stats: { added, removed, reordered } }`
2. Reuse `captureCheckboxState()` from `src/application/checkbox-operations.ts` as building block

#### Phase 2: Templates ‚Äî Edit-mode prompt
5. Rewrite `DEFAULT_PLAN_TEMPLATE` in `src/domain/defaults.ts`:
   - Tell the model: "Edit the file directly to add missing TODO items"
   - Allow inserting items at logical positions (not just end)
   - Allow reordering unchecked items for better flow
   - Prohibit checking off items or removing checked items
   - Provide the file path as context

#### Phase 4: Application ‚Äî Rewrite plan-task scan loop
6. In `src/application/plan-task.ts` scan loop:
   - Before worker: `latestDocumentSource` already holds the snapshot (in memory)
   - Run worker with prompt that references the file path
   - After worker exits: read the file, validate diff against in-memory snapshot
   - If valid: continue to next scan
   - If invalid: `writeText(source, latestDocumentSource)` to restore, emit error
   - Convergence: file unchanged after worker exit
7. Remove stdout parsing pipeline (parsePlannerOutput, insertPlannerTodos, applyPlannerOutput, etc.)
8. Adjust prompt building ‚Äî no longer include full document source in prompt (model reads the file)

#### Phase 5: Deep mode
9. Deep mode can also use direct editing ‚Äî prompt scoped to "add child items for task X"
10. Validation still checks no checked items were modified

#### Phase 6: Cleanup
11. Remove dead code: `validatePlannerStdoutContract`, `detectNonAdditivePlannerOutput`, `appendPlannerTodosToExistingList`, `appendPlannerTodosAtEnd`, `normalizePlannerTodoAdditions` (keep if used elsewhere)
12. Update all plan-task tests

### Relevant files
- `src/domain/planner.ts` ‚Äî New `validatePlanEdit()`, remove stdout-parsing functions
- `src/domain/defaults.ts` ‚Äî Rewrite `DEFAULT_PLAN_TEMPLATE`, `DEFAULT_DEEP_PLAN_TEMPLATE`
- `src/application/plan-task.ts` ‚Äî Rewrite scan loop: snapshot in memory ‚Üí run ‚Üí validate ‚Üí restore/continue
- `src/application/checkbox-operations.ts` ‚Äî Reuse `captureCheckboxState()` / `detectCheckboxMutations()`
- `__tests__/application/plan-task.test.ts` ‚Äî Rewrite tests

### Verification
1. Unit tests for `validatePlanEdit()`: mid-insertion, reorder, check-off rejection, checked-item removal rejection
2. Test backup/restore: validation failure ‚Üí file restored to original
3. Test convergence: no changes ‚Üí early exit
4. Test deep mode with direct editing

### Decisions
- No backwards compat needed (unreleased)
- Worker edits file directly (no stdout capture for plan)
- Backup in artifacts folder for rollback on validation failure
- Checked items protected, unchecked items freely editable

### Open questions
- Worker prompt transport: currently prompt is passed via file ($file substitution). The prompt would now say "edit /path/to/tasks.md" ‚Äî does the worker pattern need changes?
- File locking: advisory lock only affects other rundown processes, not the worker. Keep lock held throughout ‚Äî no change needed.

## Feature Specification

The `end` tool is a new built-in handler tool that allows a task to short-circuit the current group of sibling tasks. When a task prefixed with `end:` is executed and the AI worker determines the condition is satisfied, rundown should:

1. Mark the `end:` task itself as checked (`[x]`).
2. Mark all remaining unchecked sibling tasks (same depth level, appearing after the `end:` task in document order) as checked.
3. For each skipped sibling, insert a sub-item annotation explaining why it was skipped (e.g., `- skipped: <reason from the end condition>`).

This introduces a fundamentally new control-flow primitive: the ability for a tool to affect tasks beyond the single task it is attached to. Today, no such mechanism exists in the codebase.

## Naming and Aliases

Decision: for the initial implementation, `return`, `skip`, and `quit` are aliases of `end` and resolve to the same built-in handler.

Rationale:

- The requested behavior is single and specific: short-circuit remaining sibling tasks and add `- skipped: <reason>` annotations.
- Introducing separate control-flow semantics now (especially `quit` as whole-run termination) would expand scope beyond the accepted behavior and require additional orchestration contracts.
- Aliasing matches existing built-in precedent (`verify`/`confirm`/`check`) and keeps parser, resolver, and docs simple while still supporting user vocabulary.

Semantic rule for v1:

- `end: <condition>` is canonical documentation form.
- `return: <condition>`, `skip: <condition>`, and `quit: <condition>` are accepted synonyms and must execute exactly the same logic as `end`.
- No alias has extra behavior in v1; any future divergence would require a new RFC and separate tool names.

- [x] Decide whether `return`, `skip`, and `quit` should be aliases of `end` or distinct tools with different semantics, and document the decision.
- [x] If aliases, register them in the `BUILTIN_TOOLS` map pointing to the same handler (following the `verify`/`confirm`/`check` pattern).

## Existing Architecture Context

### Tool System

Tools are registered through a three-layer resolution system (`src/infrastructure/adapters/tool-resolver-adapter.ts`): project `.js` handlers > built-in tools > project `.md` templates. Built-in tools are statically registered in `src/domain/builtin-tools/index.ts` via a `BUILTIN_TOOLS` map.

Each tool has a `ToolKind` (`"modifier"` or `"handler"`) and returns a `ToolHandlerResult` from `src/domain/ports/tool-handler-port.ts`. The result interface currently supports:

- `skipExecution` / `shouldVerify` — control execution and verification phases.
- `childTasks` — insert sub-items under the current task.
- `childFile` — delegate to an included file.
- `exitCode` / `failureMessage` / `failureReason` — signal failure.
- `contextModifications` — adjust profile, template vars, worker args.

## Design Constraints

### New `ToolHandlerResult` Fields

The `end` tool needs a way to communicate back to the orchestrator which additional tasks should be marked. Possible approaches:

1. **The tool handler directly manipulates the file.** The handler has access to `fileSystem` and the task's `file` path via `ToolHandlerContext`. It could read the source, find sibling tasks, mark them checked, insert skip annotations, and write back — all within the handler function. This avoids changing `ToolHandlerResult` but concentrates markdown mutation logic inside the tool rather than the orchestrator.

2. **Extend `ToolHandlerResult` with a new signal.** Add a field like `markSiblings: "checked"` or `skipRemainingSiblings: { reason: string }` that the orchestrator processes in `tool-execution.ts` or `complete-task-iteration.ts`. This keeps the tool handler pure and delegates mutation to the orchestrator, consistent with how `childTasks` insertion already works.

3. **Hybrid: return sibling task references for the orchestrator to process.** The tool identifies the siblings to skip and returns them; the orchestrator handles the actual file mutations.

### Determining the `end` Condition

The `end:` payload is a natural-language condition (e.g., "If there is no output to process"). The tool needs to evaluate this condition. Options:

- **Worker-based evaluation:** Send the condition to the AI worker as a prompt, asking it to return a yes/no verdict. This follows the existing pattern where tools like `memory:` invoke the worker executor.
- **Context-based evaluation:** The tool examines available context (e.g., prior task output, file state) to decide without calling a worker.

The worker-based approach is more flexible and consistent with rundown's agentic design.

- [x] Design the worker prompt for condition evaluation: send the `end:` payload as a yes/no question to the worker, parsing the response to determine whether to trigger sibling skipping.
- [x] Handle ambiguous worker responses gracefully (e.g., if the worker cannot determine yes/no, default to not skipping siblings).

### Line Number Stability

When the `end` tool marks multiple siblings as checked and inserts sub-items, it must be careful about line number shifts. Inserting sub-items under one sibling shifts the line numbers of subsequent siblings. Processing must happen bottom-to-top (last sibling first) or the tool must re-parse after each mutation.

- [x] Implement bottom-to-top processing order in the sibling-marking logic: iterate siblings in reverse document order so that line-number insertions don't invalidate subsequent operations.

### Interaction With `--all` Mode

In `--all` mode, the run loop continues until no unchecked tasks remain. If the `end` tool marks siblings as checked, the loop will skip them on the next `selectNextTask()` call. This should work naturally with no special handling, provided the file is written before the next selection pass.

### Interaction With Nested Tasks

If a skipped sibling has unchecked child tasks, those children also need to be handled. Options:

- Mark children as checked too (cascading skip).
- Leave children unchecked but mark the parent checked (the parent would never have been selected because `filterRunnable` requires no unchecked descendants, but now the parent is forcibly checked — this may leave orphaned unchecked children).
- Add skip annotations to children as well.

The cascading approach is safest.

- [x] Implement cascading skip for nested children: when a sibling is skipped, also mark all its unchecked descendants as checked and annotate them with skip sub-items.

### Verification of Skipped Tasks

Skipped sibling tasks should NOT be verified — they were never executed. The skip annotation provides an audit trail explaining why verification was skipped.

- [x] Ensure skipped siblings bypass the verification phase — the skip annotation serves as the audit trail instead of a verification step.
- [x] Review `src/application/iteration-mode.ts` (`resolveIterationVerificationMode`) to confirm the `end` tool's frontmatter flags produce the correct execution and verification behavior (the `end:` task itself should execute worker-based condition evaluation but should not trigger a separate verification pass).

## Scope

**In scope:**

- A new `end` built-in tool handler registered in `src/domain/builtin-tools/`.
- Mechanism for the tool to identify and mark same-level sibling tasks as checked.
- Sub-item annotation (e.g., `- skipped: <reason>`) inserted under each skipped sibling.
- Registration in the `BUILTIN_TOOLS` map with appropriate aliases.
- Integration with the prefix-chain system.

**Out of scope (for initial implementation):**

# 38. Fix output for every command

## Research Summary

Architecture context enriched with:
- Detailed `ApplicationOutputEvent` kind reference table with all fields, terminal rendering, and log level
- `ApplicationProgressPayload` and `Task` type field details
- How `cliOutputPort` renders each event (symbols, colors, group nesting, animations)
- How `createLoggedOutputPort` mirrors events to JSONL
- How commands are registered via Commander.js and wired through `withCliAction`
- Complete event-kind-by-command usage matrix

Key files:
- `src/presentation/cli-output-port.ts` — TTY rendering of output events
- `src/application/logged-output-port.ts` — JSONL log writer
- `src/presentation/cli-invocation-log.ts` — fatal error bypass
- `src/application/log-runs.ts` — ANSI-in-application-layer violation
- `src/presentation/intro.ts` — outlier bypassing output port
- `src/application/run-task-utils.ts` — shared `formatTaskLabel()`
- `src/application/revert-task.ts` — duplicated `formatTaskLabel()`
- `src/application/list-tasks.ts` — local `pluralize()` helper
- `src/application/run-task-iteration.ts` — only command using group-start/group-end

## Phase 1: Foundation — Output port and event system cleanup

- [x] Unify duplicated `formatTaskDetailLines()` between `logged-output-port.ts` and `output-port.ts` into a single shared implementation
- [x] Add missing event kinds (`cli-fatal`, `commander`) to the `GlobalOutputLogKind` type union so JSONL entries are fully typed
- [x] Remove `picocolors` import from `src/application/log-runs.ts` and move all ANSI formatting to the presentation layer (5 call sites: `pc.dim`, `pc.green`, `pc.red`, `pc.yellow`, `pc.blue`)
- [x] Ensure `createLoggedOutputPort` strips any residual ANSI escape sequences before writing to JSONL
- [x] Fix `emitCliFatalError` in `cli-invocation-log.ts` to emit through the standard `ApplicationOutputPort` instead of bypassing it
- [x] Audit `configureCommanderOutputInterception()` in `cli-invocation-log.ts` — it sets Commander's `configureOutput()` to write directly through `process.stdout.write`/`process.stderr.write` (lines ~130, ~136); route these through the output port or document as intentional bypass

## Phase 4: Standardize per-command output patterns

- [x] Audit `src/infrastructure/runner.ts` direct `process.stdout.write`/`process.stderr.write` for child process passthrough in "show" runner mode — determine if it should route through the output port or remain as raw passthrough

- [x] Add a summary/footer line to list-style commands (`log`, `artifacts` list mode, `memory-view`, `memory-validate`) showing item counts
- [x] Ensure `--verbose` flag is respected consistently across all worker commands (plan, research, discuss, reverify, make) — suppress per-task info lines by default, show them when `--verbose` is set
- [x] Ensure `--show-agent-output` flag is available and functional on all commands that invoke an LLM agent (notably missing from `reverify`)

- [x] Audit `src/presentation/animation.ts` — it has 15+ direct `process.stdout.write` calls for spinner/progress rendering that bypass the output port; determine whether animation rendering should be routed through the output port or remain as a low-level presentation primitive
- [x] Add `--quiet` / `-q` flag support as the complement to `--verbose` — suppress info-level output events (success messages, progress indicators) while still showing errors and warnings

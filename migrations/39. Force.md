# 39. Force — outer retry on task failure

## TL;DR

Add a `force:` prefix modifier that wraps the entire task iteration in an outer retry loop. When a `force:`-prefixed task fails (execution failure, verification failure, or template CLI failure), the system retries the full iteration from scratch instead of terminating the run. This creates a two-layer retry model: inner (verify-repair loop) and outer (force retry around the whole iteration).

## Research Summary

- Three failure paths currently terminate a run: execution failure in `run-task-iteration.ts:483`, verification failure in `complete-task-iteration.ts:208`, and template CLI failure in `complete-task-iteration.ts:183`
- The verify-repair loop in `verify-repair-loop.ts:66` handles inner retries (re-verify + repair) but cannot restart the entire execution phase
- The prefix chain system in `prefix-chain.ts:119` already supports modifiers (`kind: "modifier"`) that patch context and pass through — `profile:` is the existing example
- `force:` would register as a modifier in `builtin-tools/index.ts:14`, parsing with zero changes to `parsePrefixChain()`
- Bootstrap problem: the prefix chain is parsed inside `runTaskIteration()` at `iteration-mode.ts:64`, but `force:` needs to wrap the entire `runTaskIteration()` call. Requires pre-extraction at the `runTaskLoop` level in `run-task-execution.ts:499`
- Existing `--force-execute`, `--force-unlock`, and `--force` (revert) CLI flags are unrelated to this feature

Key files:
- `src/application/run-task-execution.ts` — task loop, `failRun()`, `runTaskIteration()` call site
- `src/application/run-task-iteration.ts` — single iteration orchestration, dispatch failure handling
- `src/application/complete-task-iteration.ts` — verification failure and completion
- `src/application/verify-repair-loop.ts` — inner repair retry loop
- `src/application/iteration-mode.ts` — intent classification and prefix chain parsing
- `src/domain/prefix-chain.ts` — modifier/handler chain parsing
- `src/domain/task-intent.ts` — legacy intent classification (fast, verify, memory)
- `src/domain/builtin-tools/index.ts` — tool registry
- `src/application/tool-execution.ts` — `executeToolChain()` modifier application

## Phase 1: Pre-parse force extraction and retry wrapper

- [x] Add a `extractForceModifier(taskText: string, toolResolver?: ToolResolverPort)` function that detects a leading `force:` prefix and returns `{ isForce: boolean; maxAttempts: number; strippedText: string }` — parse optional attempt count from payload (e.g., `force: 3, implement feature` → 3 attempts, `force: implement feature` → default attempts)
- [x] Define precise parsing grammar for `extractForceModifier()` — require a comma separator for the attempt count (e.g., `force: 3, text` sets 3 attempts, but `force: 3` alone is treated as task text "3" not as 3 attempts with empty payload), and reject or document nested `force: force:` prefixes
- [x] Define a `--force-attempts` CLI flag in `src/presentation/cli.ts` to set the default outer retry limit (default: 2) for `force:`-prefixed tasks
- [x] Wire `forceAttempts` through `cli-command-actions.ts` into the `run-task-execution.ts` execution options
- [x] In `run-task-execution.ts`, before calling `runTaskIteration()` at line 499, call `extractForceModifier()` on `result.task.text` to detect force mode
- [x] When force mode is detected, wrap the `runTaskIteration()` call in a retry loop: on failure (`continueLoop: false` with non-zero `exitCode`), retry up to `maxAttempts` times before propagating the failure to `failRun()`
- [x] Decide whether to create a new `Task` object with stripped text or mutate the existing one — the `task` object is used downstream for checkbox operations, hook environment variables, and trace context, so stripping `force:` from `task.text` affects all those consumers
- [x] Pass the stripped task text (without `force:` prefix) into `runTaskIteration()` so the inner prefix chain parser sees the remaining modifiers/handlers (e.g., `force: verify: tests pass` → inner chain sees `verify: tests pass`)
- [x] Classify which failure types are retryable: only retry for worker non-zero exit, verification failure, and template CLI failure — skip force retry for early validation failures (empty payload, missing worker command at `run-task-iteration.ts:239-267`), detached mode exits, and tool module validation errors, since these will fail identically on retry
- [x] Ensure `currentTaskIndex` in the `--all` task loop only increments after the force retry loop completes (success or all retries exhausted), not after each individual retry attempt — the retry wrapper must sit between the loop body and the `if (!iterationResult.continueLoop)` check at `run-task-execution.ts:572`

## Phase 3: Artifact and state cleanup between force retries

- [x] Call `resetArtifacts()` between force retry attempts to clear per-iteration artifact context, trace writer, and enrichment context
- [x] Clear `verificationStore` for the current task between force retries via `dependencies.verificationStore.remove(task)` — stale verification failure details from a prior attempt would leak into the next retry's repair loop prompt at `verify-repair-loop.ts:228`
- [x] Clear `state.deferredCommitContext` between force retries if it was set by a prior failed attempt — `resetArtifacts()` does not reset this field
- [x] Reset `runFailed` flag and `state.runCompleted` between force retries, OR restructure the retry loop to avoid calling `failRun()` on non-final retry attempts — `failRun()` sets both `runFailed` (used in the `finally` block to gate deferred commits) and `state.runCompleted`, which would prevent subsequent retries from functioning correctly
- [x] Decide whether `finishRun()` → `runTraceEnrichment()` should execute on intermediate force retry failures or only on the final failure — trace enrichment invokes the worker and is computationally expensive; running it per-attempt may be wasteful
- [x] Clear the cached CLI block executor (`cachedCommandExecutor`) between force retries when `--cache-cli-blocks` is enabled — if a prior failed attempt partially modified files, stale cached CLI block outputs would produce incorrect prompts on the next attempt
- [x] Document that each force retry creates a separate artifact run with a distinct `runId` (since `artifactContext` is recreated at `run-task-iteration.ts:322-333` after `resetArtifacts()`) — trace consumers will see N separate runs rather than one run with N sub-attempts; the `force-retry` trace event should cross-reference the prior `runId`
- [x] Emit a new `force-retry` trace event (or reuse `repair-attempt` with a distinguishing field) before each retry so trace consumers can differentiate outer retries from inner repair attempts
- [x] Emit a user-visible `warn` or `info` message between retries: "Force retry N of M — restarting task iteration from scratch"
- [x] Ensure `afterTaskFailed()` hooks still fire on each failed attempt (not just the final one), so `on-fail` commands run per-attempt
- [x] Ensure that the task checkbox is NOT checked between force retries — only check on final success (the existing `checkTaskUsingFileSystem()` call in `complete-task-iteration.ts:214` already only runs on the success path)

## Phase 5: Edge cases

Detached mode note: `force:` is effectively a no-op when `--detached` is enabled. In detached mode, `runTaskIteration()` exits through the detached branch with `{ continueLoop: false, exitCode: 0 }` (`run-task-iteration.ts:501-508`), which is treated as a successful iteration, so the outer force retry path never triggers.

- [x] Handle source file mutations between force retries — clarify the task re-selection strategy: either re-call `taskSelector.selectNextTask()` to get updated task content (which may select a different task if the file changed), or only refresh the task text while keeping the same task identity (file + line) — also handle the case where the task was already checked by a partial attempt
- [x] Handle `--dry-run` interaction — force retries should still be dry-run safe (no checkbox changes, no real execution); note that `--dry-run` returns exit code 0 at `run-task-iteration.ts:396-404` so no retries would ever trigger (verify and document this); also verify `--print-prompt` behaves similarly
- [x] Ensure group-start/group-end events are properly paired across force retries — each retry should emit its own group-start/group-end pair, or emit a single outer group with nested retry groups
- [x] Handle the case where a force-retried task succeeds on retry but the `on-complete` hook fails — should this trigger another force retry or propagate the hook failure?
- [x] Ensure `force:` does not interact with `--force-execute` in confusing ways — they are independent features (`--force-execute` overrides verify-only intent, `force:` retries on failure)
- [x] Handle `force:` with tool expansion tasks that insert children (`complete-task-iteration.ts:220-223`) — if the tool expansion partially inserts child tasks before failing, a force retry would attempt to re-insert them, causing duplicates
- [x] Document that `force:` is a no-op with detached mode — detached tasks return `{ continueLoop: false, exitCode: 0 }` (success) at `run-task-iteration.ts:501-508`, so force retries never trigger
- [x] Consider whether `totalTasks` count (calculated at `run-task-execution.ts:471-479`) should be recalculated on force retry if a prior attempt modified the source file and changed the task count — affects the `[current/total]` counter in `group-start` events

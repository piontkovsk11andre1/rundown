# Decompose `run-task-iteration.ts`

## Goal

Reduce `src/application/run-task-iteration.ts` (currently ~1,327 lines) to a lean iteration coordinator of ~200–300 lines by extracting clearly bounded concerns — prompt preparation, execution dispatch, post-execution completion, dry-run handling, and trace enrichment — into focused internal modules. The file is the single largest application module after steps 1–3 and its primary problem is a "god function" shape: one exported entry point (`runTaskIteration`) orchestrating too many abstraction levels through a ~50-field parameter bag.

## Background

Step 3 ([migrations/3. Decompose run-task-execution.md](3.%20Decompose%20run-task-execution.md)) unified four redundant execution paths in `run-task-execution.ts` by moving the per-iteration logic into `run-task-iteration.ts`. That was a correct structural decision — individual task execution *is* a distinct concern from the outer run loop — but the extracted code still carries all the sub-concerns it originally mixed: prompt rendering, CLI block expansion, dry-run display, task dispatch, verification, checkbox marking, trace enrichment, and lifecycle hooks.

The module is structured as six top-level functions:

| Function | Lines | Responsibility |
|---|---|---|
| `runTaskIteration` | ~210 | Main entry: resolves worker, classifies intent, validates requirements, creates artifact context, coordinates remaining functions, handles dispatch result |
| `prepareTaskPrompts` | ~150 | Loads templates, expands CLI blocks in source/task-template/verify-template (3 near-identical expand+trace pipelines), renders prompts, builds execution option variants |
| `handleDryRunOrPrintPrompt` | ~70 | Decision matrix: 6 early-return branches for `(printPrompt|dryRun) × (onlyVerify|inlineCli|rundownTask|normal)` |
| `dispatchTaskExecution` | ~200 | Routes to one of 4 executors (only-verify, inline-CLI, rundown-task, normal-worker), wraps each in phase tracing, normalizes into discriminated union result |
| `completeTaskIteration` | ~90 | Runs verification loop, marks checkbox, calls lifecycle hooks, handles repair failures |
| `runTraceEnrichment` | ~110 | Renders trace template with phase artifacts, invokes enrichment worker, queues analysis summary |
| `runVerification` | ~30 | Thin wrapper forwarding to `runVerifyRepairLoop` |

## Primary problems

### 1. Massive parameter surface

`runTaskIteration` accepts ~50 destructured fields. Downstream functions (`prepareTaskPrompts`, `dispatchTaskExecution`, `completeTaskIteration`) each accept 20–40 fields of their own, most of which are just threaded through from the top. This signals that the function is a coupling relay — it doesn't use most of the values it receives, it just passes them to children. Extracting modules would narrow each interface to the fields it actually needs.

### 2. Triple-repeated CLI expansion pipeline

`prepareTaskPrompts` runs the same expand-cli-blocks-with-trace-and-artifact-options pattern three times:
1. Source file CLI expansion (~20 lines, with artifact phase label "cli-source")
2. Task template CLI expansion (~20 lines, with artifact phase label "cli-task-template")  
3. Verify template CLI expansion (~20 lines, with artifact phase label "cli-verify-template")

Each repetition builds almost identical `cliExecutionOptions` variants (add artifact context, add trace, add failure abort, add phase label). The duplication inflates the function and makes the pipeline fragile — any change to the expansion protocol must be made three times.

### 3. Dispatch preamble mixed into orchestrator

Before calling `dispatchTaskExecution`, the orchestrator manually builds `taskContextMetrics`, calls `traceRunSession.startRun(...)`, flushes pending reset trace events, and sets up `emitExecutionWorkerOutput`. These are pre-dispatch setup that belongs with the dispatch logic, not in the outer coordinator.

### 4. Trace enrichment is a standalone concern

`runTraceEnrichment` is already export-ready and has no dependency on the iteration function's closure. It takes a self-contained params object (trace flag, artifact context, trace session, enrichment context, dependencies, transport, emit). It could live in its own module (e.g., `trace-enrichment.ts`) or be folded into `trace-run-session.ts`, decoupling the iteration module from trace template rendering, phase artifact collection, and analysis summary parsing.

### 5. Verification thin wrapper adds no value inline

`runVerification` (~30 lines) is a mechanical adapter from positional args to `runVerifyRepairLoop`'s named params. It's only called from `completeTaskIteration`. Either inline it at the call site or move it alongside `completeTaskIteration`.

## Identified extraction targets

### 1. Extract prompt preparation → `prepare-task-prompts.ts`

Move `prepareTaskPrompts` and the `PrepareTaskPromptsResult` interface to a new internal module. Internally, deduplicate the triple CLI expansion pattern into a single private helper:

```
expandCliBlocksWithOptions(content, executor, cwd, baseOptions, phaseLabel, traceWriter, runId, nowIso)
```

called three times with different phase labels. This reduces the function from ~150 lines to ~80. The module's public surface would be:

- `prepareTaskPrompts(params): Promise<PrepareTaskPromptsResult | { earlyExitCode: number }>`
- `PrepareTaskPromptsResult` interface

Dependencies: `domain/cli-block`, `domain/parser`, `domain/template`, `./cli-block-handlers`, `./project-templates`, `./task-context-resolution`.

### 2. Extract dry-run dispatch → `dry-run-dispatch.ts`

Move `handleDryRunOrPrintPrompt` and `HandleDryRunOrPrintPromptParams` to a new module. This is a pure synchronous function with no side effects beyond `emit()`. The decision matrix is self-contained and independently testable.

Dependencies: `./rundown-delegation` (for `parseRundownTaskArgs`, `buildDelegatedRundownArgs`).

### 3. Extract execution dispatch → `task-execution-dispatch.ts`

Move `dispatchTaskExecution` and `TaskExecutionDispatchResult` to a new module. Bundle the pre-dispatch setup (metrics, `traceRunSession.startRun`, pending reset flush, worker output emitter) into the dispatch function so the caller doesn't need to know about it. The dispatcher normalizes all four execution paths into the `TaskExecutionDispatchResult` discriminated union.

Dependencies: `domain/ports`, `./rundown-delegation`, `./task-context-resolution`, `./trace-run-session`, `./run-task-worker-command`.

### 4. Extract completion logic → `complete-task-iteration.ts`

Move `completeTaskIteration` and inline `runVerification` into it. The completion function handles: verification loop, checkbox marking, lifecycle hooks (`afterTaskComplete`, `afterTaskFailed`), artifact finalization, loop-control return values. It can absorb `runVerification` since that's its only caller.

Dependencies: `./verify-repair-loop`, `./checkbox-operations`, `./run-lifecycle`, `./cli-block-handlers`.

### 5. Extract trace enrichment → `trace-enrichment.ts`

Move `runTraceEnrichment` and `TraceEnrichmentContext` to a new module. It's already a self-contained exported function. Moving it removes the trace template rendering, phase artifact collection, and analysis summary parsing imports from the iteration module.

Dependencies: `domain/template`, `./trace-artifacts`, `./trace-run-session`.

### 6. Simplify the residual `run-task-iteration.ts`

After extractions, the file retains only:
- `runTaskIteration(...)` — resolves worker, classifies intent, validates requirements, creates artifact context, calls `prepareTaskPrompts`, calls `handleDryRunOrPrintPrompt`, calls `dispatchTaskExecution`, calls `completeTaskIteration`. No template rendering, no CLI expansion, no trace enrichment, no verification logic.
- `RunExecutionState` interface — mutable state bag for the iteration.

The parameter count should drop substantially because most fields only feed one extracted module. The orchestrator can construct module-specific param objects.

## Expected outcome

| Module | Estimated size |
|---|---|
| `run-task-iteration.ts` (residual) | ~200–300 lines |
| `prepare-task-prompts.ts` | ~80–100 lines |
| `dry-run-dispatch.ts` | ~80–90 lines |
| `task-execution-dispatch.ts` | ~220–250 lines |
| `complete-task-iteration.ts` | ~120–140 lines |
| `trace-enrichment.ts` | ~110–120 lines |

Total: ~810–1,000 lines across 6 files vs. 1,327 lines in one file. The line count may stay similar but the cognitive load per file drops sharply, each module becomes independently testable, and the main orchestrator's parameter surface shrinks.

## Feature specification

The `loop` command provides an outer infinite (or bounded) iteration loop that wraps the existing `call` command. Each iteration runs a full `call` pass (equivalent to `run --all --clean --cache-cli-blocks`) on the target source. After each successful iteration, the command waits for a configurable cooldown period before starting the next one.

The primary use case is continuous task processing: a developer points `loop` at a Markdown file (or directory/glob) and the tool keeps processing tasks indefinitely — useful for runbooks, monitoring checklists, or any workflow where new tasks may appear between passes (for example, from external tools appending TODOs, or from planning passes that seed new work items).

## Existing codebase context

### Architecture layers

The project (`@p10i/rundown`) follows a clean architecture with four layers:

- **Presentation** (`src/presentation/`): CLI command definitions (`cli.ts`), action factories (`cli-command-actions.ts`), option parsing (`cli-options.ts`), and output formatting.
- **Application** (`src/application/`): Use cases like `run-task-execution.ts`, `run-task-iteration.ts`, `plan-task.ts`, `research-task.ts`.
- **Domain** (`src/domain/`): Core types, task selection, markdown parsing, verification.
- **Infrastructure** (`src/infrastructure/`): File system adapters, process runners, git clients, lock management.

### Command registration pattern

All CLI commands are registered in `src/presentation/cli.ts` using Commander.js:

1. Define command with `.command("name")`, `.description()`, `.argument()`, `.option()`.
2. Wrap the action factory with `withCliAction()` (handles app initialization, animation drain, and `terminate()`).
3. Action factories live in `src/presentation/cli-command-actions.ts` and map CLI options to application-layer request contracts.
4. Use cases are composed in `src/create-app.ts` via `createDefaultUseCaseFactories()`.

### The `call` command (wrapping target)

`call` is defined at `src/presentation/cli.ts:123-135` and is a thin wrapper around `run` that force-sets `{ all: true, clean: true, cacheCliBlocks: true }` (`src/presentation/cli-command-actions.ts:195-212`). It shares all run-like options via `configureRunLikeCommandOptions()`.

### The rounds pattern (closest existing iteration model)

The `--rounds <n>` option (`cli.ts:674`) provides bounded repetition of clean execution cycles within a single `run`/`call` invocation. The rounds loop lives in `run-task-execution.ts:585-632`:

- Runs `for (let round = 0; round < rounds; round++)`.
- Resets checkboxes between rounds (requires `--clean` or `--redo --reset-after`).
- Emits round status messages.
- Fails fast if any round exits non-zero.
- No delay/cooldown between rounds.

### Existing delay/timeout patterns

- **CLI block timeout**: `DEFAULT_CLI_BLOCK_EXEC_TIMEOUT_MS = 30_000`, configurable via `--cli-block-timeout <ms>`. Uses `setTimeout()` + process kill.
- **Hook timeout**: Hardcoded 60s for `--on-complete` and `--on-fail` hooks.
- **Animation delays**: `sleep(ms)` in `animation.ts` wraps `setTimeout` in a Promise; cosmetic only.
- **No inter-round or inter-task cooldown exists anywhere in the codebase.**

### Signal handling and graceful shutdown

`terminate()` is called at process exit and handles lock cleanup, deferred commits, and trace finalization. SIGINT/SIGTERM handlers exist for graceful shutdown. The `loop` command must integrate with this lifecycle to ensure clean shutdown mid-cooldown.

## Design considerations

### Relationship to `call`

The `loop` command should compose `call` at the CLI action level, similar to how `do` composes `make` + `run`. Each loop iteration invokes `call`'s underlying logic (i.e., `app.runTask()` with `{ all: true, clean: true, cacheCliBlocks: true }`). This avoids duplicating execution wiring.

### Cooldown mechanism

A `--cooldown <seconds>` option controls the delay between iterations. Implementation should use a cancellable `setTimeout`-based promise so that SIGINT/SIGTERM can interrupt the cooldown cleanly. The existing `sleep(ms)` pattern in `animation.ts` is a reference, but the loop cooldown needs to be cancellable (not purely cosmetic).

### Iteration bounds

Two modes to consider:

- **Infinite loop** (default): runs until interrupted with Ctrl+C, exiting `0` on graceful shutdown.
- **Bounded loop** (`--iterations <n>`): runs exactly `n` iterations, then exits. This complements `--rounds` which repeats *within* a single invocation; `--iterations` repeats the entire `call` invocation.

### Failure behavior

Options for how failures are handled:

- **Stop on failure** (default, consistent with `--all` semantics): if an iteration exits non-zero, the loop stops and the exit code propagates.
- **Continue on failure** (`--continue-on-error`): log the failure and proceed to the next iteration after cooldown. This is useful for long-running monitoring scenarios where transient failures should not halt the loop.

### Options surface

The `loop` command should accept:

- `<source>` — same file/directory/glob argument as `call` and `run`.
- `--cooldown <seconds>` — delay between iterations (required or with a sensible default, e.g., 60 seconds).
- `--iterations <n>` — optional iteration cap (default: unlimited).
- All `configureRunLikeCommandOptions()` options forwarded to the inner `call` invocation (verification, repair, commit, hooks, worker, etc.).
- `--continue-on-error` — optional flag to continue looping after iteration failures.

### Output and observability

- Each iteration should emit a clear status message (e.g., "Loop iteration 3 starting..." or "Loop iteration 2 completed — cooling down for 60s...").
- The cooldown period should display a visible countdown or status indicator.
- Trace events should capture loop-level lifecycle (`loop.iteration.started`, `loop.iteration.completed`, `loop.cooldown.started`, `loop.cooldown.interrupted`).

### File locking

Each `call` iteration acquires and releases per-source locks within its execution scope. The loop itself should not hold locks across the cooldown period — this allows other processes to interact with the source files between iterations (e.g., `plan` appending new TODOs).

### Exit codes

- `0` — all iterations completed (bounded) or graceful shutdown via SIGINT (infinite).
- `1` — execution error in an iteration (when `--continue-on-error` is not set).
- `2` — validation failure in an iteration (when `--continue-on-error` is not set).

## Scope

### In scope

- New `loop` CLI command definition in `cli.ts`.
- New `createLoopCommandAction()` factory in `cli-command-actions.ts`.
- Cooldown timer with cancellation support for graceful shutdown.
- Iteration counter with optional bounds.
- Forwarding of all run-like options to the inner call execution.
- Output messages for loop lifecycle events.
- Documentation in `docs/cli.md`.

### Out of scope

- File-watching / filesystem-event-driven triggering (this is a timer-based loop, not `inotify`/`chokidar`).
- Application-layer changes to `run-task-execution.ts` or `run-task-iteration.ts` (the loop operates at the CLI composition level).
- New trace event types (can be added later; initial implementation uses existing `info`/`success`/`warn` output kinds).

## Integration notes

- The `loop` command should be registered in `cli.ts` after `call` and before `discuss`, following the existing command ordering convention.
- The `withCliAction()` wrapper handles app initialization and teardown; the loop action must work within this lifecycle.
- The `parseCliArgs()` function in `cli.ts:496-551` handles alias rewriting and worker separator splitting; no changes needed there unless `loop` needs alias support.
- Unit tests should cover option parsing, iteration counting, cooldown timing (with mocked timers), and failure propagation. Integration tests should verify end-to-end loop behavior with a small Markdown fixture.

## Acceptance criteria outline

- Running `rundown loop <source> --cooldown 10 -- opencode run` processes all tasks, waits 10 seconds, then processes again indefinitely.
- Running `rundown loop <source> --cooldown 5 --iterations 3 -- opencode run` performs exactly 3 iterations with 5-second gaps.
- Pressing Ctrl+C during cooldown exits `0` cleanly without orphaned processes or stale locks.
- Pressing Ctrl+C during task execution triggers standard graceful shutdown behavior.
- A failed iteration (exit code 1 or 2) stops the loop by default; with `--continue-on-error` it logs the failure and continues after cooldown.
- All run-like options (`--commit`, `--verify`, `--repair-attempts`, `--worker`, etc.) pass through to the inner execution correctly.

I want `rundown` to start a live help session when it is launched with no command-line parameters.

Instead of only printing static help, the CLI should open an interactive TUI assistant that can explain commands, suggest next steps, and answer "how do I do X?" questions in the context of this repository.

## Feature specification

When the user runs `rundown` with no subcommand and no positional arguments:

1. Detect whether a TUI-capable worker is configured.
2. If a worker is available and the terminal is interactive, open a help discussion session.
3. Seed the session with project docs and CLI usage context so the assistant can answer immediately.
4. If the worker/config is unavailable, fall back to normal Commander help output.

This behavior is an additive convenience mode. It should not change explicit command behavior (`rundown run ...`, `rundown plan ...`, etc.).

## Current codebase context

### CLI boot and no-arg behavior

- The root command is configured in `src/presentation/cli.ts` via Commander.
- `parseCliArgs(argv)` normalizes aliases, splits worker args, validates special cases, then calls `program.parseAsync(...)`.
- `shouldInitializeAppBeforeParse(argv)` already returns `true` for the root/no-args path, so app/config access is available before command execution.
- There is currently no `program.action(...)` handler on the root command; with no args, Commander prints static help.

This makes the root action in `cli.ts` the natural integration point for live help.

### Existing interactive precedent

- `discuss` already provides an interactive worker-driven flow (`src/application/discuss-task.ts`).
- It resolves worker config, renders a template, and runs worker in `mode: "tui"`.
- It uses artifact context and trace wiring that can be reused for observability and debugging.

Although `discuss` is task-focused (requires source/task selection), its worker resolution and execution pipeline are the closest implementation template for live help.

### Template and config system

- Default templates are defined in `src/domain/defaults.ts` and loaded through template loader ports.
- `.rundown/config.json` is already used for worker configuration resolution.
- CLI already supports `--config-dir` and global invocation setup through `createAppForInvocation(...)`.

This means live help can remain configuration-driven and consistent with existing command behavior.

## Worker resolution options

### Option A (recommended): reuse `discuss` worker config

Use the same worker resolution path as `discuss` for the root live-help mode.

Pros:
- No config schema change required.
- Fastest path to implementation.
- Keeps one interactive worker contract.

Cons:
- Couples help behavior to discuss worker settings.

### Option B: add dedicated help worker config

Introduce a dedicated key (for example `help`) in worker config.

Pros:
- Clear separation of concerns.
- Users can tune help and discuss independently.

Cons:
- Requires config schema and docs updates.

Recommended rollout: implement Option A first, keep Option B as non-breaking follow-up.

## Prompt and content model

The live-help prompt should include:

1. The current CLI version and working directory.
2. A concise command index (run/call/plan/research/make/do/discuss/reverify/revert/etc.).
3. References to repository docs under `docs/`.
4. Guidance to ask follow-up questions before suggesting risky actions.

The prompt should be template-backed (for consistency with other phases), for example a dedicated `help` template that can be overridden in `.rundown/`.

## Runtime behavior details

### Interactive gating

Live help should launch only when:

- stdout/stderr are TTY-capable (interactive terminal), and
- worker resolution succeeds.

If either condition fails, print standard help and return a normal non-error exit code.

### Artifacts and trace

For consistency with other worker flows:

- Create an artifact context for help sessions.
- Capture prompt and optional stdout/stderr depending on keep-artifacts policy.
- Support trace output if enabled.

This keeps supportability high when users report "help session did something unexpected" issues.

### Exit behavior

- Normal session completion returns `0`.
- User cancellation (Ctrl+C in TUI) should map to graceful termination behavior already used by CLI signal handling.
- Worker launch/config errors should degrade to static help (not hard failure) unless explicitly requested by strict mode in the future.

## Scope boundaries

### In scope

- Root no-args entry behavior (`rundown` only).
- Live help session launch via existing worker infrastructure.
- Template-backed help prompt with docs/context references.
- Graceful fallback to Commander help when live mode cannot run.

### Out of scope

- Replacing `--help` output format.
- Auto-executing commands from help chat.
- Persistent conversation memory across separate CLI invocations.
- New non-interactive help API.

## Edge cases and compatibility

- `rundown --help` must remain deterministic and non-interactive.
- `rundown <invalid-command>` should keep existing Commander error/help semantics.
- CI/non-interactive shells should never hang waiting for TUI input.
- Missing `.rundown/` or missing worker config should not produce a fatal error for no-arg invocation.
- `--config-dir` overrides must apply to worker/config/template discovery for live help.

## Acceptance criteria outline

- Running `rundown` with no args in an interactive terminal launches a TUI help session when worker is configured.
- Running `rundown` with no args without interactive terminal prints static help and exits `0`.
- Running `rundown --help` prints static help (no TUI launch).
- Misconfigured/missing worker for no-arg launch falls back to static help with an informational message.
- Help prompt includes command overview + docs context and supports natural-language Q&A about rundown usage.

# Plan never stops

## Problem statement

`rundown plan` currently runs a bounded scan loop controlled by `--scan-count` (default `3`).

In practice, with the direct file-editing planner flow, each scan often finds some additional rewrite or reordering opportunity, so the file keeps changing and convergence (`editedDocumentSource === scanBaseline`) does not trigger early. The command then runs to the scan cap every time, which feels like planning never naturally finishes.

The desired behavior is to support a mode where planning continues until it truly converges, while still preserving safety against runaway loops.

## Current behavior in code

Top-level scan behavior is implemented in `src/application/plan-task.ts`:

- Scans execute in a `for (scanIndex = 1; scanIndex <= scanCount; scanIndex += 1)` loop.
- Each pass runs the worker with scan context from `buildPlanScanPrompt(...)`.
- After each pass, the edited file is validated via `validatePlanEdit(scanBaseline, editedDocumentSource)`.
- Convergence is currently defined as a byte-identical no-op edit (`editedDocumentSource === scanBaseline`).
- If no convergence is reached, outcome is marked as `scan-cap-reached`.

Prompt constraints come from:

- `src/domain/defaults.ts` (`DEFAULT_PLAN_TEMPLATE`) for planning rules.
- `buildPlanScanPrompt(...)` in `src/application/plan-task.ts` for per-scan instructions.

CLI option parsing currently requires a positive integer scan count:

- `parseScanCount(...)` in `src/presentation/cli-options.ts` sets `min: 1` and disallows open-ended semantics.

## Why this does not converge reliably

The current convergence check only detects "no textual changes at all." In direct-edit mode, the worker is explicitly allowed to reorder unchecked items and insert items in the middle of lists, which makes textual no-op convergence harder to hit even when the plan is already sufficient.

This means scan-cap is functioning as the de facto stop condition rather than true planning convergence.

## Desired outcome

Introduce an unlimited planning mode as the default path so top-level scans continue until convergence is reached by planner behavior, not by a fixed small cap.

At the same time, preserve explicit finite scan control for users who want deterministic bounded runs.

## Design constraints

- Preserve existing safety invariants from `validatePlanEdit(...)` (no unchecked-to-checked transitions, no completed-item removal).
- Keep deep-pass (`--deep`) behavior separate from top-level scan convergence changes.
- Maintain backward compatibility for users who explicitly pass `--scan-count <n>`.
- Keep artifacts/trace metadata expressive so users can distinguish "converged" from "bounded by cap" outcomes.

## Implementation direction

### 1) Add explicit scan mode semantics

Extend scan control from a single numeric concept into two modes:

- Bounded mode: `--scan-count <n>` keeps current finite behavior.
- Unlimited mode: no hard cap; loop terminates on convergence signal.

Recommended default: unlimited mode when scan count is not explicitly provided.

- [x] Update `PlanTaskOptions` to accept an optional `scanCount` (undefined means unlimited) instead of always requiring a numeric default
- [x] Update `parseScanCount` in `src/presentation/cli-options.ts` to allow undefined/absent `--scan-count` (remove `min: 1` hard requirement when no value is provided) and return `undefined` instead of a numeric default
- [x] Update the duplicate `DEFAULT_PLAN_SCAN_COUNT` constant in `src/presentation/cli.ts` (line 60) used for Commander option display, so it reflects the new unlimited default
- [x] Update or remove the duplicate `DEFAULT_PLAN_SCAN_COUNT` constant in `src/presentation/cli-options.ts` (line 32) to stay consistent with the new unlimited default
- [x] Change the scan loop in `planTask` from a bounded `for` loop to a convergence-driven `while` loop that respects both bounded and unlimited modes

### 2) Strengthen convergence signal

Retain byte-identical check as one convergence signal, but add a semantic signal based on validated additions:

- If a scan produces no net new actionable TODO additions (`validationResult.stats.added === 0`), treat this as convergence for top-level scanning.

This avoids endless rewrites where text changes but no additional planning value is created.

- [x] Add a semantic convergence check in the scan loop: treat `validationResult.stats.added === 0` as convergence even when the file text changed
- [x] Extend `PlanConvergenceOutcome` to distinguish convergence reasons (e.g. `"converged-no-change"`, `"converged-no-additions"`, `"scan-cap-reached"`, `"emergency-cap-reached"`)
- [x] Define a typed interface for plan convergence metadata instead of the current `Record<string, unknown>` return shape from `buildPlanConvergenceMetadata`
- [x] Export `PlanConvergenceOutcome` from `src/application/plan-task.ts` so tests and consumers can reference the convergence reason values
- [x] Update `buildPlanConvergenceMetadata` to include the specific convergence reason in artifact metadata

### 3) Guardrails for unlimited mode

Unlimited should still include hard safety rails to avoid pathological loops:

- Internal emergency max-iteration ceiling (high default, not user-facing primary control).
- Stable warning when emergency ceiling is hit, with outcome distinct from normal convergence.
- Existing non-zero worker exit handling remains fail-fast.

- [x] Define an internal emergency max-iteration constant (e.g. 20-50 scans) used only in unlimited mode
- [x] Emit a distinct warning message when the emergency ceiling is hit (not a fatal error)
- [x] Record emergency-ceiling termination as a distinct outcome in convergence metadata (separate from normal convergence and user-defined cap)

### 4) Prompt alignment

Update plan-scan prompt language so each scan aims to add only genuinely missing tasks and stop editing when coverage is sufficient. This should reduce cosmetic rewrites across scans.

- [x] Update `buildPlanScanPrompt` to instruct the worker to avoid cosmetic rewrites and only make edits that add genuinely missing TODO items
- [x] Add prompt guidance telling the worker to leave the file unchanged if plan coverage is already sufficient
- [x] Update `buildPlanScanPrompt` to handle unlimited mode where total scan count is unknown (currently passes `scanCount` to the prompt text)
- [x] Update the `buildPlanScanPrompt` function signature to accept `scanCount: number | undefined` since the total is unknown in unlimited mode
- [x] Provide a sensible fallback for the `{{scanCount}}` template variable (plan-task.ts line 285) in unlimited mode, since user templates may reference it

## CLI and UX considerations

- Keep `--scan-count <n>` available for bounded runs.
- Clarify docs that default planning behavior is convergence-driven.
- Ensure run output clearly reports whether stop reason was:
  - Converged by no file change,
  - Converged by no new additions,
  - User-defined scan cap reached,
  - Emergency guardrail reached.

- [x] Update CLI help text for `--scan-count` to document that omitting it enables convergence-driven unlimited mode
- [x] Ensure run summary output displays the specific stop reason to the user (converged-no-change, converged-no-additions, scan-cap-reached, emergency-cap-reached)
- [x] Update `buildPlanScanLabel` to handle dynamic/unknown total scan counts in unlimited mode (e.g. `plan-scan-03` instead of `plan-scan-03-of-03`)

## Scope boundaries

In scope:

- Top-level `plan` scan-loop stopping behavior.
- Scan-loop metadata and user-visible stop reason reporting.
- CLI/default semantics for scan control.

Out of scope:

- Reworking deep child-generation pass semantics.
- Changes to `research` command behavior.
- Changing validation rules that protect completed items.

## Acceptance criteria outline

- Default `rundown plan <file>` behaves as convergence-driven and does not depend on a small fixed scan cap.
- Explicit `--scan-count <n>` still enforces bounded scans.
- Planning terminates when no new TODO additions are produced, even if incidental text rewrites occur.
- Metadata/output clearly identifies termination reason.
- Existing plan safety validations remain enforced.

- [x] Add tests verifying bounded mode (`--scan-count <n>`) still enforces the exact scan cap
- [x] Add tests verifying the emergency ceiling stops unlimited mode and records the correct outcome
- [x] Verify existing `validatePlanEdit` safety rules (no checked-item removal, no unchecked-to-checked transitions) remain enforced under both modes

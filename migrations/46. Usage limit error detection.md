## Background

Rundown does not call LLM APIs directly. It spawns external CLI processes (e.g. `opencode run`, `claude -p`) and captures `exitCode`, `stdout`, and `stderr` from those processes. This means rundown cannot inspect HTTP status codes, API error objects, or provider-specific headers (like `x-ratelimit-remaining`). All detection must rely on the observable behavior of the spawned worker — specifically its exit code and textual output.

## Problem Statement

When an LLM provider hits a usage limit (rate limit, billing quota, daily cap, context window exceeded, etc.), the spawned worker may still exit with code 0 and return text to stdout. That text is typically an error message from the provider rather than a legitimate task response. Because rundown treats any zero-exit-code output as a valid result, it will proceed to verification, which also spawns a worker — which is also likely to hit the same limit and return a similar error message. The repair phase then does the same. This wastes time, burns additional quota if any remains, and silently produces incorrect results.

## Detection Strategy: Output Similarity Across Phases

The author's original insight is sound: if the execution, verification, and repair phases all return the same (or suspiciously similar) output, that output is almost certainly not a legitimate task response. The reasoning is:

- **Execution** receives a "do the work" prompt.
- **Verification** receives a "check if the work was done correctly" prompt that expects `OK` or `NOT_OK: <reason>`.
- **Repair** receives a "fix the issues found by verification" prompt.

These three prompts are fundamentally different in purpose and expected output format. A real LLM response would differ across phases. An API limit error message would be identical (or nearly identical) because the provider returns the same canned error regardless of prompt content.

### What "equal response" means in practice

Exact string equality (`===`) would catch the most common case — e.g. the provider returns `"You have exceeded your usage limit"` verbatim every time. However, some providers include timestamps, request IDs, or slight variations, so a normalized comparison (trimmed, lowercased, or a simple similarity threshold) would be more robust.

## Current Architecture — Where Output Flows

### Execution phase

In `task-execution-dispatch.ts`, the default worker execution branch calls `workerExecutor.runWorker()` and receives `{ exitCode, stdout, stderr }`. The stdout content is:
- Forwarded to the event emitter (if `showAgentOutput` is true).
- Recorded in trace artifacts.
- Written to the memory store (for memory-capture tasks).
- **Not passed back** to the caller in `TaskExecutionDispatchResult`. The `ready-for-completion` variant carries verification config but not the execution stdout.

### Verification phase

In `verification.ts`, `parseVerificationResult()` extracts the verdict from the last line of stdout. The full stdout is available inside the function but only the parsed `sidecarContent` (verdict or failure reason) is persisted to the `VerificationStore`. The raw stdout is not returned to the verify-repair loop.

### Repair phase

In `repair.ts`, the repair worker's `{ exitCode, stdout, stderr }` is **discarded entirely** — the `await runWorker(...)` result is not assigned. Only the subsequent re-verification result matters.

### Implication

To compare outputs across phases, the raw stdout from each phase must be threaded back to a common comparison point. Currently, none of the three phases expose their raw stdout to the orchestration layer (`complete-task-iteration.ts` / `verify-repair-loop.ts`).

## Feasibility Assessment

**Yes, this is feasible.** The detection approach is architecturally clean and does not require changes to the spawned workers themselves. The changes are confined to the application and infrastructure layers:

1. **Execution phase** — extend `TaskExecutionDispatchResult` (the `ready-for-completion` variant) to carry execution stdout.
2. **Verification phase** — extend `VerifyResult` to include raw stdout alongside the parsed verdict.
3. **Repair phase** — capture the repair worker's stdout (currently discarded) and return it in `RepairResult`.
4. **Comparison logic** — introduce a comparison utility (simple string equality or a lightweight similarity check) at the orchestration level.
5. **Decision point** — the verify-repair loop (or complete-task-iteration) compares outputs and short-circuits with a clear error message if a usage limit is detected.

## Design Constraints

### False positives

The comparison must avoid false positives. Legitimate scenarios where outputs could be similar:
- Very short tasks where execution output is minimal (e.g. a single "Done" line).
- Verification that returns `OK` on first try — but this is a success path, so comparison is irrelevant (we only need to check on failure paths or when verification unexpectedly passes with non-standard output).

Mitigation: only trigger detection when verification fails or when the comparison is against non-trivial output (e.g. more than a few words).

### False negatives

Some providers may return *different* error messages per request (e.g. varying wording, included request IDs). A strict equality check would miss these.

Mitigation: apply normalization before comparison (trim whitespace, strip common noise like timestamps/UUIDs). Optionally, maintain a list of known usage-limit error patterns (e.g. `"rate limit"`, `"quota exceeded"`, `"billing"`, `"usage limit"`, `"too many requests"`) as a secondary detection heuristic.

### Performance

String comparison is negligible. No external dependencies are needed for basic equality/similarity checks.

### Existing error handling patterns

The codebase uses discriminated unions for results (`TaskExecutionDispatchResult`) and `{ ok, message, reason }` tuples. A new error kind (e.g. `"usage-limit-detected"`) would fit naturally into the `TaskExecutionDispatchResult` union or as a new `VerifyRepairLoopResult` state.

## Scope

### In scope

- Detecting usage limit errors by comparing raw stdout across execution, verification, and (optionally) repair phases.
- Surfacing a clear, actionable error message when a usage limit is detected.
- Short-circuiting the verify-repair loop to avoid wasting further quota.
- Normalizing output strings before comparison to handle minor provider-side variations.
- **Fallback worker integration**: when a usage-limit or connection error is detected, load `workers.fallbacks` from `WorkerConfig` and try each fallback command in order until one succeeds or all are exhausted.

### Out of scope

- Detecting usage limits from non-zero exit codes (these are already handled as execution failures).
- Provider-specific error parsing (e.g. parsing JSON error responses from OpenAI/Anthropic APIs) — the spawned CLI tools may or may not expose these in their stdout.
- Automatic retry with backoff (this is a separate concern from detection).
- Modifying the spawned worker processes themselves.

## Fallback Worker Design

When usage-limit detection triggers, the system should:

1. Read `workers.fallbacks` from the loaded `WorkerConfig` (a `string[][]` — ordered list of fallback worker commands).
2. Re-attempt the current task execution with the first fallback command.
3. If the fallback also hits a usage-limit or connection error, try the next fallback in order.
4. On fallback exhaustion (all fallbacks tried and failed), surface a clear error and stop the run.
5. Thread the fallback array through `verify-repair-loop` and `task-execution-dispatch` so both execution and verification/repair phases can use the active fallback.
6. When a fallback succeeds, continue the run with that fallback for subsequent tasks (do not revert to the primary worker for the remainder of the current run).

### Config example

```json
{
  "workers": {
    "default": ["opencode", "run", "--file", "$file", "$bootstrap"],
    "tui": ["opencode", "$bootstrap"],
    "fallbacks": [
      ["claude", "-p", "$bootstrap"],
      ["aider", "--message-file", "$file"]
    ]
  }
}
```

## Acceptance Criteria Outline

- When all three phases (execution, verification, repair) return identical or near-identical stdout, rundown detects this and reports a usage limit error instead of silently continuing.
- The error message clearly indicates the suspected cause ("Possible API usage limit: identical responses across execution and verification phases").
- The verify-repair loop short-circuits on detection — no further repair attempts are made.
- Detection does not trigger for legitimate short outputs like `OK` on a passing verification.
- The comparison logic is unit-testable in isolation.
- No changes to the spawned worker processes are required.

## Edge Cases

- **Detached mode**: no stdout is captured at all — detection is impossible and should be skipped.
- **TUI mode without capture**: stdout passes through to the terminal — detection is impossible unless `captureOutput` is enabled.
- **Empty stdout from all phases**: could indicate a crash rather than a usage limit — should be handled as a separate failure path (already partially handled by verification's "empty output" detection).
- **Tool expansion tasks**: stdout is parsed for sub-item insertion, not for comparison — detection should be limited to the default worker execution path.
- **Inline CLI tasks**: these are shell commands, not LLM calls — skip detection for this branch.
- **First verification passes (returns `OK`)**: no repair phase runs, so two-phase comparison (execution vs verification) is sufficient if the execution output also looks like an error message.

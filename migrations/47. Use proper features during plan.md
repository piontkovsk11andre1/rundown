## Problem statement

The plan prompt (`.rundown/plan.md` and the built-in `DEFAULT_PLAN_TEMPLATE` in `src/domain/defaults.ts:226`) instructs the planning agent to insert `- [ ]` TODO items, but gives the agent zero guidance about rundown's built-in task prefixes, directive parents, or profile system. As a result, the planner produces plain `- [ ]` items for every task — even when some tasks would benefit from verify-only execution, fast (no-verify) execution, profile assignment, or directive grouping. The operator then has to manually rewrite planned items to use those features.

## Rundown features the planner should be aware of

### Verify-only prefix (`verify:` / `confirm:` / `check:`)

When a task only needs to confirm that something is already true (tests pass, linting is clean, docs are up to date), it should use a verify-only prefix. This skips the execute phase entirely and goes straight to verify.

Single-task form:
```md
- [ ] verify: all tests pass
- [ ] confirm: linting is clean
- [ ] check: docs are up to date
```

Directive-parent form (groups multiple verify-only children):
```md
- check:
  - [ ] All tests pass
  - [ ] Linting clean
```

Source references:
- Built-in handler registration: `src/domain/builtin-tools/index.ts:14-32`
- Intent classification regex: `src/domain/task-intent.ts:29` (`EXPLICIT_VERIFY_PREFIX`)
- Iteration mode resolution: `src/application/iteration-mode.ts:64-96`
- Directive parent pattern: `src/domain/parser.ts:107`

### Fast execution prefix (`fast:` / `raw:`)

When a task is small, mechanical, or low-risk and does not need a separate verification pass, it should use the fast prefix. This executes the task but skips verification entirely — the inverse of verify-only.

Single-task form:
```md
- [ ] fast: update changelog headings
- [ ] raw: regenerate API docs summary
```

Directive-parent form (groups multiple fast children):
```md
- fast:
  - [ ] Fix typo in release notes
  - [ ] Update one CLI flag example
```

Source references:
- Intent classification regex: `src/domain/task-intent.ts:33` (`FAST_EXECUTION_PREFIX`)
- Directive parent pattern: `src/domain/parser.ts:109`
- Iteration mode skip logic: `src/application/iteration-mode.ts:93`

### Profile modifier (`profile:`)

Profiles select alternate worker configurations (model, args) from `.rundown/config.json`. Tasks that need a more capable model (complex reasoning) or a faster/cheaper model (trivial work) should declare a profile.

Three supported forms:
1. File-level frontmatter: `profile: complex` (YAML frontmatter, applies to all tasks)
2. Directive parent: `- profile: fast` with child checkboxes
3. Inline prefix modifier: `- [ ] profile: fast, verify: tests pass`

Resolution cascade (lowest to highest priority):
1. `config.defaults`
2. `config.commands.<command>`
3. File frontmatter `profile:`
4. Directive parent `profile:`
5. Prefix modifier `profile:`
6. CLI `--worker` / `-- <command>`

Source references:
- Profile modifier tool: `src/domain/builtin-tools/profile.ts`
- Worker config resolution cascade: `src/domain/worker-config.ts:158` (`resolveWorkerConfig`)
- Profile directive pattern: `src/domain/parser.ts:105`

### Prefix chain composition

Modifiers and handlers can be composed in a single task using `, ` or `; ` separators:
```md
- [ ] profile: fast, verify: release checks pass
- [ ] profile: complex; memory: capture migration constraints
```

Composition rules:
- Segments split on `, ` or `; ` only when the next segment starts with a known tool prefix
- Modifier tools (e.g. `profile:`) apply left-to-right and patch context
- Handler tools (e.g. `verify:`, `fast:`, `memory:`) are terminal and execute behavior
- Mixed explicit intent prefixes follow first-prefix precedence (`verify: fast: ...` -> verify-only)

Source reference: `src/domain/prefix-chain.ts:111` (`parsePrefixChain`)

### Directive parent grouping

Plain (non-checkbox) list items can provide context inheritance to child checkbox tasks. This avoids repeating the same prefix on every child.

Supported directive parents:
- `- profile: <name>` — children inherit that profile
- `- verify:` / `- confirm:` / `- check:` — children are verify-only
- `- fast:` / `- raw:` — children skip verification

Source references:
- Parser directive patterns: `src/domain/parser.ts:105-115`
- Documentation: `docs/configuration.md:143-174`

### Memory capture prefix (`memory:` / `memorize:` / `remember:` / `inventory:`)

Tasks that should capture and persist knowledge for future runs use memory prefixes.

```md
- [ ] memory: capture migration constraints
```

### Include prefix (`include:`)

Tasks that delegate to another markdown file's task list:

```md
- [ ] include: ./release-subtasks.md
```

## Design considerations

### How much feature detail belongs in the plan prompt?

The prompt needs to be concise enough that the planner agent doesn't get confused by excessive documentation, but detailed enough that it can correctly apply prefixes. A good approach is:
- A brief catalog of available prefixes with one-line descriptions
- Clear guidance on when to use each (verify-only for checks, fast for trivial tasks, profile for model selection)
- A few examples showing single-task and directive-parent forms
- A note about prefix composition syntax (`, ` or `; ` separators)

### Should the planner read config.json for available profiles?

The planner could be made config-aware by injecting available profile names as a template variable (e.g., `{{availableProfiles}}`). However, this would require a code change to the plan template rendering pipeline. A simpler first step is to hardcode guidance about the profile system and let the document author specify available profiles in the task document body or frontmatter.

### Template variable injection approach

An alternative to hardcoding feature docs in the prompt is injecting them via a template variable (e.g., `{{featureReference}}`). This would keep the plan prompt clean and allow the feature reference to be maintained separately. However, this requires infrastructure changes and is more complex than the direct approach.

### Consistency between project-local and built-in templates

Any changes to `.rundown/plan.md` should also be reflected in `DEFAULT_PLAN_TEMPLATE` (and `DEFAULT_DEEP_PLAN_TEMPLATE`) so that projects without a local config still benefit. The two should stay in sync.

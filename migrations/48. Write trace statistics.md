## Feature overview

When a task completes, rundown should optionally insert human-readable trace statistics as indented sub-items beneath the checked task in the source Markdown file. The statistics summarize execution cost and timing and are written inline so the author can review run characteristics without inspecting JSONL artifacts.

The feature is opt-in, configured via `.rundown/config.json`, and should produce sensible output with zero configuration when a user simply enables it.

## Existing architecture context

### Task completion flow

When a task passes verification (or verification is skipped), `completeTaskIteration` at `src/application/complete-task-iteration.ts:214` calls `checkTaskUsingFileSystem(task, dependencies.fileSystem)` to mark the checkbox. This delegates to `markChecked` in `src/domain/checkbox.ts:12`, which performs a line-targeted `[ ]` to `[x]` substitution on the source file.

After the checkbox is marked, control flows to `afterTaskComplete` (from `src/application/run-lifecycle.ts:95`) for hooks and optional commits. The trace statistics insertion should happen between the checkbox mark and the lifecycle hooks so the committed file already includes the statistics.

### Sub-item insertion precedent

The codebase already has a mature sub-item insertion mechanism used by the planner and tool-expansion flows:

- `insertSubitems` in `src/domain/planner.ts:185` splices indented child lines immediately after the parent task line. It computes a two-space child indent via `computeChildIndent` (`src/domain/planner.ts:171`) and uses `Array.prototype.splice` on the split source lines.
- `src/infrastructure/planner-io.ts` and `src/application/task-execution-dispatch.ts:406` both demonstrate the read-insert-write pattern: read the source, call `insertSubitems`, write back.
- The `SubItem` interface (`src/domain/parser.ts:56`) already models plain (non-checkbox) list items nested under tasks, with `text`, `line`, and `depth` fields.

The trace statistics feature can reuse `insertSubitems` or a similar helper to write formatted statistics lines below the newly checked task.

### Trace data already collected

The trace subsystem (`src/domain/trace.ts`) defines 22 event types with strongly typed payloads. The following events carry data directly useful for statistics output:

- `task.completed` (`TaskCompletedPayload`): `total_duration_ms`, `phases_count` â€” available at `src/application/trace-run-session.ts:576`.
- `timing.waterfall` (`TimingWaterfallPayload`): per-phase `duration_ms`, `idle_time_ms`, `total_wall_time_ms`, `total_worker_time_ms`.
- `prompt.metrics` (`PromptMetricsPayload`): `char_count`, `estimated_tokens` (chars / 4 heuristic at `src/application/trace-run-session.ts:439`).
- `phase.completed` (`PhaseCompletedPayload`): `duration_ms`, `stdout_bytes`, `stderr_bytes` per phase.
- `verification.efficiency` (`VerificationEfficiencyPayload`): `total_verify_attempts`, `total_repair_attempts`.

The `TraceRunSession` object (created by `createTraceRunSession` in `src/application/trace-run-session.ts:300`) accumulates this data in `ActiveTraceRun` state and already computes the timing waterfall at `emitTimingWaterfall` (line 607). The statistics feature needs to read from this same state rather than from the JSONL file after the fact.

### Configuration system

Worker configuration lives in `.rundown/config.json` and is loaded by `createWorkerConfigAdapter` (`src/infrastructure/adapters/worker-config-adapter.ts:129`). The file is parsed by `validateWorkerConfig` (line 109), which currently accepts three top-level keys: `defaults`, `commands`, and `profiles` (plus `toolDirs` mentioned in docs).

The config schema is validated at `src/infrastructure/adapters/worker-config-adapter.ts:109` and typed via `WorkerConfig` in `src/domain/worker-config.ts:43`. Adding a `traceStatistics` key to the config requires:

1. Extending the `WorkerConfig` interface (or creating a separate config type for non-worker settings).
2. Extending the validator to accept and validate the new key.
3. Providing defaults when the key is absent.

Currently `config.json` is exclusively a worker-configuration file. The trace statistics setting is not worker configuration â€” it controls post-completion markdown output. This may warrant either expanding the config.json schema to cover runtime behavior settings, or adding a separate top-level key (e.g., `traceStatistics`) that the validator passes through.

### Trace flag (`--trace`)

The `--trace` CLI flag (`src/presentation/cli.ts:149`) is a boolean that enables JSONL trace output to `.rundown/runs/<id>/trace.jsonl`. It controls whether the `TraceRunSession` emits events and whether the trace-enrichment pass runs.

The inline statistics feature is conceptually separate from `--trace`. A user might want inline statistics without full JSONL tracing, or vice versa. The design should support both independently:

- `--trace` controls JSONL event output and trace-enrichment.
- `config.json` `traceStatistics` (or a CLI flag) controls inline Markdown statistics.

## Proposed configuration schema

A new top-level `traceStatistics` key in `config.json`:

```json
{
  "traceStatistics": {
    "enabled": true,
    "fields": [
      "total_time",
      "execution_time",
      "verify_time",
      "repair_time",
      "tokens_estimated",
      "phases_count",
      "verify_attempts",
      "repair_attempts"
    ]
  }
}
```

### Default behavior

When `traceStatistics` is absent or `enabled` is not explicitly set:

- **If `--trace` is active**: statistics are written with a sensible default field set (e.g., `total_time` and `tokens_estimated`).
- **If `--trace` is not active**: no statistics are written.

This provides useful output without any configuration and avoids surprising users who did not opt in.

### Field registry

Each field maps to a known data source in the trace session:

| Field name | Source | Format example |
|---|---|---|
| `total_time` | `task.completed.total_duration_ms` | `total time: 522s` |
| `execution_time` | Sum of `phase.completed` where phase = `execute` | `execution: 5s` |
| `verify_time` | Sum of `phase.completed` where phase = `verify` | `verify: 12s` |
| `repair_time` | Sum of `phase.completed` where phase = `repair` | `repair: 8s` |
| `idle_time` | `timing.waterfall.idle_time_ms` | `idle: 3s` |
| `tokens_estimated` | Sum of `prompt.metrics.estimated_tokens` | `tokens estimated: 429391` |
| `phases_count` | `task.completed.phases_count` | `phases: 3` |
| `verify_attempts` | `verification.efficiency.total_verify_attempts` | `verify attempts: 2` |
| `repair_attempts` | `verification.efficiency.total_repair_attempts` | `repair attempts: 1` |

Phase-specific times (`execution_time`, `verify_time`, `repair_time`) should be rendered as nested sub-items under `total_time` for visual hierarchy, matching the example format in the original specification.

## Output format specification

Statistics are written as indented plain list items (matching existing `SubItem` conventions) immediately after the checked task line:

```markdown
- [x] Do this.
    - total time: 522s
        - execution: 5s
        - verify: 12s
    - tokens estimated: 429391
```

Indentation follows the two-space child indent rule from `computeChildIndent`. Phase breakdowns nest one additional level under their parent metric.

### Formatting rules

- Durations display as whole seconds (`Math.round(ms / 1000)`) with an `s` suffix. Sub-second runs show `<1s`.
- Token counts display as plain integers with no separator.
- Fields that have a zero or null value are omitted to keep output clean.
- The order of fields matches the `fields` array in config (or the default order).

## Integration points

### Where to insert statistics

The insertion should happen in `completeTaskIteration` (`src/application/complete-task-iteration.ts`) immediately after `checkTaskUsingFileSystem` (line 214) and before `afterTaskComplete` (line 237). At this point:

1. The checkbox is already marked `[x]`.
2. The `traceRunSession` has accumulated all phase timings and prompt metrics.
3. The source file is still in a mutable window before lifecycle hooks and commits.

### Data collection

The `TraceRunSession` already tracks phase timings in `ActiveTraceRun.phaseTimings` and computes the waterfall. A new method (e.g., `collectStatistics()`) on the session can return a structured snapshot without modifying the existing event-emission logic.

Prompt metrics (token estimates) are currently only emitted as trace events. The session would need to also accumulate prompt metric totals in the `ActiveTraceRun` state for statistics access.

### File write coordination

After `markChecked` writes the updated source, the statistics insertion reads the source back, inserts formatted lines, and writes again. This is the same read-modify-write pattern used by `insertSubitems` in the planner flow. File locking is not needed because rundown processes tasks sequentially.

## Scope

### In scope

- New `traceStatistics` config key with `enabled` and `fields` properties.
- Default field set when config is absent but `--trace` is active.
- Statistics line formatting and insertion after checkbox marking.
- Accumulation of prompt metrics totals in `TraceRunSession` state.
- A `collectStatistics()` method on `TraceRunSession` that returns data needed to format statistics lines.
- Statistics insertion helper (possibly reusing or extending `insertSubitems`).

### Out of scope

- Changing JSONL trace output format or adding new trace event types.
- Statistics for `discuss`, `plan`, or `research` commands (only `run` tasks).
- Per-round statistics in `--redo` / `--rounds` mode (only the final completion gets statistics).
- Retroactive statistics insertion for already-completed tasks.
- Interactive/streaming statistics display.

## Edge cases and constraints

- **Dry-run mode**: Statistics should not be written when execution is simulated or detached (`status === "detached"`).
- **Tool expansion**: When `insertSubitems` inserts child TODOs from tool expansion (at `src/application/task-execution-dispatch.ts:406`), the loop continues without marking the parent complete. Statistics are only relevant for the final completion.
- **Inline CLI tasks**: These complete without a worker phase. Statistics should still capture `total_time` and `execution_time` from the CLI block duration.
- **Pre-existing sub-items**: The parser already handles sub-items under tasks. Newly inserted statistics lines must not collide with existing sub-items. Insertion should place statistics after any existing children and sub-items.
- **Round-based execution (`--rounds`)**: Each round resets checkboxes. Statistics should only appear on the final round's completion to avoid accumulation of stale sub-items.
- **Checkbox reset (`--redo`)**: Pre-run reset calls `resetAllCheckboxes` but does not remove sub-items. A cleanup step may be needed to strip prior statistics before re-running.
- **Config validation**: The `fields` array should reject unknown field names at config load time with a clear error message, consistent with the existing `validateWorkerConfig` pattern.
- **Concurrent file access**: Not a concern â€” rundown runs tasks sequentially within a single process.

## Compatibility notes

- The `insertSubitems` function assumes items are unchecked TODO lines and strips bullet markers. The statistics insertion needs a lower-level helper or a variant that preserves arbitrary formatted text.
- The `SubItem` interface in the parser (`src/domain/parser.ts:56`) will parse the statistics back as plain sub-items on subsequent runs. This is benign â€” the statistics lines carry no semantic meaning to the task execution system.
- The config schema extension must remain backward-compatible: `config.json` files without `traceStatistics` should behave identically to today.

# Decompose `cli.ts`

## Goal

Reduce `src/presentation/cli.ts` (currently ~1,054 lines) to a slim command registry of ~200–300 lines by extracting option parsers, app initialization, lock lifecycle management, invocation logging, and CLI infrastructure helpers into focused modules. The file is the only presentation-layer module with substantial size, and its problem is horizontal bloat — 30+ repetitive parser functions, 13 command definitions with duplicated option-parsing patterns, and unrelated infrastructure concerns all sharing a single namespace.

## Background

`cli.ts` is the presentation entry point for all CLI commands. It uses `commander` to define the command tree, parse argv, validate options, initialize the application container, and dispatch to use cases via `getApp().<useCase>(...)`. The file also manages process signal handlers for lock cleanup, session ID generation, global invocation logging, argv alias rewriting, and test-mode termination.

The file has no architecture violations — it stays strictly in the presentation layer and delegates all business logic through the application ports. The issue is purely organizational: unrelated presentation concerns are packed together.

## Current structure

### Constants and types (~30 lines, L1–30)

Imports, allowed-value arrays (`RUNNER_MODES`, `PROMPT_TRANSPORTS`, `SORT_MODES`, `REVERT_METHODS`), sentinel constants, `CliExitSignal` class.

### Module-level state and helpers (~50 lines, L31–80)

`readCliVersion()`, `createSessionId()`, mutable module state (`workerFromSeparator`, `cliInvocationLogState`, `cliInvocationArgv`, `program`, `app`), `getApp()`.

### Lock release signal handlers (~90 lines, L80–170)

`LOCK_RELEASE_SIGNAL_HANDLER_MARKER`, `LOCK_RELEASE_EXIT_HANDLER_MARKER`, `LOCK_RELEASE_STATE_SYMBOL`, `LockReleaseState`, `getLockReleaseState()`, `releaseHeldFileLocksBestEffort()`, `hasRegisteredLockReleaseHandler()`, `hasRegisteredLockReleaseExitHandler()`, `registerLockReleaseSignalHandlers()`. This is a self-contained concern responsible for ensuring file locks are released on SIGINT/SIGTERM/exit.

### App initialization for invocation (~60 lines, L170–230)

`createAppForInvocation()`, `resolveConfigDirForInvocation()`, `resolveExplicitConfigDirFromArgv()`. Creates the application container with logged output port and config dir resolution.

### Command definitions (~440 lines, L230–670)

13 commands registered on the `commander` program:
- `run` (~110 lines) — largest, ~30 options, action handler parses all options and delegates to `app.runTask()`
- `discuss` (~60 lines) — ~20 options, similar pattern
- `reverify` (~50 lines) — ~15 options
- `revert` (~30 lines) — ~8 options
- `next` (~10 lines) — minimal
- `list` (~10 lines) — minimal
- `artifacts` (~15 lines) — minimal
- `log` (~15 lines) — minimal
- `plan` (~60 lines) — ~15 options, similar to `run`
- `unlock` (~5 lines) — trivial
- `init` (~5 lines) — trivial
- `runall` alias handled by argv rewriting, not a separate command

Every action handler follows the same pattern: parse each `opts` field → validate → pass to `getApp().<method>(...)`. The option parsing is inline and repetitive.

### Option parsers and validators (~250 lines, L670–920)

30+ small functions, each validating a single option type:
- Enum parsers: `parseRunnerMode`, `parsePromptTransport`, `parseSortMode`, `parseRevertMethod`
- Integer parsers: `parseRepairAttempts`, `parseScanCount`, `parseLastCount`, `parseLimitCount`, `parseCliBlockTimeout`
- Boolean resolvers: `resolveVerifyFlag`, `resolveNoRepairFlag`, `resolveIgnoreCliBlockFlag`
- String utilities: `normalizeOptionalString`, `resolvePlanMarkdownFile`
- Compound: `resolveExplicitConfigDirOption`, `validateExplicitConfigDirOption`, `allowsMissingExplicitConfigDir`
- Collector: `collectOption`

Many parsers share the same pattern: take optional string → default → regex test → `Number()` → `isSafeInteger` check → throw descriptive error. This pattern repeats 5 times for integer parsing alone.

### Action wrapper and logging (~40 lines, L870–910)

`withCliAction` — wraps each command action with config-dir validation, lazy app init, lock state registration, exit code handling, and fatal error emission.

### Entry point and argv processing (~60 lines, L910–970)

`parseCliArgs` (exported) — rewrites runall alias, splits worker separator, initializes invocation log state, conditionally pre-creates app, registers signal handlers, calls `program.parseAsync`. Also `shouldInitializeAppBeforeParse`, auto-parse block.

### Invocation logging infrastructure (~50 lines, L970–1020)

`createCliInvocationLogState`, `resolveConfigDirPathForInvocation`, `emitCliFatalError`, `appendCliFatalErrorToGlobalLog`, `configureCommanderOutputHandlers`, `appendCommanderFrameworkOutputToGlobalLog`. These connect the CLI to the global output log writer.

### CLI utility functions (~35 lines, L1020–1054)

`splitWorkerFromSeparator`, `rewriteRunallAlias`, `resolveInvocationCommand`, `terminate`, `isCliExitSignal`.

## Primary problems

### 1. Option parsers are repetitive and test-unfriendly

The 5 integer parsers (`parseRepairAttempts`, `parseScanCount`, `parseLastCount`, `parseLimitCount`, `parseCliBlockTimeout`) are nearly identical: `string → regex → Number → isSafeInteger → throw`. They differ only in the option name for the error message, whether zero is allowed, and whether `undefined` returns the default or `undefined`. A single generic `parseIntOption(value, name, { allowZero, default, min })` could replace all five.

The 4 enum parsers (`parseRunnerMode`, `parsePromptTransport`, `parseSortMode`, `parseRevertMethod`) all follow `cast → includes check → throw`. A generic `parseEnumOption(value, name, allowed, default)` could replace them.

### 2. Lock release handlers are an independent concern

The ~90 lines of signal handler machinery (`LOCK_RELEASE_*` markers, `LockReleaseState`, `getLockReleaseState`, handler registration, guard checks) have zero coupling to command definitions or option parsing. They interact with the module only through `app?.releaseAllLocks()` and `terminate()`.

### 3. Invocation logging is infrastructure plumbing

`createCliInvocationLogState`, `appendCliFatalErrorToGlobalLog`, `configureCommanderOutputHandlers`, `appendCommanderFrameworkOutputToGlobalLog` are logging adapters that don't belong alongside command definitions. They depend on `createGlobalOutputLogWriter` and `createNodeFileSystem` — infrastructure-layer imports that are otherwise absent from the presentation module.

### 4. Module-level mutable state complicates testability

`workerFromSeparator`, `cliInvocationLogState`, `cliInvocationArgv`, `app` are module-scope variables mutated across `parseCliArgs` and within action handlers. Grouping them into a single state struct and passing it explicitly would make the module testable without module reloading.

## Identified extraction targets

### 1. Extract option parsers → `cli-options.ts`

Move all option parsing and validation functions (~250 lines) to a presentation-layer module. Deduplicate integer parsers into a private `parseIntOption` helper and enum parsers into a `parseEnumOption` helper. Export each named parser for backward compatibility with tests. This is the highest-value extraction: it removes half the file's line count at zero risk.

Public surface:
- `parseRunnerMode`, `parsePromptTransport`, `parseSortMode`, `parseRevertMethod`
- `parseRepairAttempts`, `parseScanCount`, `parseLastCount`, `parseLimitCount`, `parseCliBlockTimeout`
- `resolveVerifyFlag`, `resolveNoRepairFlag`, `resolveIgnoreCliBlockFlag`
- `normalizeOptionalString`, `resolvePlanMarkdownFile`, `collectOption`

Dependencies: `domain/ports` (type-only, for `ProcessRunMode`, `PromptTransport`), `domain/sorting` (type-only, for `SortMode`).

### 2. Extract lock release handlers → `cli-lock-handlers.ts`

Move all lock release signal handler machinery (~90 lines) to its own presentation module. The module exports `registerLockReleaseSignalHandlers()` and `getLockReleaseState()`, receives `terminate` as a callback (to avoid the circular dependency on `cli.ts`).

Dependencies: none beyond Node.js process API and the application container type.

### 3. Extract invocation logging → `cli-invocation-log.ts`

Move `createCliInvocationLogState`, `resolveConfigDirPathForInvocation`, `emitCliFatalError`, `appendCliFatalErrorToGlobalLog`, `configureCommanderOutputHandlers`, `appendCommanderFrameworkOutputToGlobalLog`, `LoggedOutputContext` reference, and all `createGlobalOutputLogWriter`/`createNodeFileSystem` imports (~100 lines) to a dedicated module.

Public surface:
- `createCliInvocationLogState(argv): CliInvocationLogState`
- `emitCliFatalError(error, state?): void`
- `configureCommanderOutputHandlers(command, state?): void`

Dependencies: `infrastructure/adapters/global-output-log-writer`, `infrastructure/adapters/fs-file-system`, `infrastructure/runtime-artifacts`, `presentation/logged-output-port`.

### 4. Extract app initialization → `cli-app-init.ts`

Move `createAppForInvocation`, `resolveConfigDirForInvocation`, `resolveExplicitConfigDirFromArgv`, `resolveExplicitConfigDirOption`, `validateExplicitConfigDirOption`, `allowsMissingExplicitConfigDir`, and `CliInvocationLogState` interface (~100 lines) to a module that encapsulates application container creation for CLI invocations.

Public surface:
- `createAppForInvocation(argv, logState): ReturnType<typeof createApp>`
- `resolveConfigDirForInvocation(argv?, cwd?): ConfigDirResult | undefined`
- `validateExplicitConfigDirOption(argv, program): void`

Dependencies: `create-app`, `infrastructure/adapters/config-dir-adapter`, `presentation/logged-output-port`, `presentation/output-port`.

### 5. Extract CLI utilities → `cli-argv.ts`

Move `splitWorkerFromSeparator`, `rewriteRunallAlias`, `resolveInvocationCommand`, `terminate`, `isCliExitSignal`, `CliExitSignal`, `readCliVersion`, `createSessionId` (~80 lines) to a module for argv preprocessing and process lifecycle.

### 6. Simplify the residual `cli.ts`

After extractions, the file retains only:
- Imports from extracted modules
- `program` creation and global option (`--config-dir`)
- 13 command definitions (`.command(...)...action(withCliAction(...))`)
- `withCliAction` wrapper
- `parseCliArgs` (exported entry point)
- Module-scope state and auto-parse block

The command definitions themselves are unavoidably verbose due to `commander`'s API — each option is a method call chain. But the action handlers become concise once option parsing is imported from `cli-options.ts`.

## Expected outcome

| Module | Estimated size |
|---|---|
| `cli.ts` (residual) | ~250–300 lines |
| `cli-options.ts` | ~150–180 lines (after deduplication) |
| `cli-lock-handlers.ts` | ~80–90 lines |
| `cli-invocation-log.ts` | ~90–100 lines |
| `cli-app-init.ts` | ~80–100 lines |
| `cli-argv.ts` | ~70–80 lines |

Total: ~720–850 lines across 6 files vs. 1,054 lines in one file. Deduplication of integer/enum parsers saves ~50 lines. Each module has a single cohesive responsibility and can be tested independently. The residual `cli.ts` becomes a pure command registry — only `commander` API calls and delegation.

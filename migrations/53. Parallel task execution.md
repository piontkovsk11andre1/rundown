# Plan: Parallel Task Execution in Rundown

## TL;DR

Add a `parallel:` prefix that marks a parent task's direct children for concurrent execution. The parent itself becomes the synchronization barrier — it can't run until all children complete, which is already enforced by `filterRunnable()`. No new `wait:/sync:` primitives needed.

## Markdown Syntax

```markdown
- [ ] parallel: Set up all environments
  - [ ] cli: npm install
  - [ ] cli: pip install -r requirements.txt  
  - [ ] Configure database schema
- [ ] Run integration tests  ← waits for all above to finish (sequential resumes)
```

Nested parallelism is supported naturally:
```markdown
- [ ] parallel: Phase 1
  - [ ] parallel: Setup group A
    - [ ] Task A1
    - [ ] Task A2
  - [ ] parallel: Setup group B
    - [ ] Task B1
    - [ ] Task B2
- [ ] Phase 2  ← runs after Phase 1 completes
```

## Design

### New Intent: `parallel-group`

- Register `parallel:` as a new prefix in `task-intent.ts` (alongside `verify:`, `memory:`, `fast:`)
- Intent type: `"parallel-group"` 
- The `parallel:` task itself is NOT executed by a worker — it's a structural grouping marker
- When all children are checked, the parent is auto-checked (no worker invocation needed)

### Task Selection Changes

Current: `selectNextTask()` returns a single `Task | null`
New: `selectNextTask()` returns `Task[] | null` (batch of 1+ tasks)

Logic change in task-selection.ts / selector:
1. Find first runnable task as before
2. Check if that task's parent has intent `parallel-group`
3. If yes: return ALL runnable siblings under that parent (not just the first)
4. If no: return `[task]` (array of one — backward compatible)

### Execution Loop Changes (run-task-execution.ts)

Current: `while(true)` → select one → execute → complete → repeat
New: `while(true)` → select batch → execute batch concurrently → complete all → repeat

```
const batch = selectNextTask(...)
if (batch === null) break

if (batch.length === 1) {
  // Existing sequential path (no behavior change)
  await runTaskIteration(batch[0])
} else {
  // Parallel path
  await Promise.allSettled(batch.map(task => runTaskIteration(task)))
}
```

### Concurrency Safety

**File write serialization:**
- Multiple parallel tasks may need to update checkboxes in the same .md file
- Add a per-file async mutex (e.g., `AsyncMutex` keyed by file path)
- `checkbox-operations.ts` acquires mutex before read-modify-write
- Parser re-reads file content inside the mutex to avoid stale state

**Artifact isolation:**
- Each parallel task already gets its own artifact context (created per iteration)
- No changes needed — artifact directories are task-scoped

**Trace writer:**
- Each parallel task needs its own `TraceWriter` instance
- The shared `tasksCompleted` counter needs atomic increment (or post-join aggregation)

**Shared state partitioning:**
- `tasksCompleted`: aggregate after batch completes
- `runCompleted`: only set after batch + re-selection yields null  
- `deferredCommitContext`: only meaningful after final task; aggregate at batch boundary

### Completion of parallel: Parent

When all children of a `parallel:` parent are checked:
- `filterRunnable()` makes the parent eligible
- The parent has intent `parallel-group` → auto-check without worker execution
- Similar to how `end:` / `skip:` work — control flow, not work

### Error Handling

Strategy: **run-all** — All children attempt execution regardless of sibling failures. Failed ones stay unchecked. Parent stays unchecked since it has unchecked descendants. On re-run, only failed children retry. (Workers are expensive to start; tasks are independent by design.)

## Decisions

- Parent task IS the sync barrier — no `wait:` / `sync:` prefix needed
- `parallel:` parent auto-completes when children finish (no worker invocation)
- Error strategy: run-all (don't cancel siblings on failure)
- Concurrency is unbounded within a parallel group (all runnable siblings run at once)
- Children of a non-parallel parent still execute sequentially (backward compatible)
- **Git commits**: Only commit at the sync point (when `parallel:` parent auto-completes), not per-child. Children's changes accumulate and are committed together.
- **TUI workers**: Fall back to sequential execution when TUI-mode workers are detected in a parallel group.
- **CLI output**: Keep output handling simple for now — no buffering or labeling. More CLI output changes are coming soon and will address this holistically.

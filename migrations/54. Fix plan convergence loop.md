# Fix plan convergence loop

## Problem statement

The `rundown plan` command goes into a loop and never naturally converges. In unlimited mode (the default since migration 43), the scan loop runs until the emergency cap of 30 scans instead of stopping when plan coverage is sufficient.

### Root cause

Two reinforcing problems prevent convergence:

**1. Prompt permits rewording and reordering**

The `DEFAULT_PLAN_TEMPLATE` Rules section in `src/domain/defaults.ts` (line ~360) says:

```
- Insert new items where they fit best; you may add them in the middle of existing lists.
- You may reorder unchecked `- [ ]` items when it improves execution flow.
```

This was never the intended behavior. The plan operation should be **strictly additive**: each scan reviews whether the existing TODO list fully covers the workload described in the document, and if coverage is insufficient, appends new items after the last existing TODO. It should never reorder, reword, rephrase, or insert between existing items.

The same problem exists in `DEFAULT_DEEP_PLAN_TEMPLATE` (line ~422):

```
- Insert new child items where they fit best under the parent; you may reorder unchecked child `- [ ]` items when it improves execution flow.
```

And in the user template `.rundown/plan.md` (line ~72) which mirrors the same rules.

**2. Convergence detection counts gross additions, not net**

The convergence check in `src/application/plan-task.ts` (line ~831) uses `validationResult.stats.added === 0`. The stats are computed by `computeUncheckedStats()` in `src/domain/planner.ts` (line ~81), which counts **gross** additions using identity-based matching via `normalizeTodoCheckboxLine()`.

When the worker rewords an item (e.g., `- [ ] Fix bug` → `- [ ] Fix the critical bug`), the old normalized identity disappears (`removed += 1`) and a new one appears (`added += 1`). So `stats.added === 1 > 0` and convergence never fires.

**Self-sustaining cycle**: each scan reads existing TODOs → finds "improvements" → rewords/reorders → `added > 0` → loop continues → next scan sees reworded items → finds more "improvements" → repeat until emergency cap at scan 30.

### Original intent of the plan operation

1. Worker reviews the document and its existing TODO items.
2. Worker evaluates whether the existing TODOs fully cover the described workload.
3. If coverage is insufficient, worker appends new `- [ ]` items after the last existing TODO.
4. If coverage is already sufficient, worker leaves the file unchanged.
5. Next scan re-reads the file with the newly added items and repeats the coverage check.
6. When a scan finds nothing to add, convergence is detected and the loop stops.

The worker must never reorder, reword, rephrase, rewrite, or move existing items (checked or unchecked).

## Implementation

### 1. Fix plan template rules (`src/domain/defaults.ts`)

In `DEFAULT_PLAN_TEMPLATE` (line ~358), replace the current Rules section:

```
Rules:
- Add only unchecked TODO items using `- [ ]` syntax.
- Insert new items where they fit best; you may add them in the middle of existing lists.
- You may reorder unchecked `- [ ]` items when it improves execution flow.
- Do not change any `- [ ]` item to `- [x]`.
- Do not remove, rewrite, or move any completed `- [x]` item.
- Do not output a proposed list on stdout; apply edits to `{{file}}` directly.
```

With strictly additive rules:

```
Rules:
- Add only unchecked TODO items using `- [ ]` syntax.
- Append new items after the last existing TODO item in the list. Do not insert between existing items.
- Do not reorder, reword, rephrase, or rewrite any existing TODO item (checked or unchecked).
- Do not change any `- [ ]` item to `- [x]`.
- Do not remove or move any existing item (checked or unchecked).
- Do not output a proposed list on stdout; apply edits to `{{file}}` directly.
- If plan coverage is already sufficient, leave the file unchanged.
```

### 4. Strengthen convergence detection (`src/application/plan-task.ts`)

As defense-in-depth, change the convergence check to use **net additions** instead of gross additions. This ensures convergence even if the worker violates prompt rules and rewords items.

At line ~831 in `src/application/plan-task.ts`, the current check is:

```typescript
if (validationResult.stats.added === 0) {
  convergenceOutcome = "converged-no-additions";
```

Change to detect net-zero additions (rewrites produce `added = N, removed = N`, net = 0):

```typescript
const netAdditions = validationResult.stats.added - validationResult.stats.removed;

if (netAdditions <= 0 && validationResult.stats.removed > 0) {
  // Worker rewrote existing items without adding new ones — revert file to pre-scan state.
  dependencies.fileSystem.writeText(source, scanBaseline);
  latestDocumentSource = scanBaseline;
  convergenceOutcome = "converged-rewrite-only";
  emit({ kind: "warn", message: "Planner scan " + scanIndex + " rewrote existing TODO items without net additions. File reverted. Treating as converged." });
  planSuccessCount += 1;
  emit({ kind: "group-end", status: "success" });
  continue;
}

if (netAdditions <= 0) {
  convergenceOutcome = "converged-no-additions";
```

When the worker only rewrites items (net additions ≤ 0 but file changed), **revert the file** to the pre-scan baseline. This output was never desired, so discarding it is correct and prevents text drift across scans.

### 5. Add diminishing-returns safeguard (`src/application/plan-task.ts`)

As an additional safety net, track consecutive scans that produce very few new items. If 2 consecutive scans each add ≤ 1 net item, declare convergence. This catches tail-end single-item churn where the model keeps discovering exactly one more thing per scan.

- [x] Add a `consecutiveLowAdditionScans` counter to the scan loop state variables (near line ~660, next to `insertedTotal`)
- [x] After successful scan insertion, increment the counter when net additions ≤ 1 and reset to 0 otherwise
- [x] When `consecutiveLowAdditionScans >= 2`, set convergence to `"converged-diminishing"` and break the loop
- [x] Add `"converged-diminishing"` to `PlanConvergenceOutcome` in `src/application/plan-task.ts` and in `src/domain/trace.ts`
- [x] Update `buildPlanConvergenceMetadata()` to include `"converged-diminishing"` in the `planConverged` boolean check

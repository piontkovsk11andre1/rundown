## Context

The current flow reaches a state where task execution is successful, verification passes, and the task is marked done, but the commit step still runs and fails because there are no staged or unstaged changes. This is operationally noisy and makes an otherwise successful run look broken.

The observed output confirms that this is not a Git repository health issue. The branch is valid and clean; the failure is specifically caused by invoking `git commit` in a no-op state.

## Problem framing

The system currently treats `git commit` as an unconditional post-task action under `--commit`/`--commit-mode file-done`, even when the work performed in that step does not mutate tracked files. That behavior needs reinforcement so "nothing to commit, working tree clean" is handled as an expected branch of control, not an error condition.

This should preserve strict failure behavior for real commit errors while downgrading no-change commits to a clean, intentional outcome.

## Expected behavior

When commit mode is active and the run reaches commit:

- If there are committable changes, commit proceeds as normal.
- If there are no committable changes, the run should not be marked failed.
- The UI/log should communicate that commit was skipped because the working tree is clean.
- Task success and verification success should remain intact and visible.
- Exit status should reflect success for this no-op commit case.

## Scope boundaries

In scope:

- Detection and handling of empty-commit conditions in the commit pipeline.
- Message/telemetry normalization so clean-tree commit attempts are not surfaced as warnings/errors.
- Compatibility with existing commit modes, especially `file-done`.

Out of scope:

- Automatic push behavior changes.
- Rewriting commit semantics for unrelated Git failures.
- Changing task verification rules.
- Forcing empty commits by default.

## Technical considerations

Likely control-point characteristics to account for in implementation planning:

- Pre-flight check should rely on Git state before invoking commit (index + working tree relevant to current mode).
- Defensive fallback should still recognize and normalize Git’s "nothing to commit" response if pre-flight misses an edge case.
- Classification should distinguish:
  - clean-tree/no-op commit (expected),
  - recoverable process issues (if any policy exists),
  - true hard failures (must still fail run).
- Output formatting should remain consistent with existing rundown phase logs (informational rather than warning for no-op commit).

## Edge cases to preserve

- Repository clean because only ephemeral artifacts changed and are ignored.
- Repository clean after prior step already committed equivalent changes.
- Mixed state where unrelated untracked files exist but no staged tracked diff for commit mode.
- Branch ahead/behind status present; this should not be conflated with commit eligibility.
- Commit hooks/config interactions should continue to fail appropriately when there are real changes and commit is attempted.

## Acceptance criteria outline

A planning/implementation pass should ensure:

- Clean-tree scenario with `--commit` does not produce a failed run.
- Log line explicitly indicates commit skipped due to no changes.
- Existing successful commit path remains unchanged.
- Real commit failures still surface as failures with actionable error output.
- Regression coverage exists for at least one no-change file-done flow and one normal commit flow.

## Integration notes

This change sits at the boundary between task completion orchestration and Git execution. It should be introduced where commit decisions are made, not only where errors are rendered, so behavior and exit codes remain coherent.

If there is already a reusable Git helper abstraction, the no-op classification should live there (or be exposed there) to avoid duplicating string matching/parsing logic across phases.

## Risks

- Over-broad error normalization could hide legitimate Git failures.
- Under-broad matching could still leak no-op commit failures in some environments/locales.
- Message changes without status-code changes could still break automation expectations.

Mitigation is to make no-op detection explicit and test both positive and negative branches with stable assertions on status and phase reporting.

## Commit decision point located (`--commit` + `--commit-mode file-done`)

Primary decision point for deferred file-level commit is in:

- `src/application/run-task-execution.ts`
  - `deferCommitUntilPostRun` is computed as:
    - `commitAfterComplete && (resetAfter || (effectiveRunAll && commitMode === "file-done"))`
  - This is the gate that routes commit timing from per-task into end-of-file/end-of-run behavior when `file-done` is active in effective run-all flows.

Commit execution implementation is in:

Git staging/commit plumbing is in:

  - `commitCheckedTaskWithGitClient()` performs:
    - `git add -A -- :/ ...excludes`
    - `git commit -m <message>`

## Current clean-tree control flow (as-is)

Observed behavior for clean-tree runs with commit enabled is explained by the current two-stage flow:

1. **Pre-run repository gate (`run-task-execution.ts`)**
   - If `--commit` is enabled, runner checks repo presence and cleanliness up front via `isWorkingDirectoryClean()`.
   - `isWorkingDirectoryClean()` uses `git status --porcelain` with rundown artifact/lock exclusions.
   - This check enforces "start from clean tree" (rejects dirty start), not "there are changes to commit later".

2. **Post-task deferred commit (`run-task-execution.ts` -> `run-lifecycle.ts`)**
   - In `file-done` mode, commit is deferred and executed after successful run completion.
   - `afterTaskComplete()` does not run a "committable diff exists" preflight before commit.
   - It unconditionally attempts `git add -A` then `git commit -m`.

3. **No-change outcome classification (`run-lifecycle.ts`)**
   - If `git commit` returns "nothing to commit, working tree clean", the thrown Git error is caught by broad `catch` and emitted as:
     - `warn: --commit failed: <error>`
   - Control then continues (warning-only), but log semantics present this expected no-op branch as failure text.

4. **Why this appears noisy in `file-done`**
   - `file-done` commit happens after success states (task checked, verify passed), so warning is emitted at the end of an otherwise successful flow.
   - Result: operationally successful run with misleading commit-failed warning for clean-tree no-op.

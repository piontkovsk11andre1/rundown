# Plan: `rundown query <text>` Command

## TL;DR

Add a new `query` command that takes a natural language question, researches the codebase, generates a step-by-step plan to answer it, executes each step with intermediate files in a workdir, and aggregates results into formatted output (markdown or JSON). Architecturally: **make (ephemeral) → research → plan → run → aggregate/format**. Reuses existing explore pipeline with workdir management and output formatting layered on top.

## Design Decisions

- **Ephemeral source document**: query creates a temp `.md` file inside `.rundown/runs/<run-id>/query.md`, never pollutes user workspace
- **Workdir per run**: `.rundown/runs/<run-id>/workdir/` — each execution step can write intermediate output files there. Path is communicated via template variable `{{workdir}}`, not special infrastructure
- **`--dir <path>`** with cwd default: sets the analysis root directory
- **`--format markdown|json|yn|success-error`**: output format (markdown default)
  - `markdown` — full prose response
  - `json` — structured `{ query, steps[], output }`
  - `yn` — final answer is a single `Y` or `N` token. Useful for scripting/CI checks ("does this codebase have SQL injection vulnerabilities?")
  - `success-error` — outputs `success` or `failure: <reason>`, matching the verification verdict contract. Useful for CI gates and composing with other rundown commands ("does the migration plan cover all breaking changes?")
- **`--output <file>`**: redirect final output to file instead of stdout
- **Two output modes — stream vs file**:
  - **Stream mode** (default, no `--output`/`--format json`): Each step's findings print directly to CLI stdout as the step completes. Worker writes to stdout, output is captured and echoed to terminal in real-time. No intermediate workdir files. Interactive and immediate.
  - **File mode** (triggered by `--output <file>` or `--format json`): Steps write findings to `{{workdir}}/step-{{taskIndex}}.md`. After all steps complete, aggregate and format the result, then write to file or stdout. Needed when structured output (JSON) or a single coherent file is required.
  - The execution template switches behavior based on mode: stream template tells worker to output findings to stdout; file template tells worker to write to `{{workdir}}/step-{{taskIndex}}.md`.
- **Reuse explore pipeline**: query chains make → research → plan → run → aggregate, following the same patterns as `do` but with query-specific templates
- **Template-based seed (not worker-generated)**: The seed document is rendered from `DEFAULT_QUERY_SEED_TEMPLATE` with `{{query}}` and `{{dir}}` placeholders. This frames the intent as research/investigation rather than implementation. The research phase does the enrichment. Users override via `.rundown/query-seed.md`.
- **`question:` tasks for user clarification**: Plan phase can generate early `question:` prefix tasks that use the question/answer tool to prompt the user for clarification before heavy investigation begins. This narrows scope, confirms output preferences, or resolves ambiguity. Answers are captured to memory so subsequent steps reference them.

## Seed Template Design

The seed template is critical because it frames the entire pipeline. Unlike `make` (which writes user text verbatim), `query` wraps the user's question in research-oriented framing so that:
- **Research** phase analyzes the codebase rather than prescribing implementation
- **Plan** phase generates investigation steps ("Trace the message handler chain") not implementation steps ("Add retry logic")
- **Execution** steps produce findings/content written to workdir, not code changes

Example rendered seed for `rundown query "describe communication patterns between TDS.MT components" --dir C:\Work\tds-mt`:

```markdown
# Query: describe communication patterns between TDS.MT components

## Objective

Research the codebase and produce a comprehensive answer to the query above.
This document describes what to investigate — not what to build.

## Analysis directory

`C:\Work\tds-mt`

## Output directory

Each investigation step writes findings to `<workdir path>`.
```

## Steps

### Phase 1: Domain Layer — Templates & Variables

1. **Add query templates to `src/domain/defaults.ts`**
   - `DEFAULT_QUERY_SEED_TEMPLATE`: Generates the initial markdown document from query text. Wraps it in research-oriented framing with `{{query}}`, `{{dir}}`, `{{workdir}}` placeholders.
   - `DEFAULT_QUERY_EXECUTION_TEMPLATE` (file mode): Instructs worker to write step output to `{{workdir}}/step-{{taskIndex}}.md`. Worker produces *content*, not code changes.
   - `DEFAULT_QUERY_STREAM_EXECUTION_TEMPLATE` (stream mode): Instructs worker to output findings to stdout directly. No workdir writes.
   - `DEFAULT_QUERY_YN_SEED_TEMPLATE`: Frames the query as a yes/no question. Seed includes "conclude with a single Y or N verdict".
   - `DEFAULT_QUERY_SUCCESS_ERROR_SEED_TEMPLATE`: Frames as pass/fail check. Seed includes "conclude with `success` or `failure: <reason>`".
   - `DEFAULT_QUERY_AGGREGATION_TEMPLATE` (file mode only): Reads all `workdir/*.md` files and assembles into a coherent response.

2. **Add `workdir` and `dir` to template variables** in `src/domain/template.ts` (*parallel with step 1*)
   - Add `workdir?: string` and `dir?: string` to the `TemplateVars` interface
   - Wire through template rendering so `{{workdir}}` and `{{dir}}` resolve correctly

### Phase 2: Application Layer — Query Orchestration

3. **Create `src/application/query-task.ts`** (*depends on 1, 2*)
   - `createQueryTask(dependencies: QueryTaskDependencies)` → `(options: QueryTaskOptions) => Promise<number>`
   - Dependencies: same as `ResearchTaskDependencies` + `RunTaskExecutionDependencies` (superset needed for all phases)
   - Options: `QueryTaskOptions` with `queryText`, `dir`, `format`, `output`, plus inherited options from research/plan/run
   - **Orchestration flow**:
     a. Resolve `dir` to absolute path (default: `process.cwd()`)
     b. Select seed template based on `format` (`yn` → YN seed, `success-error` → success-error seed, else → default seed)
     c. Create run directory: `.rundown/runs/<run-id>/`
     d. Create workdir: `.rundown/runs/<run-id>/workdir/` (file mode only)
     e. Render seed template with `{{query}}`, `{{dir}}`, `{{workdir}}` → write ephemeral `.rundown/runs/<run-id>/query.md`
     f. Run research phase: `app.researchTask({ source: queryDoc, cwd: dir, ... })`
     g. Run plan phase: `app.planTask({ source: queryDoc, cwd: dir, ... })`
     h. Select execution template based on output mode (stream vs file)
     i. Run execution phase: `app.runTask({ source: queryDoc, runAll: true, cwd: dir, ... })` — pass `dir` as cwd so workers analyze the target directory
     j. **File mode only**: aggregate `workdir/*.md` → format (markdown/json) → write to `--output` file or stdout
     k. **yn/success-error**: parse final step's stdout for verdict, set exit code (0=Y/success, 1=N/failure)
   - Pass `dir` as `cwd` to all worker/research/plan invocations — this is already supported through `WorkerExecutorPort.cwd`
   - Pass `workdir` via extra template vars

4. **Wire into `src/application/create-app.ts`** (*depends on 3*)
   - Add `queryTask` method to the app object, following the same pattern as `researchTask`, `planTask`, `runTask`

### Phase 3: Presentation Layer — CLI Wiring

5. **Add `query` command to `src/presentation/cli.ts`** (*parallel with 3, 4*)
   - Command: `query <text>`
   - Arguments: `<text>` — the query string (required)
   - Options:
     - `--dir <path>` — target directory to analyze (default: cwd)
     - `--format <format>` — output format: `markdown` | `json` | `yn` | `success-error` (default: `markdown`)
     - `--output <file>` — save output to file instead of stdout
     - `--skip-research` — skip research phase, go straight to plan
     - Inherited from explore: `--mode`, `--scan-count`, `--deep`, `--worker`, `--var`, `--vars-file`, `--keep-artifacts`, `--print-prompt`, `--dry-run`, `--verbose`
   - Follow existing command builder patterns (e.g., `addModeOption`, `addWorkerSeparator`, etc.)

6. **Add `createQueryCommandAction` to `src/presentation/cli-command-actions.ts`** (*depends on 4, 5*)
   - Parse CLI options following `createDoCommandAction` pattern
   - Resolve `--dir` to absolute path
   - Call `app.queryTask(...)` with assembled options
   - Handle exit status (including format-driven exit codes for `yn`/`success-error`)

### Phase 4: Output Formatting

7. **Create `src/application/query-output.ts`** (*parallel with 3*)
   - `aggregateQueryOutput(workdir: string): Promise<string>` — reads all `*.md` files from workdir, sorts by name (step-01, step-02...), concatenates with headers
   - `formatQueryOutput(content: string, format: "markdown" | "json" | "yn" | "success-error", query: string): string`
     - `markdown`: passthrough (content is already markdown)
     - `json`: structure as `{ query, steps: [{ title, content }], output }`
     - `yn`: extract final verdict, return `Y` or `N`
     - `success-error`: extract verdict, return `success` or `failure: <reason>`
   - `resolveQueryExitCode(format: string, output: string): number` — returns 0 for Y/success, 1 for N/failure, 0 for markdown/json
   - `writeQueryOutput(formatted: string, outputPath?: string): Promise<void>` — stdout or file

### Phase 5: Query-Specific Project Templates

8. **Support `.rundown/query-*.md` template overrides** (*depends on 1*)
   - Add to the template loader in `src/application/project-templates.ts`:
     - `.rundown/query-seed.md` → overrides `DEFAULT_QUERY_SEED_TEMPLATE`
     - `.rundown/query-execute.md` → overrides `DEFAULT_QUERY_EXECUTION_TEMPLATE` / stream variant
     - `.rundown/query-aggregate.md` → overrides `DEFAULT_QUERY_AGGREGATION_TEMPLATE`
   - Users customize any phase independently

## Decisions

- **Workdir communication via template variable, not infra**: The `{{workdir}}` path is injected into prompts. Workers write there because the prompt tells them to. No special file-watching or worker protocol needed.
- **Ephemeral by default**: Query doc + workdir live in `.rundown/runs/` and follow existing artifact retention logic (`--keep-artifacts` to persist).
- **No new domain port**: Query orchestration reuses existing ports (worker executor, file system, etc.) — no need for a `QueryPort`.
- **`--dir` flows through existing `cwd` plumbing**: `WorkerExecutorPort` → `worker-executor-adapter` → `runner.ts` → `cross-spawn` already support `cwd` option. Query just passes `--dir` value as `cwd` to all phase invocations.
- **Format-driven exit codes**: `yn` → 0=Y, 1=N. `success-error` → 0=success, 1=failure. `markdown`/`json` → always 0 on completion.
- **Stream is default**: Interactive use streams to terminal. File mode is opt-in via `--output` or `--format json`/`--format success-error` (where structured output is needed). `--format yn` uses stream mode (single token output).
- **Format-specific seed templates**: `yn` and `success-error` use different seeds that frame the query as a yes/no or pass/fail check with verdict instructions. The worker sees a different framing from the start.

## Further Considerations

1. **Aggregation strategy**: AI-driven summary by default. When output goes to a single file (`--output`) or CLI stdout, a final worker pass reads all step outputs and produces a coherent document. For `--format json`, mechanical structuring suffices. For directory output with many files, mechanical concat may be more appropriate due to volume.
2. **Step output naming convention**: `step-{{taskIndex}}.md` for file-mode workdir files. Less critical now that aggregation is AI-driven (worker reads all step outputs regardless of naming).
3. **`--skip-research` flag**: Supported. Skips the research phase and goes straight to plan. Useful when the seed document already has sufficient context or when the user wants faster iteration. `--no-plan` is NOT supported (plan is fundamental to query).
4. **`question:` task integration**: The question tool already exists (`src/domain/builtin-tools/question.ts`) and works via the standard `tool-expansion` intent. Tasks prefixed with `question:` are routed through the existing tool handler, which prompts the user interactively, captures answers, and supports defaults/options. No special handling needed for query — it uses the standard `runTask` pipeline, so `question:` tasks will work out of the box. Answers are persisted and reusable within the same run.

# Plan: Prediction System (start + migrate commands)

## TL;DR
Build a long-planning prediction system using two new top-level commands: `start` (scaffold a project) and `migrate` (generate/manage sequential migration files with satellite artifacts). Follows existing rundown architecture (ports & adapters, domain-driven, Commander CLI). `start` scaffolds a design-oriented project directory; `migrate` reads migration state and generates the next logical step or satellite artifacts (context, backlog, snapshot, review, user-experience, user-session).

## Naming Convention
- Migration: `0007-implement-feature.md` (4-digit zero-padded, single dash, kebab-case name)
- Satellite: `0007--snapshot.md` (double dash, satellite type as name)
- Satellite types: context, snapshot, backlog, user-experience, review

## Steps

### Phase 1: Domain — Migration File System
> Foundation: parse migration directories, distinguish migrations from satellites, track position.

1. Create `src/domain/migration-parser.ts`
   - `parseMigrationFilename(filename)` → `{ number, name, isSatellite, satelliteType } | null`
     - Regex: `/^(\d{4})-(?!-)(.+)\.md$/` for migrations, `/^(\d{4})--(.+)\.md$/` for satellites (satellite regex checked first, or use negative lookahead on migration regex to avoid `0007--snapshot.md` matching as migration)
   - `parseMigrationDirectory(files: string[])` → `MigrationState`
     - Groups files into migrations and satellites
     - Returns sorted by number
   - `getCurrentPosition(state)` → highest migration number
   - `getNextMigrationNumber(state)` → current + 1, zero-padded
   - `getLatestSatellite(state, type)` → most recent satellite of given type
   - `buildMigrationContext(state)` → reads latest context + latest migration for incremental context building

2. Create `src/domain/migration-types.ts` — types
   - `Migration { number, name, filePath, satellites, isApplied }` (`isApplied` = all tasks checked)
   - `Satellite { migrationNumber, type, filePath }`
   - `SatelliteType = "context" | "snapshot" | "backlog" | "user-experience" | "review"`
   - `MigrationState { projectRoot, migrationsDir, migrations, currentPosition, latestContext, latestBacklog }`
   - `projectRoot` resolved as: parent of `migrationsDir` (convention: `<project>/migrations/`)

### Phase 1b: Domain — Migration Port
> Port contract for migration file I/O (required by ports & adapters architecture).

2b. Create `src/domain/ports/migration-port.ts`
   - `MigrationPort { scanDirectory(dir): MigrationState, readFile(path): string, writeFile(path, content): void, removeFile(path): void }`
   - Keeps domain logic decoupled from filesystem access

### Phase 2: Domain — Templates
> Default prompt templates for migrate generation and each satellite type.

3. Add to `src/domain/defaults.ts`:
   - `DEFAULT_MIGRATE_TEMPLATE` — generates next migration proposal (ranked alternatives from backlog + design + context)
   - `DEFAULT_MIGRATE_CONTEXT_TEMPLATE` — incremental context generation
   - `DEFAULT_MIGRATE_SNAPSHOT_TEMPLATE` — project state snapshot
   - `DEFAULT_MIGRATE_BACKLOG_TEMPLATE` — technical debt extraction
   - `DEFAULT_MIGRATE_REVIEW_TEMPLATE` — design comparison review
   - `DEFAULT_MIGRATE_USER_EXPERIENCE_TEMPLATE` — UX scenarios with `- [ ] question: ...` TODO items
   - `DEFAULT_UNDO_TEMPLATE` — AI undo: receives task text, execution artifacts, current state; must produce reversal actions
   - Template vars: `{{design}}`, `{{latestContext}}`, `{{latestMigration}}`, `{{latestBacklog}}`, `{{migrationHistory}}`, `{{position}}`, `{{executionOutput}}`, `{{taskText}}`

### Phase 3: Application — Start Project
> Scaffold a prediction project directory with design docs, agent config, and migration dir.

4. Create `src/application/start-project.ts`
   - `createStartProject(deps: StartProjectDependencies)` → async function
   - **Scaffold structure:**
     ```
     <dir>/
       Design.md          ← seeded with user's description
       AGENTS.md           ← agent definitions template
       migrations/         ← empty dir
       migrations/0001-initialize.md  ← first migration
       specs/              ← AI-verified specifications
       .rundown/           ← config (via existing initProject)
     ```
   - Flow:
     1. Create target dir if needed (if `--dir` specified and doesn't exist)
     2. **Init git** if not inside a git repo (`git init` via `git-operations.ts`)
     3. Call existing `initProject()` to create `.rundown/` config
     4. Write `Design.md` with the user's seed description as H1 + body
     5. Write `AGENTS.md` with a default agent template
     6. Create `migrations/` directory
     7. Write `migrations/0001-initialize.md` with initial migration content
     8. Run `explore` on `Design.md` (research + plan) to flesh out the design
     9. Run `explore` on `0001-initialize.md` to enrich the first migration
     10. Git commit scaffold: `git add -A && git commit -m "rundown: start project"`
   - *Depends on*: Phase 1 (migration naming), Phase 2 (templates)

### Phase 4: Application — Migrate Task
> Core migration logic: generate next migration, generate satellites, with optional confirmation.

5. Create `src/application/migrate-task.ts`
   - `createMigrateTask(deps: MigrateTaskDependencies)` → async function
   - **Dependencies** (following existing port pattern):
     - `workerExecutor`, `fileSystem`, `templateLoader`, `sourceResolver`
     - `workerConfigPort`, `artifactStore`, `outputPort`
     - `interactiveInput` (existing `InteractiveInputPort`) — for `--confirm` flow
   - **Options include `confirm: boolean`** — when true, every write goes through review gate

   **Confirmation flow** (applies to all sub-commands when `--confirm` is set):
   - After worker produces output, emit the generated content to stdout via `outputPort`
   - Call `interactiveInput.request({ kind: "confirm", message: "Write <filename>?", default: true })`
   - If confirmed → write file; if rejected → skip write, emit "Skipped" info, return success
   - Non-TTY fallback: uses default (true) — confirm auto-approves in CI/scripts
   - Helper: `confirmBeforeWrite(outputPort, interactiveInput, filename, content)` — reusable within migrate-task

   **Sub-commands routed by action argument:**

   **5a. `migrate up`** — Apply migrations
   - Thin wrapper: delegates to `runTask()` with `source: migrationsDir`, `runAll: true`, `commitAfterComplete: true`, `commitMode: "per-task"`
   - Equivalent to: `rundown all migrations/ --commit`
   - Natural sort ensures migrations execute in 0001→0002→... order
   - Checked tasks are skipped automatically (existing behavior)
   - Partially applied migrations continue from where they left off

   **5b. `migrate down [n]`** — Undo migrations
   - Alias for `rundown undo` (see Phase 4b below)
   - `migrate down` = `rundown undo` (undo last task)
   - `migrate down 5` = `rundown undo --last 5`

   **5c. `migrate` (no sub-command)** — Generate next migration
   - Scan migration dir → build `MigrationState`
   - Read: `Design.md`, latest `--context.md`, latest `--backlog.md`, last migration file
   - Render `DEFAULT_MIGRATE_TEMPLATE` with all context vars
   - Execute worker → parse output as ranked migration proposals
   - Present proposals to user (interactive selection or first if non-interactive)
   - **[confirm gate]** → Write `NNNN-name.md` migration file
   - Run `explore` on new migration (reuse `runExplorePhases` pattern)
   - **[confirm gate]** → Generate `NNNN--context.md` (incremental from previous context + new migration)

   **5d. `migrate snapshot`** — Generate snapshot satellite
   - Read all migrations up to current position + latest context
   - Render snapshot template → worker generates → **[confirm gate]** → write `NNNN--snapshot.md`

   **5e. `migrate backlog`** — Generate backlog satellite
   - Read design + all migrations + latest context
   - Render backlog template → worker generates → **[confirm gate]** → write `NNNN--backlog.md`

   **5f. `migrate context`** — (Re)generate context satellite
   - Read previous context + latest migration
   - Render context template → worker generates → **[confirm gate]** → write `NNNN--context.md` (removes previous if exists)

   **5g. `migrate review`** — Generate review satellite
   - Read design + latest context + latest snapshot
   - Render review template → worker generates → **[confirm gate]** → write `NNNN--review.md`

   **5h. `migrate user-experience`** — Generate UX satellite
   - Render UX template → worker generates → **[confirm gate]** → write `NNNN--user-experience.md`
   - Output contains `- [ ] question: ...` TODO items for user to fill

   **5i. `migrate user-session`** — Interactive TUI discussion
   - Open TUI (reuse `discuss-task.ts` pattern with tui mode)
   - Target: the latest migration file at current position (highest-numbered migration)
   - After session ends, summarize discussion and append/update content in that migration file
   - **[confirm gate]** on summary write → then trigger backlog rebuild (`migrate backlog`)

### Phase 4b: Application — Undo Task
> New top-level command: AI-powered task reversal. Also serves as `migrate down` backend.

5j. Create `src/application/undo-task.ts`
   - `createUndoTask(deps: UndoTaskDependencies)` → async function
   - **How it works**:
     1. Resolve target task(s): last checked task (default), `--last N`, or specific task by source+line/name
     2. For each target task, load its run artifact (execution output, file diffs, context)
     3. Render `DEFAULT_UNDO_TEMPLATE` — includes: task text, execution output from artifacts, current file state, instructions to generate reversal actions
     4. Execute worker → AI analyzes what the task did and produces reversal actions
     5. Verify reversal: run verification worker to confirm the undo is correct
     6. If verified: uncheck the checkbox (`- [x]` → `- [ ]`), commit if `--commit`
     7. If verification fails: crash with error (no silent partial undo)
   - **Dependencies**: `workerExecutor`, `fileSystem`, `templateLoader`, `artifactStore`, `taskVerification`, `outputPort`
   - **Differs from `revert`**: 
     - `revert` = git-level, mechanical, requires commits
     - `undo` = AI-level, semantic, works without git history
   - **Options**: `--last <n>`, `--commit`, shared worker options

### Phase 4c: Application — Test Command
> AI-verified specifications: verify assertions against predicted migration state, not actual implementation.

5k. Create `src/application/test-specs.ts`
   - `createTestSpecs(deps: TestSpecsDependencies)` → async function
   - **New execution mode** — not a wrapper around `run` or `do`. Each spec file is a single assertion (the full file content), not a checklist of tasks.
   - **`rundown test` (no args)** — Run all specs
     - Resolve specs dir (default `./specs/`, configurable with `--dir`)
     - Collect all `.md` files from specs dir (natural sort)
     - Load migration context: Design.md, latest `--context.md`, latest `--snapshot.md`
     - For each spec file:
       1. Read file content — this IS the assertion text
       2. Render `DEFAULT_TEST_VERIFY_TEMPLATE` with `{{assertion}}` = file content + migration context vars
       3. Execute worker in verify mode (single call, no execute phase)
       4. Parse output: `OK` = pass, `NOT_OK: <reason>` = fail
       5. Emit result per spec: pass/fail with reason
     - Summary: total passed / total failed, exit code 1 if any fail
   - **`rundown test new "User can upload files"`** — Create a new spec
     - Slugify prompt → `user-can-upload-files.md`
     - Write file with the assertion as content
     - If `--run` flag: immediately verify the new spec (call test logic on it)
   - **`rundown test new --mode tui`** — Interactive spec definition via TUI
   - **Dependencies**: `workerExecutor`, `fileSystem`, `templateLoader`, `migrationPort`, `outputPort`

5l. Add to `src/domain/defaults.ts`:
   - `DEFAULT_TEST_VERIFY_TEMPLATE` — receives `{{assertion}}`, `{{design}}`, `{{latestContext}}`, `{{latestSnapshot}}`, `{{migrationHistory}}`; must evaluate whether the assertion holds true given the predicted migration state; responds `OK` or `NOT_OK: <reason>`

## Decisions

- **Double dash convention confirmed**: `0007-name.md` = migration, `0007--type.md` = satellite
- **Top-level commands**: `start` and `migrate` are peers of `run`, `plan`, `explore` (not subcommands)
- **`--dir` flag**: Configurable directory, defaults to cwd for `start`, `./migrations` for `migrate`
- **Project root convention**: parent of migrations dir. `migrate` resolves `Design.md` etc. from `<migrationsDir>/../`
- **Context is incremental**: Each `NNNN--context.md` builds from previous context + latest migration only (not full re-scan)
- **`migrate` (no args) generates next migration**: Proposes ranked alternatives, writes + runs explore
- **Satellites are individually triggered**: `rundown migrate snapshot`, `rundown migrate backlog`, etc.
- **Satellite overwrite**: Regenerating a satellite at the same position overwrites the existing file (idempotent)
- **user-session targets latest migration**: TUI discussion context is the highest-numbered migration file
- **user-session triggers backlog rebuild**: After TUI discussion, auto-regenerates backlog satellite
- **Reuses existing infrastructure**: natural sort (sorting.ts), source resolution (sources.ts), explore phases (cli-command-actions.ts), TUI mode (discuss-task.ts), init (init-project.ts)
- **`--confirm` flag on all migrate commands**: Uses existing `InteractiveInputPort` (`kind: "confirm"`); shows generated content, asks yes/no before each file write; auto-approves in non-TTY environments
- **`up` = `rundown all migrations/ --commit`**: Thin wrapper, no migration-specific execution logic
- **`down [n]` = `rundown undo [--last n]`**: AI-powered reversal, not git revert
- **`undo` vs `revert`**: `revert` = mechanical git undo (requires commits); `undo` = AI analyzes task artifacts and generates reversal actions (works without git). Crashes on verification failure — no silent partial undo
- **`undo` is a standalone top-level command**: Usable independently of migrate; `migrate down` is just an alias
- **Git required for `up`/`down`**: `start` auto-inits git; commit flag requires git repo (existing validation)
- **`up`/`down` are execution commands; everything else is prediction**: Clear conceptual split
- **`test` verifies against predicted state, not implementation**: Specs are checked against migration context/snapshots, not real code. This controls migration flow
- **`test` is a new execution mode**: Not a wrapper around `run` or `do`. Each spec file is a single assertion (full file content), verified by worker in one call. No checkboxes, no task execution
- **`test new --run` creates and immediately verifies**: Create spec then run verification on it in one command
- **Specs live in `specs/`**: Part of scaffold, contains assertion-style markdown files (one assertion per file)
- **No `migrate status`**: Use existing `rundown list <migrations/>` and `rundown next <migrations/>` to inspect migration state
- **Partially applied migrations**: `up` skips checked tasks naturally; `down` reverts via existing artifact system
- **Design.md only**: No Domain.md in scaffold. Additional project phases/structure controlled through AGENTS.md and user-created files
- **Migration proposal UX**: CLI numbered prompt first; TUI selection as future enhancement

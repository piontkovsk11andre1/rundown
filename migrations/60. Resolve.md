# Plan: Resolve Phase for Exhausted Repair Loops

## TL;DR

Add a **resolve** phase to `runVerifyRepairLoop` that triggers when `maxRepairAttempts >= 2` and all repair attempts fail. The resolve phase sends a diagnostic prompt to the same worker containing the verification message, all repair attempt outputs, and the original execution output. The AI diagnoses the root cause. If it can → a single specialized repair attempt runs with the diagnosis context. If it cannot → the task fails with exit code 2.

## Phase 1: Capture repair attempt history

Currently, `runVerifyRepairLoop` discards per-attempt outputs — only the last `repairStdout`/`verificationStdout` survive. We need to collect all of them.

### Steps

1. **Define `RepairAttemptRecord` type** in `src/application/verify-repair-loop.ts`
   - Fields: `{ attempt: number, repairStdout: string | undefined, verificationStdout: string | undefined, failureReason: string | null }`
   
2. **Accumulate records during repair loop** — push a `RepairAttemptRecord` after each `repair()` call in the `while (attempts < input.maxRepairAttempts)` loop (~line 380)

## Phase 2: Resolve phase types and output contract

3. **Define resolve output format** — Worker outputs:
   - `RESOLVED: <root cause diagnosis>` — AI identified the problem
   - `UNRESOLVED: <reason why it cannot diagnose>` — AI cannot determine cause
   
4. **Add `ResolveResult` type** — `{ resolved: boolean, diagnosis: string | null }`

5. **Add `parseResolveResult(stdout: string)` parser** in `src/infrastructure/verification.ts` or new file — extracts structured result from worker stdout, similar to `parseVerificationResult`

## Phase 3: Resolve prompt template

6. **Add default resolve template** in `src/domain/defaults.ts` — includes:
   - Task metadata (file, line, text)
   - Source context
   - Original execution stdout
   - Verification failure message
   - Each repair attempt's repair stdout + verification stdout + failure reason
   - Instructions to output `RESOLVED:` or `UNRESOLVED:` verdict
   
7. **Add template loading** in `src/application/project-templates.ts` — load `.rundown/resolve.md` override, fallback to built-in default

## Phase 4: Resolve execution

8. **Add `resolve()` function** in `src/infrastructure/repair.ts` (or new `src/infrastructure/resolve.ts`)
   - Takes: task, template, worker pattern, attempt history, execution stdout, artifact context
   - Runs worker with resolve prompt
   - Parses output via `parseResolveResult`
   - Returns `ResolveResult`

9. **Add resolve-aware repair function** — a single repair attempt that receives diagnosis context in its template vars (e.g., `resolvedDiagnosis` variable)

## Phase 5: Integration into verify-repair-loop

10. **Add resolve phase after repair exhaustion** in `runVerifyRepairLoop` (~line 500, after the "All repair attempts failed" block):
    - Guard: only trigger when `input.maxRepairAttempts >= 2`
    - Call `resolve()` with collected `RepairAttemptRecord[]` and execution stdout
    - If `UNRESOLVED` → return `{ valid: false, failureReason }` as today
    - If `RESOLVED` → run single specialized repair attempt with diagnosis context → re-verify → return result

11. **Add `"resolve"` to phase type unions**:
    - `TracePhase` in `src/domain/trace.ts`
    - `RuntimePhase` in `src/infrastructure/runtime-artifacts.ts`
    - `ArtifactStorePhase` in `src/domain/ports/artifact-store.ts`

## Phase 6: Trace events

    - `ResolveOutcomeEvent` — `{ resolved: boolean, diagnosis: string | null }`

13. **Emit trace events** from the resolve code path in `runVerifyRepairLoop`

## Phase 7: CLI wiring

## Decisions

- Resolve uses same worker (opencode) as execution/repair, just different template
- Resolve triggers only when `maxRepairAttempts >= 2` and all attempts exhausted
- Resolve gets: original execution stdout + all repair attempt outputs + verification messages
- On successful diagnosis: single repair attempt with diagnosis as template var, then re-verify
- On failed diagnosis: fail with exit code 2 (current behavior)
- No new CLI flags — resolve is automatic when conditions are met
- Output contract: `RESOLVED: <reason>` / `UNRESOLVED: <reason>` (similar to verification's `OK`/`NOT_OK`)

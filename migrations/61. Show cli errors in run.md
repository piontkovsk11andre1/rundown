- Show CLI task errors in console output for `rundown run` and `rundown run --all`.

When a task uses `- [ ] cli: <command>` and that command fails, the user should see the actual stderr/stdout failure details directly in the console (not just a generic failure message), so the cause is immediately diagnosable.

## Background and current behavior

`rundown run` and `rundown run --all` share the same execution pipeline in `src/application/run-task-execution.ts`, with `--all` represented by `effectiveRunAll` and a loop in `runTaskLoop`. This means failure visibility for a single task and full-run mode is controlled by the same lower-level dispatch path.

For inline CLI tasks (`- [ ] cli: ...`), execution is handled in `src/application/task-execution-dispatch.ts`:

- The command is run via `workerExecutor.executeInlineCli(...)`.
- The result includes `exitCode`, `stdout`, and `stderr`.
- On non-zero exit, dispatch currently returns a normalized failure like `Inline CLI exited with code N`.
- Detailed `stdout`/`stderr` is only emitted when `showAgentOutput` is enabled (`emitExecutionWorkerOutput`), which is not the default.

`run-task-iteration.ts` then surfaces only the normalized failure text via `handleDispatchFailure`, so users typically see a generic failure line without the command’s actual output.

## Why this matters

When an inline CLI command fails, the actionable information is almost always in stderr/stdout (missing executable, bad flags, file/path issues, permission errors, script stack traces). Requiring `--show-agent-output` to diagnose failures creates an unnecessary second run and slows troubleshooting.

The expected behavior is: on failure, emit the command output automatically in that same run so the user can diagnose immediately.

## Affected modules and boundaries

- `src/application/task-execution-dispatch.ts`
  - Source of inline CLI execution result (`stdout`/`stderr` + exit code).
  - Current gating behavior tied to `showAgentOutput`.
- `src/application/run-task-iteration.ts`
  - Converts dispatch failure into emitted error/group-end messages.
- `src/presentation/output-port.ts`
  - `text` events go to stdout, `stderr` events go to stderr, `error`/failed `group-end` go to stderr.
  - Important for stream correctness and UX consistency.
- `src/application/run-task-execution.ts`
  - Shared run loop for single and `--all` execution paths (integration point, not necessarily implementation site).

## Scope

In scope:

- Failure-time output visibility for inline `cli:` task execution in `run` flows.
- Ensuring behavior applies equally to `rundown run` and `rundown run --all`.
- Preserving existing success-path behavior (no noisy output by default on successful tasks).

Out of scope:

- Changing default verbosity of successful worker output.
- Reworking non-inline worker execution output policy globally.
- Artifact schema changes or trace format redesign.

## Design constraints and compatibility considerations

- Keep existing `--show-agent-output` semantics for successful execution; this migration is specifically about failure diagnosability.
- Do not remove existing high-level error lines (for test stability and quick scanability); augment with detailed stderr/stdout content.
- Preserve stream routing conventions:
  - diagnostics/errors to stderr where appropriate,
  - plain command stdout details can remain stdout unless intentionally mirrored as stderr-formatted diagnostics.
- Maintain compatibility with grouped output (`group-start` / `group-end`) so failure detail appears in the right task context.
- Avoid ANSI/logging regressions in persisted output logs (existing logging wrappers and stream mapping should remain consistent).

## Edge cases to account for

- Failure with only stderr, only stdout, both, or neither.
- Commands that fail with multi-line output (preserve readability and line breaks).
- Very large output payloads (avoid rendering corruption; consider bounded presentation if needed, while still exposing root cause).
- Non-zero exit with empty streams should still show the generic exit-code error.
- `run --all` should continue to stop/fail according to current lifecycle semantics; this change is about visibility, not control flow.
- TTY and non-TTY runs should both show details in plain, stable output.

## Integration notes for planning

Likely integration point is the inline CLI failure branch in `task-execution-dispatch.ts` where `cliResult` is available before returning `execution-failed`. Emitting captured output there on failure avoids plumbing extra fields through multiple layers and naturally covers both `run` and `run --all` through shared flow.

A second viable approach is to carry failure stdout/stderr through `TaskExecutionDispatchResult` and emit in `run-task-iteration.ts` during `handleDispatchFailure`. That is more explicit but touches more types/callers.

Whichever approach is selected, the final console output should include:

- existing structured failure indicator (`Inline CLI exited with code N`), and
- actual command stderr/stdout content for the failed inline task.

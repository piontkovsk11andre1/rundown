When `rundown` is launched from a directory linked to another actual rundown workspace (`rundown start` link mode), we must pass linked-workspace context to the agent via template variables.

Intent: if harness/worker execution effectively runs against the linked target workspace (cwd switch behavior), the agent should explicitly know both the invocation directory and the resolved source/workspace directory through variables. This may already be implemented; verify and close gaps only if missing.

## Context

This migration builds directly on `67. workspace.link for rundown start dir.md`, which defines `.rundown/workspace.link` as the bridge from a linked invocation directory to the actual workspace root.

The core risk is context drift: runtime may correctly switch execution cwd to the resolved workspace, while prompt templates still omit where the command was originally invoked. In linked mode, that ambiguity can mislead agent decisions about paths, edits, and command targeting.

## Current implementation shape (relevant findings)

- `run`, `plan`, and `research` already support explicit cwd override (`options.cwd`) and consistently derive `executionCwd = options.cwd ?? workingDirectory.cwd()`.
- Worker execution and template-var file loading already consume this effective cwd in:
  - `src/application/run-task-execution.ts`
  - `src/application/plan-task.ts`
  - `src/application/research-task.ts`
- Prompt rendering pipeline already supports arbitrary extra template keys:
  - `ExtraTemplateVars` is `Record<string, string>` in `src/domain/template-vars.ts`.
  - `TemplateVars` supports ad-hoc keys via index signature in `src/domain/template.ts`.
  - Prompts receive merged vars via `templateVarsWithUserVariables`.
- Default templates currently expose generic `## Variables` (`{{userVariables}}`) but do not provide first-class linked-workspace fields (no dedicated invocation/workspace path vars).

Implication: plumbing exists to carry additional path variables without changing renderer contracts, but linked-workspace context is not yet explicitly modeled as stable template variables.

## Feature specification

When linked-workspace resolution is active, agent-facing templates must receive explicit workspace context variables that disambiguate:

- where the command was invoked, and
- where execution is actually rooted.

Recommended stable variable set:

- `invocationDir`: original CLI invocation directory (pre-link resolution).
- `workspaceDir`: resolved effective workspace directory used for execution.
- `workspaceLinkPath`: absolute path to `.rundown/workspace.link` when used, otherwise empty.
- `isLinkedWorkspace`: `"true"` / `"false"`.

Compatibility rule: always provide all variables (with sensible fallback values), not only in linked mode.  
For non-linked runs, `invocationDir` and `workspaceDir` should match, `workspaceLinkPath` should be empty, and `isLinkedWorkspace` should be `"false"`.

## Integration boundaries

This migration should stay focused on prompt/template context propagation, not workspace-link discovery itself (that belongs to migration 67). It should integrate at the point where execution context is already finalized and extra template vars are assembled.

Primary touchpoints:

- runtime option shaping and cwd selection path,
- per-command template-var merge layer (`file vars` + `CLI vars` + runtime-injected vars),
- worker-facing prompt construction for commands that run harness/worker flows.

At minimum, ensure consistency across `run`, `plan`, and `research` paths, since all three already render templates and run workers with cwd-sensitive behavior.

## Constraints and edge cases

- Relative link targets must be normalized to absolute paths before surfacing to templates.
- Broken/stale link targets should not silently emit misleading `workspaceDir`; fallback behavior should be explicit and deterministic.
- Windows and POSIX separators should be preserved consistently with existing path conventions in prompts.
- User-provided `--var` keys must not be allowed to override authoritative runtime workspace-context variables.
- Existing behavior when `.rundown/workspace.link` is absent must remain unchanged.

## Acceptance criteria outline

- In linked mode, prompts include both invocation and resolved workspace directory values.
- In non-linked mode, the same variables exist with non-linked fallback values.
- Worker runs continue to execute against effective cwd with no regression.
- Template rendering remains backward compatible for existing templates and custom vars.
- Existing commands continue to function when no link file is present.

## Out of scope

- Redesigning `.rundown/workspace.link` format or resolution algorithm (migration 67 scope).
- Introducing task decomposition/TODO authoring changes.
- Changing checkbox semantics or completion flow.
- Adding new worker command resolution rules unrelated to linked workspace context.

Continuation of migration 67: the agent must explicitly know which directory it is currently operating on when the current directory is linked to another rundown source/workspace.

If linked mode is active, agent context should clearly expose both:

- the invocation/current directory, and
- the resolved linked source/workspace directory.

If this is already implemented via workspace-link context variables, confirm behavior and add missing coverage/docs only where needed. If not, implement the minimum missing propagation so prompts are unambiguous.

## Context and motivation

This follows the linked-workspace foundation introduced in migration 67 and the prompt-context direction in migration 68. The risk is prompt ambiguity: runtime execution can happen in one directory while the agent reasons as if it is in another.

When `rundown` is started from a linked directory, two locations can diverge:

- invocation directory (where the user launched the command),
- effective workspace directory (where source resolution and worker/harness execution are rooted).

Without explicitly surfacing both values, the agent can produce path-sensitive mistakes (wrong relative paths, edits in unintended tree, incorrect shell assumptions).

## Current implementation findings (codebase)

Relevant flow already exists for cwd propagation, but linked-workspace identity is not explicitly surfaced as first-class prompt context variables.

- `run` computes `executionCwd = options.cwd ?? workingDirectory.cwd()` in `src/application/run-task-execution.ts`.
- `run` passes that cwd through prompt and dispatch paths in `src/application/run-task-iteration.ts` and `src/application/prepare-task-prompts.ts`.
- Prompt templates support arbitrary additional keys through:
  - `ExtraTemplateVars = Record<string, string>` in `src/domain/template-vars.ts`,
  - index-signature-friendly `TemplateVars` in `src/domain/template.ts`.
- Built-in templates currently expose generic variables (`{{userVariables}}`) and memory metadata, but no dedicated linked-workspace identity fields in `src/domain/defaults.ts`.
- Source resolution itself (`src/infrastructure/sources.ts`) resolves file/directory/glob inputs but does not annotate link-state metadata.

Observed gap: there is no explicit `invocation vs resolved workspace` variable contract visible to the worker prompt by default.

## Desired behavior contract

In any worker-facing prompt path, linked-workspace context must be unambiguous and deterministic.

Recommended canonical fields:

- `invocationDir`: absolute directory where CLI invocation started.
- `workspaceDir`: absolute effective workspace directory used for execution.
- `workspaceLinkPath`: absolute path to `.rundown/workspace.link` when active; empty otherwise.
- `isLinkedWorkspace`: string `"true"` or `"false"`.

Compatibility defaults for non-linked mode:

- `invocationDir === workspaceDir`
- `workspaceLinkPath` is empty
- `isLinkedWorkspace === "false"`

Linked mode expectation:

- `invocationDir` and `workspaceDir` may differ
- `workspaceDir` is the resolved linked target
- `workspaceLinkPath` points to the concrete link file used
- `isLinkedWorkspace === "true"`

## Scope boundaries

This migration should stay narrow and propagation-focused.

In scope:

- confirm whether linked workspace context variables are already injected into prompt vars,
- add only missing propagation required for prompt clarity,
- add/update tests proving prompt-visible context in linked and non-linked runs,
- add/update docs/template references so behavior is discoverable.

Out of scope:

- redesigning `.rundown/workspace.link` format or resolution algorithm (migration 67 concern),
- broad worker-execution refactors unrelated to prompt context,
- TODO decomposition or checkbox-state behavior changes.

## Integration points

Use one authoritative resolved workspace context object and inject it once per command path, instead of recomputing deep in prompt rendering.

Primary seams:

- command-level cwd/link resolution layer (before prompt assembly),
- template var merge layer (file vars + CLI vars + runtime vars),
- worker-facing prompt construction paths (`run`, and parity checks for `plan`/`research` where applicable).

Precedence rule:

- authoritative runtime workspace context keys must not be overridden by user `--var` values.

## Constraints and edge cases

- stale or broken link targets must produce deterministic fallback values (no misleading `workspaceDir`),
- all exposed paths should be absolute and normalized according to existing project conventions,
- behavior must remain unchanged when no `.rundown/workspace.link` is present,
- Windows and POSIX path behavior must remain consistent with existing cwd/path rendering in prompts and logs.

## Acceptance criteria outline

- prompts always include both invocation and effective workspace context fields,
- linked mode clearly distinguishes invocation and resolved workspace paths,
- non-linked mode still emits the same fields with fallback-equal values,
- runtime execution cwd behavior remains unchanged (no regressions in harness/task execution),
- template compatibility remains backward-safe for existing custom templates,
- docs describe the new/existing variables and fallback semantics clearly,
- test coverage includes linked, non-linked, and stale-link fallback scenarios.

When resetting TODO items (checkbox reset flow), also remove/reset known runtime-added stale annotations under those tasks (for example metrics-like execution artifacts).

Today, reset primarily toggles checkbox state, but execution can append extra subitems/metadata that become stale after reset. Reset should return tasks to a clean actionable state by clearing known generated additions that should not survive a restart.

## Feature Specification

Reset behavior should be treated as a two-part cleanup pass over task trees:

- Normalize task state: convert checked task markers back to unchecked.
- Normalize task context: remove runtime-generated child annotations that are execution byproducts rather than author-authored plan content.

The intent is that a post-reset document looks like a fresh, actionable TODO file, not a historical run log embedded under each task.

## Existing Implementation Context

Current reset entry points are in `src/application/checkbox-operations.ts`:

- `maybeResetFileCheckboxes(...)` orchestrates dry-run reporting and real reset writes.
- `resetFileCheckboxes(...)` performs file mutation under serialized file locking.
- `resetFileCheckboxes(...)` already calls:
  - `resetAllCheckboxes(...)` from `src/domain/checkbox.ts` (checkbox state only), then
  - `stripTraceStatisticsFromSource(...)` (removes known trace-statistics blocks).

Important existing behavior:

- Trace-statistics cleanup is already pattern-based and indentation-aware (child/grandchild lines), using label regexes and descendant traversal.
- Reset count reporting is based on checked tasks, not removed annotations.
- Dry-run currently reports checkbox reset intent only.

This migration extends the same cleanup concept from “trace statistics only” to a broader “known stale runtime additions” set.

## Runtime-Added Additions in This Codebase

Known runtime-generated subitems currently written by execution flows include:

- Trace statistics lines (already stripped on reset), inserted by `insertTraceStatisticsUsingFileSystem(...)` in `src/application/checkbox-operations.ts`.
- Verification failure annotations, inserted as `fix: ...` by `writeFixAnnotationToFile(...)` in `src/application/checkbox-operations.ts`.
- End/skip annotations, inserted as `skipped: ...` by `skipRemainingSiblingsUsingFileSystem(...)` in `src/application/checkbox-operations.ts`.
- Tool-generated child TODOs can be inserted through `insertSubitems(...)` call sites (for example in `src/application/task-execution-dispatch.ts` and `src/application/tool-execution.ts`), but these are actionable planning artifacts and should not be treated as stale metrics/log metadata by default.

## Scope Boundaries

In scope:

- Remove/reset known runtime-added non-actionable annotations beneath tasks during checkbox reset.
- Keep cleanup deterministic and safe under mixed indentation and nested descendants.
- Preserve user-authored task text and user-authored non-runtime notes.

Out of scope:

- Reformatting or reordering task trees.
- Deleting actionable child TODO items.
- Rewriting planner-produced structure unrelated to runtime annotations.
- Broad heuristic deletion of arbitrary child bullets without positive identification.

## Identification Strategy for “Known Stale” Lines

Use explicit allowlisted label detection, similar to existing trace-stat handling, instead of loose heuristics.

Candidate known-stale prefixes/labels to support:

- Trace statistics labels already recognized (for example `total time:`, `tokens estimated:`, `verify attempts:` and nested `execution:`, `verify:`, `repair:` lines).
- `fix:` annotation lines generated by verification failure handling.
- `skipped:` annotation lines generated by sibling-skip flow.

Design preference:

- Match list-item label content after bullet marker normalization.
- Apply indentation rules relative to parent task depth.
- Remove contiguous trailing/descendant stale blocks, while leaving unrelated child content untouched.

## Integration Notes

Likely implementation surface is centralized in `src/application/checkbox-operations.ts`, where reset currently composes:

1. Checkbox reset (`resetAllCheckboxes`)
2. Trace cleanup (`stripTraceStatisticsFromSource`)

This should evolve into a single cleanup pipeline that:

- Parses tasks once per file pass (or at least avoids conflicting rewrites).
- Applies stale-line stripping for all known runtime-added annotation categories.
- Preserves newline style and serialized mutation guarantees already present in `withSerializedFileMutation(...)`.

Keep reset behavior consistent whether invoked as:

- Pre-run reset (`--redo` flow),
- Post-run reset (`--reset-after` flow),
- Multi-round execution resets in `src/application/run-task-execution.ts`.

## Edge Cases and Compatibility Considerations

- Mixed content under one task: if both user notes and runtime annotations coexist, only known stale lines are removed.
- Nested stale blocks: remove nested runtime metric children when their parent stale label is removed.
- Already-clean files: no-op write avoidance should be preserved when possible.
- Dry-run: messaging may need to remain checkbox-centric or be expanded to mention stale annotation cleanup intent.
- Parser compatibility: cleanup must operate correctly with current task parsing and depth semantics from `src/domain/parser.ts`.
- Idempotency: repeated reset should yield stable output with no further diffs after first cleanup.
- Safety: avoid deleting checked/unchecked child TODO lines unless explicitly classified as runtime-generated stale artifacts by design.

## Acceptance Criteria Outline

- After reset, checked tasks are unchecked as today.
- After reset, known runtime-added stale annotations (`trace stats`, `fix:`, `skipped:`) are removed where applicable.
- Author-authored task lines and actionable child TODOs remain intact.
- Behavior is stable across nested tasks, multiple files, and repeated resets.
- Existing trace-stat cleanup behavior is preserved (no regressions).
- Reset flows invoked by `--redo`, `--reset-after`, and round-based execution produce clean actionable documents.

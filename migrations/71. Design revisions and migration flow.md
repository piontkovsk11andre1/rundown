Introduce design revision management as a directory-based workflow, not a single hardcoded `Design.md` file.

Intent:

- `docs/current/` is the editable draft area (working version).
- We need commands to save the current draft as an immutable revision snapshot (for example `docs/rev.N/` or archive equivalent).
- After saving a revision, we should be able to diff current vs previous revision and generate/preview migrations from that diff.

Key correction: `Design.md` should be treated as default content within a managed design directory workflow, not as the only design file/path.

Behavior goals:

- Edit happens in `docs/current/`.
- A dedicated command saves a new revision from current draft state.
- A dedicated command compares revisions and prepares migration output/preview.
- Migration/research agents must be aware revisions may live in revision/archive directories and use them as valid context sources.

## Context and current implementation baseline

Today, prediction flows are still anchored to a single root file path in multiple places:

- `src\application\start-project.ts` scaffolds `Design.md` at project root.
- `src\application\migrate-task.ts` reads design context from `<projectRoot>/Design.md` in `buildTemplateVars(...)`.
- `src\application\test-specs.ts` reads design context from `<projectRoot>/Design.md` in `buildTestContext(...)`.
- CLI/help/docs also describe prediction start state as `Design.md` (`src\presentation\cli.ts`, `README.md`, `docs\cli.md`).

This means migration generation and predicted-state testing currently depend on one canonical design file, while the desired model is a design workspace rooted at `docs/` with revision history.

## Problem statement

Long-horizon prediction work needs two distinct properties at once:

- a mutable working draft for active edits;
- immutable historical snapshots for reproducibility, comparison, and migration reasoning.

A single mutable `Design.md` cannot represent both roles cleanly. It erases history unless users manually duplicate files, and it gives migrate/research/test flows no explicit revision semantics.

## Target design model

### Directory contract

Use a managed design directory under project root:

- `docs/current/` — editable working draft.
- `docs/rev.N/` (or equivalent archive layout) — immutable snapshots.
- `docs/current/Design.md` — default primary design document inside the working draft (not the only allowed file).
- Additional files inside `docs/current/` are valid design context and should be included in revision snapshots.

### Revision semantics

A saved revision is a point-in-time copy of `docs/current/`:

- revision directories are immutable after creation;
- revision numbering is monotonic and deterministic;
- metadata needed for ordering (revision index, timestamp, optional label) should be derivable from directory naming and/or sidecar metadata.

### Diff and migration semantics

After creating a revision, the system can compare:

- `docs/rev.(N-1)/` vs `docs/current/`, or
- `docs/rev.(N-1)/` vs `docs/rev.N/` (if current is promoted first).

The diff result becomes structured context for migration generation/preview, instead of relying only on latest migration text and a single mutable design file.

## CLI and workflow integration expectations

## start workflow

`start` should scaffold prediction docs as a directory workflow:

- create `docs/current/` as the default edit target;
- place default seed content (including `Design.md`) inside `docs/current/`;
- preserve compatibility by treating root `Design.md` as legacy/fallback input during transition, not as the forward canonical source.

## migrate workflow

`migrate` should operate with revision awareness:

- design context source resolves from managed docs workflow (current + optionally revision-aware context), not only root `Design.md`;
- dedicated revision-save command runs before diff-based migration preview when requested;
- migration prompt/template vars should include revision-aware fields (for example current revision id, previous revision id, and revision diff summary/source references).

## research/agent behavior

Research and migration agents should treat these as valid first-class context locations:

- `docs/current/**`
- revision/archive directories (`docs/rev.*/**` or equivalent)

Agents should not assume that all authoritative context lives in a single file path.

## test workflow impact

Predicted-state test context currently includes `design`, `latestContext`, `latestSnapshot`, and migration history. Under the new model:

- `design` should resolve from managed docs context (defaulting to `docs/current/Design.md`);
- legacy root `Design.md` remains fallback-compatible during migration period;
- tests should remain deterministic when revision directories exist.

## Source and module boundaries to touch

Likely implementation seams based on current structure:

- application layer:
  - `src\application\start-project.ts` (scaffold layout)
  - `src\application\migrate-task.ts` (template vars and design context loading)
  - `src\application\test-specs.ts` (predicted-state design context)
- presentation/docs:
  - `src\presentation\cli.ts` (help strings and command wording)
  - `README.md`, `docs\cli.md` (user-facing workflow contract)
- domain/template contract:
  - template vars in migrate/test prompts should reflect revision-aware sources without breaking existing templates.

## Compatibility and transition constraints

To avoid breaking existing projects:

- keep backward compatibility for repositories that only have root `Design.md`;
- prefer additive resolution order during migration period (new docs workflow first, legacy fallback second);
- preserve existing migrate/test behavior when no `docs/` structure exists;
- avoid forcing immediate data migration for existing users.

## Edge cases and operational constraints

- Missing `docs/current/`: commands should surface clear guidance and/or bootstrap behavior.
- Empty current draft: revision save should still be valid if directory exists, but generation commands should handle low-context gracefully.
- Non-sequential or malformed revision directories: ignore safely with deterministic selection rules.
- Repeated revision save with no content changes: should still produce predictable behavior (either no-op with message or explicit new immutable snapshot policy).
- Cross-platform path handling must remain consistent with existing path operations and CLI output conventions.

## Acceptance criteria outline

- Design source resolution supports managed docs workflow and no longer assumes root `Design.md` as the only path.
- `docs/current/` is treated as canonical editable draft area for prediction design work.
- A dedicated revision-save flow exists and produces immutable revision snapshots.
- A dedicated diff/preview flow can compare current vs previous revision and feed migration generation context.
- Migration/test/research prompt context can include revision/archive directories as valid sources.
- Existing projects using only root `Design.md` continue to work via compatibility fallback.
- User-facing docs and CLI help reflect directory-based design revision workflow clearly and consistently.

## Scope and non-goals

In scope:

- design revision workflow contract;
- revision-aware context resolution for migrate/research/test flows;
- command/documentation updates needed to make this behavior explicit and usable.

Out of scope for this migration:

- changing core migration filename conventions (`NNNN-name.md`, `NNNN--satellite.md`);
- redesigning unrelated run/plan/reverify mechanics;
- introducing breaking removal of legacy `Design.md` support in one step.

Reinforce runtime resiliency for worker failures caused by usage limits and timeout/connection issues, with automatic fallback worker/model selection.

Current intent extends beyond a single timeout retry:

- Detect known usage-limit responses reliably.
- Detect timeout/no-response/connection-killed failure modes.
- Mark worker/model as temporarily unhealthy and switch to configured fallback.
- Reuse unhealthy workers only after cooldown.

If `--commit` flow is active, preserve in-progress git state safely before retry/fallback (stash or equivalent safe mechanism), then continue from a clean retry boundary.

Likely requires config evolution to track worker health policy, for example:

- per-worker status/health metadata (healthy, cooling-down, unavailable),
- cooldown durations before retrying a failed worker,
- fallback chain/priority for automatic failover.

Goal: avoid repeated failures on the same exhausted/unstable worker and keep long runs progressing automatically.

State management must be persistent and handled properly between runs:

- If a worker run fails due to usage limits, apply configured cooldown and keep that state for subsequent runs.
- If failure indicates connection/unavailable transport, mark unavailable and retry with fallback now, but allow reevaluation on a later run according to policy.
- Status tracking should support worker-level and profile-level resolution (since profile choice may map to different workers/models).

Add CLI visibility for this runtime state:

- command/output that shows current worker/profile health status,
- active cooldown timers / next-eligible times,
- fallback eligibility/order used by failover logic.

## Context in current codebase

Rundown already has several pieces that are close to this feature, but they are not connected into persistent worker health/failover yet:

- Usage-limit detection exists in verify/repair orchestration (`src/application/verify-repair-loop.ts`) via known-pattern matching and cross-phase similarity checks.
- A canonical run reason already exists for usage limit failures (`usage-limit-detected` in `src/domain/run-reasons.ts`) and is persisted in run metadata extras through run finalization (`runReason`, `failureExitCode` in `src/application/run-task-execution.ts`).
- Worker config supports `workers.default`, `workers.tui`, and `workers.fallbacks` plus command/profile overrides (`src/domain/worker-config.ts`, `src/infrastructure/adapters/worker-config-adapter.ts`).
- Worker resolution currently picks a single effective worker command for the invocation (`src/application/resolve-worker.ts`) and does not track runtime health/cooldown state.
- Runtime metadata persistence infrastructure already exists for runs/phases (`src/infrastructure/runtime-artifacts.ts`) but currently stores run outcomes, not a reusable worker-health registry.

This migration should build on those primitives rather than re-implementing detection logic from scratch.

## Problem framing

Today, repeated failures can loop on the same worker command/profile across tasks and runs, especially when:

- provider quota/rate-limit is exhausted,
- transport is degraded (timeouts, killed sessions, connection resets),
- the configured worker for a profile is temporarily unavailable.

Without stateful health tracking, each new task/run can select the same failing worker again, causing avoidable repeated failures and operator intervention.

## Failure classification model

Define explicit runtime failure classes used by failover decisions:

- `usage_limit`: known quota/rate-limit/billing exhaustion signals (already partially detectable in verify/repair).
- `transport_unavailable`: timeout, no response, connection killed/reset/unavailable channel.
- `execution_failure_other`: non-matching failures that should not automatically poison worker health long-term.
- `success`: clears transient penalties for the worker/profile used.

Design note: existing usage-limit heuristics in verify/repair should become one of the primary inputs for `usage_limit`, while transport classification should include process/runner-level timeout/no-output conditions.

## Health state model (persistent)

Represent health at both worker-command and profile resolution levels:

- `healthy`: eligible immediately.
- `cooling_down`: temporarily ineligible until `nextEligibleAt`.
- `unavailable`: hard-ineligible until reevaluation policy allows probing again.

Suggested persisted fields per entry:

- `key`: stable identity (normalized worker command and optional profile/command context).
- `status`
- `lastFailureClass`
- `lastFailureAt`
- `cooldownUntil` (for `cooling_down`)
- `failureCountWindow` (optional, for policy escalation)
- `lastSuccessAt`
- `source` (worker-level vs profile-level record)

## Fallback selection behavior

When execution is about to start for a task:

1. Resolve primary worker as today.
2. Consult health state for primary (worker-level and profile-level).
3. If ineligible, skip directly to fallback candidate ordering.
4. Candidate ordering should use:
   - explicitly configured fallback chain (`workers.fallbacks`),
   - eligibility filtering (exclude cooling/unavailable),
   - deterministic first-eligible selection.
5. On failure:
   - classify failure,
   - update health state for the failing candidate,
   - retry with next eligible fallback within policy limits.
6. On success:
   - record success and continue run using the successful worker for current task scope (and optionally subsequent tasks in same run, per policy).

Policy should prevent retry storms against the same exhausted provider during a single long run.

## Commit-mode safety requirements (`--commit`)

Current `run` flow enforces clean worktree before task execution when `--commit` is enabled (`src/application/run-task-execution.ts`). This migration adds a new requirement during failover retry boundaries:

- preserve in-progress git state safely before switching worker/retrying (stash or equivalent),
- retry from a clean boundary to avoid partial mixed state,
- restore/apply state consistently so final task completion/commit semantics remain intact.

Important interaction points:

- existing deferred-commit flows (`--commit-mode file-done`) in run lifecycle should remain correct,
- no corruption of artifact/trace expectations during stash/unstash boundaries,
- failure paths must not silently drop user changes.

## Persistence and storage strategy

Worker health must survive process boundaries and future runs. Persistence should be independent from transient in-memory run state.

Implementation options (to be decided in planning phase):

- dedicated health store file under `.rundown/` (recommended for clarity),
- extension to existing runtime artifacts metadata index (less clean for lookup),
- hybrid: dedicated store + trace linkage.

Requirements regardless of storage format:

- atomic write/update semantics,
- tolerant read behavior on missing/corrupt file,
- versionable schema for future policy evolution.

## CLI visibility requirements

Add operator-facing visibility for runtime worker health state, including:

- current health status by worker/profile,
- cooldown remaining and absolute next-eligible timestamp,
- fallback order and eligibility evaluation snapshot.

Output should be readable in normal CLI mode and scriptable (JSON mode is likely useful if aligned with existing command conventions).

## Integration boundaries and module touchpoints

Likely impacted modules:

- `src/application/resolve-worker.ts` (selection now health-aware and fallback-aware)
- `src/application/run-task-iteration.ts` / `src/application/task-execution-dispatch.ts` (retry/failover loop boundary)
- `src/application/verify-repair-loop.ts` (already emits usage-limit signals; feed classification)
- `src/domain/worker-config.ts` + adapter validation (health policy config schema evolution)
- new persistence adapter/service for worker health state
- `src/presentation/cli.ts` + `src/presentation/cli-command-actions.ts` (new visibility command/surface)

## Configuration evolution (direction)

Extend worker configuration with a health policy section, e.g. conceptual fields:

- per-failure-class cooldown durations,
- optional max failover attempts per task/run,
- fallback behavior knobs (strict order vs policy-based),
- reevaluation/probe policy for `unavailable` workers.

This should remain backward-compatible: absence of new config should preserve current behavior unless a safe default is explicitly chosen.

## Edge cases and constraints

- Detached/TUI nuances: output-capture limitations may weaken classification confidence for certain failures.
- Inline CLI/tool-expansion tasks: ensure failover logic only applies to worker-driven phases where appropriate.
- Parallel task groups: avoid race conditions when multiple tasks update shared health state concurrently.
- Cross-platform command identity normalization: worker keys must be stable across equivalent token arrays.
- Avoid destructive retry loops: cap retries/failovers and emit clear terminal reason when exhausted.

## Acceptance criteria outline

- Usage-limit and transport-unavailable failures trigger health updates and immediate fallback selection when eligible.
- Cooling-down/unavailable workers are not reused until policy allows.
- Health state persists across runs and influences subsequent worker resolution.
- `--commit` mode preserves in-progress git state safely across retry/failover boundaries.
- CLI exposes health status, cooldown timers, and fallback eligibility/order.
- Existing non-failover flows remain backward-compatible when policy/config is absent or disabled.

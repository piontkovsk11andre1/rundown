## Slug/name flow map (current state)

### 1) Task creation (`rundown migrate` with no action)

1. `createMigrateTask` resolves `resolvedWorker.workerPattern` via `resolveWorkerPatternForInvocation`.
2. `generateNextMigration` calls `workerExecutor.runWorker(..., artifactPhaseLabel: "migrate-next")` using that worker pattern.
3. Worker stdout is parsed by `parseProposals`.
4. Proposal `name` values are normalized by `toKebabCase` and then used to build `migrationFilename` as `${NNNN}-${name}.md`.
5. `createMigrationDocument` also embeds the chosen worker-derived `selected.name` in the migration title line (`# NNNN <name>`).

Result: migration naming is currently coupled to whichever worker resolved for this invocation, because the canonical name string originates in worker output before local kebab normalization.

### 2) Migration path (`rundown migrate up` prediction reconciliation)

1. `createMigrateTask` resolves `resolvedWorker.workerPattern` and passes it into `reconcilePendingMigrationPredictions`.
2. When pending predictions are stale, `reResolvePendingPredictionSequence` invokes the worker with `buildPendingReResolutionPrompt`.
3. Worker JSON `migrations[].name` is parsed in `parseReResolvedPlanItems`.
4. Each parsed name is normalized by `toKebabCase` and persisted as `migrationName`.
5. `reconcilePendingPredictedItemsAtomically` writes migration files as `${NNNN}-${migrationName}.md` and snapshot files as `${NNNN}--snapshot.md`.

Result: reconciliation can rename/rewrite pending migration filenames based on worker-produced names; this is another slug/name path tied to execution worker context.

### 3) Test-spec generation (`rundown test new`)

1. `createTestSpecs` takes assertion text from `options.prompt`.
2. `slugifySpecName(assertion)` performs local deterministic normalization.
3. Spec file is written to `${slug}.md`.
4. If `--run` is set, worker execution only verifies assertions and does not participate in slug generation.

Result: spec slugging is deterministic and local; unlike migrate flows, it is not derived from worker output.

## Call-site inventory: name derivation from worker context

- `src/application/migrate-task.ts` `parseProposals` parses worker stdout proposal lines and normalizes proposal names with `toKebabCase(match[1])`.
- `src/application/migrate-task.ts` `parseProposals` fallback path derives a name from first stdout line via `toKebabCase(...)` when no numbered proposals are parsed.
- `src/application/migrate-task.ts` `generateNextMigration` -> `parseProposals` -> `toKebabCase` -> `${nextNumber}-${name}.md`.
- `src/application/migrate-task.ts` `createMigrationDocument` uses worker-selected proposal name in migration heading.
- `src/domain/prediction-reconciliation.ts` `parseReResolvedPlanItems` -> `toKebabCase(name)` from worker JSON.
- `src/domain/prediction-reconciliation.ts` `reconcilePendingPredictedItemsAtomically` writes `${NNNN}-${migrationName}.md` from parsed worker names.

## Worker-context entry points that feed naming

- `src/application/migrate-task.ts` `createMigrateTask` resolves `resolvedWorker.workerPattern` and passes it to both `generateNextMigration` (task creation path) and `reconcilePendingMigrationPredictions` (migrate up reconciliation path).
- `src/domain/prediction-reconciliation.ts` `buildPendingReResolutionPrompt` requires worker JSON items with `"name"`; those values are later normalized and persisted as migration names.

## Non-worker deterministic slug paths (for contrast)

- `src/application/test-specs.ts` `slugifySpecName` -> `${slug}.md` from user assertion text.
- `src/application/migrate-task.ts` `generateSatellite` writes `${NNNN}--${satelliteType}.md` (type-driven, not worker-derived name).

## Intent

Add a loop-style task prefix so one checkbox task can execute its child checklist repeatedly for a precomputed set of values. The behavior should work for `for:`, `each:`, and `foreach:` as equivalent user-facing spellings, with one canonical internal tool name.

The loop task should behave like a control-flow task (similar shape to `parallel:` / `include:` / `end:`): it orchestrates execution and completion of nested checkbox child tasks, instead of being a normal single-pass worker prompt.

## Behavior model (expanded)

The examples above describe the lifecycle:

- loop parent starts unchecked
- loop candidates are pre-baked once at start into `for-item:` lines
- one active value is exposed as `for-current:`
- nested child checkbox tasks run for the active value
- when advancing to next value, child checkbox states reset for the next pass
- when all values complete, loop parent becomes checked and `for-current:` is removed or no longer required

Important detail: this is not a dynamic stream. It snapshots iteration items first, then iterates that stable list.

## Representation in Markdown

Planned on-disk representation from runtime behavior:

- Parent checkbox task text contains loop prefix (`for:` or alias).
- Iteration metadata is stored as nested non-checkbox list items:
  - `for-item: <value>` for each pre-baked candidate
  - `for-current: <value>` for active item
- Actual executable work remains nested checkbox child items under the same parent.

This aligns with current parser behavior in `src/domain/parser.ts`:

- nested checkbox list items are parsed as child tasks
- nested plain list items are preserved as `subItems`
- directive-like non-checkbox lines already exist as a supported pattern

So `for-item:` and `for-current:` naturally fit as sub-items rather than tasks.

## Integration points in current architecture

Existing code already has a prefix-chain + built-in tool model:

- prefix parsing: `src/domain/prefix-chain.ts`
- built-in tool registry: `src/domain/builtin-tools/index.ts`
- task iteration orchestration: `src/application/run-task-iteration.ts`
- parser task/sub-item extraction: `src/domain/parser.ts`

Loop prefix should plug into this model as a built-in handler tool (not a modifier), likely next to other control-flow handlers.

Expected integration shape:

- add built-in aliases (`for`, `each`, `foreach`) mapping to one loop handler
- handler receives payload and execution context, resolves item list, writes/updates sub-items, and drives per-item child execution flow
- respect current safety rules for checkbox mutation and source updates owned by orchestrator layer

## Prefix semantics

User-level semantics:

- `for: <query-or-source>` is the canonical documented form
- `each:` and `foreach:` are compatibility aliases
- aliases normalize to canonical behavior, no semantic drift between names

Prefix composition should stay compatible with current chain parser (comma/semicolon boundaries), e.g. profile selection before loop handler should remain valid when used.

## Execution semantics

Execution is effectively nested:

- outer: loop over pre-baked items
- inner: run unchecked checkbox children under loop parent

Per-item rules:

- before each item run, set exactly one `for-current:`
- child checkbox tasks start unchecked for that item
- after all children pass for current item, advance
- if a child fails verification/repair policy, preserve current item context for retry behavior

Completion rules:

- all items done => mark loop parent checked
- keep `for-item:` lines as execution trace/context
- final doc should reflect completed children (as shown in example)

## State transition and reset expectations

The note "So it resets and changes current" is a core contract:

- reset scope is child checkbox state for the next item pass
- `for-current:` must always match the active item while in progress
- transition should be atomic enough to avoid ambiguous file state when reloaded between iterations

If process is interrupted mid-loop, document should still indicate resumable state via existing `for-item:` list and current cursor.

## Error handling and edge cases

Edge cases to define and keep deterministic:

- zero resolved items: either no-op complete or explicit failure, but must be consistent
- duplicate item values: allowed vs deduplicated must be explicit
- item text containing `:` or markdown-sensitive chars must round-trip safely in sub-item text
- child task list missing: loop task behavior should be explicit (skip vs fail)
- existing manual `for-item:` / `for-current:` in source before run: whether they are trusted, replaced, or regenerated
- alias mixed usage (`each:` parent with `for-item:` metadata) should still normalize to one metadata format

## Compatibility constraints

Must preserve current guarantees used across run/verify flows:

- no mutation of fenced code block content
- no accidental promotion of metadata sub-items into executable tasks
- no unexpected checkbox toggles outside active loop parent scope
- maintain existing CLI flags behavior (`--redo`, `--reset-after`, dry-run/print-prompt semantics)

And should not break existing built-in prefixes (`verify`, `fast`, `include`, `parallel`, `end`, `profile`, `force`).

## Observability and artifacts

Runtime artifact and trace output should make loop progress inspectable:

- log pre-baked item count and values (or summarized values)
- log `for-current` transitions
- log per-item child completion
- log final parent completion

This keeps loop runs debuggable in the same way current task iteration phases are debuggable.

## Acceptance criteria outline

A planning/implementation pass should treat the feature as complete when:

- a loop-prefixed parent task runs child checkboxes once per pre-baked item
- source doc receives/updates `for-item:` and `for-current:` metadata consistently
- child checkbox reset between items works as specified
- parent checkbox only completes after all items are done
- aliases `for:`, `each:`, `foreach:` behave equivalently
- interrupted runs can resume without corrupting loop metadata
- existing non-loop task behavior and tests remain green

## Scope boundaries

In scope:

- loop prefix recognition and alias normalization
- runtime loop orchestration over child tasks
- metadata sub-item materialization (`for-item`, `for-current`)
- deterministic reset/advance semantics

Out of scope for this migration:

- introducing a new markdown syntax beyond current list/sub-item conventions
- changing global checkbox engine contracts unrelated to loop behavior
- redesigning runtime artifact storage format outside what loop observability needs

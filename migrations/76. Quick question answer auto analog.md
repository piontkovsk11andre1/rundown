## Intent

Add a lightweight question/answer extraction pattern that lets one task ask for a concrete list and persist the returned values directly under that task as structured sub-items. The core goal is speed and determinism: ask once, capture results in-place, and let later tasks consume those values without re-asking.

This is similar in spirit to existing built-in control-flow/tool-prefixed tasks (`verify:`, `include:`, `parallel:`, `question:`), but scoped to quick data collection where the output is a list of discovered names.

## Expected behavior

The `get:` task behaves as a single-pass extraction request:

- Task text declares the query, for example: `get: All current names of this and that`.
- Execution resolves the query and writes one or more `get-result:` sub-items under the same task.
- The task checkbox is marked complete only after extraction succeeds and results are persisted.
- Result lines are plain nested list sub-items (not nested checkbox tasks), so they are treated as context, not executable work.

Target shape after execution:

```md
- [x] get: All current names of this and that
  - get-result: This
  - get-result: That
```

## Data contract in markdown

Result representation should stay stable and machine-readable:

- Prefix: `get-result:`
- One value per line
- Stored as direct child sub-items of the `get:` parent task
- Preserve order returned by extraction unless explicit normalization is requested
- Avoid transforming values beyond minimal whitespace normalization

This follows parser conventions in `src/domain/parser.ts`, where nested non-checkbox list items are captured as `subItems` and remain separate from executable child tasks.

## Integration context in current architecture

Relevant existing patterns in the codebase suggest this should be implemented as a built-in handler tool:

- Prefix parsing is handled via `parsePrefixChain(...)` in `src/domain/prefix-chain.ts`.
- Built-in tool registration and aliases are centralized in `src/domain/builtin-tools/index.ts`.
- Tool handlers can skip worker execution and/or verification based on frontmatter behavior.
- A close behavioral analog exists in `src/domain/builtin-tools/question.ts`, which:
  - reads existing sub-items,
  - collects or reuses an answer,
  - writes structured sub-items back into the same markdown task block.

That makes `get:` a natural fit for a handler that updates task-local sub-items, likely with `skipExecution: true` when extraction is fully handled by the tool path, or normal worker execution when the model must perform discovery. Either way, persisted output format should remain `get-result:` lines.

## Resolution model

To keep this "quick extraction" reliable, resolution rules should be explicit:

- If results already exist (`get-result:` lines present), decide whether to:
  - reuse existing values (idempotent default), or
  - refresh/replace on rerun (explicit mode).
- If no results are found:
  - fail with clear reason, or
  - succeed with an explicit empty-state marker policy.
- Duplicate values:
  - either preserve as returned, or deduplicate deterministically.
- Sorting:
  - either preserve discovery order, or sort lexically; whichever is chosen should be consistent.
- Escaping:
  - values containing `:` or markdown-significant characters must round-trip without corrupting structure.

## Scope boundaries

In scope:

- Recognize `get:` as a task prefix for quick extraction.
- Materialize `get-result:` sub-items beneath the task.
- Keep extraction output lightweight and directly reusable by following tasks.

Out of scope:

- Converting results into new checkbox TODO tasks automatically.
- Multi-phase planning/decomposition (that belongs to `research`/`plan` flows).
- Custom markdown syntax outside existing task/sub-item conventions.

## Acceptance criteria outline

Feature is considered complete when:

- A task written as `get: <query>` can execute and persist `get-result:` lines under the same task.
- Persisted results are parsed as sub-items, not executable tasks.
- Checkbox completion reflects successful extraction and persistence.
- Re-run behavior for pre-existing `get-result:` lines is deterministic.
- Existing prefix/tool behavior (`verify`, `include`, `parallel`, `question`, etc.) remains unaffected.

## Compatibility and operational notes

- Must not mutate fenced code blocks.
- Must preserve existing markdown structure and indentation conventions used by current sub-item upsert logic.
- Should remain compatible with dry-run/print-prompt semantics and artifact/trace workflows in run iteration.
- Should produce clear runtime logs indicating whether results were generated, reused, replaced, or empty.

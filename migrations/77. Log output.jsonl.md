Log output.jsonl.

## Intent

Add and standardize a cumulative JSONL output log at `<config-dir>/logs/output.jsonl` so every CLI invocation writes structured, append-only runtime output that is easy to ingest by line-oriented log collectors (for example Promtail) and easy to correlate per command/session.

The goal is observability without changing normal CLI UX: users still see normal terminal output, while the same events are mirrored into a stable machine-readable stream.

This stream is a baseline operational telemetry channel and should be available independently of trace mode or artifact retention choices.

## Current architecture context

The codebase already separates output concerns across presentation/domain/infrastructure layers:

- Presentation output events are emitted through `ApplicationOutputPort` and rendered by `src/presentation/output-port.ts`.
- Structured global logging types and sanitization live in `src/domain/global-output-log.ts`.
- File append behavior is provided by `src/infrastructure/adapters/global-output-log-writer.ts`.
- Invocation metadata and pre-app parse/error logging are coordinated in `src/presentation/cli-invocation-log.ts`.
- Runtime artifact path conventions and config-dir-relative log placement are defined in `src/infrastructure/runtime-artifacts.ts` (`GLOBAL_OUTPUT_LOG_RELATIVE_PATH = "logs/output.jsonl"`).
- App wiring for invocation-scoped logged output happens in `src/presentation/cli-app-init.ts` via `createLoggedOutputPort(...)`.

This migration should preserve that layering (domain schema + presentation mapping + infrastructure writer).

Additional implementation context in the existing architecture:

- `createLoggedOutputPort(...)` wraps the concrete CLI output port, writes a structured entry first (best-effort), then forwards the original event so terminal UX remains authoritative.
- Event-to-log mapping is centralized in presentation (`src/presentation/logged-output-port.ts`) via explicit `resolveLogKind/resolveLogLevel/resolveLogStream/resolveLogMessage` helpers.
- Domain serialization (`serializeGlobalOutputLogEntry`) guarantees one JSON object per line with trailing newline and delegates ANSI stripping via `sanitizeGlobalOutputLogEntry(...)`.
- Infrastructure writing is synchronous append (`appendFileSync` with `flag: "a"`), with lazy parent-directory creation and swallowed failures by design.
- Invocation metadata (`command`, `argv`, `cwd`, `pid`, `version`, `session_id`) is captured once per invocation in `createCliInvocationLogState(...)` and attached to every line.
- Commander framework output is intercepted with `command.configureOutput(...)` so parse-time stdout/stderr can be mirrored into the same global stream, including paths before full app wiring.

## Target behavior

`output.jsonl` should behave as one cumulative stream per resolved config directory:

- File path: `<resolved-config-dir>/logs/output.jsonl`.
- Each emitted output event becomes one JSON object per line.
- Entries append only; existing lines are never rewritten/truncated by normal command flow.
- Log writing is best-effort: failures to write logs must not fail command execution/output rendering.
- ANSI escape sequences must be stripped from persisted string fields so the stream stays parse-safe plain text.
- Logging should cover both:
  - normal application output events (task lifecycle, info/warn/error/progress/text/stderr/group events),
  - CLI/framework-level failures (for example Commander parse errors and fatal startup/runtime errors) so error paths are also represented.

Operational expectations implied by this behavior:

- The log file is shared across repeated invocations under the same effective config dir.
- Multi-line user-facing messages remain valid JSON values in a single JSONL line (escaped `\n`), not physical multi-line records.
- The stream is append-only under normal command flow; no rotation/truncation is introduced by this migration.
- Failures in logging path are non-fatal and must never mask original command outcomes.

## Stable line schema

Each line should use stable top-level fields:

- `ts` (ISO-8601 timestamp)
- `level` (`info` | `warn` | `error`)
- `stream` (`stdout` | `stderr`)
- `kind` (event-kind label, including CLI/framework kinds where relevant)
- `message` (plain text payload)
- `command` (resolved top-level command token)
- `argv` (invocation args)
- `cwd` (invocation working directory)
- `pid` (process id)
- `version` (CLI version)
- `session_id` (invocation correlation id)

Kinds should remain semantically aligned with output events (`info`, `warn`, `error`, `success`, `progress`, `task`, `text`, `stderr`, `group-start`, `group-end`) plus CLI/framework categories for non-use-case paths (for example `commander`, `cli-fatal`).

Schema and mapping notes:

- Key set is stable and flat at top level; no nested metadata envelope is required for ingestion.
- `level` and `stream` are derived from event semantics, not only from where terminal rendering happens.
- `group-end` failure maps to warning severity and stderr stream; success maps to info severity and stdout stream.
- `task`/`progress`/`group-*` messages are normalized to deterministic plain-text forms for machine consumption.
- ANSI stripping applies across all string fields (`message`, `command`, `argv[]`, `cwd`, `version`, `session_id`, and timestamp/kind string values).

## Config-dir and placement rules

Log placement should follow the same config-dir resolution behavior used by runtime features:

- Explicit `--config-dir` wins.
- Otherwise use discovered `.rundown` via upward resolution.
- If discovery is unavailable, use fallback `<cwd>/.rundown` when writing.
- In linked workspace mode, resolution should follow the invocation workspace/link policy, not local accidental cwd drift.

This ensures logs are colocated with the effective runtime configuration and not split unpredictably.

Resolution details that should remain consistent with current behavior:

- Upward discovery should treat `.rundown` as valid only when `config.json` exists, so artifact-only directories do not shadow configured parents.
- Linked workspace flow should prefer resolving from linked workspace root and only fall back to invocation directory when needed.
- Invocation log state should be initialized once early in parse flow so both parse-time and app-time outputs target the same resolved log path.

## Integration with trace streams

`output.jsonl` is for rendered CLI/application output events. It is distinct from trace streams:

- Per-run trace artifacts remain at `.rundown/runs/<id>/trace.jsonl`.
- When tracing is enabled, cumulative trace mirroring is handled separately at `.rundown/logs/trace.jsonl`.

The output stream should not require `--trace`; it should exist independently as baseline operational telemetry.

Integration boundary notes:

- Output logging is invocation-global and always-on.
- Trace logging remains execution-phase instrumentation and opt-in where currently designed.
- No schema coupling is required between output and trace lines beyond shared operational correlation patterns (for example timestamp/session-level diagnostics).

## Design constraints

- Do not change existing terminal output behavior to users as a side effect of logging.
- Do not require artifact retention (`--keep-artifacts`) for global output log capture.
- Do not introduce ANSI/control sequences in persisted records.
- Do not make log persistence a hard dependency (must remain fail-safe).
- Preserve stream semantics (`stdout` vs `stderr`) and severity mapping consistency.
- Keep one-line-per-entry JSONL formatting with trailing newline termination for ingest compatibility.

Additional constraints from current patterns:

- Preserve write ordering guarantees expected by append-only readers.
- Avoid introducing buffering semantics that could drop tail events on process exit unless explicit flush lifecycle is added and safely wired.
- Keep parse-time behavior intact (Commander output still appears as before) while adding mirror writes.
- Keep command failure handling unchanged: logged failures are additive observability, not behavioral changes.

## Edge cases to cover

- Parse-time command failures before app initialization.
- Invalid option errors emitted by Commander.
- Runtime execution failures (for example failing `cli:` task command).
- Multi-line messages (must remain valid JSON as single line with escaped newlines).
- Repeated consecutive CLI invocations (strict append-only growth).
- Commands with no artifact preservation still producing output log entries.
- ANSI in task text or subprocess output being stripped in persisted fields.
- Missing parent log directory created lazily.

Additional edge-case clarifications:

- Empty framework writes should not emit empty records.
- Root-command/help flows that initialize app early should still preserve consistent invocation metadata in output lines.
- Failures while creating parent log directory and failures during append must both be swallowed.
- Group lifecycle output should stay pairable through `group-start`/`group-end` kind events for downstream reconstruction.
- Mixed invocations (`next`, `list`, `run`, etc.) in same workspace should append to same file while preserving per-invocation `command` and `session_id`.

## Scope boundaries

In scope:

- Global JSONL output sink behavior and schema stability.
- Event mapping from output events to structured log entries.
- Invocation metadata capture and correlation fields.
- Config-dir-based log path resolution and append semantics.
- Coverage for normal and failure paths.

Out of scope:

- Built-in rotation/compression/retention policy for `output.jsonl`.
- Backfilling historical run artifact output into this file.
- Redesign of trace schema or replacing per-run artifacts.
- Changes to command lifecycle semantics unrelated to logging.

## Validation and test surface

Existing tests already cover the intended behavior categories and should remain representative:

- Domain serialization tests for single-line JSON + trailing newline and ANSI stripping.
- Infrastructure writer tests for lazy directory creation, append behavior, and best-effort failure swallowing.
- Presentation mapping tests for event-kind/level/stream/message normalization, including group and task formatting.
- CLI invocation tests for file creation, append-only growth across commands, config-dir placement, parse-time commander errors, runtime failures, and independence from `--keep-artifacts`.

Validation outcomes expected from those categories:

- Every parsed line is valid JSON object with stable key set.
- No persisted line contains ANSI escape sequences.
- Consecutive invocations preserve prior content exactly and only append new lines.
- Failure-path logs are observable without altering process exit semantics.

## Acceptance criteria outline

Feature is complete when:

- Running CLI commands creates/appends `<config-dir>/logs/output.jsonl`.
- Entries are valid single-line JSON objects with the stable schema keys.
- Logs remain append-only across multiple invocations.
- ANSI sequences are absent from persisted entries.
- Both success-path output and key failure-path output are captured.
- Log capture works regardless of `--keep-artifacts`.
- Config-dir resolution determines log location consistently (including upward discovery / explicit config dir).
- Logging failures do not break user-facing command execution/output.

Acceptance interpretation details:

- “Creates/appends” includes first-write lazy directory creation and subsequent no-truncate appends.
- “Success-path output captured” includes normal event kinds emitted through `ApplicationOutputPort`.
- “Failure-path output captured” includes Commander parse errors and fatal CLI errors represented with framework/fatal kinds.
- “Does not break execution/output” means output rendering and command exit behavior stay functionally equivalent when log writes fail.

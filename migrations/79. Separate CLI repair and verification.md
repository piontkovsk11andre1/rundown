Separate CLI repair and CLI verification flows.

## Intent

Clarify and decouple verification and repair behavior at the CLI contract level so each phase can be configured, resolved, and observed independently without changing the existing default behavior for users who do not opt in to separation.

Today, verification and repair are exposed as separate concepts in user messaging and flags, but they still share parts of the same runtime path and worker resolution assumptions. This migration should make those boundaries explicit.

## Current behavior and coupling

The current flow is centered in `runVerifyRepairLoop` (`src/application/verify-repair-loop.ts`):

- Initial verification runs through `taskVerification.verify(...)`.
- Repair attempts run through `taskRepair.repair(...)`.
- Each repair call performs repair + immediate re-verification internally (`src/infrastructure/repair.ts`), so verification retries are partly owned by the repair implementation.
- Resolve/resolve-informed repair is layered on top of the same loop and reuses the same worker pattern and most runtime options.

CLI entry points (`run`, `do`, `loop`, `reverify`) all normalize to shared behavior flags via `resolveRunBehavior` (`src/domain/run-options.ts`) and currently expose:

- `--repair-attempts <n>`
- `--resolve-repair-attempts <n>`
- `--no-repair`
- `--only-verify` / `--no-verify` (run-like commands)

Worker resolution already supports intent-level override for verify-only intent (`commands.verify`) via `resolveWorkerPatternForInvocation` (`src/application/resolve-worker.ts`), but there is no first-class repair intent override in the same style.

## Target separation model

Separate flows means verification and repair should be independently controllable in runtime orchestration, not just conceptually different in templates/messages.

Planned model:

1. Verification flow owns verification execution policy and result parsing.
2. Repair flow owns repair execution policy and correction strategy.
3. The loop coordinator composes both flows explicitly rather than hiding verification retries inside repair internals.
4. Default behavior remains backward-compatible when no new separation options are provided.

This keeps existing user-facing semantics (`verification failed`, `repair attempt`, `resolve-informed repair`) while making implementation boundaries more deterministic and testable.

## Module boundaries and integration points

Primary modules affected:

- `src/application/verify-repair-loop.ts`
  - Keep as orchestration layer.
  - Shift phase ownership so verify and repair are called as distinct steps with explicit transitions.
- `src/infrastructure/repair.ts`
  - Remove or reduce implicit re-verification ownership inside `repair(...)` path if separation requires orchestration-level reverify calls.
- `src/infrastructure/verification.ts`
  - Remains source of verification parsing and normalization (`OK` / `NOT_OK:` contract).
- `src/application/reverify-task.ts`
  - Must preserve parity with run behavior for separated flows.
- `src/application/run-task-execution.ts` and `src/application/task-execution-dispatch.ts`
  - Continue to feed normalized run behavior and failure reason surfaces into completion.
- `src/application/resolve-worker.ts` and `src/domain/worker-config.ts`
  - If separation introduces distinct worker resolution (for verification vs repair), extend command/intent mapping without breaking current cascades.
- `src/presentation/cli.ts`, `src/presentation/cli-command-actions.ts`, `src/presentation/cli-options.ts`
  - Wire and validate any new/updated CLI options consistently across run-like and reverify commands.
- `docs/cli.md`
  - Keep option docs and behavior notes aligned with actual CLI behavior.

## Compatibility constraints

- Preserve existing defaults:
  - `--repair-attempts 1` behavior remains effectively unchanged for existing users.
  - `--no-repair` still disables repair even when attempts are set.
  - `--only-verify` still skips execution, then runs verification.
- Preserve existing exit-code contracts:
  - verification/repair exhaustion remains verification-failure exit path.
  - resolve unresolved/exhausted behavior remains consistent with current status handling.
- Preserve trace/artifact semantics for verify/repair/resolve phases.
- Preserve grouped CLI output readability and current summary lines.

## Edge cases to keep deterministic

- `--repair-attempts 0` with verification enabled.
- `--no-repair` combined with non-zero repair attempts.
- verify pass on first attempt (repair flow never invoked).
- repair success followed by re-verification pass.
- resolve path entry after normal repair exhaustion.
- resolve-informed repair exhaustion when `--resolve-repair-attempts` is set.
- mode differences (`wait`, `tui`, `detached`) where usage-limit detection and output capture rules already differ.
- inline `cli:` tasks and tool-expansion tasks, where verification/repair behavior is already constrained by task type.

## Acceptance criteria outline

Feature is complete when:

- Verification and repair flows are represented as separate runtime phases with explicit orchestration boundaries.
- CLI behavior remains backward-compatible by default across `run`, `do`, `loop`, and `reverify`.
- Reverify preserves parity with run for verification/repair sequencing and failure messaging.
- Worker/profile resolution behavior for verification and repair is unambiguous and documented.
- Existing integration tests for verification/repair/resolve continue to pass with updated architecture.
- New coverage confirms no regression in edge cases listed above.

## Out of scope

- Redesigning verification verdict format (`OK` / `NOT_OK:`).
- Changing core resolve diagnosis contract (`RESOLVED:` / `UNRESOLVED:`).
- Reworking unrelated command pipelines (plan/research/query/discuss) beyond shared option plumbing required for parity.
- Changing artifact storage format beyond what is necessary to keep phase attribution accurate.

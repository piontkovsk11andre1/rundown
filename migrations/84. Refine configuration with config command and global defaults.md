Refine configuration UX with a dedicated `config` command and user-level global defaults applied across rundown workspaces.

## Intent

Configuration should be discoverable, editable, and layered without requiring manual JSON edits in every repository.

This migration introduces:

1. A first-class `rundown config` command surface.
2. User-level (global) config defaults that apply to every rundown project unless overridden locally.
3. Safer init behavior that does not clobber existing config files.

## Product goal

- Users can inspect and update config via CLI, not only by editing files directly.
- A global default config provides baseline behavior across all rundown directories.
- Local project config remains authoritative for project-specific overrides.
- `init` is idempotent and preserves existing config content by default.

## Proposed config layering model

Recommended precedence (low -> high):

1. Built-in defaults.
2. User-level global config (for example in user home config dir).
3. Repository `.rundown/config.json`.
4. Command-specific overrides (existing behavior such as CLI flags).

Notes:

- Global config should be optional; absence means current behavior remains unchanged.
- Merge semantics should be explicit and deterministic (object deep-merge policy vs replace-by-key where appropriate).

## Scope

- Add `config` command group for reading/writing effective and scoped config values.
- Add global config file discovery/loading from user scope.
- Integrate layered resolution into existing config load pipeline.
- Preserve and document `init` behavior so existing config is not overwritten.

## Behavior constraints

- Existing projects without global config remain fully backward compatible.
- Invalid global config should surface clear diagnostics without corrupting local config.
- Effective config output should show source attribution (global vs local vs flag) where practical.
- No destructive overwrite of `.rundown/config.json` during `init` when file already exists.

## Acceptance criteria

- `rundown config` provides a usable interface for inspect/set/unset/list operations.
- Global config is discovered from a user-level location and applied to all rundown projects.
- Local `.rundown/config.json` overrides global defaults predictably.
- `init` preserves existing config by default and communicates when files are reused.
- CLI/docs explain layering, precedence, and global config path conventions.

Prefix test:

## Intent

Define a deterministic, repo-backed test contract for prefix behavior so parser intent inheritance, prefix-chain tool routing, and execution/verification mode selection remain aligned as prefix aliases evolve.

This migration is about **testing and specification clarity**, not introducing new prefix semantics.

## Why this is needed

Prefix behavior is currently implemented across multiple layers, and regressions are easy when one layer changes without synchronized tests:

- Parse layer intent detection and directive inheritance in `src/domain/parser.ts`
- Legacy intent classification in `src/domain/task-intent.ts`
- Unified tool-prefix parsing in `src/domain/prefix-chain.ts`
- Built-in prefix aliases and canonical mapping in `src/domain/builtin-tools/index.ts`
- Verification-mode selection in `src/application/iteration-mode.ts`
- Worker selection precedence in `src/application/resolve-worker.ts` and `src/domain/worker-config.ts`

A “prefix test” migration should lock the expected cross-layer behavior so future prefix additions or alias changes do not drift.

## Current implementation context

### Prefix families already present

- Verify-only aliases: `verify`, `confirm`, `check`
- Fast-execution aliases: `fast`, `raw`, `quick`
- Memory-capture aliases: `memory`, `memorize`, `remember`, `inventory`
- Parallel-group aliases: `parallel`, `concurrent`, `par`
- Loop aliases: `for`, `each`, `foreach`
- Modifier prefixes (tool chain): `profile=...`, `force: ...`
- Tool-style generic prefixes: `<toolName>: payload` via resolver

### Canonicalization and routing model

- Parser (`parseTasks`) handles directive inheritance (`verify:`, `fast:`, `parallel:` with empty payload forms) and task-level metadata extraction.
- Intent classifier (`classifyTaskIntent`) still defines precedence for legacy flow: verify > memory > parallel > fast > tool-expansion > default.
- Prefix-chain parser (`parsePrefixChain`) handles composable tool syntax and boundary splitting at `,` / `;` only when next segment starts with a known tool.
- Built-in registry normalizes some aliases to canonical tool names (for example `each` and `foreach` resolve as handler name `for`).
- Iteration mode suppresses verify for fast intent and preserves verify-only behavior unless force-execute is enabled.

## Scope for this migration

### In scope

- Document and test expected behavior for all supported prefix families.
- Validate alias equivalence and canonicalization where applicable.
- Validate precedence and composition rules across parser, intent classifier, and prefix-chain layers.
- Validate empty payload handling paths for prefixed tasks.
- Validate that logs/messages remain semantically consistent for alias-equivalent prefixes.
- Validate that worker resolution hooks (intent command names and `tools.<name>`) respect prefix-derived task intent/tool identity.

### Out of scope

- Introducing new prefixes or changing runtime execution semantics.
- Refactoring away legacy intent classification.
- CLI UX redesign for prefix authoring.
- Non-prefix task planning behavior.

## Prefix behavior contract

### 1) Alias parity

Each alias set must behave identically within its family:

- `verify|confirm|check` -> verify-only semantics
- `fast|raw|quick` -> execution without per-task verification
- `for|each|foreach` -> same loop handler behavior and metadata format

### 2) Precedence stability

Mixed-prefix strings must resolve by current precedence contracts, including first explicit recognized intent/tool position rules where applicable. Behavior must remain stable for examples such as:

- `verify: quick: ...` remains verify-only
- `quick: verify: ...` remains fast-execution
- `profile=fast, verify: ...` applies modifier then verify handler
- Unknown prefixes must fall back to regular execute-and-verify task text behavior

### 3) Empty payload determinism

Prefix tasks with missing payload must follow existing family-specific handling without silent behavior drift:

- Fast/quick/raw empty payload handling remains identical across aliases.
- Tool-expansion empty payload classification remains explicit and test-covered.
- Directive forms with no payload remain directive-only when syntax requires it.

### 4) Syntax compatibility and guardrails

- Case-insensitive and whitespace-tolerant forms remain supported (`Quick:`, `quick :`, `VERIFY :`, etc.) where already accepted.
- Legacy rejected syntax (`profile: name`) remains explicitly invalid where enforced.
- `profile=<name>` remains the valid profile modifier/directive shape.

## Integration notes by module boundary

- `src/domain/parser.ts` is the source of truth for directive inheritance and task/sub-item separation; tests must assert it does not over-classify plain sub-items as executable tasks.
- `src/domain/task-intent.ts` still influences iteration mode and worker intent resolution paths; regression tests should remain until full migration to tool-only intent is complete.
- `src/domain/prefix-chain.ts` and `src/domain/builtin-tools/index.ts` define composable tool behavior and alias canonicalization; tests should assert chain segmentation and terminal handler selection.
- `src/application/iteration-mode.ts` is where configured verify flags interact with prefix-derived intent; tests should assert verify suppression and emitted info messages for fast-family tasks.
- `src/application/resolve-worker.ts` must continue honoring intent/tool-derived override keys (`verify`, `memory`, `tools.<toolName>`) without breaking base `commands.run` fallback behavior.

## Edge cases to lock with tests

- Prefix-like text in plain prose does not trigger intent classification.
- Fenced code blocks containing prefix-like lines do not affect task extraction.
- Payload text containing `:` after a valid prefix is preserved and not re-parsed unexpectedly.
- Separator parsing in chained prefixes (`profile=fast, ...` and `profile=fast; ...`) only splits when next segment is a known tool.
- Alias/canonical name differences do not break downstream worker/tool resolution.
- Directive inheritance applies to checkbox descendants only and does not mutate non-checkbox metadata lines.

## Acceptance criteria outline

- Prefix families listed above have deterministic tests proving alias parity.
- Mixed-prefix precedence behavior is covered and stable.
- Empty-payload handling is explicitly asserted for major families.
- Parser, intent classifier, prefix-chain parser, and iteration mode tests collectively cover the same behavior surface to prevent cross-layer drift.
- Worker resolution tests confirm prefix-derived intent/tool routing still selects the intended command overrides.
- No existing prefix behavior regresses for current migrations and integration scenarios.

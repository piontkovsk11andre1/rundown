start in a dir with files will create first migration that will research on target documents and materials and create 0 revision target

## Intent

When `rundown start` is executed in a directory that already contains files, the first migration must be seeded for discovery-first onboarding, not greenfield scaffolding. The seed should explicitly direct research of existing materials and creation of a revision-0 baseline target.

This is the migration that defines the initial protocol for bringing an existing project under prediction workflow.

## Current behavior and related surfaces

The current `start` flow already distinguishes empty vs non-empty target directories and creates `migrations/0001-initialize.md` during bootstrap.

Relevant implementation and behavior context:
- `src/application/start-project.ts` detects `targetDirectoryHadFilesBeforeStart` and switches the initial migration seed.
- In non-empty directories, the seed includes:
  - `Research target documents and existing project materials`
  - `Create revision 0 target baseline from design/current/Target.md`
- Integration coverage exists in `__tests__/integration/start-project.integration.test.ts` (`"seeds first migration for existing directories with research and revision-0 baseline tasks"`).
- `start` also runs explore enrichment on both `design/current/Target.md` and `0001-initialize.md`, then creates an initial scaffold commit.

## Design and revision context

The canonical design workspace is `design/current/**`, with revision snapshots in `design/rev.*`. Revision commands rely on managed design paths and support legacy fallback (`docs/current/**`, root `Design.md`) only when canonical paths are unavailable.

For this migration, revision `0` is treated as the initial baseline snapshot for the target state. This must be explicitly captured so later migrations can be interpreted as deltas from a known starting point.

## Scope

In scope:
- Ensure first-migration seeding for existing directories is research-first and revision-baseline oriented.
- Ensure wording and structure in the seeded migration clearly guide document/material discovery and baseline creation.
- Ensure compatibility with current `start` bootstrap flow and explore integration.

Out of scope:
- Executing the migration tasks.
- Changing migrate execution semantics (`migrate up/down`, reconciliation).
- Redesigning docs/design command naming in this migration.

## Constraints and compatibility notes

- Preserve current behavior for empty directories (greenfield seed remains different).
- Keep compatibility with configurable workspace directories (`--design-dir`, `--specs-dir`, `--migrations-dir`) and persisted mapping in `.rundown/config.json`.
- Avoid assumptions that only `design/current/Target.md` exists when custom design dir is configured; migration wording should remain valid with configured directory mapping.
- Keep migration naming/numbering conventions compatible with parser expectations (`0001-*.md`).

## Acceptance criteria outline

- Starting in a non-empty target directory generates a first migration focused on:
  - researching existing target documents/materials;
  - defining revision-0 baseline target.
- The generated first migration remains parseable by existing migration tooling.
- Existing integration expectations for non-empty directory seeding remain satisfied.
- No regressions in `start` for empty directory bootstrap behavior.
- Resulting flow remains compatible with managed design context resolution and revision workflow.

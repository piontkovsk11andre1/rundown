Implement prefix test

## intent

Add focused, regression-safe test coverage for task prefix behavior so prefix parsing, intent classification, and execution routing remain stable as built-in tools and aliases evolve.

## current context

The codebase currently has two active prefix interpretation surfaces that both influence runtime behavior:

- `src/domain/task-intent.ts` classifies legacy and compatibility intents (`verify-only`, `memory-capture`, `fast-execution`, `parallel-group`, `tool-expansion`, default).
- `src/domain/prefix-chain.ts` parses unified tool chains (`modifier` + terminal `handler`) and supports composition boundaries such as `, ` and `; `.
- `src/domain/parser.ts` still provides directive-level inheritance patterns (`verify:`, `fast/raw/quick:`, `parallel:` and `profile=`) that affect child tasks before execution.
- `src/application/iteration-mode.ts` combines inherited directive intent, task-intent classification, and prefix-chain handler metadata to derive `onlyVerify` and `shouldVerify`.
- `src/application/task-execution-dispatch.ts` routes tool-handled chains first, then falls back to legacy branches (verify-only, memory-capture, inline CLI, default execution).

This means prefix regressions can appear in domain parsing, intent precedence, or application dispatch even when one layer appears correct in isolation.

## feature specification

Define and implement a prefix test suite that validates behavior at the right abstraction levels:

- Unit-level prefix parsing and normalization:
  - explicit prefix detection (`verify`, `memory`, `fast/raw/quick`, `parallel` and aliases),
  - payload trimming and empty payload handling,
  - mixed-case and spacing tolerance,
  - first-recognized-prefix precedence.
- Prefix-chain parsing behavior:
  - modifier/handler composition,
  - boundary splitting at `, ` and `; ` only when next segment is a known prefix,
  - no false splitting inside payload text,
  - legacy-invalid forms rejected where intended (for example `profile: <name>` task syntax).
- Integration-level execution semantics:
  - verify-only prefixes skip execution and run verification,
  - fast/raw/quick prefixes execute and suppress per-task verification,
  - memory prefixes persist capture output through configured writer,
  - tool-handler prefixes route through tool execution path without breaking fallback behavior.

## scope boundaries

### in scope

- Expanding or refining tests in:
  - `__tests__/domain/task-intent.test.ts`
  - `__tests__/domain/prefix-chain.test.ts`
  - relevant parser/iteration/dispatch integration tests where prefix behavior is asserted indirectly.
- Adding parity assertions for alias groups where behavior must remain identical.
- Capturing precedence and composition rules in tests as executable spec.

### out of scope

- Redesigning prefix grammar.
- Changing canonical prefix naming policy in prompts/docs.
- Refactoring execution pipeline architecture unless required to make tests deterministic.

## compatibility and edge cases

Test coverage should explicitly guard:

- Alias parity:
  - `verify|confirm|check`,
  - `memory|memorize|remember|inventory`,
  - `fast|raw|quick`,
  - `parallel|concurrent|par`.
- Prefix order interactions:
  - `verify: quick: ...` remains verify-only,
  - `quick: verify: ...` remains fast-execution semantics.
- Empty payload behavior:
  - prefix-only tasks with no payload are handled consistently and do not silently drift.
- Non-prefix false positives:
  - plain sentences containing words like “verify”, “memory”, “fast”, or “parallel” do not trigger prefix intent.
- Tool resolution interactions:
  - built-in explicit prefix precedence remains stable when tool resolver contains same-name tools.
- Directive inheritance interactions:
  - parser-derived intent inheritance still applies only where expected (task hierarchy and checkbox scope).

## module boundaries and test placement

- Domain parser/classifier tests should stay in `__tests__/domain/*` and validate pure logic without worker execution.
- Application behavior tests should validate `resolveIterationVerificationMode` and dispatch outcomes where prefix decisions change execution/verification flow.
- Built-in tool registry interactions can be verified via `resolveBuiltinTool` / `listBuiltinToolNames` usage to avoid brittle fixture setup.
- Integration tests should cover end-to-end observable behavior, not internal private helpers.

## acceptance criteria outline

- Prefix classification tests cover canonical forms, aliases, spacing/case normalization, precedence, and empty payloads.
- Prefix-chain tests cover composition delimiters, modifier + handler sequencing, and non-splitting payload text.
- Iteration/dispatch tests confirm effective execution mode (`onlyVerify`, `shouldVerify`) for representative prefixed tasks.
- Alias parity is enforced with shared assertions so behavior changes fail loudly across all aliases.
- Existing behavior for non-prefixed tasks remains unchanged and verified by regression tests.

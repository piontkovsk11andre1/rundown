Rename docs save -> docs release

## Goal

Align the docs revision lifecycle verb with release terminology by renaming the docs snapshot action from `save` to `release` across CLI surface, help text, docs, and tests, while preserving deterministic behavior and workspace-resolution semantics.

## Current context

The repository already separates docs revision lifecycle from migrate lifecycle:

- `rundown docs ...` is for revision lifecycle (`publish`, `diff`) in current docs/CLI.
- `rundown migrate ...` explicitly rejects old revision actions like `save`, `diff`, and `preview`.
- Design snapshots are immutable `design/rev.N/` directories, sourced from `design/current/` (with legacy fallback support for `docs/current/Design.md` and root `Design.md`).

Relevant code paths and contracts:

- `src/presentation/cli.ts` defines docs subcommands (`publish`, `diff`).
- `src/presentation/cli-command-actions.ts` routes docs subcommands and validates diff selectors.
- `src/application/docs-revision-task.ts` executes docs actions and delegates snapshot/diff operations.
- `src/application/design-context.ts` implements revision discovery, save logic, metadata sidecars, and diff context.
- `docs/cli.md`, `docs/examples.md`, `docs/overview.md`, and `README.md` contain user-facing command references and examples.
- Tests cover routing and integration behavior in `__tests__/presentation/cli-command-actions.test.ts` and `__tests__/integration/migrate-task.integration.test.ts`.

## Scope

### In scope

- Rename user-facing docs snapshot command to `docs release`.
- Replace `save` terminology in docs revision command UX where it refers to the command/action name.
- Keep snapshot mechanics unchanged (monotonic `rev.N`, immutability, no-op when unchanged, metadata sidecar behavior).
- Update help text, docs, examples, and test expectations to match renamed command.
- Ensure command-boundary guidance still clearly separates `docs` lifecycle from `migrate` lifecycle.

### Out of scope

- Changing diff selector semantics (`current`/`preview`, `--from/--to` constraints).
- Changing revision storage format, metadata schema, or directory naming.
- Expanding migrate actions or restoring removed `migrate save`.
- Altering workspace link resolution behavior.

## Desired command behavior

Primary intent:

- `rundown docs release` becomes the canonical command to snapshot `design/current/` into next immutable `design/rev.N/`.

Behavioral expectations (unchanged from existing snapshot flow):

- If `design/current/` matches latest revision byte-for-byte, command reports no-op and does not create a new revision.
- If no design draft workspace exists, command fails with guidance on creating canonical/legacy paths.
- Optional label metadata continues to be supported and written to `rev.N.meta.json`.
- `docs diff` behavior remains unchanged.

Compatibility consideration:

- Decide whether to provide a temporary compatibility alias (`docs publish` and/or `docs save`) or enforce hard rename.
- If alias is retained, output/help should steer users to `docs release` as preferred canonical verb.
- If alias is removed, error messages should be explicit and actionable.

## Integration touchpoints

CLI/API surface likely affected:

- Command declaration and description for docs snapshot subcommand in `src/presentation/cli.ts`.
- Action creator naming and routing in `src/presentation/cli-command-actions.ts` (currently publish-oriented naming).
- Action union/types in `src/application/docs-revision-task.ts` (currently `"publish" | "diff"`).

User-facing docs likely affected:

- `docs/cli.md` docs command section and synopsis blocks.
- `docs/examples.md` docs revision examples.
- `docs/overview.md` prediction workflow bullets.
- `README.md` command examples and recommended structure comments.

Test coverage likely affected:

- Routing tests that assert docs publish action payloads.
- Integration tests invoking `rundown docs publish`.
- Any golden/help text assertions that mention `publish` or old `save` wording.

## Design and compatibility constraints

- Preserve canonical-first design path strategy: `design/current/**` and `design/rev.*/**` remain first-class.
- Preserve legacy fallback readability for older project layouts.
- Preserve deterministic workspace resolution via `.rundown/workspace.link` and `--workspace`.
- Preserve immutable snapshot guarantees and conflict errors for pre-existing revision paths/metadata sidecars.
- Keep docs and CLI examples synchronized with actual command parser behavior to avoid drift.

## Edge cases to account for

- Workspaces configured with non-default design/migrations directories in `.rundown/config.json`.
- Linked workspace ambiguity requiring `--workspace`.
- Existing scripts or docs still invoking deprecated verbs.
- Case/format validation paths in CLI parsing and tests.
- No-change snapshot flow and empty draft warning messages.

## Acceptance criteria outline

- `rundown docs release` executes the existing snapshot workflow successfully.
- All official docs reference `docs release` as canonical snapshot command.
- CLI help/examples no longer present stale `save`/`publish` naming as primary command.
- Snapshot/diff semantics and outputs remain behaviorally consistent with current implementation.
- Tests pass with updated command naming and expected messaging.
- Migration/migrate command boundaries remain explicit: docs lifecycle under `docs`, migration lifecycle under `migrate`.

migrate: argument to output agents.md help to use rundown? - This means `rd --agents >> AGENTS.md` to inject rundown

## intent

Add a dedicated CLI help output mode that prints AGENTS.md-ready guidance for using rundown in a repository, so users can bootstrap agent instructions with a single command and redirection flow.

Primary UX target:

- `rd --agents >> AGENTS.md`
- equivalent canonical form: `rundown --agents >> AGENTS.md`

## current context

Relevant repository behavior and conventions already in place:

- Executable alias parity exists: `rd` is a first-class alias for `rundown` (migration `88` established this direction and codebase conventions assume parity).
- Project scaffolding currently creates `AGENTS.md` via `start`:
  - `src/application/start-project.ts` writes `AGENTS.md` using `DEFAULT_AGENTS_TEMPLATE`.
- CLI root command behavior is already special-cased for no-args and `--help`:
  - root `rundown` action in `src/presentation/cli.ts`
  - live-help/static-help fallback logic in `src/presentation/cli-command-actions.ts`
- Command alias rewriting in `src/presentation/cli-argv.ts` is for command tokens (`all`, `migrate down`), not root global-output flags.
- User-facing CLI docs currently mention `AGENTS.md` in scaffold output (`docs/cli.md`) but do not provide a direct “emit AGENTS guidance” mode.

Gap today: there is no explicit flag that outputs inject-ready AGENTS guidance text to stdout.

## feature specification

Introduce a global/root CLI option:

- `--agents` prints canonical AGENTS guidance for rundown usage to stdout and exits successfully.
- Output is intended for shell redirection/append workflows (`>> AGENTS.md`) and should be deterministic and plain-text/Markdown-safe.
- The output should be usable as a standalone AGENTS section (or full file body) without additional transformation.
- Behavior must be identical for both executables:
  - `rundown --agents`
  - `rd --agents`

Expected usage examples:

- create/overwrite:
  - `rd --agents > AGENTS.md`
- append:
  - `rd --agents >> AGENTS.md`
- inspect in terminal:
  - `rundown --agents`

## scope boundaries

### in scope

- Add root-level CLI option parsing and handling for `--agents`.
- Define and centralize a deterministic AGENTS output template.
- Ensure root option precedence and exit behavior are explicit and test-covered.
- Document the new mode in CLI docs/help examples, including redirection usage.
- Preserve alias parity (`rd` and `rundown`).

### out of scope

- Changing task execution semantics (`run`, `plan`, `migrate`, etc.).
- Replacing project-specific `AGENTS.md` authoring with fully dynamic repository introspection.
- Altering existing `start` scaffold behavior beyond optional template reuse alignment.
- Introducing interactive TUI behavior for `--agents`.

## integration touchpoints

Likely code surfaces:

  - wire action precedence relative to no-arg help behavior
- `src/presentation/cli-command-actions.ts`
  - root action logic should short-circuit for `--agents` and print content deterministically
- `src/presentation/cli-argv.ts`

- `src/application/start-project.ts`
  - optional alignment/reuse of AGENTS template source to avoid drift between scaffolded content and emitted content
- `docs/cli.md` (and optionally README/examples)
  - document `--agents` behavior and shell redirection examples

## behavior and precedence expectations

Root invocation precedence should remain deterministic:

- `--agents` is non-interactive and should not invoke live help worker flows.
- `--agents` should succeed in TTY and non-TTY contexts equally.
- `--help` remains static help behavior.
- If both `--agents` and `--help` are provided, precedence must be defined and tested (recommended: `--agents` takes precedence for explicit content emission, or fail with clear guidance; choose one and keep deterministic).
- `rundown <subcommand> ... --agents` should be rejected unless intentionally supported as global passthrough; preferred behavior is root-only semantics with clear error/help guidance if misused.

## output contract

AGENTS output should follow stable formatting rules:

- Plain UTF-8 text, newline-terminated.
- Markdown-safe content intended for direct file append/write.
- No ANSI color codes, spinner output, or extra status lines when `--agents` is used.
- Deterministic section ordering to keep diffs stable and script-friendly.
- Versioning strategy should be explicit:
  - either stable evergreen text
  - or optionally include version marker in content (if included, do so intentionally and test it)

## compatibility and edge cases

- Existing repositories with custom `AGENTS.md`:
  - append workflow should not destroy existing content.
- Repeated appends:
  - duplicate content risk should be documented (command is intentionally append-friendly but not dedupe-aware).
- Cross-shell behavior:
  - examples should work for POSIX and PowerShell equivalents where possible.
- CI/non-interactive environments:
  - command must not require worker configuration or TTY.
- Alias parity:
  - `rd --agents` and `rundown --agents` must produce byte-identical output and exit codes.

## testing implications

Coverage should include:

- CLI unit/integration tests for:
  - `rundown --agents` exits `0`
  - `rd --agents` parity
  - no worker invocation side effects
  - no interactive help fallback path activation
- Snapshot/golden test for emitted AGENTS content (or equivalent exact-string assertion).
- Precedence tests for flag combinations (`--agents`, `--help`, invalid mixes).
- Regression checks that existing root behaviors (`rundown`, `rundown --help`) remain unchanged.

## acceptance criteria outline

- `--agents` is available as a root/global invocation path and emits AGENTS guidance to stdout.
- Output is deterministic, Markdown-safe, and suitable for `> AGENTS.md` or `>> AGENTS.md`.
- `rd --agents` and `rundown --agents` are behaviorally identical.
- No worker/TUI dependency exists for `--agents`.
- CLI docs describe the feature with direct injection examples.
- Existing root help/no-arg behaviors remain stable outside the new flag path.

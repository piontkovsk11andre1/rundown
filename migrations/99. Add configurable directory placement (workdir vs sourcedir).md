migrate: Add configurable directory placement (workdir vs sourcedir)

## intent

Allow prediction workspace buckets (`design`, `migrations`, `specs`) to be placed either in the effective source workspace directory or in the invocation/work directory, with deterministic defaults and no regression for existing projects.

The request is about placement strategy, not only directory names. Directory-name overrides already exist; this migration adds directory-root selection.

## current context

The codebase already supports configurable bucket names (migration 82), linked-workspace resolution (migrations 67-69), and runtime workspace context variables:

- Bucket names are read from `.rundown/config.json` via `resolvePredictionWorkspaceDirectories(...)` in `src/application/prediction-workspace-paths.ts`.
- Path-sensitive commands resolve effective workspace root through `.rundown/workspace.link` + `--workspace` selection.
- Runtime prompt context already exposes both directories (`invocationDir`, `workspaceDir`) and linked-state metadata in `src/application/runtime-workspace-context.ts`.
- CLI/docs already document custom bucket names (`--design-dir`, `--specs-dir`, `--migrations-dir`) and linked workspace behavior.

Observed gap:

- Bucket path resolution currently assumes one root (`workspaceRoot` / source workspace) and does not let users choose placement root per bucket.
- In linked mode, invocation directory and source workspace directory can diverge, but bucket placement cannot currently be configured to follow invocation/workdir.

## terminology contract

Define stable placement terms used by config, docs, and prompt context:

- `sourcedir`: effective workspace/source directory resolved for command execution (today's default behavior).
- `workdir`: invocation/working directory where command was launched (or command-local working root when explicitly set).

For linked workspaces, these may differ:

- `sourcedir` -> resolved linked workspace root.
- `workdir` -> invocation directory.

For non-linked workspaces, they are equal.

## feature specification

Add placement configuration for each prediction bucket:

- `design` placement: `sourcedir` or `workdir`
- `migrations` placement: `sourcedir` or `workdir`
- `specs` placement: `sourcedir` or `workdir`

Configuration should be persisted in project config and consumed by all path-sensitive flows that currently depend on bucket directories.

Recommended config shape extension (conceptual):

- Keep existing `workspace.directories.<bucket>` for relative directory names.
- Add `workspace.placement.<bucket>` with enum values `sourcedir | workdir`.
- Default placement remains `sourcedir` for backward compatibility.

Effective path rule for each bucket:

- Resolve selected root (`sourcedir` or `workdir`).
- Join selected root + configured bucket directory relative path.
- Enforce traversal safety and conflict checks under the selected root.

## scope

In scope:

- Placement config model and validation for design/specs/migrations buckets.
- Runtime path resolver updates so commands can choose bucket root by placement.
- Propagation across all prediction flows that consume workspace bucket paths.
- Prompt/template context additions so workers can see placement decisions and effective paths.
- CLI/docs/test updates for discoverability and behavior guarantees.

Out of scope:

- Redesigning linked-workspace schema format.
- Renaming docs/design command family in this migration.
- New migration lifecycle semantics unrelated to directory placement.

## behavior constraints

- Backward compatible by default: existing repositories with no placement config behave exactly as today (`sourcedir`).
- Placement values must be explicit enum values; invalid values fail fast with actionable config/CLI errors.
- Path safety rules remain strict:
  - bucket directory values remain relative
  - no project-root escape
  - no duplicate/overlapping bucket targets under the same selected root
- Linked workspace resolution remains deterministic:
  - placement uses already-resolved `invocationDir`/`workspaceDir`
  - broken/stale workspace links must not produce misleading placement paths.

## edge cases and compatibility notes

- Mixed placement is valid (for example: `design` in `sourcedir`, `specs` in `workdir`).
- In linked mode, `workdir` placement can intentionally route writes/reads outside source workspace; docs/help must call this out clearly.
- If two buckets choose different roots but resolve to the same absolute path, treat as conflict and fail early.
- Legacy fallback reading for managed design docs (`design/current/**`, then legacy fallbacks) should still apply within the resolved design bucket location.
- Prompt/runtime context should remain authoritative: user `--var` values must not override system-computed placement/path fields.

## documentation and template implications

- Update CLI docs with placement semantics, defaults, and linked-workspace examples.
- Update template docs to expose any new placement-aware variables (for example, per-bucket placement mode and effective absolute paths when useful).
- Keep guidance explicit on `workdir` vs `sourcedir` meaning to avoid ambiguity for agents and users.

## acceptance criteria outline

- Project config supports per-bucket placement (`sourcedir` / `workdir`) with deterministic defaults.
- Effective path resolution for design/specs/migrations honors both configured directory names and placement root.
- `migrate`, `docs`, `test`, and related flows resolve bucket paths consistently using the new placement contract.
- Linked-workspace runs behave deterministically and surface accurate placement-derived paths in prompts/context.
- Existing projects without placement config remain fully compatible.
- CLI/docs/tests are updated to reflect placement behavior, validation, and examples.

## notes for planning handoff

Prioritize a single authoritative resolver that takes:

- invocation directory,
- resolved workspace/source directory,
- bucket directory names,
- bucket placement modes,

and returns effective absolute bucket paths for all consumers. This avoids per-command drift and keeps linked-workspace behavior consistent.

Treat placement as additive configuration layered on top of existing directory-name configuration, not a replacement for it.

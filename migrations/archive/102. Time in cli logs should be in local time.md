time in cli logs should be in local time

## context

the current cli timestamp behavior is explicitly utc-oriented in both code and docs.

- `src/domain/cli-timestamp.ts` defines `CLI_TIMESTAMP_FORMAT = "UTC ISO-8601"` and `formatCliTimestamp()` normalizes values with `toISOString()`.
- `src/presentation/output-port.ts` prepends bracketed timestamps to command-level terminal lines via `withTimestamp(...)` and `formatCliTimestamp(...)`.
- `src/application/log-runs.ts` uses `formatCliTimestamp(...)` for the absolute timestamp token shown by `rundown log`.
- docs currently state utc semantics for terminal timestamps (`docs/cli.md`, terminal timestamps + global output log sections).

this means operators always see `...Z` timestamps even when working in a non-utc locale.

## affected surfaces

local-time behavior is relevant to user-facing cli output surfaces that are read by humans in terminal sessions.

primary surfaces:

- command-level terminal lines emitted by `cliOutputPort` for `info`, `warn`, `error`, `success`, `progress`, `group-start`, and `group-end`.
- compact absolute timestamp token shown by `rundown log` text output.

non-goals / likely unchanged surfaces (unless explicitly expanded later):

- global jsonl log timestamps (`<config-dir>/logs/output.jsonl` field `ts`) currently machine-oriented and documented as iso utc.
- artifact metadata timestamps (`startedAt`, `completedAt`) and trace records, which are persisted interoperability fields.
- plain `text` / `stderr` passthrough lines in terminal output that intentionally do not get prefixed timestamps.

## existing constraints from codebase

- timestamp formatting is centralized for terminal/log-runs display through `formatCliTimestamp(...)`, so changing that function can fan out to multiple presentation paths.
- invalid timestamp inputs currently fall back to original string; this behavior is used by `log-runs` tests and should remain deterministic.
- rendering tests strip ansi and often assert exact output line content, including timestamp token structure.
- group rendering in `output-port` supports tty and non-tty modes; timestamp injection must preserve grouping prefixes and line stability.

## design considerations for local time

local time must be human-readable and stable enough for test assertions while reflecting host locale offset.

key decisions to capture before implementation planning:

1. display format contract for local timestamps.
   - option a: local iso-like with numeric offset (example: `2026-04-14T01:26:01.557-07:00`).
   - option b: locale-formatted text (example: `4/14/2026, 1:26:01 AM`).
   - option c: keep iso structure but remove `Z` by converting to local with offset.
   - deterministic ci testing strongly favors a fixed structural format (a/c) over locale-dependent text (b).

2. scope boundary between human output and persisted machine logs.
   - user intent says "cli logs"; this should be interpreted carefully so machine-consumed jsonl and artifact records do not silently change unless explicitly desired.
   - if local-time applies only to terminal-facing display, correlation guidance in docs should explain that persisted timestamps may remain utc.

3. timezone offset edge cases.
   - daylight saving transitions (offset change over time).
   - non-hour offsets (`+05:30`, `+09:30`, etc.).
   - test stability across environments with different `TZ`.

## compatibility and migration notes

- this change conflicts with current docs language that says terminal timestamps are utc iso-8601.
- existing tests in:
  - `__tests__/domain/cli-timestamp.test.ts`
  - `__tests__/presentation/output-port.test.ts`
  - `__tests__/application/log-runs.test.ts`
  assert utc-oriented values and will need expectation updates aligned with the final local-time format contract.
- integration expectations in `docs/cli.md` around "terminal timestamps" and possibly "global output log" need explicit differentiation between display time and persisted time.

## acceptance criteria outline

1. user-facing cli timestamped lines render in local time instead of utc where timestamps are currently prepended.
2. `rundown log` text mode absolute timestamp token is local-time formatted using the same canonical display contract.
3. invalid/unparseable timestamp strings remain pass-through (no crash, no mutation).
4. persisted structured logs and artifacts remain unchanged unless this task explicitly expands scope.
5. documentation reflects the new local-time behavior and clarifies utc retention (if retained) for machine-oriented records.
6. test suite updates cover timezone-offset rendering and keep deterministic assertions.

## out of scope

- adding new command flags to switch between utc/local display modes.
- changing artifact id generation, run ordering semantics, or relative-time calculations.
- altering worker transcript payload formatting for `text`/`stderr` streams.

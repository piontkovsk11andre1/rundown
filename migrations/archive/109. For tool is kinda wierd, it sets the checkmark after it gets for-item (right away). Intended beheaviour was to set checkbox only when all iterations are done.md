migrate: For tool is kinda wierd, it sets the checkmark after it gets for-item (right away). Intended beheaviour was to set checkbox only when all iterations are done.

## Intent

Keep `for:` / `each:` / `foreach:` as loop-control tasks where the parent checkbox is only evidence of full loop completion, not loop initialization.

Expected lifecycle:

- loop parent starts unchecked
- `for-item:` metadata can be materialized early
- `for-current:` points to active item during iteration
- child checkbox tasks run per active item
- child checkboxes reset between items
- parent checkbox flips to checked only after the final item is completed

## Problem statement (expanded)

Current report: parent gets checked right after items are baked (`for-item`) instead of waiting for all iterations.

This is a contract break against the core runtime rule documented in `docs/overview.md`: checkbox completion should happen only after successful work completion. For loop tasks, “successful work” means all per-item child passes are done, not just item discovery.

## Current architecture and control flow

Relevant files and responsibilities:

- `src/domain/builtin-tools/for-loop.ts`
  - resolves loop items (metadata first, otherwise research worker)
  - returns `forLoopItems` and `skipExecution: true`
- `src/application/task-execution-dispatch.ts`
  - executes tool chain
  - for `for` handler, syncs metadata and calls `advanceForLoopUsingFileSystem(...)`
  - emits `forLoopAdvanced` / `forLoopCompleted` into completion stage
- `src/application/complete-task-iteration.ts`
  - loop-specific gate before generic checkbox write
  - if advanced: returns early (`continueLoop: true`) without calling `checkTaskUsingFileSystem`
  - if completed: falls through to `checkTaskUsingFileSystem`
- `src/application/checkbox-operations.ts`
  - `advanceForLoopUsingFileSystem(...)` owns cursor transitions and per-item child reset
  - `rewriteForLoopMetadataLines(...)` rewrites `for-item` and `for-current` lines
- `src/domain/task-selection.ts`
  - special-case for loop parents with no `for-current` to allow loop parent to run as orchestrator

## Existing safeguards already in code

There is already explicit logic intended to prevent early parent completion:

- loop-advanced path in `complete-task-iteration.ts` returns before `checkTaskUsingFileSystem(...)`
- parent completion should happen only when transition state is `completed`
- child reset and `for-current` updates are handled atomically in the same serialized mutation flow (`withSerializedFileMutation`)

So this migration should treat the bug as either:

- a path where loop signals (`forLoopAdvanced` / `forLoopCompleted`) are missing or dropped
- a stale task/state mismatch causing loop branch to be skipped
- a fallback path that reaches generic `checkTaskUsingFileSystem(...)` too early

## Likely risk points to investigate

- signal handoff mismatch between `dispatchTaskExecution(...)` and `completeTaskIteration(...)`
- mismatch between task text alias normalization and `isForLoopTaskText(...)` checks
- edge case where no child tasks are detected at runtime and flow falls into non-loop completion behavior
- metadata rewrite side effects that remove/lose `for-current` unexpectedly
- resume/interruption state where loop task identity relocation fails and wrong branch runs
- interactions with `skipExecution` + `autoComplete` frontmatter expectations (even if `autoComplete` is not currently active in application flow)

## Repro/diagnostic guidance

Minimal repro shape:

```md
- [ ] for: All controllers
  - [ ] Do this
  - [ ] Do that
```

Observe after first run pass:

- parent must remain `- [ ]`
- `for-item:` lines may appear
- `for-current:` should be set to first item
- child tasks should represent active-item execution state

Parent `- [x]` is valid only when:

- no remaining items
- no active `for-current`
- all intended item passes were executed

## Compatibility and invariants

Must keep these invariants stable:

- no fenced-code mutations
- no metadata line promotion into executable tasks
- no out-of-scope checkbox toggles
- aliases `for:` / `each:` / `foreach:` stay behaviorally equivalent
- run/retry/repair semantics stay consistent with non-loop tasks where applicable

## Test coverage status and gap

Current tests already exist for loop metadata and transition primitives (`src/__tests__/application/checkbox-operations.test.ts`, `src/__tests__/domain/for-loop.test.ts`), but this bug suggests an integration-level timing gap across:

- tool execution dispatch
- completion decision
- final checkbox mutation

This migration should lock parent-check timing with end-to-end run-path assertions, not only helper-level transition tests.

## Acceptance criteria outline

Feature is correct when:

- baking `for-item:` never checks parent by itself
- parent stays unchecked while there are pending loop items
- parent checks exactly once, after final item completion
- `for-current:` accurately tracks active item during progress and is finalized deterministically
- interruption/resume cannot cause premature parent check
- alias forms remain identical in behavior and persisted metadata format

## Scope boundary

In scope:

- parent checkbox timing for for-loop tasks

- tests that prevent regression of “checked too early” behavior

Out of scope:

- redesign of loop syntax
- unrelated checkbox engine changes outside loop completion timing
- non-loop task selection semantics except where shared logic directly causes this bug

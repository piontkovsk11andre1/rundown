migrate: Planner must use memory (remember:) if task says something about 'doing research for next tasks' only and doesn't specify target file to write. If intention is to save document in filesystem or edit it -- it's not a memory. Memory = temporary result needed for next tasks, only for target md file.

## Intent

Define a stricter planning rule for research-only TODO generation:

- when a planned task is about gathering context for later steps (research, inventory, notes, constraints, references) and does not require editing/writing a target file, planner should express it as memory capture intent;
- when a task explicitly says to write/update/create content in the filesystem, planner must keep it as normal execution (not memory).

This migration is about planning semantics (how TODOs are authored), not about changing run-time memory execution mechanics.

## Why this migration is needed

Current planning guidance already mentions `memory:` as an available prefix, but it is optional and broad. In practice, research-oriented tasks are often emitted as plain `- [ ]` execution items, which causes two problems:

- the intent is ambiguous (is this research temporary context, or a document edit task?);
- results that should be source-local reusable context are not consistently captured in memory artifacts.

The requested behavior clarifies intent: research-for-next-steps without a file write target should be represented as memory capture.

## Existing implementation context

### Runtime intent classification already supports memory capture

`src/domain/task-intent.ts` classifies task text with `memory|memorize|remember|inventory:` as `memory-capture`.  
Aliases are supported at execution time, and empty payload is rejected by `run-task-iteration` with a clear error.

### Memory handling is source-local and designed for reusable context

`src/domain/builtin-tools/memory.ts` executes the worker and persists output through the memory writer.  
Memory artifacts are source-scoped (source markdown file + `.rundown` memory files/index), matching this migration’s requirement that memory is temporary context for the same target markdown workflow.

### Planner prompt rules currently prefer canonical prefix names

`src/domain/defaults.ts` plan/deep-plan templates already state:

- use `memory:` for tasks that should capture reusable context;
- normalize aliases to canonical names, and do not use alias forms in authored TODO text.

That means this migration should target planner decision heuristics, while keeping emitted prefix canonical as `memory:` (even if user wording mentions `remember:`).

## Required planning behavior

### Classification rule for planner-generated tasks

Planner should treat a candidate TODO as memory-capture when all are true:

- task objective is research/context capture for later tasks (examples: gather constraints, collect references, inventory unknowns, capture findings);
- task outcome is temporary reusable context, not a committed source-file change;
- no explicit target file to create/edit is specified in the task itself.

Planner should treat as normal execution when any are true:

- task explicitly asks to write/edit/create/update a file;
- task output is a document artifact intended to remain in repository/workspace;
- task implies code/config/docs modification as primary deliverable.

### Prefix form requirement

For planner output, canonical prefix should be `memory:`.  
`remember:`/`memorize:`/`inventory:` remain execution-compatible aliases but should not be newly authored by planner unless a separate migration changes canonical naming policy.

### Decision boundary (important)

“Do research for next tasks” is memory.  
“Research and write findings into `X.md`” is not memory.

This is the core boundary to enforce in prompt guidance and acceptance tests.

## Prompt/template integration notes

Primary integration surface is plan prompt guidance in `src/domain/defaults.ts`:

- `DEFAULT_PLAN_TEMPLATE`
- `DEFAULT_DEEP_PLAN_TEMPLATE`

Both templates need stronger, explicit heuristics so the model consistently chooses `memory:` for research-only/no-target-file tasks, and avoids `memory:` for write/edit tasks.

No change is required to research command templates for this migration unless they include planning-style TODO synthesis behavior (they currently do not).

## Compatibility and safety constraints

- Keep existing add-only planner safety behavior unchanged (`validatePlanEdit` and related plan flow protections).
- Do not alter checkbox state rules or insertion ordering behavior.
- Do not change memory runtime persistence contracts.
- Preserve existing alias parsing at execution level (`remember:`, etc.) for backward compatibility.
- Keep canonical planner output guidance aligned with current “use canonical prefix names” rule.

## Edge cases to specify explicitly

- Research phrasing with implicit write target (“prepare notes section in this doc”) should be treated as file edit, not memory.
- Tasks that mention “collect data” plus a later explicit “save to file” in same line should be non-memory.
- Parent directive + child tasks: if a parent implies memory capture intent, child tasks still must be checked for explicit file-write language before inheriting/keeping memory prefix.
- Mixed tasks should split semantically during planning when possible (capture context via memory, then separate write/edit task), while preserving add-only/no-rewrite constraints on existing items.

## Acceptance criteria outline

- Planner guidance explicitly states the memory-vs-write decision boundary.
- For research-only/no-target-file work, planner outputs `memory:`-prefixed TODOs.
- For tasks with explicit filesystem edit/write intent, planner outputs non-memory execution TODOs.
- Planner output continues to use canonical prefix names.
- Existing runtime handling for memory aliases remains compatible and unchanged.
- Regression tests cover heuristic examples for both plan and deep-plan prompt-driven behavior.

## Goal

Align migration and revision naming with one clear, canonical vocabulary and layout so command names, directories, and generated artifacts all use the same terms.

## Canonical naming target

Use this structure consistently:

```text
design/rev.0/Target.md             # Last released revision (rd design release)
design/rev.1/Target.md
design/current/Target.md           # Current design document

migrations/1. Initialization.md    # First migration
migrations/2. Add Feature.md       # Feature migration (relative to released design)
migrations/2.1 Snapshot.md         # Predicted state up to this migration
migrations/2.2 Backlog.md          # Deferred follow-up work
migrations/2.3 Review.md           # Prediction vs design comparison

specs/feature-tests.md
specs/end-to-end-tests.md
specs/ux-tests.md
```

## Naming rules

- Use `design` as the canonical command and directory vocabulary for revision lifecycle.
- Keep `migrations` for change steps applied to design state.
- Keep `specs` for test-like TODO documents (`test:` prefix).
- Preserve numbered migration filenames (`N. Title.md`) and satellite naming (`N.1`, `N.2`, `N.3`).
- Prefer canonical wording in help text, docs, and generated output; legacy aliases may remain only as compatibility shims.

## Scope

In scope:
- Normalize command/docs wording to canonical names.
- Ensure generated paths and examples match canonical structure.
- Verify migration numbering and satellite naming remain deterministic.

Out of scope:
- Behavioral changes to release/prediction algorithms.
- Rewriting historical migration files unless required for compatibility.

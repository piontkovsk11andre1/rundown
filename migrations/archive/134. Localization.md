# Plan: Rundown Localization Feature

## TL;DR

Make rundown 100% translatable via two new surfaces: `rundown init --language "Chinese"` (sets language at init time) and a standalone `rundown localize --language "Chinese"` (re-runnable). When triggered, an AI worker translates all 35 `.rundown/` template files and generates localized intent keyword aliases (e.g., `记忆:` → `memory:`). Locale data is stored in a new `.rundown/locale.json`. Intent classification gains a pre-normalization step that maps locale keywords to canonical English before routing.

CLI output messages are out of scope for v1 — documented as a follow-up that requires a message-ID catalog.

---

## Phase 1: Locale Storage Schema

**Goal:** Define where locale data lives and ensure it loads at startup.

**Steps:**
1. Define `.rundown/locale.json` schema:
   ```json
   { "language": "Chinese", "aliases": { "记忆:": "memory:", "验证:": "verify:", "快速:": "fast:" } }
   ```
2. Create `src/infrastructure/adapters/locale-adapter.ts` — reads/writes `locale.json` from config dir; returns a `LocaleConfig | null`.
3. Add `LocaleConfig` interface to `src/domain/ports/` (or inline in adapter) with `language: string` and `aliases: Record<string, string>`.
4. Load `locale.json` in `create-app.ts` (or wherever config is assembled) and pass `localeAliases` through app dependencies.

**Files:**
- `src/infrastructure/adapters/locale-adapter.ts` — NEW
- `src/domain/ports/` — new `LocaleConfig` type (or inline)
- `src/application/create-app.ts` — load locale at startup

---

## Phase 2: Translation Prompt Templates

**Goal:** Two new prompt templates in `defaults.ts`.

**Steps:**
1. Add `DEFAULT_LOCALIZE_PROMPT_TEMPLATE` — translates a single `.rundown/*.md` prompt file:
   - Inputs: `{{language}}`, `{{content}}`
   - Rules: preserve `{{double-brace vars}}`, `OK`/`NOT_OK:`, `RESOLVED:`/`UNRESOLVED:`, `- [ ]`/`- [x]`, all code blocks
   - Output: only the translated prompt content, no commentary
2. Add `DEFAULT_LOCALIZE_ALIASES_TEMPLATE` — generates locale intent aliases:
   - Inputs: `{{language}}`
   - Lists all canonical keywords: `memory:` (+ memorize:, remember:, inventory:), `verify:` (+ confirm:, check:), `fast:` (+ raw:, quick:), `parallel:` (+ concurrent:, par:)
   - Output: valid JSON object only — `{ "localeKeyword:": "canonical:" }`

**Files:**
- `src/domain/defaults.ts` — add `DEFAULT_LOCALIZE_PROMPT_TEMPLATE` and `DEFAULT_LOCALIZE_ALIASES_TEMPLATE`

---

## Phase 3: `localizeProject` Application Use Case

**Goal:** New `src/application/localize-project.ts` that orchestrates the full translation.

**Steps:**
1. Accept `{ language: string }` as options plus standard dependencies (fileSystem, workerExecutor, configDir, output, templateLoader).
2. Load all 35 template files from `.rundown/` (the ones `init` writes). For each:
   a. Read current content (from disk if exists, else from default constant)
   b. Build a prompt using `DEFAULT_LOCALIZE_PROMPT_TEMPLATE` with `{{language}}` + `{{content}}`
   c. Run AI worker in `wait` mode — stdout is the translated template
   d. Write result back to `.rundown/<name>.md` (overwrite)
   e. Emit progress: `success "Translated <name>.md"`
3. Run AI worker once with `DEFAULT_LOCALIZE_ALIASES_TEMPLATE` — stdout is a JSON object
4. Parse JSON, write to `.rundown/locale.json` with `{ language, aliases }`
5. If worker not configured, emit error + return non-zero early (fail gracefully).
6. Export `LocalizeProjectDependencies` and `LocalizeProjectOptions` types.
7. Export `createLocalizeProject(deps)` factory returning async executor.

**Files:**
- `src/application/localize-project.ts` — NEW
- *Depends on Phase 1 (locale.json schema) and Phase 2 (templates)*

## Phase 4: CLI Wiring

**Goal:** Surface the feature as `init --language` and `localize` command.

**Steps:**
1. **`localize` command** in `src/presentation/cli.ts`:
   - `program.command("localize")` with `--language <lang>` (required-ish, prompt if missing?), standard options (`--worker`, `--dry-run`, `--print-prompt`, `--verbose`, `--quiet`)
   - Delegates to `createLocalizeCommandAction`
2. Add `createLocalizeCommandAction` in `src/presentation/cli-command-actions.ts` — resolves options, calls `getApp().localizeProject({ language })`
3. **`init --language <lang>`** option in `src/presentation/cli.ts` init command.
4. In `init` action handler (`cli-command-actions.ts`), after `app.initProject()` succeeds, if `--language` passed, also call `app.localizeProject({ language })`.
5. Expose `localizeProject` on the app interface (`src/application/create-app.ts`).

**Files:**
- `src/presentation/cli.ts` — add `localize` command + `--language` option on `init`
- `src/presentation/cli-command-actions.ts` — `createLocalizeCommandAction` + init handler update
- `src/application/create-app.ts` — expose `localizeProject` on app
- *Parallel with Phase 3*

## Phase 5: Intent Keyword Normalization

**Goal:** Allow users to write `记忆: do something` in task files and have it work like `memory: do something`.

**Steps:**
1. Add `normalizeLocaleKeyword(text: string, aliases: Record<string, string>): string` in `src/domain/task-intent.ts` (or a new `src/domain/locale-normalize.ts`):
   - Checks if task text starts with any alias key (case-insensitive prefix match)
   - Replaces the matched alias prefix with the canonical English one
   - Returns normalized text (or original if no match)
2. In `src/application/iteration-mode.ts`, in `resolveIterationVerificationMode`:
   - Accept `localeAliases?: Record<string, string>` in the options
   - Before calling `classifyTaskIntent`, call `normalizeLocaleKeyword(task.text, localeAliases)`
   - Pass normalized text to `classifyTaskIntent`
3. Thread `localeAliases` from app dependencies through `run-task-iteration.ts` → `resolveIterationVerificationMode`.

**Files:**
- `src/domain/task-intent.ts` — add `normalizeLocaleKeyword` function
- `src/application/iteration-mode.ts` — normalize before classifyTaskIntent + accept localeAliases
- `src/application/run-task-iteration.ts` — thread localeAliases in deps
- `src/application/run-task-execution.ts` — include localeAliases in deps assembly

---

## Out of Scope (v1)

**CLI output message translation** — requires:
- Extracting ~35+ hardcoded English strings from emit() calls across 15+ files into a message-ID map
- Adding a `messages` section to `locale.json`
- A `getMessage(id: string, fallback: string)` helper that checks locale first
- Touching every `emit({ kind: "success", message: "..." })` site

Recommended as a follow-up migration once Phase 1–5 are proven.

2. Create a task `- [ ] 记忆: capture this context` → run `rundown run` → verify it resolves as `memory-capture` intent
3. Run `rundown localize --language "Kawaii~~~~"` → verify prompts have kawaii style
4. Run `rundown localize` on a project with no worker configured → verify error message + non-zero exit
5. Unit test `normalizeLocaleKeyword` with a sample alias map
6. Integration test: `classifyTaskIntent` on a locale-normalized text returns correct intent

## Key Decisions

- **locale.json not config.json** — kept separate to avoid polluting worker config schema
- **Overwrite on re-localize** — `localize` command always overwrites; `init --language` also overwrites since the user is explicitly asking for it
- **English protocol tokens preserved** — `OK`, `NOT_OK:`, `RESOLVED:`, `UNRESOLVED:`, `{{vars}}`, checkboxes never touched by translation AI
- **Style works as language** — "Kawaii~~~~" is valid as `language` parameter; prompt treats it as style spec
- **CLI messages out of scope v1**

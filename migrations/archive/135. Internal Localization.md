# Internal Localization — Message Catalog and CLI Output Translation

## Goal

Extract every hardcoded user-facing string from all `emit()` call sites across the application into a typed message catalog (`src/domain/messages.ts`), introduce a `msg()` helper that performs locale lookup with English fallback, enhance `locale.json` with a `messages` block, and wire the translated messages through `localize-project.ts` so that all CLI output — not just AI prompt files — can be expressed in any target language.

This is the follow-up to migration 134 (Localization). Phases 1–5 of that migration cover AI prompt template translation and intent alias normalization. This migration covers Phase 6: the CLI message layer.

## Background

After migration 134, `locale.json` stores `language` and `aliases`. AI prompt templates in `.rundown/` can be translated. But all `emit({ kind: "success"|"info"|"warn"|"error", message: "..." })` calls throughout the application still produce English output, because those strings are inline literals scattered across ~15 source files with no extraction layer.

This migration adds that layer in three moves: extract → catalog → translate.

## Architecture

### `src/domain/messages.ts` — typed message catalog

A single source-of-truth file exports:

```ts
export const MESSAGES = {
  "init.created-tools-dir": "Created {{configDir}}/tools/",
  "init.skip-existing":     "{{filePath}} already exists, skipping.",
  // ...
} as const;

export type MessageId = keyof typeof MESSAGES;
```

Each key is a stable dot-namespaced ID. Values are English template strings using `{{var}}` placeholders (same convention already used in AI prompt templates). IDs are never changed once published — adding IDs is the only allowed mutation.

### `src/domain/locale.ts` — `msg()` helper

```ts
export function msg(
  id: MessageId,
  vars: Record<string, string> = {},
  localeMessages: Record<string, string> = {}
): string {
  const template = localeMessages[id] ?? MESSAGES[id];
  return template.replace(/\{\{(\w+)\}\}/g, (_, k) => vars[k] ?? `{{${k}}}`);
}
```

- Locale lookup first; English fallback if key absent.
- `{{var}}` substitution is identical for both branches.
- No runtime exceptions for missing locale IDs — English fallback is always safe.

### `locale.json` schema enhancement

```jsonc
{
  "language": "Chinese",
  "aliases": { "记忆:": "memory:", "验证:": "verify:" },
  "messages": {
    "init.created": "已创建 {{filePath}}",
    "init.success": "已初始化 {{configDir}}/ 默认模板。",
    "run.no-tasks-found": "未找到未完成的任务。"
  }
}
```

`messages` is optional. If absent, all output falls back to English.

### Translation call in `localizeProject`

After translating `.rundown/*.md` files and generating `aliases`, a third AI call runs:

> *Here is a message catalog where each entry is `"id": "English text with {{placeholders}}"`. Translate all values to `{{language}}`. Keep `{{placeholder}}` tokens verbatim. Output valid JSON only, same keys, translated values.*

The stdout JSON is parsed and written as the `messages` block in `locale.json`.

### NEW: `src/domain/locale.ts`

The `msg()` helper. Accepts message ID, variable map, and optional locale messages map.

- Import `MESSAGES` and `MessageId` from `./messages`.
- Export `msg(id: MessageId, vars?: Record<string, string>, localeMessages?: Record<string, string>): string`.
- Locale lookup: `localeMessages[id] ?? MESSAGES[id]`.
- `{{var}}` substitution regex: `/\{\{(\w+)\}\}/g`.
- No exceptions: if `MESSAGES[id]` is somehow missing (type-safe so impossible), return `id` as last resort.
- Export `LocaleMessages = Record<string, string>` type alias.

### NEW: `DEFAULT_LOCALIZE_MESSAGES_TEMPLATE` in `src/domain/defaults.ts`

Inputs: `{{language}}`, `{{catalog}}` (full JSON of `MESSAGES`).

Rules to embed in the prompt:
- Translate only the values (right-hand side); keys are code identifiers and must not change.
- `{{placeholder}}` tokens inside values must be preserved verbatim (they are substituted at runtime).
- Do not add, remove, or rename any keys.
- Output must be valid JSON only — no markdown, no code fences, no commentary.
- If a value has no natural translation analog in `{{language}}`, keep the English value rather than inventing something wrong.

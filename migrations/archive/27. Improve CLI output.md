# Improve CLI Output

## Background & Current State

### Output Architecture

All application output flows through a single `ApplicationOutputPort` interface defined in `src/domain/ports/output-port.ts`. Events are discriminated unions:

```
info | warn | error | success | progress | task | text | stderr
```

The presentation layer implements the port in two files:

- **`src/presentation/output-port.ts`** — `cliOutputPort`: renders events to the terminal with color and structure. This is the single render site for all 16 CLI commands.
- **`src/presentation/logged-output-port.ts`** — wraps `cliOutputPort` and mirrors every event to a JSONL persistence log via `GlobalOutputEntryWriter`. Log writes are best-effort (errors swallowed). Structured logs must stay ANSI-free — stripping happens in `src/domain/global-output-log.ts`.

Flow:

```
Application use case → output.emit(event)
    → createLoggedOutputPort (wrapper)
        ├→ GlobalOutputEntryWriter (JSONL persistence, ANSI-free)
        └→ cliOutputPort (terminal rendering, color)
```

### Color / Styling

`picocolors` (^1.1.1) is the sole color dependency. Used in 4 source files:

- `src/presentation/output-port.ts` — event rendering (cyan, yellow, red, green, blue, dim, bold)
- `src/presentation/intro.ts` — animation slides
- `src/presentation/cli-invocation-log.ts` — invocation logging
- `src/application/log-runs.ts` — log command formatting

### Current Event Rendering

| Event Kind | Prefix | Destination |
|------------|--------|-------------|
| `info`     | `ℹ` (blue) | stdout |
| `warn`     | `⚠` (yellow) | stdout |
| `error`    | `✖` (red) | stderr |
| `success`  | `✔` (green) | stdout |
| `progress` | `⏳` (blue) or interactive spinner | stdout |
| `task`     | `file:line [#idx] text` (cyan/yellow/dim) | stdout |
| `text`     | raw | stdout |
| `stderr`   | raw | stderr |

### TTY / CI Detection

`isInteractiveProgressEnabled()` in `output-port.ts`:

1. `process.stdout.isTTY` must be truthy.
2. `process.env.CI` must be absent, empty, `"0"`, or `"false"`.

Effect:
- **Interactive** (TTY + non-CI): uses `renderInteractiveProgress()` — spinner + in-place carriage-return updates.
- **Non-interactive** (non-TTY or CI): uses `formatProgressLine()` — stable one-line output, no overwrite.

### Two Spinner Implementations

**CLI Spinner** (`src/presentation/output-port.ts`):
- 4 ASCII frames: `- \ | /`
- Increments per progress event (no timer).
- Renders with `\r` overwrite: `# Label [====        ] 5/10 units - detail`

**Intro Spinner** (`src/presentation/intro.ts`):
- 10 braille frames: `⠋ ⠙ ⠹ ⠸ ⠼ ⠴ ⠦ ⠧ ⠇ ⠏`
- 80ms frame timer.
- Cyan prefix, `\r` overwrite.
- Only used in the animated `intro` command.

### Two Progress Bar Implementations

**CLI Progress Bar** (`output-port.ts` / `renderBoundedProgress`):
- Characters: `=` (filled), ` ` (empty), brackets `[ ]`
- Width: 16 chars
- Format: `Label [========        ] 5/10 units - detail`

**Intro Progress Bar** (`intro.ts` / `progressBar`):
- Characters: `█` (filled), `░` (dimmed unfilled)
- Width: 32 chars, 35ms fill delay
- Animated frame-by-frame fill
- Non-TTY fallback: instant full bar

### Animation Primitives in `intro.ts`

The `intro` command has a rich set of animation functions used only for the 6-act slide presentation:

| Function | Behavior |
|----------|----------|
| `typeText(text, ctx, charDelay=18)` | Character-by-character typing |
| `typeLine(text, ctx, charDelay=18)` | typeText + newline |
| `revealLines(lines[], ctx, lineDelay=60)` | One line per interval |
| `dramatic(text, ctx, charDelay=45)` | Bold, slow typing |
| `whisper(text, ctx, charDelay=30)` | Dim, intimate text |
| `announce(text, ctx)` | Bold with pre/post pauses |
| `cascade(lines[], ctx, lineDelay=80)` | Alias for revealLines (vertical stagger) |
| `drawBox(title, lines[])` | Unicode box with `─ │ ┌ ┐ └ ┘` |
| `spinner(message, durationMs, ctx)` | Braille spinner with done message |
| `progressBar(label, ctx, width=32, fillDelay=35)` | Animated `█░` fill |
| `sleep(ms, ctx)` / `pause(ms=400, ctx)` | Async delays, bypassed if non-TTY or skipped |

All animations respect a `SlideContext.skipped` flag and `isTTY()`, falling back to instant output when non-interactive. Keypress handling (`waitForKeypress()`, `runSlide()`) provides skip-on-any-key behavior. These primitives are **private to `intro.ts`** — not shared.

### Progress Events Are Unused

`ApplicationProgressPayload` and the `progress` event kind are defined and fully rendered, but **no application use case currently emits progress events**. The render path exists and is tested, but it's dead code in production. The `docs/progress-lifecycle-hooks.md` document maps out future hook points across `run`, `plan`, `research`, `discuss`, and delegated `rundown:` flows, but none are implemented yet.

## Scope

### In Scope

1. **Upgrade CLI spinner** — Replace 4-frame ASCII (`- \ | /`) with the braille set (`⠋⠙⠹⠸⠼⠴⠦⠧⠇⠏`) already proven in `intro.ts`. Add a timer-based frame advance instead of per-event increment.
2. **Upgrade CLI progress bar** — Replace `[====    ]` with `[████░░░░]` characters. Consider increasing width from 16 to 24–32.
3. **Add task reveal** — When `run`/`call`/`do` selects a task, render it with a brief typing or cascade effect in TTY mode instead of a flat `info` line.
4. **Typing-style milestone messages** — Apply a subtle typewriter effect to key success/info milestones (e.g., "All tasks completed", "Planning document: ...") in TTY mode.
5. **Improve `list` output** — The `task` event renders a dense tree. Add spacing, numbering, or grouping to make scan-reading easier.
6. **Improve `log` output** — Currently pipe-delimited text per run. Add column alignment, color-coded status, and relative time emphasis.
7. **Improve `artifacts` output** — Currently plain text lines per run. Add structured grouping with indented metadata.
8. **Improve `next` output** — Currently a bare `task` event. Add context (e.g., file count, position in sequence).
9. **Extract shared animation module** — Move reusable primitives (`typeText`, `spinner`, `progressBar`, `drawBox`, `cascade`, `sleep`, TTY detection) from `intro.ts` into a new `src/presentation/animation.ts`. `intro.ts` and `output-port.ts` both import from it.

### Out of Scope

- **Domain interface changes** — `ApplicationOutputEvent` and `ApplicationOutputPort` stay as-is.
- **New runtime dependencies** — No new npm packages.
- **Structured log format** — JSONL schema and `logged-output-port.ts` behavior unchanged.
- **Progress lifecycle hook implementation** — Wiring actual progress events (from `docs/progress-lifecycle-hooks.md`) is a separate migration.
- **Intro command content** — The 6-act slide content stays; only the shared primitives move.

---

## Design Constraints

1. **Container / CI compatibility** — All output must degrade to plain text when `isInteractiveProgressEnabled()` returns false. No ANSI codes in non-TTY output.
2. **ANSI stripping** — `logged-output-port.ts` must remain animation-free. Structured logs written via `GlobalOutputEntryWriter` must never contain escape sequences.
3. **Performance** — Character delay for typing effects ≤ 12ms in regular command output (not intro). Spinner frame rate ≤ 100ms. No perceptible lag on task selection or milestone messages.
4. **No new runtime deps** — `picocolors` only. Animation primitives use raw `process.stdout.write` and `setTimeout`.
5. **Backwards-compatible test surface** — Tests that assert on output messages (e.g., `"All tasks completed"`, `"No unchecked tasks found"`) must not break. New animation characters are stripped or skipped in non-TTY test mode.

## Command-by-Command Output Audit

### 1. `run` — Find & execute next unchecked TODO

**Current output:**
- `ℹ` info lines for task selection, force-unlock, round progress
- `✔` success for completion ("All tasks completed (N total)")
- `✖` error for lock conflicts, option validation
- `⚠` warn for no-git, lock cleanup failures
- Long silence during prompt preparation + worker execution (no progress events)

**Target:**
- Task selection line rendered with typing effect in TTY mode.
- Spinner shown during worker execution phase (requires progress event wiring or local animation).
- Completion message with brief success animation.

### 2. `call` — Full clean pass with CLI block caching

**Current output:** Same as `run` (shared execution path).

**Target:** Same improvements as `run`.

### 3. `discuss` — Interactive discussion session

**Current output:**
- `ℹ` "Next task: ..." then silence during worker execution.
- `ℹ` worker start/finish messages.
- `✔`/`✖` final status.

**Target:**
- Task announcement with typing effect.
- Worker phase spinner.

### 4. `plan` — Synthesize actionable TODOs

**Current output:**
- `ℹ` "Planning document: ..." and TODO count.
- Long silence during each scan + worker execution.
- `✖` errors for worker/CLI block failures.

**Target:**
- Per-scan progress indicator with scan number (e.g., `Scan 2/3 ...`).
- Spinner during worker execution.

### 5. `make` — Create file from seed + research + plan

**Current output:** Combines research + plan output. Same patterns.

**Target:** Phase labels (research → plan) with transitions.

### 6. `do` — Bootstrap + execute all tasks

**Current output:** Combines make + run output. Round indicators when `--rounds > 1`.

**Target:** Phase transitions (bootstrap → execution) clearly labeled. Round progress bar for multi-round.

### 7. `research` — Enrich document with context

**Current output:**
- Mostly error/dry-run info lines.
- Silence during worker execution.

**Target:** Spinner during worker phase. Phase labels for prompt build → worker → write.

### 8. `next` — Show next task

**Current output:** Single `task` event or `ℹ No unchecked tasks found.`

**Target:** Add file/position context (e.g., "Task 3 of 12 in project.md"). Consider subtle task reveal.

### 9. `list` — List all unchecked tasks

**Current output:** One `task` event per task, rendered as dense tree with children/sub-items. Or `ℹ No tasks found.`

**Target:**
- Numbered tasks (1., 2., 3...).
- File grouping headers when tasks span multiple files.
- Spacing between task entries.
- Summary line at end (e.g., "12 tasks across 3 files").

### 10. `artifacts` — Manage runtime artifacts

**Current output:**
- Per-run summary: `{id} [status=...] [command=...] [mode=...] [transport=...] {path}`
- Indented metadata: `task: ...`, `worker: ...`, `started: ...`
- Or `ℹ No runtime artifacts found.`

**Target:**
- Color-coded status badges.
- Relative timestamps.
- Cleaner grouping with box or separator lines.

### 11. `log` — Show completed run history

**Current output:**
- Pipe-delimited text per run: `{id} | {relativeTime} | {status} | {taskSummary} | source=... | command=... | sha=... | revertable=yes/no`
- Color via `picocolors` (already present in `log-runs.ts` directly).

**Target:**
- Column-aligned table or structured rows.
- Color-coded status (green=completed, red=failed, yellow=reverted).
- Emphasize relative time and task summary; dim metadata fields.

### 12. `reverify` — Re-run verification on saved artifacts

**Current output:**
- `ℹ` "Re-verify task: ..." then silence during verification.
- `✔`/`✖` pass/fail result.
- `✖` errors for missing/invalid metadata.

**Target:** Spinner during verification phase. Typing effect on result message.

### 13. `revert` — Undo task runs via git

**Current output:**
- `ℹ` per-commit revert messages.
- `✔` "Reverted N runs successfully."
- Dry-run info lines listing what would happen.

**Target:** Progress counter for multi-commit revert. Color-coded dry-run output.

### 14. `unlock` — Release stale lockfile

**Current output:** Single `ℹ`/`✔`/`✖` line.

**Target:** No changes needed — output is already minimal and clear.

### 15. `init` — Create .rundown config directory

**Current output:** One `✔`/`⚠` line per file created/skipped. Final `✔` summary.

**Target:** Cascade reveal for file creation list. Brief typing effect on summary.

### 16. `intro` — Animated introduction

**Current output:** Full 6-act slide presentation with all animation primitives.

**Target:** No content changes. Underlying primitives move to shared `animation.ts`.

## Feature Overview

A new command (or a mode of the existing `discuss` command) that opens an interactive TUI session for discussing the outcome of a **completed** task. Unlike the existing `discuss` command — which targets the next unchecked task for pre-execution refinement — this feature targets a task that has already been executed, verified, and possibly repaired. The session prompt instructs the AI worker to read the execution log, verification logs, and repair logs from the run's artifact directory so it can answer questions about what changed, how, and why.

## Existing Architecture Context

### The Existing `discuss` Command

The current `discuss` command (`src/application/discuss-task.ts`) selects the **next unchecked task** via `taskSelector.selectNextTask()` and opens a TUI session for pre-execution refinement. Key characteristics:

- Always runs in TUI mode (`mode: "tui"` is hardcoded at `discuss-task.ts:370`).
- Uses the `discuss.md` template (default at `src/domain/defaults.ts:112-153`, overridable via `.rundown/discuss.md`).
- Snapshots and restores checkbox state — the worker can edit task text but must not toggle checkboxes.
- Creates its own artifact context (`commandName: "discuss"`) with status `"discuss-completed"` or `"discuss-cancelled"`.
- Does **not** reference any prior run artifacts, execution logs, or verification results.

### Run Artifact Structure

Every `run` execution produces a directory under `.rundown/runs/run-<timestamp>-<hex>/` containing:

- `run.json` — run-level metadata: `runId`, `commandName`, `status`, `task: { text, file, line, index, source }`, `extra: { commitSha?, commitMessage? }`, `startedAt`, `completedAt`.
- `<NN>-execute/` — execution phase: `prompt.md`, `stdout.log`, `stderr.log`, `metadata.json`.
- `<NN>-verify/` — verification phase: `prompt.md`, `stdout.log`, `stderr.log`, `metadata.json` (includes `verificationResult` field: `"OK"` or `"NOT_OK: <reason>"`).
- `<NN>-repair/` — repair phase (one directory per attempt): `prompt.md`, `stdout.log`, `stderr.log`, `metadata.json` (includes `extra.attempt` number).
- Optionally `cli-block-<N>-stdout.txt` / `cli-block-<N>-stderr.txt` for CLI block artifacts.

Note: stdout.log and stderr.log are only written when non-empty. They are only captured when the worker runs in `mode: "wait"` or when `mode: "tui"` with `--keep-artifacts` (piped + mirrored). If the original run used `mode: "wait"` (the default for `run`), logs are captured. If the original run used `mode: "tui"` without `--keep-artifacts`, no stdout/stderr logs exist.

### Run Resolution Precedent

The `reverify` command (`src/application/reverify-task.ts`) already implements the pattern for locating a finished run and re-resolving its task:

- Accepts `--run <id|latest>` to select a completed run.
- Uses `artifactStore.find(runId)` or `artifactStore.latest()` to locate the run.
- Reads `selectedRun.task` metadata from `run.json`.
- Uses `resolveTaskContextFromRuntimeMetadata()` (`src/application/task-context-resolution.ts:120-148`) to re-resolve the task in the current file state using `findTaskByFallback()` — tries exact line+text match, then index+text, then unambiguous text-only match.

This is the closest existing precedent and can serve as the pattern for run selection in this feature.

### Task Selection for Checked Tasks

The `TaskSelectorPort` already exposes `selectTaskByLocation(filePath, line)` (`src/infrastructure/selector.ts:84-102`) which can select **any** task (checked or unchecked) by file and line. This method exists on the port interface but is not currently used by any command flow. It could be leveraged for selecting a specific checked task for discussion.

### Template System

Templates use `{{variable}}` interpolation syntax and are rendered via `renderTemplate(template, vars)`. Templates can be overridden by placing a file in `.rundown/` (e.g., `.rundown/discuss.md`). The infrastructure loads user overrides with fallback to built-in defaults (`src/infrastructure/templates-loader.ts`, `src/application/project-templates.ts`).

Current template types: `execute`, `verify`, `repair`, `plan`, `research`, `trace`, `discuss`. A new template would follow the same pattern.

### TUI Mode Details

In TUI mode with `captureOutput: false` (default when no `--keep-artifacts`), the runner spawns the worker with `stdio: "inherit"` — fully interactive terminal passthrough (`src/infrastructure/runner.ts:249-261`). With `captureOutput: true`, stdout/stderr are piped, captured in buffers, AND mirrored to the terminal in real-time.

## Specification

### Entry Point

The feature needs a way to select a finished run. Two approaches are possible:

**Option A — New subcommand or mode of `discuss`**: Add a `--run <id|latest>` option to the existing `discuss` command (similar to `reverify`). When `--run` is provided, the command operates on the finished task from that run instead of the next unchecked task.

**Option B — Separate command**: A dedicated command (e.g., `discuss-run` or extending `discuss` with a required `--run` flag for finished-task mode). This avoids overloading the existing `discuss` semantics.

Either way, the command should accept:
- A run identifier (`--run <runId|prefix|latest>`) to select which completed run to discuss.
- Optionally, the `<source>` argument could be omitted when `--run` is provided (the source is already stored in `run.json`).

### Prompt Template

A new template (or variant) is needed — tentatively `discuss-finished.md` or an extended `discuss.md` — that tells the AI worker to read the run's artifact files. The prompt should include:

1. **Run metadata** — run ID, status, task text, source file, timestamps, commit SHA if available.
2. **Execution log references** — paths to `<NN>-execute/stdout.log` and `<NN>-execute/prompt.md`.
3. **Verification log references** — paths to `<NN>-verify/stdout.log`, `<NN>-verify/metadata.json` (for the `verificationResult` field).
4. **Repair log references** — paths to all `<NN>-repair/` directories and their contents (there may be multiple repair attempts).
5. **Instructions** — tell the worker to read these files and be ready to discuss what was changed, why, whether verification passed or failed, what repairs were attempted, etc.

The prompt should use the worker's file-reading capabilities (tool use) to load the logs, rather than embedding potentially large stdout contents directly in the prompt. This keeps the prompt concise while giving the worker access to full details.

Template variables that would be needed:
- `{{runId}}` — the run identifier.
- `{{runStatus}}` — completed/failed/execution-failed.
- `{{runDir}}` — absolute path to the run artifact directory (so the worker can read files).
- `{{taskText}}` — the original task text.
- `{{taskFile}}` — the source file containing the task.
- `{{taskLine}}` — line number of the task.
- `{{executionPhaseDir}}` — path to the execute phase directory.
- `{{verifyPhaseDirs}}` — paths to all verify phase directories.
- `{{repairPhaseDirs}}` — paths to all repair phase directories.
- `{{commitSha}}` — git commit SHA if the run created a commit.
- `{{phaseSummary}}` — a structured summary of all phases (sequence, label, exit code, verification result).

### Phase Discovery

The run artifact directory may contain multiple phases in varying combinations. The feature needs to scan the run directory for all `<NN>-<label>/` subdirectories, read each `metadata.json`, and categorize them by phase type (`execute`, `verify`, `repair`). The `metadata.json` files contain:
- `phase` — the phase type string.
- `exitCode` — worker exit code.
- `verificationResult` — for verify phases, `"OK"` or `"NOT_OK: <reason>"`.
- `stdoutFile` / `stderrFile` — `"stdout.log"` / `"stderr.log"` or `null`.
- `promptFile` — `"prompt.md"` or `null`.

### Run Selection Flow

Following the `reverify` precedent:
1. Parse `--run <id>` option.
2. Use `artifactStore.find(runId)` for ID/prefix match, or `artifactStore.latest()` for `--run latest`.
3. Validate the run exists and has a `task` field in its metadata.
4. Optionally validate the run status is a terminal state (`completed`, `failed`, `execution-failed`, etc.) — not an in-progress run.

### Checkbox Protection

The same checkbox snapshot/restore mechanism from the existing `discuss` command should be applied. Since the target task is already checked, the feature must prevent any checkbox state changes — the worker should not uncheck it or toggle other tasks.

## Edge Cases and Design Constraints

- **Missing logs**: If the original run used TUI mode without `--keep-artifacts`, stdout/stderr logs will not exist. The prompt/template should handle this gracefully — the worker should be informed that execution output was not captured.
- **Multiple verify/repair phases**: A run may have gone through several verify-repair cycles. The prompt should enumerate all phases so the worker can understand the full execution timeline.
- **Deleted or moved source files**: The task file referenced in `run.json` may have been moved or deleted since the run. The feature should handle this gracefully, perhaps showing the task text from `run.json` even if the file no longer exists.
- **Purged artifacts**: Runs without `--keep-artifacts` have their artifact directories deleted after completion (`finalizeRuntimeArtifacts` deletes the directory when `preserve` is false at `runtime-artifacts.ts:577`). The feature can only discuss runs whose artifacts still exist. A clear error message is needed when the requested run has no saved artifacts.
- **`selectTaskByLocation` for checked tasks**: The existing `selectNextTask` filters to unchecked tasks only. For this feature, either `selectTaskByLocation` (which accepts any task state) or direct task metadata from `run.json` should be used instead.
- **Locking**: The existing discuss command locks the source file. The discuss-finished-task feature should do the same since the worker could potentially edit the file.

## Scope

### In Scope
- New command or mode to discuss a finished task's execution artifacts.
- Run selection by ID, prefix, or `latest`.
- New prompt template that references execution/verification/repair logs.
- Phase directory scanning and metadata collection.
- TUI session launch with artifact-aware prompt.
- Graceful handling of missing logs, purged artifacts, moved files.
- Checkbox state protection (same as existing discuss).

### Out of Scope
- Automatic linking of checked tasks back to their runs (no new indexing system).
- Embedding log file contents directly in the prompt (the worker should use its file-reading tools).
- Modifications to the existing `discuss` command's behavior for unchecked tasks.
- Non-interactive (wait/detached) modes — this feature is inherently interactive.

## Acceptance Criteria Outline

- Running the command with a valid run ID opens a TUI session.
- The prompt instructs the worker to read execution, verification, and repair logs from the run's artifact directory.
- The worker can answer questions about what was changed, verification results, repair attempts, and outcomes.
- Missing log files (no captured output) are handled with informative messaging in the prompt.
- Purged artifact directories produce a clear error message.
- Checkbox state is preserved after the discussion session.
- The command follows existing patterns for CLI registration, option parsing, and artifact management.

## Feature specification

The `question` tool is a new built-in handler that enables interactive prompts during task execution. Unlike every other built-in tool (which delegates work to a worker process or modifies execution context), `question` needs to pause the execution pipeline, interact directly with the user via stdin/stdout, and write the answer back into the source Markdown file before continuing.

### Syntax

The tool uses the standard `prefix: payload` syntax already established by existing tools:

```markdown
- [ ] question: Which module we are going to improve?
```

The payload after `question:` is the prompt text displayed to the user.

### Answer format

The answer is persisted as a non-checkbox sub-item beneath the question task:

```markdown
- [ ] question: Which module we are going to improve?
    - answer: CliResourceModule
```

This follows the existing `SubItem` structure that the parser already recognizes -- `src/domain/parser.ts` extracts non-checkbox list items as `task.subItems[]` with `{ text, line, depth }`. The `answer:` prefix distinguishes the response from other potential sub-items.

### Options syntax

When the question offers predefined choices, they should be specified as sub-items beneath the question task. A suggested format:

```markdown
- [ ] question: Which module we are going to improve?
    - option: CliResourceModule
    - option: ParserModule
    - option: RunnerModule (default)
```

The `(default)` marker identifies which option is selected when running non-interactively. When the user answers, the options remain and the answer is appended:

```markdown
- [ ] question: Which module we are going to improve?
    - option: CliResourceModule
    - option: ParserModule
    - option: RunnerModule (default)
    - answer: CliResourceModule
```

### Re-run behavior

If a question task already has an `- answer:` sub-item when execution reaches it, the tool should use that existing answer without prompting. This allows re-running documents where answers were previously provided, and supports manual pre-filling of answers before execution.

## Existing tool architecture context

### Built-in tool registration pattern

Built-in tools live in `src/domain/builtin-tools/`. Each tool exports a `ToolHandlerFn`:

```
(context: ToolHandlerContext) => Promise<ToolHandlerResult>
```

Static tools (no dependencies) are registered in `src/domain/builtin-tools/index.ts` via the `BUILTIN_TOOLS` record. Tools that require injected dependencies (like the memory tool needs `MemoryWriterPort`) are registered dynamically in `src/infrastructure/adapters/tool-resolver-adapter.ts`.

The `question` tool needs access to `stdin`/`stdout` for interactive prompting and needs to be TTY-aware. These are infrastructure concerns not currently available in `ToolHandlerContext`. This means either:
1. The `question` handler is registered dynamically (like memory) with an injected prompt port, or
2. `ToolHandlerContext` is extended with a new port for interactive input.

### ToolHandlerContext fields available today

Key fields from `src/domain/ports/tool-handler-port.ts`:
- `task: Task` -- the parsed task including `subItems[]` for reading existing options/answers
- `payload: string` -- the text after `question:` (the prompt text)
- `source: string` -- the source file path's content context
- `fileSystem: FileSystem` -- read/write operations for modifying the TODO file
- `emit: (event) => void` -- output port for emitting UI events
- `pathOperations: PathOperationsPort` -- path resolution utilities
- `cwd: string` -- working directory

Missing: There is no stdin/interactive-input port. A new port will be needed.

### ToolHandlerResult fields

The result from `src/domain/ports/tool-handler-port.ts`:
- `skipExecution?: boolean` -- should be `true` (the question tool does not need a worker)
- `shouldVerify?: boolean` -- should be `false` (nothing to verify after a question)
- `childTasks?: string[]` -- could potentially be used to insert the answer, but this inserts child checkbox items, not plain sub-items
- `exitCode?: number` -- for error reporting
- `failureMessage?: string` / `failureReason?: string` -- for error reporting

### How child items get inserted today

`src/application/tool-execution.ts` lines 114-124 handle child task insertion after a tool handler completes:

```typescript
if (result.childTasks && result.childTasks.length > 0) {
  const source = context.fileSystem.readText(context.task.file);
  const updatedSource = insertSubitems(source, context.task, result.childTasks);
  context.fileSystem.writeText(context.task.file, updatedSource);
}
```

This uses `insertSubitems()` from `src/domain/planner.ts` which inserts lines with `- ` prefix beneath the parent task at the correct indentation. However, this always strips list markers and re-adds them, and it inserts as checkbox-agnostic plain list items. This mechanism could work for inserting `answer: <value>` sub-items, but the tool handler itself would need to do the file write directly since it needs to control the exact format.

### Task intent classification

`src/domain/task-intent.ts` classifies task prefixes in priority order. The `question` prefix would be classified as `"tool-expansion"` intent -- the generic fallback for any `<name>:` prefix where `toolResolver.resolve(name)` succeeds. No changes to task-intent.ts are needed.

### Prefix chain composition

The `question` tool is a `"handler"` (terminal action), not a `"modifier"`. It could be composed with modifiers:

```markdown
- [ ] profile: detailed, question: Which module?
```

This works automatically via `src/domain/prefix-chain.ts` chain parsing -- modifiers are applied left-to-right, then the handler executes.

## TTY and interactive input context

### Existing TTY detection

`src/presentation/animation.ts` line 7 provides the canonical TTY check:

```typescript
export const isTTY = (): boolean => Boolean(process.stdout.isTTY);
```

`src/presentation/output-port.ts` adds a CI-aware variant:

```typescript
function isInteractiveProgressEnabled(): boolean {
  if (!process.stdout.isTTY) return false;
  const ci = process.env["CI"];
  // ... returns false if CI=true
}
```

### Existing interactive input pattern

The only existing interactive input lives in `src/presentation/intro.ts` with `waitForKeypress()`:

```typescript
function waitForKeypress(): Promise<void> {
  if (!isTTY()) return Promise.resolve();
  return new Promise((resolve) => {
    const stdin = process.stdin;
    const wasRaw = stdin.isRaw;
    stdin.setRawMode(true);
    stdin.resume();
    const handler = (data: Buffer): void => {
      stdin.removeListener("data", handler);
      stdin.setRawMode(wasRaw);
      stdin.pause();
      if (data[0] === 3) process.exit(0); // Ctrl+C
      resolve();
    };
    stdin.on("data", handler);
  });
}
```

This is a single-keypress mechanism. The `question` tool needs a richer interaction: display a prompt, show numbered options, and accept typed input or a selection number. This requires readline-style input, not raw-mode keypress.

### New interactive input port

A new `InteractiveInputPort` (or `QuestionPromptPort`) should be introduced following the clean architecture pattern:

- **Domain port** (`src/domain/ports/`): Defines the interface for prompting the user.
- **Infrastructure adapter** (`src/infrastructure/adapters/`): Implements the port using `process.stdin`/`process.stdout` with readline or raw terminal APIs.
- **Composition root** (`src/create-app.ts`): Wires the adapter and passes it through to the tool resolver.

The port interface needs to support:
- Displaying a prompt string
- Optionally displaying numbered options
- Accepting free-text input or option selection
- Returning the selected answer
- Falling back to a default when not in TTY mode

## File modification pattern

The `question` tool modifies the source Markdown file directly by inserting an `- answer: <value>` sub-item beneath the question task. This is similar to how `insertSubitems()` works in `src/domain/planner.ts`, but with a specific format.

Key considerations:
- The indentation must match existing sub-items under the task. `computeChildIndent()` from `src/domain/planner.ts` computes the correct two-space child indent from the parent line.
- Line endings must be preserved (CR+LF vs LF), consistent with `markChecked()` in `src/domain/checkbox.ts`.
- If the question already has an `- answer:` sub-item, it should be updated rather than duplicated.
- If options are present as sub-items, the answer should be inserted after the last option.
- The file write must happen before the task is marked as checked, so subsequent re-parsing reflects the answer.

## Execution flow integration

### Where execution pauses

The `question` handler runs inside `executeToolChain()` in `src/application/tool-execution.ts`. The handler is an async function, so it naturally pauses the execution loop while awaiting user input. The `while(true)` task loop in `runTaskLoop()` (`src/application/run-task-execution.ts` line 467) awaits each `runTaskIteration()`, so a blocking prompt inside the tool handler will pause the entire pipeline as expected.

### `rundown all` and `rundown do` propagation

- `rundown run --all` uses the same `runTaskLoop` and iterates through all unchecked tasks in sequence. Interactive prompts work naturally since each iteration is awaited.
- `rundown do` composes `make` (research + plan) and then `run --all`. The question tool would only be relevant during the `run` phase, which uses the same execution loop.
- `rundown call` wraps `run --all` with `clean: true`. Same execution path.

The key requirement is that stdin must not be consumed by the worker subprocess when the `question` tool fires. Since the tool sets `skipExecution: true`, no worker process is spawned, so stdin remains available for the prompt.

### Nested execution via `include:`

When a file containing `question:` tasks is referenced via `include:`, it gets cloned and executed as a child. The inline-rundown delegation (`src/infrastructure/inline-rundown.ts`) spawns a child `rundown` process. If the parent runs in TTY mode, the child inherits stdin (runner `wait` mode uses `stdio: ["inherit", "pipe", "pipe"]`), so interactive prompts should work through include chains. However, if the child is spawned with `stdio: "pipe"` for stdin, the interactive prompt would fail. This needs verification.

## Edge cases and constraints

### No default available, no TTY

When running non-interactively (piped input, CI, `!isTTY()`):
- If options are provided and one is marked `(default)`, use that option silently.
- If no options are provided or no default is marked, the handler must return an error (`exitCode: 1`) with a clear failure message. This prevents silent incorrect execution.

### Answer already present

If the task already has an `- answer:` sub-item (from a previous run or manual entry), the tool should skip prompting and return success. This enables idempotent re-runs.

### Empty payload

If the prefix has no payload (`- [ ] question:` with no text), the tool should fail with a clear error, similar to how `include:` fails on empty payloads.

### Multiple questions in sequence

Multiple question tasks in a document should each pause independently. Since the execution loop is sequential, this works naturally.

### Ctrl+C during prompt

The interactive prompt must handle Ctrl+C gracefully -- either by exiting the process (matching `waitForKeypress` behavior) or by returning a failure that stops the execution loop.

### Answer value with special characters

The answer text could contain colons, markdown formatting, or other characters. The `- answer:` format should treat everything after `answer: ` as the raw answer value.

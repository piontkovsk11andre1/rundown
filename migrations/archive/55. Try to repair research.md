## Problem framing

Current repair behavior is not reliably producing a valid research artifact when `rundown research` is executed through inline CLI tasks. The failure mode in the logs shows two coupled issues:

1. The generated file content is invalid for the phase (`worker chatter` instead of enriched Markdown body output).
2. The task completion signal is not applied correctly (`Selected task is still unchecked`).

This means the repair system is attempting to fix a content-shape violation and a task-state violation at the same time, but the repair prompt likely does not make the output contract explicit enough for either.

## Observed behavior and implications

From the log, verification runs immediately after worker execution and blocks progress if either output format or checkbox state is wrong. Because repair consumes previous chat output plus prompt, it can inherit noisy assistant text and keep failing the same validator.

Implications:

- Repair retries can become deterministic failures if prompt constraints are not strict.
- A single bad completion format can poison follow-up repair attempts.
- CLI automation (`rundown all`) stalls on one task because task state cannot be transitioned to checked.

## Likely root causes

### 1) Output contract is underspecified for repair

The research phase expects: full updated Markdown document body only, no chatter. If repair template reuses generic assistant framing, model may prepend explanations or status lines.

### 2) Checkbox semantics are not anchored strongly enough

The validator expects checked syntax `[x]` in the controlling task file. If the prompt says "do not change checkbox state" in one context but runner expects completion marking in another context, model can follow the wrong rule.

### 3) Validation target mismatch

The failed file is `migrations/38. Fix output for every command.md`, while the state error references `migrations/Research.md` line 5. This indicates output file validation and orchestrator task-list validation are separate checks that can fail independently.

## Scope boundaries

### In scope

- Clarify repair prompt contract for research command failures.
- Ensure repair path can produce strict Markdown-only output.
- Define deterministic completion rules for checkbox updates in orchestrator-managed files.
- Make per-command repair templates explicit where behavior differs (`research`, potentially others).

### Out of scope

- Reworking the entire rundown execution architecture.
- Changing document authoring style in migration notes.
- Implementing model-specific hacks tied to one provider only.

## Proposed template strategy

Keep the current idea and formalize it: introduce `research-repair.md` as a dedicated template for research-phase repair. Use command-specific templates when validator contracts differ.

Template design goals:

- State exact output shape at top: "Return only the full updated Markdown document." 
- Include explicit forbidden output: no analysis preface, no fenced meta commentary, no status lines.
- Include explicit completion behavior expected by orchestrator, separated from content-writing rules.
- Include minimal failure context (last validation error + selected task metadata) so repair has precise target.

## Validation contract to codify

Repair should satisfy all of the following in one pass:

- Target content file contains enriched Markdown document body, not chat transcript.
- Any required selected task checkmark transition is applied in controlling task file using `[x]`.
- No unintended checkbox mutations elsewhere.
- No contradictions with source author text; only augmentation.

## Compatibility and edge cases

- If memory context is unavailable, repair must still proceed using source + full document only.
- If first failure reason is content-only, repair still needs to re-evaluate task state before returning.
- If selected task is in a different file than output artifact, repair instructions must name both paths explicitly.
- If command runs from nested cwd (`migrations`), path normalization should remain stable between validator and prompt text.

## Integration notes

- Add a command-to-repair-template mapping (for example, `research -> research-repair.md`) instead of one generic repair prompt.
- Include `Last validation error` as structured input in repair prompt so the model optimizes for the real failing assertion.
- Keep retry count small, but make each retry stricter by appending validator diffs/errors rather than full previous assistant chatter.

## Acceptance criteria outline

- A failed `rundown research ...` run can be repaired successfully without manual edits.
- Repaired output contains only valid enriched Markdown for the target document.
- Selected task is marked checked with `[x]` when run expectations require completion.
- `rundown all .\migrations\Research.md` continues past this step after repair.
- Repair behavior is reproducible across at least two different research documents.

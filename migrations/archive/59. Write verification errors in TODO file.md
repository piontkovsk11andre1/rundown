## Feature Specification

When the verify-repair loop exhausts all repair attempts and the task still fails verification, the system should **mark the task as checked** (`[x]`) and **insert a `- fix:` annotation** as an indented sub-item beneath the task line. The annotation contains the verification failure reason so the author can see exactly what went wrong.

Currently, verification failure terminates the run with exit code 2 and leaves the task **unchecked**. The desired behavior changes this: instead of blocking the run on a failed verification, the task is annotated with the failure reason and marked complete, allowing the run to either continue (in run-all mode) or stop gracefully.

## Current Verification Flow

The verification pipeline lives across three layers:

1. **`runVerifyRepairLoop`** (`src/application/verify-repair-loop.ts`) — Orchestrates initial verification and up to `maxRepairAttempts` repair cycles. Returns `{ valid: boolean, failureReason: string | null }`. When all attempts are exhausted, it returns `valid: false` with the last cumulative failure reason.

2. **`completeTaskIteration`** (`src/application/complete-task-iteration.ts`, lines 140–211) — Calls `runVerifyRepairLoop` and handles the result. On `!valid` (line 192), it currently:
   - Emits an error event with the full verification failure message (line 197).
   - Calls `afterTaskFailed` lifecycle hook (line 198).
   - Returns `continueLoop: false` with `exitCode: 2` via `failRun` (lines 206–209).
   - **Does not** modify the source markdown file — the task remains unchecked.

3. **`checkTaskUsingFileSystem`** (`src/application/checkbox-operations.ts`, line 9) — Called only on the success path (line 214), after verification passes. Reads the source file, calls `markChecked`, and writes it back.

## Existing Domain Primitives

Two domain functions in `src/domain/` are directly relevant:

- **`markChecked(source, task)`** (`src/domain/checkbox.ts`, line 12) — Replaces `[ ]` with `[x]` on the task's line (identified by 1-based line number). Returns updated source string. Preserves EOL style.

- **`insertSubitems(source, task, subitemLines)`** (`src/domain/planner.ts`, line 185) — Inserts sub-item lines directly beneath a parent task line. It:
  - Computes correct child indentation from the parent line via `computeChildIndent`.
  - Strips leading list markers (`[-*+] `) from each sub-item, then re-adds `- ` with proper indentation.
  - Splices the new lines into the source array at `parentLineIndex + 1`.

These two primitives can be composed: first `markChecked` to toggle the checkbox, then `insertSubitems` to append the `- fix:` annotation. Since `markChecked` only modifies the content of the task line (not line count), the task's line number remains valid for the subsequent `insertSubitems` call.

## Sub-item Format

The `insertSubitems` function strips list markers and re-adds them, so passing `"fix: <reason>"` (without a leading `- `) would produce correctly indented output like:

```
    - fix: Cannot be verified because this is missing
```

The indentation depth is derived from `computeChildIndent`, which reads the parent line's leading whitespace and list marker to determine the correct child indent level.

## Integration Point

The change belongs in `completeTaskIteration` at the `if (!valid)` block (line 192). Before calling `afterTaskFailed` and `failRun`, the system should:

1. Mark the task checked in the source file (analogous to the success-path call at line 214).
2. Insert the `- fix:` sub-item with the failure reason.

A new application-level helper (e.g., `writeFixAnnotationToFile` in `checkbox-operations.ts`) would compose `markChecked` and `insertSubitems` into a single file read/write cycle.

## Scope

**In scope:**
- Writing `- fix: <reason>` annotation and checking the task when verification fails.
- Composing existing `markChecked` and `insertSubitems` domain functions.
- Adding the new application helper function.
- Modifying `completeTaskIteration` to call the helper on verification failure.
- Unit tests for the new helper.
- Integration-level tests for the verification-failure annotation path.

**Out of scope:**
- Changing whether the run continues or stops after a verification failure (the existing `continueLoop: false` / exit code 2 behavior should be preserved unless explicitly changed later).
- Modifying the `runVerifyRepairLoop` return contract.
- Changing the `- fix:` format to something else (e.g., `- error:`, `- warning:`).
- Stripping or rotating old `- fix:` annotations on re-runs.

## Edge Cases and Constraints

- **No failure reason available:** `failureReason` can be `null` when the verification store has no details. The annotation should still be written, potentially as `- fix: Verification failed (no details).` to match the existing fallback pattern used in `verify-repair-loop.ts`.
- **Line number stability:** `markChecked` does not add or remove lines, so the task's line number stays valid for the subsequent `insertSubitems` call. Both operations can use the same `task` object.
- **EOL preservation:** Both `markChecked` and `insertSubitems` detect and preserve `\r\n` vs `\n` line endings. The composed operation should use a single read/modify/write cycle to avoid inconsistency.
- **File system atomicity:** The current `checkTaskUsingFileSystem` does a read-modify-write cycle. The new helper should follow the same pattern — one read, both mutations applied to the in-memory string, one write.
- **Existing `- fix:` annotations:** If a task already has a `- fix:` annotation from a previous run, inserting a new one will add it between the task line and the old annotation. Whether to deduplicate or stack annotations is a design choice; stacking is simpler and preserves history.
- **Checked task with fix annotation on reset:** The `resetAllCheckboxes` function resets `[x]` to `[ ]` but does not remove `- fix:` sub-items. This is acceptable — the annotation serves as a historical record.

## Acceptance Criteria

- When verification fails after all repair attempts, the task line in the source markdown file changes from `- [ ] Do this.` to `- [x] Do this.` with a `- fix: <reason>` sub-item inserted immediately below.
- The `failureReason` from `runVerifyRepairLoop` is used as the annotation text.
- When `failureReason` is null, a fallback message is written.
- Existing run termination behavior (exit code 2, `continueLoop: false`) is preserved.
- The `afterTaskFailed` lifecycle hook is still called.
- Unit tests cover the new helper function with various indentation levels and EOL styles.
- Integration tests verify the annotation appears in the source file after a failed verification.

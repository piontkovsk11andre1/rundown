Add planner support for optional user-defined instruction prompts:

- `.rundown/plan-prepend.md`
- `.rundown/plan-append.md`

These files are not literal task snippets to copy. They are AI guidance inputs for planning behavior, describing what kinds of tasks should be added at the beginning/end when appropriate.

Example intent: if planned work changes source files, planner may append `- [ ] cli: npm test` or similar validation tasks based on append guidance.

Behavior requirements:

- Planner template context should include prepend/append guidance text when files exist.
- Guidance is advisory and interpreted by AI (semantic), not deterministic text insertion.
- Default behavior should be safe when files are absent (treated as empty guidance).
- Planning should still respect existing convergence/add-only rules.

## Feature intent and expected outcome

Introduce optional, project-local planner guidance files that influence how the planner thinks about task ordering and coverage without hardcoding deterministic insertions.

The goal is to let teams express planning preferences such as:

- what kind of setup/discovery tasks should appear early;
- what kind of verification/handoff tasks should appear late;
- when those patterns are appropriate;
- when they should be skipped.

This is guidance for model behavior, not a mechanical prepend/append macro system.

## Current planning architecture context

Top-level planning is orchestrated in `src/application/plan-task.ts` and currently:

- loads the plan template from project templates (`.rundown/plan.md`) or falls back to built-in defaults;
- renders a template variable map (`TemplateVars`) and executes the worker in scan loops;
- enforces add-only safety and convergence via `validatePlanEdit(...)` and convergence outcome rules;
- relocates inserted TODOs to the end of the document with `relocateInsertedTodosToEnd(...)`.

Default planner prompt content is defined in `src/domain/defaults.ts` (`DEFAULT_PLAN_TEMPLATE` and `DEFAULT_DEEP_PLAN_TEMPLATE`) and already emphasizes add-only behavior, prefix normalization, and convergence-safe edits.

This migration extends prompt context, not core planner mutation permissions.

## Proposed guidance model

### Guidance sources

Use two optional files under the active config directory root:

- `.rundown/plan-prepend.md`
- `.rundown/plan-append.md`

If a file is missing, unreadable, or empty, treat its guidance as an empty string.

### Guidance semantics

Guidance is interpreted semantically by the model. It should influence prioritization and ordering decisions but must not be interpreted as:

- mandatory task text to inject verbatim;
- a required minimum task count;
- an override of planner safety constraints.

### Placement intent

- `plan-prepend.md`: advice about what belongs near the beginning of planned work (for example discovery, setup, or prerequisite checks).
- `plan-append.md`: advice about what belongs near the end (for example final validation, packaging, release checks, docs sync, cleanup).

These are intent hints, not insertion directives.

## Integration boundaries and module touchpoints

Likely implementation seams:

- `src/application/plan-task.ts`
  - read optional guidance files relative to resolved config root;
  - add new template vars to the rendered planner context;
  - keep behavior unchanged for scan loop, deep passes, and convergence accounting.
- `src/domain/template.ts` / template var typing surface
  - ensure added keys are represented consistently in `TemplateVars` usage.
- `src/domain/defaults.ts`
  - include neutral sections in default plan/deep-plan templates so guidance is visible to the planner even when users do not override `.rundown/plan.md`.
- Optional docs/help updates
  - mention these files in user docs for planning customization.

## Compatibility and resolution rules

Guidance file lookup should follow existing config-root behavior:

- if a config dir is resolved, read from `<configDir>/plan-prepend.md` and `<configDir>/plan-append.md`;
- do not attempt to read from task-source-relative paths unless that already matches config resolution;
- absence of either file is a non-error no-op.

This keeps backward compatibility for repositories without these files and avoids breaking existing `plan`, `explore`, and `make` flows.

## Safety and invariants (must remain unchanged)

This feature must not weaken current planner guardrails:

- add-only constraints for TODO generation/edit validation remain enforced;
- checked-state mutations remain disallowed;
- convergence behavior and scan stop semantics remain unchanged;
- deep planning child-scope rules remain unchanged;
- no deterministic literal insertion pipeline is introduced.

Guidance can change what the model chooses to add, but not what edits are permitted.

## Edge cases and design constraints

- Guidance contains example TODO lines: planner may use the idea but should not blindly copy literal examples.
- Guidance conflicts with document intent: source document workload remains primary; irrelevant guidance should be ignored.
- Guidance encourages non-additive rewrites: edit validator should still reject unsafe rewrites as today.
- Only one guidance file exists: missing counterpart resolves to empty guidance with no warnings required.
- Large guidance content: pass through as plain template context; planner still constrained by existing prompt and validation safeguards.
- Unlimited or multi-scan mode: guidance remains static per run unless files are re-read by design; do not introduce hidden scan-to-scan mutation semantics.

## Suggested prompt contract shape

Expose explicit template fields such as:

- `planPrependGuidance`
- `planAppendGuidance`

and render them in plan/deep-plan prompts with clear framing like:

- “Optional prepend guidance (advisory)”
- “Optional append guidance (advisory)”
- “Interpret semantically; do not copy literally.”

This makes behavior explicit and keeps custom `.rundown/plan.md` templates able to opt in to these variables.

## Scope and non-goals

In scope:

- loading optional guidance files;
- surfacing their contents in planner template context;
- documenting advisory semantics and no-file defaults.

Out of scope:

- deterministic prepend/append insertion engine;
- new hardcoded TODO synthesis rules tied to exact file text;
- modifying convergence/add-only mechanics;
- introducing new task prefixes or execution semantics.

## Acceptance criteria outline

- Planner receives prepend/append guidance text in template vars when corresponding files exist.
- Missing guidance files are treated as empty values without failing planning.
- Planner behavior remains advisory and semantic; no direct literal copy requirement is introduced.
- Existing convergence and add-only validation rules continue to gate final file edits.
- Behavior is consistent across `plan`, `explore` (plan phase), and `make` (plan phase), since they share planning orchestration.

## Test coverage guidance

Add focused tests near existing plan-task/template tests to verify:

- guidance files present: prompt/template context includes both guidance values;
- one or both files missing: values resolve to empty and no error is emitted;
- existing planning safety tests still pass unchanged (no regressions in convergence/add-only behavior);
- deep planning prompt context receives the same advisory fields if deep prompt is expected to honor them.

Also validate default-template behavior so built-in prompts expose guidance context when project templates are not overridden.

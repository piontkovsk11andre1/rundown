Rename end/return/break to optional/skip.

## Goal

Shift control-flow naming from the current `end`-family wording to `optional`/`skip` wording, while preserving existing runtime behavior for task execution and sibling short-circuiting.

## Current behavior baseline

The control-flow handler currently lives behind `endHandler` and is registered as built-in aliases in `src/domain/builtin-tools/index.ts`:

- `end`
- `return`
- `skip`
- `quit`
- `break`

Behavior today:

- The handler evaluates a natural-language condition with a worker.
- If condition is true, it marks execution as control-flow complete and skips remaining sibling tasks.
- If condition is false or ambiguous, it does not skip siblings.
- The current task is still marked complete as part of normal iteration finalization.
- Skipped siblings and their unchecked descendants are marked checked and receive `skipped: <reason>` sub-items.

## Target naming model

Adopt `optional`/`skip` as the public naming model for this control-flow feature.

Suggested semantic framing:

- `optional: <condition>` is the canonical “conditional short-circuit” prefix.
- `skip: <condition>` remains valid as a concise alias for the same behavior.
- Legacy prefixes (`end:`, `return:`, `break:`) should be treated as compatibility aliases unless explicitly removed in a breaking release.
- `quit:` should be explicitly decided: keep as alias for backward compatibility or remove if it no longer matches intended language.

## Compatibility and migration constraints

Because task files, docs, and tests already reference legacy names, this rename should be compatibility-first:

- Keep old aliases resolving to the same handler during transition.
- Update docs and help text to present `optional`/`skip` first.
- Avoid behavior changes in execution, verification gating, or sibling selection.
- If deprecation is desired, emit non-fatal warnings when legacy aliases are used.
- Avoid introducing a new control-flow meaning for `skip`; it already has established semantics in this codebase.

## Affected module boundaries

Implementation context to account for:

- Built-in tool registry: `src/domain/builtin-tools/index.ts` (alias map and canonical names).
- Control-flow logic: `src/domain/builtin-tools/end.ts` (may keep implementation location even if public name changes).
- Tool result contract: `src/domain/ports/tool-handler-port.ts` (`skipRemainingSiblings` payload).
- Tool-chain propagation: `src/application/tool-execution.ts`.
- Iteration completion side-effects: `src/application/complete-task-iteration.ts`.
- File mutation for sibling/descendant skipping: `src/application/checkbox-operations.ts`.
- Task discovery/classification path: `src/domain/task-intent.ts` + resolver-based prefix resolution.
- User-facing help/default templates: `src/domain/defaults.ts`.
- Product docs: `docs/overview.md`, `docs/configuration.md`, `docs/cli.md`, `docs/examples.md`.

## Design decisions to settle before planning

- Canonical handler key: whether to register `optional` as the primary built-in name and retain `end` as alias, or keep `end` internally and expose `optional` only in docs.
- Alias policy timeline: immediate soft deprecation vs long-term permanent aliases.
- Messaging policy: whether to surface “legacy prefix” notices in run output.
- Scope of rename: runtime only vs runtime + docs + tests + migration notes in one change.

### Decision: canonical handler key

- Register `optional` as the canonical built-in control-flow name in `src/domain/builtin-tools/index.ts`.
- Keep `end` as a compatibility alias that resolves to the same handler behavior.
- Keep the implementation location/function (`src/domain/builtin-tools/end.ts` / `endHandler`) for now to minimize churn; only public prefix naming becomes `optional`-first.

## Edge cases and behavior guarantees

Must remain stable after rename:

- Empty payload still fails predictably (non-zero with clear message).
- Ambiguous worker response still defaults to “do not skip siblings.”
- Skipping logic still cascades to unchecked descendants.
- Bottom-to-top mutation order must remain intact to avoid line-number drift.
- `--all` mode behavior must remain unchanged (next selection naturally bypasses newly checked siblings).
- Verification behavior must remain unchanged (control-flow task itself does not introduce unexpected verify passes).

## Acceptance criteria outline

- `optional:` and `skip:` resolve to the same control-flow handler behavior.
- Existing `end:`/`return:`/`break:` tasks continue to work (unless explicitly out of scope as breaking change).
- All user-facing alias lists show `optional`/`skip` as preferred naming.
- Control-flow execution outcomes (true/false/ambiguous/error) match current semantics.
- Sibling and descendant skip annotations remain identical in structure and placement.
- Test coverage reflects new preferred names and preserved legacy compatibility where intended.

## In scope

- Rename/alias updates for built-in control-flow prefixes.
- Documentation/help text refresh to preferred naming.
- Tests adjusted to validate preferred names and compatibility behavior.
- Optional deprecation messaging (non-breaking) if chosen.

## Out of scope

- Changing underlying skip-siblings algorithm or checkbox mutation strategy.
- Redefining `skip` to mean anything other than current control-flow semantics.
- Altering unrelated prefix systems (`verify`, `memory`, `fast`, `include`, `profile`).
- Removing legacy aliases without an explicit breaking-change decision.

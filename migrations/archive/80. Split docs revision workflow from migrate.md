Separate docs revision lifecycle commands from migration execution commands.

## Intent

The current UX mixes two different concerns under `rundown migrate`:

1. Managing design-document revisions (`save`, `diff`, `preview`), and
2. Managing migration planning/execution (`migrate`, `up`, `down`, satellites, user session).

This migration makes docs revision management first-class under a dedicated `rundown docs` command surface and removes `migrate save` entirely.

## Product goal

Support both user workflows cleanly:

- Docs-first workflow: edit `docs/current/`, publish a new immutable revision, inspect diffs, then generate migrations.
- Migration-first workflow: generate/iterate migrations and satellites, then publish aligned docs revisions.

The command model should match user mental model:

- `rundown docs ...` for revision lifecycle of design docs.
- `rundown migrate ...` for migration lifecycle.

## Required command surface

### New docs command group

Add a top-level `docs` command group with:

- `rundown docs publish`
  - Publishes `docs/current/` into the next immutable `docs/rev.N/` snapshot.
  - Supports optional revision label metadata.
  - Preserves existing no-change no-op behavior when current equals latest revision.

- `rundown docs diff`
  - Supports both forms:
    - explicit selector form: `--from <rev|current> --to <rev|current>`
    - shorthand form: `rundown docs diff [target]`
  - Provides deterministic default when selectors are omitted.
  - Produces clear summary and file-level change listing suitable for humans and migration context.

### Migrate command cleanup

Remove docs-revision actions from `migrate`:

- remove `save`
- remove `diff`
- remove `preview`

`migrate` remains responsible for:

- migration proposal generation,
- migration execution (`up`, `down`),
- satellite generation,
- user-session flow,
- prediction reconciliation.

No backward-compatibility aliases are required for removed actions.

## Implementation boundaries

The implementation should preserve existing core revision logic by reusing current design-revision primitives and only changing command boundaries and routing.

Expected touchpoints:

- CLI command definitions and help text for new `docs` group and updated `migrate` action set.
- CLI action routing and validation for docs subcommands.
- Application use-case boundary for docs revisions (publish + diff).
- App wiring/factory registration for new docs use case.
- Documentation pages that currently present revision actions under `migrate`.

## Behavior constraints

- Revision snapshots remain immutable once created.
- Revision numbering remains monotonic (`rev.1`, `rev.2`, ...).
- Diff behavior remains deterministic for default and explicit target selection.
- Existing migration generation continues to consume revision-aware diff context internally.
- Linked-workspace resolution rules remain unchanged.

## Acceptance criteria

- `rundown docs publish` is the canonical revision publish command and correctly snapshots `docs/current/`.
- `rundown docs diff` supports shorthand and `--from/--to` selector forms.
- `rundown migrate save`, `rundown migrate diff`, and `rundown migrate preview` are removed from CLI routing and help output.
- `rundown migrate` (no action) behavior remains intact, including migration proposal generation and follow-up enrichment/satellite flow.
- User-facing docs consistently describe docs lifecycle under `rundown docs` and migration lifecycle under `rundown migrate`.

## Out of scope

- Renaming migration file conventions.
- Changing prediction baseline/reconciliation rules.
- Changing artifact storage format.
- Redesigning satellite content contracts.

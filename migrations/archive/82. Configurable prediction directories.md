Make prediction workspace directory names configurable instead of hardcoding `specs/`, `migrations/`, and `design/`.

## Intent

Today, prediction workflows assume fixed folder names in project layout and command behavior. This migration introduces user-configurable directory names, with safe defaults, so teams can align with their repository conventions.

Primary entrypoint: `rundown start` should accept explicit path arguments for these workspace buckets.

## Product goal

- Keep current defaults for zero-config usage.
- Allow users to override workspace directory names at project bootstrap time.
- Persist chosen paths so subsequent commands resolve the same layout consistently.

## Scope

- Add `start` options for configuring prediction workspace directories (design/specs/migrations).
- Persist resolved directory config in project metadata/config.
- Update path-resolution helpers so runtime commands consume configured directories instead of literals.
- Keep fallback compatibility for existing projects that still use default names.

## Behavior constraints

- Defaults remain: `design/`, `specs/`, `migrations/` when no overrides are provided.
- Overrides must be relative to project root (or clearly validated if absolute paths are disallowed).
- Invalid/conflicting paths should fail with actionable CLI error messages.
- Existing linked-workspace behavior must remain intact when custom directory names are used.

## Acceptance criteria

- `rundown start` supports user-specified directory names for design/specs/migrations.
- Chosen directory names are persisted and reused by migrate/docs/test/prediction flows.
- Existing projects with default layout continue to work without migration steps.
- Help/docs clearly describe defaults, overrides, and validation rules.

Add explicit commands to remove/unlink workspaces from the current directory and clean up related files with confirmation.

## Intent

Users need a safe, first-class way to detach a linked workspace and optionally remove stale workspace artifacts from disk. This should be command-driven, confirmable, and reversible through clear previews.

This migration defines unlink/remove flows for workspace records and their associated files.

## Product goal

- Users can remove a workspace link record from current source context.
- Users can optionally delete corresponding workspace files/directories when requested.
- Destructive cleanup actions always require explicit confirmation (or explicit force flag).

## Scope

- Add command surface for unlinking/removing workspaces.
- Support target selection by path/id when multiple records exist.
- Add dry-run/preview behavior before file deletion.
- Remove stale/orphan workspace metadata safely.

## Proposed command surface

- `rundown workspace unlink [--workspace <dir|id>] [--all] [--dry-run]`
  - Removes link record(s) only; does not delete workspace files by default.
- `rundown workspace remove [--workspace <dir|id>] [--all] [--delete-files] [--dry-run]`
  - Removes link record(s), and when `--delete-files` is set, deletes linked workspace files/directories after confirmation.

Naming can be adjusted, but semantics should keep `unlink` (metadata only) separate from `remove` (potential file deletion).

## Behavior constraints

- If multiple workspace records exist and no selector is passed, command must prompt/list and fail safely with guidance.
- File deletion requires explicit confirmation unless `--force` is provided.
- `--dry-run` prints exactly what records/files would be removed.
- Deleting files outside allowed workspace boundaries must be blocked.
- Legacy single-record link format remains supported.

## Acceptance criteria

- Users can unlink one or many workspace records without touching files.
- Users can remove a linked workspace with optional file deletion guarded by confirmation.
- Orphan/stale records are detectable and cleanable through command flow.
- Output clearly distinguishes invocation dir, selected workspace, and deleted targets.

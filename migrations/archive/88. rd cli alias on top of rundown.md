rd cli alias on top of rundown

## Intent

Add a first-class `rd` executable as a CLI alias for `rundown`, so users can invoke the same runtime with a shorter command without changing behavior.

## Background and current state

- The package currently exposes only one binary in `package.json`:
  - `"bin": { "rundown": "dist/cli.js" }`
- The CLI implementation is rooted in `src/presentation/cli.ts` and is command-driven via Commander, with root command name set to `rundown`.
- Argument alias rewriting exists in `src/presentation/cli-argv.ts`, but it is command-level aliasing (`all -> run --all`, `migrate down -> undo ...`), not executable-name aliasing.
- User-facing docs/README already imply short invocation ergonomics (`rd` appears in the harness usage snippet), so adding a true binary alias aligns runtime behavior with published usage expectations.

## Feature specification

`rd` must behave as a strict executable alias to `rundown`:

- `rd <args...>` routes to the same compiled entrypoint as `rundown <args...>`.
- All commands, global flags, exit codes, output formatting, and runtime side effects are identical.
- Alias support applies everywhere Node package bins are installed (global install, local `node_modules/.bin`, workspace tool runners).
- No command semantics change is introduced; only invocation name availability is expanded.

## Scope

### In scope

- Expose `rd` in package bin mapping, pointing to the same entry script as `rundown`.
- Ensure help/docs mention `rd` as a supported alias where appropriate.
- Confirm runtime behavior parity between `rundown` and `rd` across representative commands.

### Out of scope

- Renaming the canonical product/command from `rundown` to `rd`.
- Introducing separate `rd`-specific flags, behavior, branding, or output.
- Changing existing command-level alias rewriting logic in `cli-argv.ts` unless required for executable-name parity.

## Technical context and integration notes

- Binary aliasing in this codebase is packaging-level, not Commander-level. The right integration layer is `package.json` `bin`, not argument rewriting.
- `program.name("rundown")` in `src/presentation/cli.ts` can remain canonical for help and usage text; executable aliasing should not force internal command name changes.
- Existing invocation analysis (`resolveInvocationCommand`) inspects command argv tokens and defaults to `"rundown"` for no positional command; this should remain unaffected by executable filename.
- Any docs that present startup shortcuts should stay consistent with actual shipped binaries to avoid drift.

## Compatibility and edge cases

- `rd --help` should produce the same static help contract as `rundown --help`.
- Bare `rd` should follow the same root-mode behavior as bare `rundown` (including interactive help fallback logic already defined in CLI flow).
- `rd <invalid-command>` should preserve current Commander error/help semantics.
- Works in non-interactive and CI environments exactly as `rundown` does today.
- Cross-platform bin shims (npm/pnpm/yarn/bun) should resolve both names when installed through supported package managers.

## Acceptance criteria outline

- `rd` is installed as an executable alongside `rundown` from the same package version.
- `rd` and `rundown` execute the same entrypoint and return matching exit codes for equivalent inputs.
- Representative parity checks hold for:
  - root invocation (`rd`, `rundown`)
  - help (`rd --help`, `rundown --help`)
  - command execution (`rd run ...`, `rundown run ...`)
  - known command aliases (`rd all ...`, `rd migrate down ...`) behaving identically to `rundown`.
- Documentation no longer implies unsupported shorthand; `rd` is explicitly documented as a supported alias where shorthand is shown.

## Risks and constraints

- Primary risk is documentation/runtime mismatch if alias is documented but not shipped (or vice versa).
- Packaging metadata changes must not disrupt existing `rundown` consumers.
- Any test updates should validate parity, not duplicate full CLI behavior coverage already tested for `rundown`.

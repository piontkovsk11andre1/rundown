## intent

Add `quick:` as a first-class alias of `fast:` for task intent classification and directive inheritance, with no behavioral differences in execution, verification suppression, logging category, or precedence handling.

## current context

- `fast:` / `raw:` already represent fast-execution intent (execute task, skip verification).
- verification intent and prefix precedence are already established (`verify` > `memory` > `fast/raw` > tool-expansion > default).
- directive parents already support `fast:` / `raw:` inheritance for child checkbox tasks.
- docs and examples already contain user-facing `quick:` usage, so this migration closes behavior/docs drift.

## feature specification

- treat `quick:` exactly as `fast:` anywhere fast intent is recognized:
  - inline checkbox prefix (`- [ ] quick: update headings`)
  - directive parent (`- quick:` with child checkbox tasks inheriting fast-execution intent)
  - mixed-case and whitespace-tolerant variants (`Quick:`, `quick :`, etc.) consistent with existing prefix parsing behavior
- `quick:` must suppress per-task verify phase exactly like `fast:` and `raw:`.
- empty payload behavior for `quick:` must match `fast:`/`raw:` (same skip/error/warning path currently used for fast-execution empty payload tasks).
- runtime logs/messages for fast-execution should not diverge semantically when source prefix is `quick:`.

## scope boundaries

### in scope

- extend fast-execution alias recognition to include `quick` in intent classification and parser directive detection.
- preserve existing prefix-chain and task-intent precedence semantics.
- update docs/help/examples that enumerate fast aliases so `quick:` is listed where relevant.
- add/adjust tests that already cover `fast`/`raw` alias behavior to include `quick`.

### out of scope

- changing canonical authoring guidance away from `fast:` unless explicitly desired later.
- changing execution dispatch architecture, worker routing, or built-in tool ordering.
- introducing new intent categories or new verification modes.

## integration notes

- update all fast-alias recognition surfaces consistently (task intent regex, parser directive pattern, prefix detection helpers, docs/help text, integration tests).
- keep compatibility with composed prefixes and modifiers (for example `profile=fast, quick: ...` should still resolve by existing first-explicit-intent rules in task text).
- ensure parent directive propagation remains stable for nested checkbox tasks and does not affect non-checkbox lines.
- keep behavior identical under global `--verify`: `quick:` tasks still run execution and skip per-task verify.

## edge cases and compatibility checks

- mixed prefix text:
  - `verify: quick: ...` stays verify-only (first explicit prefix wins)
  - `quick: verify: ...` stays fast-execution
- empty payload:
  - `quick:` with no payload follows same handling as empty `fast:`/`raw:`
- unknown prefix fallback must remain unchanged for non-recognized prefixes.
- existing `fast:` and `raw:` behavior must remain unchanged (strict regression guard).

## acceptance criteria outline

- `quick:` inline tasks produce fast-execution intent and skip verification.
- `quick:` directive parents propagate fast-execution intent to child checkbox tasks.
- all `fast`/`raw` behavior remains unchanged after adding `quick`.
- docs/help/examples that list fast-execution aliases include `quick:` consistently.
- targeted and integration tests confirm parity across `fast`, `raw`, and `quick`, including precedence and empty-payload handling.

in document version and migration tool we want to add this: - If revision is 0 that means initial state - But if revision 1 is the first one that handled as 'need to migrate from nothing to this'

## Intent

Clarify and lock revision semantics for the document version + migration flow so revision numbering is interpreted consistently by diff/migration logic, CLI output, and agent prompts.

The key rule we want:

- `revision 0` is the explicit initial baseline state.
- If `revision 1` is the first available revision (no `revision 0` exists), it must still be treated as migration **from nothing -> revision 1**.

## Current implementation context

Relevant behavior already exists in the design revision pipeline:

- `src/application/design-context.ts` parses revision directories as `rev.<n>` with `n >= 0` (`parseDesignRevisionDirectoryName`), so `rev.0` is a valid revision id.
- Diff preparation (`prepareDesignRevisionDiffContext`) already supports an initial comparison mode when no previous revision exists, producing summary text in the form `Compared nothing -> <target> ...`.
- Tests in `src/__tests__/application/design-context.test.ts` already cover:
  - `rev.0` parsing and ordering;
  - first revision treated as migration from nothing;
  - `rev.0 -> rev.1` comparison when both exist and `rev.1` is selected.
- Revision save flow (`saveDesignRevisionSnapshot`) currently computes the next index monotonically and starts from `1` when no revisions exist, so first saved snapshot is typically `rev.1` unless `rev.0` was created externally.

## Problem to solve

Even though parts of the behavior exist, the rule is not yet explicitly documented as a full contract for the migration tool:

- We need unambiguous semantics for how `rev.0`, `rev.1`, and missing predecessors are interpreted.
- We need deterministic behavior regardless of whether the repo starts from `rev.0` or directly from `rev.1`.
- We need this represented as planning-level requirements so future changes do not regress first-revision handling.

## Target behavior specification

### Revision meaning

- `rev.0` means baseline initial state (explicit initial snapshot).
- Any revision with no valid predecessor in the discovered ordered set must be treated as an initial migration target from nothing.
- `rev.1` is **not** required to have `rev.0`; if `rev.1` is first, comparison source is `nothing`.

### Comparison selection rules

For a target revision `rev.N`:

- If a previous discovered revision exists with lower index, compare from the nearest lower index revision.
- If no lower revision exists, compare from nothing.
- For `target=current`, use latest revision as source; if no revisions exist, compare `nothing -> current`.

### Output and prompt consistency

- Diff summary and structured change context must consistently represent initial comparisons as `nothing -> target`.
- Source references for initial comparisons should include only target source path(s), since no prior revision directory exists.
- Migration/research prompt context should preserve this semantic so agents do not hallucinate a missing predecessor revision.

## Scope

In scope:

- Explicit contract definition for `rev.0` initial semantics and `rev.1`-as-first semantics.
- Alignment of diff behavior and migration context generation with this contract.
- Tests and documentation updates that prevent regression.

Out of scope:

- Renaming revision directory format (`rev.<n>`).
- Changing migration file naming conventions.
- Introducing destructive backfill/mutation of existing revision directories.

## Constraints and compatibility notes

- Must remain backward compatible with repositories that currently begin at `rev.1`.
- Must not require users to create `rev.0` manually to get correct first migration behavior.
- Must preserve deterministic sorting and safe handling of malformed revision directory names.
- Must preserve immutable revision snapshot behavior.

## Edge cases to keep explicit

- Only `rev.0` exists -> comparing `target=rev.0` is `nothing -> rev.0`.
- Only `rev.1` exists -> comparing `target=rev.1` is `nothing -> rev.1`.
- `rev.0` and `rev.1` both exist -> `target=rev.1` compares `rev.0 -> rev.1`.
- Non-sequential revisions (example: `rev.1`, `rev.4`) -> `target=rev.4` compares from nearest lower discovered revision (`rev.1`).
- Missing target path -> keep current graceful “diff unavailable” response contract.
- Case variants (`REV.1`) and leading zeros (`rev.001`) continue to follow existing parser/discovery rules.

## Acceptance criteria outline

- Contract docs explicitly state that `rev.0` is initial baseline and first discovered revision (including `rev.1`) compares from nothing.
- Diff context generation returns `hasComparison=true` with initial-added file changes when target has no predecessor.
- Summary strings and source references are deterministic for both `rev.0`-first and `rev.1`-first repositories.
- Existing tests covering initial-from-nothing and `rev.0 -> rev.1` remain valid; missing coverage for `rev.1`-first explicit revision target is added if needed.
- No regression in revision discovery, ordering, immutability checks, or fallback behavior.

## Integration notes

Primary module boundary for this behavior is `src/application/design-context.ts` (especially predecessor selection and initial diff generation).  
Command-level exposure is via revision/diff flows in `src/application/docs-revision-task.ts` and CLI command actions in `src/presentation/cli-command-actions.ts`.  
Behavior should stay aligned with migration task context assembly in `src/application/migrate-task.ts` so revision semantics are consistent across publish/diff/migrate workflows.

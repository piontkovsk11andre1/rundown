Output timestamps in CLI

## intent

Improve CLI observability by including clear, consistent timestamps in user-visible command output so operators can correlate terminal events with run artifacts, trace logs, and global JSONL streams.

## current context

Timestamp data already exists in multiple runtime surfaces, but CLI presentation is mixed:

- `rundown log` currently renders **relative** time (`2m ago`, `just now`) via `src/application/log-runs.ts` and `src/domain/relative-time.ts`.
- Run metadata stores absolute ISO timestamps (`startedAt`, `completedAt`) in artifacts handled by `src/infrastructure/runtime-artifacts.ts`.
- Global output JSONL (`<config-dir>/logs/output.jsonl`) always records absolute event time as `ts` in ISO-8601 UTC, wired via `src/presentation/cli-invocation-log.ts` and `src/presentation/logged-output-port.ts`.
- Terminal rendering lives in `src/presentation/output-port.ts`, where most info/success/warn/error lines currently have no prefixed timestamp.

This creates a gap: machine-readable logs are timestamped, but human terminal output is often not.

## problem statement

When diagnosing failures, long-running loops, retries, or multi-command workflows, users must manually cross-reference untimestamped terminal lines with artifact metadata or JSONL entries. Relative-only timestamps in `rundown log` are useful for quick scanning but not sufficient for precise event correlation.

## feature specification

Define a consistent CLI timestamp output model for human-readable terminal events:

- Add explicit timestamp rendering for relevant user-facing CLI lines (at minimum: command-level info/warn/error/success and compact run history lines where appropriate).
- Preserve existing semantic content and ordering of messages; timestamps should be additive, not a rewrite of event text.
- Keep machine formats stable:
  - JSON outputs (for example `rundown log --json`) should retain existing fields and compatibility.
  - Global JSONL schema remains unchanged (`ts` already present).
- Standardize displayed format for terminal timestamps (UTC ISO-8601 recommended for exactness and direct parity with JSONL/artifacts).
- Avoid introducing ambiguous local-time formatting unless explicitly scoped and documented.

## relevant code surfaces

- CLI command registration/options: `src/presentation/cli.ts`
- Log command shaping and line formatting: `src/application/log-runs.ts`
- Relative-time helper: `src/domain/relative-time.ts`
- Terminal presentation pipeline: `src/presentation/output-port.ts`
- Invocation/global log timestamp emission: `src/presentation/cli-invocation-log.ts`
- Artifact timestamp persistence and run ordering: `src/infrastructure/runtime-artifacts.ts`

## scope boundaries

### in scope

- Defining where timestamps appear in human-readable CLI output.
- Aligning timestamp semantics between terminal output and existing run/log metadata.
- Updating docs (`docs/cli.md`) to describe timestamp behavior clearly.
- Adding/adjusting tests for timestamp rendering and stability.

### out of scope

- Changing artifact timestamp storage semantics.
- Changing JSONL field names/types.
- Redesigning unrelated CLI color/group formatting behavior.
- Introducing broad localization/timezone preferences unless explicitly required.

## constraints and compatibility

- Backward compatibility for scripts: text output may change shape, so machine consumers should continue using `--json`; this should be called out in docs.
- Existing tests that assert exact output strings (notably `log-runs` and integration CLI tests) will need controlled updates.
- Colorized styling in `output-port.ts` must remain compatible when timestamps are prepended/appended.
- Relative-time behavior in `rundown log` should remain available unless explicitly replaced; absolute timestamp output can coexist with relative labels.

## edge cases to account for

- Invalid or missing timestamps in persisted metadata (current fallback behavior should remain deterministic).
- Very recent and future timestamps (`just now`, `in 30s`) in relative-time paths if retained.
- Multi-line output and grouped output rendering (`group-start`/`group-end`) when timestamp prefixes are enabled.
- Quiet mode behavior (`-q, --quiet`) should not accidentally re-enable suppressed lines through timestamp plumbing.
- Cross-platform consistency (Windows/macOS/Linux) for displayed timestamp text.

## testing implications

Coverage should include:

- Application-level `log-runs` formatting tests for any changed line shape.
- Presentation tests in `__tests__/presentation/*` ensuring timestamped rendering is stable with colors/group prefixes.
- Integration tests for `rundown log` and representative commands where users rely on temporal ordering.
- Regression checks that `--json` output contracts remain unchanged.

## acceptance criteria outline

- CLI output includes deterministic, human-visible timestamps at the defined presentation points.
- Timestamp format is documented and consistent with existing artifact/JSONL time semantics.
- `rundown log` remains readable and preserves filtering behavior (`--revertable`, `--command`, `--limit`, `--json`).
- JSON outputs and global output JSONL schema remain backward compatible.
- Updated tests validate timestamp rendering without regressing non-timestamp output semantics.
